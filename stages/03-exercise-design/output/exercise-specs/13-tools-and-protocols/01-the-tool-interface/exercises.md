## Exercises

1. **Build and validate a JSON Schema for a simple tool.** Implement a Python script that defines a JSON Schema (using the `jsonschema` library) for a tool called `send_email` with the following input contract: `to` (string, required, email format), `subject` (string, required, max 120 characters), `body` (string, required). Validate two payloads against it тАФ one that passes and one that fails the email format constraint. Print a clear `PASS` or `FAIL` line for each payload, including the validation error message on failure. Run the script and confirm terminal output shows both outcomes.

2. **Trace the three-stage lifecycle with debug prints.** Implement a mock tool-call flow for a `lookup_company` tool: (1) define the input schema with `domain` (string, required) and `include_employees` (boolean, default false), (2) simulate the agent's serialization step by constructing a JSON payload, and (3) validate that payload against the schema using `jsonschema.validate`. Insert a `print` statement at each stage showing the stage name and the data being processed. Run the script and verify all three stages print in order before a final `RESULT: ACCEPTED` line.

3. **Compare a passing tool-call against a type-mismatch failure.** Implement a `calculate_arr` tool with schema requiring `monthly_revenue` (number) and `months` (integer, minimum 1). Create two payloads: one valid (`monthly_revenue: 50000`, `months: 12`) and one where `months` is passed as the string `"12"`. Validate both. Print which payload succeeded, which failed, the exact schema keyword that triggered the failure (e.g., `type`), and the value the agent serialized that caused the mismatch. Run and confirm the terminal shows the failure point clearly.

4. **Implement a schema-validated tool registry.** Build a `ToolRegistry` class that supports `register(name, input_schema, handler)` and `dispatch(name, payload)`. When `dispatch` is called, it must validate `payload` against the registered `input_schema` before invoking the handler. Register at least two tools relevant to GTM enrichment тАФ e.g., `find_company_by_domain` (input: `domain` string) and `score_lead` (input: `company_name` string, `employee_count` integer). Dispatch one valid call and one invalid call (wrong type for `employee_count`). Print the handler's return value on success and a `ValidationError`-derived message on failure. Run and confirm both outputs appear.

5. **Diagnose schema negotiation failures in a multi-provider enrichment pipeline.** Implement a script that simulates calling two enrichment providers тАФ e.g., a mock Apollo-style provider expecting `company_domain` (string) and a mock Clay-style provider expecting `website` (string) тАФ for the same logical "enrich company" operation. Define a canonical internal schema using `domain` (string, required). Write an adapter layer that maps the canonical payload to each provider's expected field name. Demonstrate one successful dispatch through both adapters and one failure where a field name is missing from the adapter (simulating a schema-negotiation bug). Print the mapped payload, the provider response, and тАФ for the failure case тАФ a one-line diagnosis stating which field mapping is missing and the fix. Save the adapter code and a 3тАУ5 line diagnosis comment at the top of the file to `handlers/enrichment_adapters.py`. Run and confirm terminal output shows both the success and the diagnosed failure.

6. **Design a complete tool interface spec for a CRM log-call tool.** Write a Python module that defines the full JSON Schema (input and output) for a `log_call_to_crm` tool: input must include `contact_id` (string), `call_outcome` (enum: `answered`, `voicemail`, `no_answer`, `follow_up`), `duration_seconds` (integer, minimum 0), and optional `notes` (string). The output schema must describe a result object with `success` (boolean) and `activity_id` (string). Implement a handler function that validates input against the schema, returns a mock activity ID on success, and raises on validation failure. Wire it into a registry (you may reuse Exercise 4's `ToolRegistry`). Demonstrate three dispatches тАФ one valid, one with an invalid enum value, one missing a required field тАФ and print the outcome of each. Save the complete module to `signals/examples/log_call_tool.py`. Run and confirm all three outcomes print with clear pass/fail diagnostics.
