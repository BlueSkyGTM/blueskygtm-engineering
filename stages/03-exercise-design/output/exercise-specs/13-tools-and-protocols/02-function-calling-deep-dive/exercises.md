## Exercises

1. **Implement a tool-definition converter.** Write a Python function that accepts a plain dictionary with `name`, `description`, and `parameters` (JSON Schema) and returns three provider-specific tool-definition objects — one for OpenAI (`{type, function: {…, strict}}`), one for Anthropic (`{name, description, input_schema}`), and one for Gemini (`FunctionDeclaration` nested in `tools`). Call it with a `lookup_company_domain` tool spec and print all three JSON objects to the terminal. Verify the structural differences (wrapper nesting, schema key name, `strict` / `additionalProperties` constraints) match what the lesson describes.

2. **Implement the complete five-phase function-calling loop with OpenAI.** Define a `get_company_info` tool that accepts a `company_name` parameter. Send a user message asking about a specific company, detect the tool call in the response, execute a local lookup (a hardcoded dict of 3–5 companies is fine), inject the result back as a `tool` role message, and call the API again for the final natural-language answer. Print the raw response at each phase so you can trace the `tool_call_id` linkage from request through execution to continuation.

3. **Build a GTM task router using Anthropic's tool use with forced tool choice.** Define three tools: `search_prospects` (calls Apollo's People Search API with a job-title filter), `enrich_company` (calls a company-enrichment endpoint), and `log_to_crm` (writes a contact to HubSpot via their API — a mock function is acceptable). Classify five different user messages (e.g., "Find VPs of Engineering at Series B startups") and use Anthropic's `tool_choice` parameter to force the correct tool for each. Print which tool was selected, the parsed arguments, and the tool result for each message.

4. **Implement parallel function-call handling with Gemini.** Configure Gemini with a `batch_lookup_companies` tool that the model may call with an array of company names. Craft a prompt that triggers parallel lookups (e.g., "Compare the tech stacks of Stripe, Plaid, and Ramp"). Parse the response, execute all lookups concurrently (use `concurrent.futures.ThreadPoolExecutor` or `asyncio`), and handle two edge cases: (a) one company name is malformed or missing a required field — catch the validation error, do not crash the loop; (b) one company returns no data — return a structured error object instead of a string so the model can reason about the gap. Inject all function responses and print the model's final synthesis.

5. **Build a production multi-provider tool execution engine at `handlers/multi_provider_tools.py`.** Create a unified `ToolEngine` class that accepts a list of tool definitions and a provider name (`openai`, `anthropic`, or `gemini`), internally converts the tools to the correct format, executes the function-calling loop with retry logic for malformed parameters (max 2 retries with error feedback injected), and supports configurable `tool_choice` steering (`auto`, `any`, or a specific tool name). Include a `compare()` method that runs the same prompt across all three providers and prints a side-by-side table showing: which tool was called, arguments parsed, number of loop iterations, and final response length. Run it against a Clay enrichment tool definition that looks up a person by email and returns their LinkedIn, title, and company. Output the comparison table to the terminal.
