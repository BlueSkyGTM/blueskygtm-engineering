## Exercises

1. **Publish an Agent Card and verify discovery.** Stand up a minimal HTTP server (any framework — `http.server`, Flask, FastAPI, or equivalent) that serves a valid Agent Card document at `/.well-known/agent.json`. The card must include `name`, `description`, `url` (your task endpoint), at least one entry in `skills` with `id`, `name`, and `description`, and a `capabilities` object with `streaming: false`. Run the server on `localhost:7800` and fetch the card with `curl http://localhost:7800/.well-known/agent.json`. Confirm the JSON prints to your terminal and validates against the field requirements above.

2. **Implement and trace the Task state machine.** Write a Python script that models the A2A Task lifecycle as a state machine. Define the five states — `submitted`, `working`, `completed`, `failed`, `canceled` — and a function `transition(current_state, event)` that returns the next state, rejecting illegal transitions (e.g., `completed` → `working` must raise an error). Run the script through a sequence of transitions that ends in `completed` and print each state change to the terminal. Then run a second sequence that ends in `failed`. Both traces must be visible in stdout.

3. **Build an A2A task server with lifecycle.** Implement a standalone HTTP server that accepts A2A task submissions via `POST /` with a JSON-RPC 2.0–style payload (`method: "tasks/send"`). When a task arrives, store it with state `working`, simulate processing for a few seconds, then update it to `completed` and attach a simple text artifact. Expose `method: "tasks/get"` so a caller can poll task status by ID. Test end-to-end with `curl`: submit a task, poll until `completed`, and print the returned artifact to your terminal. No scaffold — design the payload structure yourself based on the lesson.

4. **Build an A2A client and apply it to a GTM enrichment scenario.** Write an A2A client script that (a) discovers a remote agent by fetching its Agent Card from `http://localhost:7800/.well-known/agent.json`, (b) reads the `skills` array and prints which skills are available, (c) submits a task that simulates sending a company domain to an enrichment agent (use the enrichment skill from your card), and (d) polls until the task is `completed`, then prints the artifact. If you have an Apollo or Clearbit API key, wire the server's enrichment skill to call the real API and return enriched company data. If not, return a realistic mock payload with fields like `company_name`, `industry`, `employee_count`. The client output must show the full discovery → submit → poll → artifact flow in the terminal.

5. **Implement a multi-agent enrichment handler with authentication.** Build an A2A server that acts as a coordinator agent. It requires Bearer-token authentication (reject unauthenticated requests with HTTP 401 and an appropriate JSON-RPC error). Expose two skills in its Agent Card: `enrich_contact` (accepts an email or LinkedIn URL, returns person-level data) and `score_lead` (accepts company and contact data, returns a numeric score and rationale). When a task is submitted to `enrich_contact`, the server should internally call a second A2A agent running on `localhost:7801` (a "data-lookup" agent you also build) to fetch the raw contact record, then annotate it before returning the artifact. Persist the full handler as a runnable Python module at `handlers/enrichment_coordinator.py`. Run both servers and demonstrate the flow with `curl` — unauthenticated request returns 401, authenticated request returns a completed task with an enriched-contact artifact. Print all HTTP exchanges to the terminal.

6. **Compare A2A, MCP, and plain REST for a GTM orchestration system.** Write a design document (Markdown) that analyzes three approaches to building a multi-step GTM workflow: lead enrichment → scoring → CRM sync. For each of the three approaches (plain REST endpoints, MCP tool bindings, A2A task delegation), describe how capability discovery works, how state is tracked, how errors propagate, and what the client integration looks like. Include a concrete JSON example of the same "enrich this lead" request expressed in all three paradigms. Conclude with a recommendation for which to use when the system involves two autonomous agents vs. a single agent calling tools. Save the document to `outputs/skill-a2a-vs-mcp-vs-rest.md`. The file must exist and contain all three JSON examples plus the recommendation section.
