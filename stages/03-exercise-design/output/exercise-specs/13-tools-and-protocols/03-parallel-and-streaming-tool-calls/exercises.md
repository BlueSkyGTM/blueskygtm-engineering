# Exercises — Parallel Tool Calls and Streaming with Tools

## Exercises

1. **Implement** a non-streaming tool-use request using the Anthropic Messages API with at least two tools defined (e.g., `get_account_info` and `get_contact_info`). Send a user prompt that requires both lookups. Print the `stop_reason`, the number of `tool_use` blocks in the response content, and each block's `id`, `name`, and `input`. Verify that the model returns multiple tool calls in a single turn and that your terminal shows them all before you execute anything.

2. **Reconstruct** tool-call arguments from a streaming response by accumulating `input_json_delta` chunks keyed by content block index. Send the same two-tool prompt with `stream=True`. For each `content_block_start` event, initialize an empty string buffer at that block index. On each `input_json_delta` event, append the `partial_json` to the correct buffer. After the stream completes, parse each buffer with `json.loads` and print a dictionary mapping block index to the fully reconstructed `{name, input}` pair. Confirm the parsed inputs match what you would see in a non-streaming call.

3. **Compare** wall-clock latency of sequential versus parallel tool execution. Define two or more tools that each include a `time.sleep(1.5)` (simulating an I/O-bound enrichment call). Run the agent loop twice: once dispatching tool calls one at a time, once dispatching them concurrently with `concurrent.futures.ThreadPoolExecutor`. Use `time.perf_counter()` to measure each approach and print both elapsed times and the speedup ratio. The parallel run should be noticeably faster when the model emits multiple tool calls in one turn.

4. **Apply** the parallel streaming pattern to a GTM enrichment workflow. Define tools that call Apollo's People Search API and Accounts Search API (or two different enrichment endpoints you have access to). Prompt the model with a natural-language request like "Find contacts and company details for companies in the fintech space with 50-200 employees." Stream the response, reconstruct any parallel tool calls, execute them concurrently, and return the results. Print the final model response and the total number of tool calls executed in parallel.

5. **Build** a complete reusable streaming tool-use agent loop that handles parallel tool calls end-to-end: stream the response, reconstruct arguments from deltas across multiple block indices, dispatch all tool calls concurrently, append `tool_result` blocks for each, and continue the conversation until `stop_reason` is `end_turn`. Support at least three tools of your choosing. Include a `--verbose` flag that prints every delta event, reconstructed argument, and tool result. Save the implementation as a persistent artifact at `handlers/streaming_tool_agent.py`.

6. **Design** and document a benchmark suite that measures the latency characteristics of parallel tool calling under varying conditions. Run the streaming agent from Exercise 5 across at least five different prompts designed to elicit 1, 2, 3, 4, and 5 simultaneous tool calls. For each run, record: number of tool calls emitted by the model, number of `input_json_delta` chunks per block, total reconstruction time, parallel execution time, and total round-trip latency. Output the results as a markdown table in `outputs/skill-parallel-tools-benchmark.md` with one row per prompt. Include a short analysis paragraph explaining when parallelism provides the greatest speedup and when it does not help.
