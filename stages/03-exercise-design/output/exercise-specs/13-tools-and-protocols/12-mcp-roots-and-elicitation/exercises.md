## Exercises

1. **Implement** a scoping elicitation function that accepts a raw user prompt and returns a structured JSON object with four fields: `intent`, `constraints`, `missing_info`, and `ready` (boolean). Use your model client with a system message that forces JSON-only output — no prose, no markdown fences. Call the function with the prompt `"Write a cold outreach sequence for our new product"` and print the JSON. Then call it again with `"Write a 4-email cold outreach sequence for VPs of Engineering at Series B SaaS companies, promoting our AI code review tool. Budget is enterprise pricing."` Print both outputs and verify that the second call produces `ready: true` with an empty `missing_info` list while the first does not.

2. **Implement** an uncertainty-marker detector that scans a block of generated text for phrases such as `"I'm not sure"`, `"it depends"`, `"might be"`, `"could also"`, `"unclear"`, and `"several possible"`. The function should return a list of detected markers with their character positions. Test it against three sample outputs you write yourself — one with zero markers, one with two markers, and one with five or more. Print the marker list for each sample and confirm the counts match your expectations.

3. **Build** a complete mid-flight elicitation loop. Your function should accept a user prompt, generate a draft response, run the uncertainty detector from Exercise 2 on that draft, and if any markers are found, formulate a clarifying question (derived from the nearest detected marker's surrounding sentence), print that question to the terminal, accept user input via `input()`, and regenerate with the user's answer appended to the original prompt. If no markers are found, return the draft directly. Test with a deliberately vague prompt like `"Help me write a landing page"` and respond to the clarifying question when prompted. Print the final output and confirm it contains no remaining uncertainty markers.

4. **Apply** the scoping elicitation pattern to a GTM enrichment workflow. Write a script that reads a CSV of 10 company records (name, website, industry — invent them) and, before enriching each account, runs a scoping check that asks: does this record have enough data to run a Clay-style waterfall enrichment (email finder → phone finder → technographics)? If `missing_info` is non-empty, skip enrichment for that row and print `SKIPPED: <company> — missing <fields>`. If `ready` is true, print `ENRICHED: <company>`. Do not call any external APIs — simulate the enrichment readiness check using your model client as the scoping layer.

5. **Build** a combined pipeline that chains proactive scoping (Exercise 1) with reactive mid-flight elicitation (Exercise 3) for an account-classification task. The pipeline should accept a CRM account description, produce a scoping object, and if `ready` is false, immediately surface the missing-info questions before generation even starts. If `ready` is true, generate an account tier classification (Enterprise / Mid-Market / SMB) — but still run the uncertainty detector on the classification output. If uncertainty markers appear mid-flight, elicit clarification and regenerate. Save the final classification results for all 10 accounts from Exercise 4 to `handlers/account_classifier.py` as a runnable module. Execute the module and confirm it prints each account name alongside its tier, along with any clarifications that were requested mid-flight.

6. **Design** and implement an experiment that quantifies the trade-off between proactive scoping and reactive mid-flight elicitation across 20 GTM prompts (mix of enrichment-scoping prompts, outbound-copy prompts, and ICP-definition prompts). Run each prompt through three modes: (a) no elicitation (raw generation), (b) scoping-only, and (c) scoping-plus-mid-flight. For each mode, count the number of outputs that contain at least one uncertainty marker, and measure how many user interactions were required on average. Write a Markdown report comparing the three modes with a table of results and a 2–3 paragraph recommendation on when each pattern is necessary. Save the report to `outputs/skill-elicitation-comparison.md`. Print the path and a summary line to the terminal when finished.
