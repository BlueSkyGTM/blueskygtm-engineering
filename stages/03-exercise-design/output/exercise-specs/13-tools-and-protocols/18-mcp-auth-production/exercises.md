## Exercises

1. **Implement** an RFC 8414 authorization-server metadata endpoint. Stand up a minimal HTTP server (any framework) that serves a JSON document at `.well-known/oauth-authorization-server` containing `issuer`, `authorization_endpoint`, `token_endpoint`, `jwks_uri`, `scopes_supported` (include `mcp:tools:read` and `mcp:tools:write`), and `grant_types_supported` (include `authorization_code` and `client_credentials`). Start the server on port 8080 and `curl` the endpoint. Verify the response includes every required field and prints a human-readable summary of discovered scopes and grant types.

2. **Build** a JWKS client that fetches a remote key set, indexes every key by its `kid` claim, and caches the result in memory. Fetch from a live JWKS endpoint — use `https://www.googleapis.com/oauth2/v3/certs` or your own server from Exercise 1. Print the number of keys loaded and each key's `kid` and `kty`. Then implement re-fetch-on-miss: if a `kid` from an incoming JWT header is not in the cache, re-fetch the JWKS once and retry the lookup. Print whether the key was found on the first fetch or required a re-fetch.

3. **Implement** an MCP resource-server middleware that validates a JWT's `aud` claim against a pinned resource identifier (`https://mcp.mycompany.com/resources/tools`). Generate two JWTs with `cryptography` or `PyJWT`: one with the correct audience and one with `https://api.salesforce.com` as the audience. Feed both through your middleware. The middleware must print `ACCEPT` and the decoded subject for the matching token, and `REJECT: audience mismatch` for the replayed token. Run the script and confirm both outcomes in your terminal.

4. **Apply** JWKS key rotation to a GTM enrichment MCP server. Using a CSV of 50 Apollo-like prospect records as your "protected resource," issue access tokens signed with `kid="key-1"`. Build a verification function that checks the JWT signature against your cached JWKS and, on success, returns the slice of records the token's `scope` permits. Mid-run, swap your signing key to `kid="key-2"` (remove `key-1` from the active set but keep it resolvable in JWKS). Issue a new token under `key-2` and re-run. Both the original and rotated tokens must validate without client restart. Print which `kid` each token was verified under and how many records each returned.

5. **Build** a complete token-trace pipeline that simulates the full MCP enrollment lifecycle for a Clay-enrichment tool server. The pipeline must: (a) serve authorization-server metadata, (b) issue a client-credentials token with `aud` pinned to `clay-mcp://enrichment/v1` and `scope=clay:enrich:write`, (c) verify the token via JWKS lookup, (d) execute a mock Clay API call (enrich a company domain → return employee count) only after all checks pass, and (e) print a numbered trace of every check performed: metadata discovery, token acquisition, signature verification via `kid`, audience match, scope authorization, and tool invocation. Write the entire server and trace harness as a runnable Python program that outputs the step-by-step trace. Persist the server and verification logic to `handlers/mcp_clay_auth_server.py`.

6. **Design** and prove a key-rotation runbook for a production MCP authorization server that serves multiple resource identifiers (e.g., `mcp://crm/contacts`, `mcp://enrichment/company`, `mcp://email/sequences`). Write a rotation script that introduces a new signing key, publishes both old and new keys in the JWKS simultaneously, waits for a configurable overlap window (simulated in seconds), then removes the old key. Run a simulation that issues tokens under the old key, rotates, re-verifies old tokens during the overlap, and confirms rejection after overlap expiry. Print a timeline of key states and token validation results. Document the runbook — including overlap-window sizing guidance, rollback procedure, and monitoring checks — as a persistent reference at `outputs/skill-mcp-key-rotation.md`.
