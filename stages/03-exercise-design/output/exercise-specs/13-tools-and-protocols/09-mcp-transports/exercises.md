## Exercises

1. **Compare** the three transports across six dimensions by writing a Python script that prints a formatted table to stdout. The script should define a data structure (list of dicts or nested dict) capturing each transport's connection model, statefulness, port requirement, process lifecycle, remote accessibility, and spec status (current vs deprecated). Run the script and verify the table renders all three transports × six dimensions with no missing cells.

2. **Configure** a stdio MCP server subprocess. Write a minimal Python script that reads newline-delimited JSON-RPC messages from stdin, logs each received message to stderr with a timestamp, and writes a JSON-RPC response to stdout. Pipe a test `initialize` request into the script using `echo` or a heredoc from the terminal and confirm the response appears on stdout while the log lines appear on stderr.

3. **Implement** a Streamable HTTP MCP server endpoint using Python's `http.server` or a lightweight framework. The server must accept POST requests containing a JSON-RPC payload, extract a session ID from the `Mcp-Session-Id` header (generating a new UUID if absent), store session state in a dictionary keyed by session ID, and return the JSON-RPC response with the session ID echoed in the response header. Start the server on `localhost:8080`, send two POST requests with `curl` — one without a session header and one reusing the returned session ID — and confirm the server assigns then reuses the session correctly.

4. **Trace** the migration path from legacy HTTP+SSE to Streamable HTTP for a hypothetical server currently exposing `GET /sse` for the event stream and `POST /messages` for client-to-server JSON-RPC. Write a script that outputs a component-by-component change log: for each of the five components (endpoint structure, handshake flow, message direction mechanism, session management, reconnection strategy), print what exists in the legacy transport, what it becomes in Streamable HTTP, and which spec version introduced the change. Run the script and verify all five components are accounted for.

5. **Build** a transport selection evaluator for GTM deployments. Write a Python script that accepts a JSON configuration file describing a deployment scenario — including fields like `client_type` (IDE, web-app, CRM-integration), `network_topology` (local-only, same-VPC, internet-facing), `concurrent_sessions` (integer), and `enrichment_source` (Clay, Apollo, Clearbit, custom-API). The script applies decision rules (e.g., if `client_type` is IDE and `network_topology` is local-only, recommend stdio; if internet-facing with multiple concurrent CRM sessions, recommend Streamable HTTP) and writes a markdown recommendation report to `outputs/skill-transport-selection.md` including the chosen transport, rationale, and a list of rejected alternatives with reasons. Test with at least three different scenario JSON files covering stdio, Streamable HTTP, and a rejected-legacy-SSE case.

6. **Design** a dual-transport MCP server for a GTM enrichment tool that serves both local IDE users (via stdio) and a remote CRM webhook integration (via Streamable HTTP). Write a single Python module that exposes the same three tools — `lookup_company`, `enrich_contact`, `score_lead` — over both transports. The stdio entrypoint reads JSON-RPC from stdin; the HTTP entrypoint listens on a configurable port and handles POST-based JSON-RPC with session management. Demonstrate both entrypoints work by running the stdio path via piped input and the HTTP path via `curl`, confirming identical responses from both. Save the server module to `handlers/enrichment_mcp_server.py`.
