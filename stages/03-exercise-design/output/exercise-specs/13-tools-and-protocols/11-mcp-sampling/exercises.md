## Exercises

1. **Implement a basic sampling request.** Build a minimal MCP server that exposes a tool called `summarize_topic`. When the client calls this tool with a `topic` string argument, the server sends a `sampling/createMessage` request back to the client with a user message asking the LLM to summarize that topic in three sentences. Set `maxTokens` to 200. Run the server against a compatible MCP client (Claude Desktop or a custom test harness) and invoke the tool. Verify that the server logs the LLM's returned summary to stdout before returning it as the tool result.

2. **Configure sampling parameters and compare outputs.** Extend the server from Exercise 1 to accept two additional string arguments: `tone` (e.g., "technical", "casual") and `audience` (e.g., "executive", "engineer"). Use these to construct a dynamic `systemPrompt` for the sampling request. Set `modelPreferences` to prefer a fast, low-cost model using `hints` with `"anthropic/claude-3-5-haiku"` and `costPriority` set high. Call the tool twice with different tone/audience combinations and print both responses side-by-side in the terminal. Confirm the outputs differ in style and depth.

3. **Build a convergence-based agent loop.** Implement an MCP server tool called `extract_criteria` that takes a product description string. The server should enter a loop: on each iteration it sends a sampling request asking the LLM to extract or refine a list of evaluation criteria from the product description, then sends a second sampling request asking the LLM whether the criteria list is "complete" or "needs refinement." Terminate the loop when the LLM responds "complete" or after a maximum of 5 iterations. Print each iteration's criteria and the convergence decision to the terminal. Verify the final criteria list is returned as the tool result.

4. **Apply sampling to GTM enrichment.** Build an MCP server tool called `score_icp_fit` that accepts a JSON string representing a prospect (company name, industry, employee_count, and a short description). The server should make a sampling request to the client's LLM with a system prompt that defines an Ideal Customer Profile (e.g., B2B SaaS companies, 50–500 employees, in revenue operations). Ask the LLM to return a fit score from 0–100 and a one-sentence justification. Test the tool with at least three sample prospects — one clear fit, one clear non-fit, and one borderline. Print each prospect name alongside its score and justification to the terminal. Confirm the scores differentiate the three cases as expected.

5. **Design and implement a human-approved GTM research loop.** Build an MCP server tool called `research_account` that accepts a target company name. The server enters an agent loop with the following phases: (1) send a sampling request asking the LLM to generate three strategic questions about the company, (2) send a second sampling request per question asking the LLM to draft a hypothetical answer based on its knowledge, (3) send a final sampling request asking the LLM to synthesize the questions and answers into a brief account-research summary. Between each phase, the server must log a checkpoint message to stdout indicating it is pausing for human review, and sleep for a configurable number of seconds (default 3) to simulate an approval gate. After the loop completes, write the final summary to `outputs/skill-mcp-sampling-account-research.md` in markdown format with sections for the company name, the three Q&A pairs, and the synthesis. Run the tool against a test client and verify the file is created with all sections populated.

6. **Compare sampling vs. tool-calling architectures.** Implement two versions of a GTM lead-qualification assistant. Version A uses standard tool-calling: the client LLM calls server tools (`get_company_info`, `check_technology_stack`, `score_lead`) and orchestrates the workflow itself. Version B uses MCP sampling: a single server tool called `qualify_lead` enters an agent loop that uses sampling requests to drive the LLM at each step. Both versions should process the same input — a JSON prospect record — and produce a qualification verdict. Save a comparison document to `outputs/skill-mcp-sampling-vs-tools.md` that includes: (a) the terminal output from both runs, (b) the number of LLM round-trips in each version, (c) where the orchestration logic lives in each architecture, and (d) a recommendation for which pattern to use when building a production GTM enrichment pipeline and why.
