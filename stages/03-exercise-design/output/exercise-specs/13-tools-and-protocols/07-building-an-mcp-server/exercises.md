## Exercises

1. **Implement** a minimal MCP server using the Python FastMCP SDK that exposes a single tool `score_company(domain: str) -> dict` returning a hardcoded score and tier. Start the server in stdio mode and connect to it using the MCP inspector (`npx @modelcontextprotocol/inspector`). Verify that `tools/list` returns your tool schema and that calling the tool with a valid domain returns the expected JSON. Then call it with a missing argument and observe the raw JSON-RPC error object in the inspector's request/response panel.

2. **Trace** the full JSON-RPC lifecycle by adding structured logging to your server that prints each incoming request method and params to stderr. Connect a client, list tools, call the scoring tool, then disconnect. Capture the stderr output to a file and verify it shows the sequence: `initialize` → `notifications/initialized` → `tools/list` → `tools/call`. Confirm each request-response pair shares a matching `id` field.

3. **Implement** a tool called `enrich_company(domain: str)` that calls a real enrichment API — use Hunter.io's free email-finder endpoint or Apollo's people search API with a demo key. Wrap the HTTP call so that three failure modes return distinct JSON-RPC error responses: missing API key (code -32602, invalid params), malformed domain (code -32602), and upstream API failure (code -32000, server error). Verify each error path by triggering it and confirming the error object structure matches JSON-RPC 2.0 spec: `{"jsonrpc": "2.0", "id": <id>, "error": {"code": <code>, "message": <msg>}}`.

4. **Build** the same three-primitive server (a scoring tool, a `icp://criteria` resource returning markdown, and an `outreach_draft` prompt template) using the TypeScript MCP SDK instead of Python. Run it in stdio mode and connect with the MCP inspector. Call `resources/list`, `resources/read`, `prompts/list`, and `prompts/get` against your server. Capture the four JSON responses to a file and confirm each primitive returns correctly typed content.

5. **Package** your Python MCP server from Exercise 3 as a uvx-installable CLI tool. Write a `pyproject.toml` with a console script entry point that launches the FastMCP server. Install it locally with `uvx --from . <your-package-name>`. Create a Claude Desktop configuration entry in `~/Library/Application Support/Claude/claude_desktop_config.json` that launches your server via uvx. Restart Claude Desktop, open a conversation, and verify your enrichment tool appears and executes successfully. Produce the working server as `handlers/enrichment_mcp_server.py` and the pyproject.toml alongside it.

6. **Compare** the Python FastMCP and TypeScript MCP SDKs by implementing identical `score_company`, `enrich_company`, and `outreach_draft` primitives in both languages. Run each server through the MCP inspector and capture the `initialize` response, the `tools/list` schema output, and a sample `tools/call` response from both. Write a structured analysis to `outputs/skill-mcp-sdk-comparison.md` documenting the differences in: (a) how each SDK declares and validates JSON Schema for tool arguments, (b) how capability negotiation differs in the `initialize` handshake, and (c) how each handles transport selection (stdio vs SSE) and error propagation. Include concrete JSON snippets from both implementations as evidence.
