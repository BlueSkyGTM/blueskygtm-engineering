## Exercises

1. **Implement a minimal MCP client that connects to a stdio server, completes the `initialize` handshake, and prints the discovered tool list to stdout.** Use the MCP Python SDK's `StdioServerParameters` and `ClientSession`. Connect to any stdio-based MCP server (you may use a reference server from the SDK or one you build). After the handshake succeeds, call `tools/list` and print each tool name, description, and a summary of its input schema fields. Your terminal output should show the full list of discovered tools with their names and argument schemas.

2. **Extend your client to invoke a specific tool by name and print the returned content blocks.** Pick any tool from the discovered list, construct valid arguments based on its input schema, and send a `tools/call` request. Print the raw content block types (text, image, resource) and their payload summaries. Then deliberately pass invalid arguments to the same tool and confirm the response comes back with `isError: true` — print the diagnostic text from the content blocks so you can see how errors surface in-band rather than as exceptions.

3. **Build a client wrapper that connects to an MCP server exposing a GTM enrichment tool (e.g., company lookup, email validation, or LinkedIn profile fetch via Apollo, Clay, or a custom enrichment API wrapped as an MCP server).** Discover the tool, invoke it with a real or sample company domain and contact name, and print the structured enrichment data returned in the content blocks. If you do not have API credentials, use a mock MCP server that returns realistic JSON payloads — but the invocation path must be real end to end. Confirm your client correctly parses the returned resource content block and extracts the enriched fields.

4. **Implement session lifecycle management with keep-alive and clean shutdown.** Your client should: (a) complete initialization, (b) perform a capability negotiation check and print which server capabilities are available (tools, resources, prompts, logging), (c) set a keep-alive interval that sends a `ping` request every 10 seconds and prints a heartbeat confirmation, and (d) on `SIGINT` or `KeyboardInterrupt`, send a proper shutdown sequence and close the stdio pipe. Run the client for at least 30 seconds to observe two heartbeat cycles, then interrupt it and confirm the shutdown logs appear before the process exits.

5. **Build a multi-server MCP router that connects to two or more stdio servers simultaneously, merges their tool namespaces, and handles name collisions with a `server.tool` prefixing scheme.** Persist this as `handlers/mcp_router.py`. When a caller invokes a tool, the router parses the prefix, dispatches to the correct server session, and returns the content blocks. Include a `list_all_tools()` method that returns a merged dictionary keyed by prefixed tool name. Add collision detection: if two servers expose a tool with the same base name, log a warning and apply the prefix. Test by connecting to two servers that each expose at least one tool, listing the merged namespace, and invoking one tool from each server. Print the merged tool list and both invocation results to stdout.

6. **Design and implement a comparison harness that connects to the same MCP server via both stdio and Streamable HTTP transports, then benchmarks discovery latency and invocation latency for each.** Persist the harness as `signals/examples/transport_benchmark.py`. Run at least 5 discovery cycles and 5 invocations per transport, compute median latencies, and print a comparison table. Include a written analysis (printed to stdout or written to `outputs/skill-transport-comparison.md`) explaining which transport you would choose for three deployment scenarios: local CLI tool, shared team service, and cloud-hosted multi-tenant platform. Justify each choice based on your measured data and the protocol characteristics discussed in the lesson.
