## Exercises

1. **Configure a fallback waterfall using LiteLLM's `Router` class.** Define a model list with at least two providers (e.g., OpenAI `gpt-4o-mini` and Anthropic `claude-3-5-haiku`). Place a intentionally invalid model alias first in the fallback chain so the first attempt fails. Send a chat completion request asking the model to return the single word "OK." Print the full response object and confirm which provider ultimately handled the request by inspecting the response metadata or the `model` field in the output.

2. **Verify provider abstraction by sending the identical OpenAI-shaped request to two different backends.** Construct a single message list (`[{"role": "user", "content": "Return the word BANANA and nothing else."}]`) and route it once to an OpenAI model and once to an Anthropic or Google model through the router. Print the `choices[0].message.content` from each response and confirm both return the same structure despite hitting different provider APIs.

3. **Compare three routing strategies for a GTM enrichment workload.** Implement cost-based routing (cheapest model first), latency-based routing (fastest responded model first), and fallback routing (preferred provider then backup) using the `Router` class with different `routing_strategies`. For each strategy, send a request that classifies a company description into an ICP tier (e.g., "Enterprise SaaS, 5000 employees, Series D" → "Tier 1"). Print which model each strategy selected and the classification result. Write a one-paragraph comparison to `outputs/skill-routing-comparison.md` noting which strategy you would choose for a high-volume enrichment job processing 10,000 accounts and why.

4. **Build a token-cost attribution system for pipeline stages.** Create a Python script that wraps LiteLLM router calls and tags each request with a pipeline stage (`enrichment`, `personalization`, `scoring`). After each call, extract `usage.prompt_tokens` and `usage.completion_tokens` from the response, look up per-model pricing, and accumulate cost into a per-stage dictionary. Send at least two requests per stage using different GTM prompts (e.g., enrichment: "Summarize this company's business model"; personalization: "Write a cold email opener for this prospect"; scoring: "Score this lead 1-10 based on firmographics"). Print a summary table showing total tokens and dollar cost per stage.

5. **Build a production-grade enrichment routing handler that processes a batch of company names from a CSV and outputs enriched results.** Read a CSV containing at least 10 company names (you may generate synthetic GTM data or pull from a sample Apollo/Clay export). For each company, call the router with a prompt that generates a two-sentence business description, using a fallback waterfall (primary model → fallback model). Track per-request cost and attribute all spend to the `enrichment` stage. If a provider fails mid-batch, verify the fallback handled the request. Write the enriched company data as JSON and the cost breakdown as a separate report to `handlers/enrichment_router.py` (the runnable script) and `outputs/skill-enrichment-batch.json` (the results).

6. **Evaluate LiteLLM, OpenRouter, and Portkey against a specific GTM team's production requirements and produce a recommendation document.** Write a script at `outputs/skill-platform-evaluation.md` that addresses each of the following constraints for a 50-person GTM team running enrichment, scoring, and personalization at scale: (a) self-hosting vs. managed SaaS trade-offs, (b) observability and per-request tracing capabilities, (c) cost attribution granularity, (d) vendor lock-in risk, and (e) fallback waterfall configuration ergonomics. Include a final recommendation with justification and a migration path if the team is currently calling provider APIs directly.
