## Exercises

1. **Implement a minimal call-now/fetch-later server.** Build a small HTTP server (Flask, FastAPI, or stdlib `http.server`) with two endpoints: `POST /tasks` creates a task, assigns a UUID, stores `{"state": "pending"}` in an in-memory dict, and returns the task ID immediately. `GET /tasks/<id>` returns the stored state. Add a background thread that flips the state to `running` after 1 second, then `completed` after 3 seconds. Start the server, make a POST with `curl`, capture the task ID, then poll with `curl` at least three times to observe state transitions in your terminal.

2. **Trace the five-state lifecycle.** Extend the server from Exercise 1 so that every state transition appends a timestamped line to a log list stored alongside the task (e.g., `task["log"].append({"ts": ..., "from": "pending", "to": "running"})`). Add a `GET /tasks/<id>/history` endpoint that returns the full transition log. Create a task, poll until terminal, then fetch the history. Verify the output shows all five states represented across the lifecycle — if `cancelled` and `failed` are not exercised, add endpoints or flags (`POST /tasks/<id>/cancel`, `POST /tasks/<id>?fail=true`) to trigger them and re-trace.

3. **Build a polling client with exponential backoff and a timeout ceiling.** Write a standalone Python script (`poll_client.py`) that takes a task ID and base URL as arguments, polls `GET /tasks/<id>` starting at 0.5s intervals, doubles the interval on each non-terminal response, caps the interval at 5s, and aborts with a clear error if total elapsed time exceeds 30s. The client must treat `completed`, `failed`, and `cancelled` as terminal and exit with code 0 for `completed`, non-zero for the others. Print each poll attempt number, elapsed time, and observed state. Run it against your server from Exercise 1 with a 4-second task and confirm it terminates correctly.

4. **Apply the async pattern to a simulated Apollo-style enrichment API.** Build a mock enrichment endpoint that accepts a company domain, kicks off a "contact discovery" task (just sleep 5 seconds, then return a hardcoded list of 3 contacts), and requires the call-now/fetch-later pattern. Write a client script that submits a domain, polls for results, and prints the discovered contacts as a formatted table. Then add a configuration flag `--sync` that attempts the same enrichment via a blocking endpoint (which sleeps the full 5 seconds before responding). Run both modes through `time` and compare wall-clock time and connection behavior in your terminal output.

5. **Build a batch enrichment handler that manages multiple concurrent async tasks.** Create `handlers/batch_enrich.py` that accepts a list of 5 company domains (read from a JSON file or hardcoded), submits all 5 as async tasks to your enrichment server from Exercise 4, then polls all outstanding tasks concurrently using a single polling loop with shared backoff. As each task completes, collect its contacts. After all tasks reach a terminal state, write the combined results to `outputs/batch-enrichment.json` with one entry per domain. Handle the case where one or more tasks fail mid-batch — the handler must record the failure, continue polling the remaining tasks, and include error entries in the output file. Run the handler and inspect the JSON artifact.

6. **Compare polling vs. webhook retrieval and document the tradeoff.** Extend your enrichment server with a webhook delivery mode: when a task is created with a `webhook_url` parameter, the server POSTs the result to that URL on completion instead of requiring polling. Implement a small webhook receiver endpoint that logs incoming deliveries. Run the same 5-domain enrichment from Exercise 5 twice — once with polling, once with webhooks — measuring total HTTP requests made by the client and total elapsed time. Write your findings, including a recommendation table (latency sensitivity, infra complexity, retry semantics, NAT/firewall constraints), to `outputs/skill-async-retrieval.md`.
