## Exercises

1. **Parse and classify JSON-RPC 2.0 messages.** Write a Python script that reads newline-delimited JSON objects from `stdin` and prints, for each one, the message type (`request`, `response`, `notification`, or `invalid`) along with which required fields are missing. Feed it the three sample envelopes from the lesson plus two malformed ones (a request missing `id`, a response with both `result` and `error`). Verify your script outputs five classification lines and correctly flags the two invalid messages.

2. **Construct JSON-RPC envelopes by hand.** Using only `json.dumps` and `print`, emit three messages to the terminal: a request calling `tools/list` with `id: 1`, the matching success response containing a fabricated `result`, and a notification for `notifications/initialized`. Confirm the request and response share the same `id` and that the notification has no `id` field. Pipe your script's output through `python -m json.tool --json-lines` to validate each line is well-formed JSON.

3. **Implement a minimal MCP server that completes the three-phase lifecycle.** Write a script that reads JSON-RPC messages from `stdin` one per line and responds correctly to `initialize` (returning `protocolVersion`, `capabilities`, and `serverInfo`), echoes back a `notifications/initialized` acknowledgment, handles one operational method of your choice, and cleanly returns a response to a `shutdown` request. Drive your server by piping a sequenced input file (one JSON object per line) and confirm the output stream contains exactly the expected responses in order.

4. **Trace and enforce capability negotiation.** Extend your server from Exercise 3 to reject any method other than `initialize` if the handshake has not yet completed — return a JSON-RPC error response with code `-32002` and a message explaining the session is not initialized. Construct an input sequence that sends `tools/list` *before* `initialize` and observe the error, then send a correct sequence and observe success. Print both runs and compare the outputs side by side.

5. **Build a GTM-focused MCP server exposing multiple primitive types.** Implement a server that exposes at least one **tool** (e.g., `lookup_company` that fetches firmographic data from a local CSV of Apollo-style records), one **resource** (e.g., `account://context` returning CRM account notes as plain text that *informs* the model), and one **prompt** template (e.g., `outreach_draft` that *shapes* the conversation by returning a structured message template with arguments). Handle `initialize`, `tools/list`, `resources/list`, `prompts/list`, and the corresponding `/*call` methods. Save the server to `handlers/mcp_gtm_server.py` and include a `demo_input.jsonl` file in the same directory that exercises every primitive end-to-end. Run `cat demo_input.jsonl | python handlers/mcp_gtm_server.py` and confirm each call returns a well-formed JSON-RPC response.

6. **Design a primitive selection rationale for a real GTM workflow.** Choose a concrete GTM scenario (e.g., enriching a list of 200 stale leads in Clay, then drafting personalized outreach). Write a short design document at `outputs/skill-mcp-primitives.md` that specifies which of the six MCP primitives your server would expose for this workflow, what each one does to the context window (act, inform, or shape), and which primitives you would deliberately *not* expose and why. Include a JSON snippet showing the `capabilities` object your server would advertise during `initialize`. Conclude with one paragraph explaining how skipping `initialize` would break this workflow under a spec-compliant host.
