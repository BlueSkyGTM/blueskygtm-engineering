## Exercises

1. **Implement `resources/list` and `resources/read` handlers in a Python MCP server.** Create a server that exposes two static GTM resources: an ICP definition document at `gtm://icp/definition` and a target account list at `gtm://accounts/targets`. Start the server and use the MCP inspector (or a client script) to call `resources/list` — verify both URIs appear in the output. Then call `resources/read` on each URI and confirm the content payload is returned. Print both responses to the terminal.

2. **Implement `prompts/list` and `prompts/get` handlers in the same server.** Add two prompts to your server: a cold email draft template (accepting `company_name` and `persona` arguments) and a discovery call prep template (accepting `account_domain`). Call `prompts/list` and confirm both prompt names appear. Call `prompts/get` on each prompt with sample arguments and verify the returned structure contains a `messages` list with properly typed `user`/`assistant` roles. Print the full prompt response to the terminal.

3. **Classify and implement six GTM capabilities as the correct MCP primitive.** Given these six capabilities: (a) Apollo person search by title, (b) company firmographic snapshot from Clearbit, (c) cold email generation template, (d) CRM deal-stage update in HubSpot, (e) competitor battle card document, (f) discovery call question framework — decide for each whether it belongs as a tool, a resource, or a prompt. Write a one-line justification comment above each handler. Implement all six in a single server, then run the server and print a summary table showing each capability name, its primitive type, and its identifier (URI or prompt name).

4. **Configure Claude Desktop to mount your server and verify both primitives are discoverable.** Edit `claude_desktop_config.json` to point at the server from Exercise 3. Restart Claude Desktop, open the resource panel (paperclip icon), and confirm your resources are listed. Open the prompt picker and confirm your prompts are listed. Attach a resource to a conversation and invoke a prompt, capturing a screenshot or log output proving both primitives are accessible from the live session.

5. **Build a benchmark server that exposes a state-changing CRM pipeline as both a subscribed resource and a polling tool, then measure push vs. pull latency.** Create a server where a pipeline snapshot updates every few seconds (simulate deal-stage changes). Expose the snapshot as a subscribable resource at `pipeline://snapshot` that pushes `notifications/resources/updated` on each change, and also as a tool `get_pipeline_snapshot` that returns the current state on demand. Write a benchmark client script that measures round-trip time to receive an update via subscription (push) versus polling the tool at 1-second intervals (pull). Run at least 20 cycles and print a comparison table of mean and median latency for each pattern. Save the server implementation to `handlers/pipeline-benchmark.py`.

6. **Design a complete MCP server for a B2B account-research workflow and justify every primitive choice.**
