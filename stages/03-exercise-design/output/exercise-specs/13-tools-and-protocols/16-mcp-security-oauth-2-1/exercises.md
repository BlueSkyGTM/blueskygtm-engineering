## Exercises

1. **Implement a PKCE code verifier/challenge pair using S256 and verify the hash manually.** Generate a 43-byte random code verifier, base64url-encode it, compute the SHA-256 hash, and base64url-encode the digest to produce the code challenge. Print both values, then independently re-compute the challenge from the verifier using only `hashlib` and confirm the two strings match character-for-character. Your terminal output should show the verifier, the challenge, and a final `MATCH` or `MISMATCH` line.

2. **Decode and compare two JWTs — one issued with a `resource` parameter and one without.** Use `PyJWT` (or `python-jose`) to decode both tokens from the lesson's sample authorization server responses. Print the `aud` claim, `scope` claim, and any `resource` claim from each. Identify which token is safe to send to `https://crm.internal` and which one an attacker could replay against `https://analytics.internal`. Your terminal output should end with a single line: `SAFE_TOKEN=<index>`.

3. **Trace a token through two resource servers and pinpoint the exact claim that causes audience rejection.** You are given a JWT (paste it from the lesson or mint one with `pyjwt` containing `aud: ["https://enrichment.internal"]` and `scope: "contacts:read profiles:read"`). Write a validation function that checks the token against `https://crm.internal` and `https://enrichment.internal`. For each server, print the validation result (`ACCEPT` / `REJECT`) and, on rejection, print the specific claim name and value that caused the failure. Your output must show two lines — one per server — with the claim-level reason.

4. **Build a step-up authorization state machine for an Apollo people-search integration.** Start with a token scoped to `people:search`. Call a mock Apollo endpoint (a local function that returns `403` with `WWW-Authenticate: Bearer scope="people:enrich"` when the required scope is missing). Your state machine must intercept the 403, extract the required scope from the challenge header, request a new token with the original scope plus `people:enrich` (incremental — not a full re-authorization), and retry the call. Print the HTTP status code of the initial call, the scope extracted from the challenge, the updated token's scope claim, and the HTTP status code of the retry.

5. **Build a reusable OAuth 2.1 client handler that enforces PKCE, resource indicators, and incremental scope expansion for a Clay enrichment workflow.** Your handler must: (a) generate and store a PKCE pair per authorization attempt, (b) include the correct `resource` parameter in both the authorize and token requests, (c) cache tokens keyed by `(resource, scopes)` so that a request to a different resource or expanded scope triggers a new token — not a replay, and (d) expose a `request(url, scopes)` method that auto-handles `403 WWW-Authenticate` challenges by requesting incremental scopes. Include a `__main__` block that demonstrates the full lifecycle against mock endpoints for two Clay API resources (`https://api.clay.com/tables` and `https://api.clay.com/enrich`). Save the file to `handlers/oauth21_client.py`. Your terminal output must show: the PKCE challenge for each token request, the `aud` claim of each token, and the final HTTP status for both resources.

6. **Design and prove which blast-radius constraint mitigates which threat.** Write a formal security analysis covering three scenarios: (a) a malicious MCP server attempting token replay against a different resource server, (b) an over-privileged client accessing unrelated API endpoints, and (c) a compromised token being used for destructive operations after legitimate read access. For each scenario, identify which constraint (audience binding, scope restriction, or incremental expansion) is the primary mitigation, cite the RFC section that defines it, and write a one-paragraph proof of why the other two constraints are insufficient alone. Save the document to `outputs/skill-blast-radius-analysis.md`. Include a summary table at the top with columns: `Threat`, `Primary Constraint`, `RFC`, `Why Others Fail`.
