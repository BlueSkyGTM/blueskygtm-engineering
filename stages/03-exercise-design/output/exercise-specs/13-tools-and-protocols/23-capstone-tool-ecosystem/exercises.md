## Exercises

1. **Implement** a three-stage pipeline (`fetch`, `enrich`, `format`) that reads a list of 10 company domains, calls a mock enrichment function for each, and writes the output of each stage to `outputs/stage1_raw.json`, `outputs/stage2_enriched.json`, and `outputs/stage3_formatted.json`. Run the pipeline, then delete `outputs/stage3_formatted.json` and add a `--resume-from stage2` flag that skips stages 1 and 2 by loading `outputs/stage2_enriched.json` from disk. Confirm by printing which stages executed and the total elapsed time for a full run vs. a resumed run.

2. **Implement** a `validate_contract(record, required_fields)` function that checks whether a dictionary contains every field in `required_fields` and raises a `ContractError` with the exact missing field name if not. Feed it 5 test records where two are missing fields (`email` missing from one, `domain` missing from another). Print which records pass and which fail. Then wire it into the entry point of stage 2 from Exercise 1 so that any record failing the contract is logged and skipped before enrichment runs.

3. **Implement** a `CircuitBreaker` class that opens after N consecutive failures (configurable threshold) and stays open for a configurable cooldown period. Feed it a sequence of 15 mock API calls where calls 3–7 fail and the rest succeed. Print the call number, success/failure status, and circuit state (`closed`, `open`, `half_open`) for each call. Verify that once the breaker opens, subsequent calls are short-circuited without hitting the mock API. Confirm the breaker resets after the cooldown expires.

4. **Apply** three different degradation strategies to a Clay-style enrichment pipeline that pulls 20 companies from a mock Apollo endpoint. Configure stage 1 (fetch) to **retry** up to 3 times on transient failures, stage 2 (enrich) to **skip** records with missing LinkedIn URLs, and stage 3 (score) to use a **fallback** static score of 50 when the scoring API is unreachable. Run the pipeline against a mock server that returns deliberate errors for 30% of requests. Print a per-stage summary showing how many records were retried, skipped, and fallback-scored, and write the final dataset to `outputs/capstone_companies.json`.

5. **Build** a complete pipeline orchestrator at `handlers/pipeline_orchestrator.py` that integrates intermediate persistence, contract validation, a circuit breaker, and per-stage degradation policies into a single runnable system. The orchestrator must accept a JSON config file specifying stage names, input/output paths, required fields, circuit breaker thresholds, and degradation strategy per stage. Run it against the dataset from Exercise 4 with one stage configured to fail intentionally. The orchestrator must write structured JSON logs to `outputs/pipeline_run_log.json` containing one entry per stage with: timestamp, stage name, records in, records out, errors encountered, circuit state transitions, and degradation actions taken. Confirm the log pinpoints which stage failed and how the pipeline responded.

6. **Design** a degradation policy matrix and prove it with data. Create `outputs/skill-degradation-matrix.md` documenting which strategy (skip, retry, fallback) you would assign to each of the following GTM pipeline stages: Apollo company search, Clearbit logo fetch, OpenAI email classification, CRM webhook delivery, and LinkedIn profile scrape. For each stage, justify the choice in 2–3 sentences, then run your orchestrator from Exercise 5 against mock implementations of all five stages with a 25% failure rate injected. Include the resulting pipeline run log summary in the document and state whether the observed behavior matched your design intent.
