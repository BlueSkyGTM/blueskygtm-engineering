# Exercises — KV Cache, Flash Attention & Inference Optimization

## Exercises

1. **Compute and compare FLOP costs.** Write a Python script that takes `seq_len`, `d_model`, and `n_heads` as command-line arguments and prints (a) the total FLOP estimate for one autoregressive attention step *without* a KV cache and (b) the same step *with* a cache containing `seq_len - 1` prior tokens. Use the standard approximation: each multiply-add is 2 FLOPs. Run it at `seq_len = 512, 2048, 8192` with `d_model = 4096` and `n_heads = 32`. Print a formatted table to stdout showing the ratio of cached to uncached FLOPs at each length.

2. **Trace shape transformations.** Build a function `trace_cached_attention(batch, seq_len, n_heads, d_head)` that returns a list of `(stage_name, shape_tuple)` tuples covering: input projection of the new token, new K and V production, cache state before append, cache state after append, Q against cached-K matmul, and the final attention output. No model weights needed — just construct and manipulate `torch.zeros(...)` tensors of the correct shapes and return their `.shape` at each stage. Print each stage as `stage_name: [B, S, H, D]` to verify the flow matches the lesson's shape diagram.

3. **Implement a minimal KV cache.** Write a class `CachedAttention(d_model, n_heads)` with methods `forward(new_token_embedding, past_cache)` returning `(output, updated_cache)`. Implement the uncached variant in the same file. Generate a synthetic `[batch=2, seq_len=256, d_model=512]` tensor of random embeddings and run 50 autoregressive steps under both paths. Print wall-clock time for each and the speedup factor. Verify that the cached path produces numerically equivalent outputs (assert `torch.allclose` within tolerance).

4. **Compare HBM allocation.** Using `torch.cuda.memory_allocated()` (or `tracemalloc` on CPU), instrument a standard multi-head attention implementation and `torch.nn.functional.scaled_dot_product_attention` at sequence lengths `[1024, 4096, 16384]` with `d_model=2048, n_heads=16`. Print peak memory in MB for each variant at each length. Identify the threshold (in tokens) where the standard path exceeds 80% of available memory assuming a 24 GB GPU. Print that threshold.

5. **Model enrichment pipeline cost.** Build a CLI tool `python cost_model.py --batch-size 32 --concurrency 4 --avg-tokens 450 --hit-rate 0.15` that models a Clay-style enrichment pipeline hitting an LLM endpoint for company/lead scoring. Compute total tokens processed per hour, estimated API spend (assume $0.005 / 1K input tokens, $0.015 / 1K output tokens, 40 output tokens avg per call), and the savings a Redis response cache would provide at the given hit rate. Sweep hit rates from 0.0 to 0.9 in 0.1 increments and print a table. Mark the hit rate at which monthly API spend drops below $500 for 100K enriched rows. Write the full model as a reusable module to `signals/examples/enrichment_cost.py`.

6. **Design a cache-aware inference scheduler.** Implement a scheduler that accepts a stream of enrichment job descriptions (each with a prompt length estimate and dedup key) and decides, per job, whether to (a) serve from cache, (b) batch with other in-flight jobs, or (c) run standalone. Use a sliding window of `max_batch_tokens` and a TTL-based cache. Feed it 1000 synthetic jobs (random prompt lengths 100–800, 20% duplicate keys). Print throughput (jobs/sec), cache hit rate, average batch fill, and p95 latency estimate. Write the scheduler to `handlers/batch_scheduler.py` and a markdown summary of design tradeoffs to `outputs/skill-inference-scheduler.md`.
