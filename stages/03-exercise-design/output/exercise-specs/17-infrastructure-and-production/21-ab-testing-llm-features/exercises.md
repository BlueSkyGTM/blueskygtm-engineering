## Exercises

1. **Implement deterministic bucketing from scratch.** Write a function `assign_variant(user_id: str, num_variants: int, salt: str = "ab-test-v1") -> int` that hashes `(salt + user_id)` with SHA-256, takes the integer modulo of `num_variants * 10000`, divides by 10000, and returns the floor as the variant index. Generate 10,000 random user IDs, assign each to one of two variants, and print the count per variant. Confirm the split is within ±1% of 50/50. Re-run the script and confirm the same user IDs map to the same variants — determinism is the whole point.

2. **Compute a two-proportion z-test by hand.** Variant A received 1,200 sessions and 36 clicks (3.0% CTR). Variant B received 1,180 sessions and 52 clicks (4.4% CTR). Implement the pooled-proportion z-test from scratch — pooled p, standard error, z-score, and two-tailed p-value — without importing any statistics library beyond `math.erf`. Print each intermediate value and the final p-value. State whether you would reject the null at α = 0.05.

3. **Compute minimum detectable effect and required sample size.** Your baseline CTR is 3.0%. You want 80% power at α = 0.05 (two-tailed). Implement the sample-size formula for a two-proportion test: for effect sizes ranging from 0.5% absolute lift to 3.0% absolute lift (in 0.5% steps), print a table showing required sessions per variant. Then answer in a printed statement: if you can only collect 2,000 sessions per variant, what is the smallest lift you can reliably detect? This is the number that determines whether your test is worth running at all.

4. **Trace a vibes-based evaluation to its failure.** You are given a CSV at `data/llm_summaries_eval.csv` with columns: `user_id`, `variant` (A or B), `summary_text`, `analyst_rating` (1–5 scale from one reviewer), `clicked` (0 or 1). The analyst who rated these summaries swears Variant B is "clearly better written." Compute the mean analyst rating per variant and the actual CTR per variant. Print both. Then run a two-proportion z-test on the click data and a t-test (you may use `scipy.stats.ttest_ind`) on the analyst ratings. Report which metric shows statistical significance and which does not. Write one paragraph to `outputs/skill-vibes-vs-stats.md` explaining where the analyst's judgment diverged from measured user behavior and why.

5. **Build a feature-flag-gated A/B test pipeline with GrowthBook-style bucketing.** Implement a Python module `handlers/llm_ab_test.py` that does the following: (a) reads a JSON config file specifying two prompt templates and traffic allocation (e.g., 50/50); (b) uses deterministic hashing to assign each incoming `request_id` to a variant; (c) calls a mock LLM function (you define it — it returns a canned summary based on which prompt template is selected); (d) logs the variant assignment and a simulated outcome (click / no-click) to a JSONL file at `outputs/ab_test_events.jsonl`; (e) after N events, runs the two-proportion z-test and prints the result. Run the pipeline for 5,000 simulated requests and print the variant counts, CTR per variant, z-score, p-value, and your decision. Then write a brief evaluation of GrowthBook and Statsig to `outputs/skill-growthbook-statsig-eval.md` covering: which platform offers CUPED for pre-experiment covariate adjustment, which offers sequential testing to stop early, and which applies Bonferroni or Holm corrections for multiple metrics — and which of these matters most for LLM feature tests where output quality is noisy.

6. **Design a multi-metric experiment that accounts for multiple comparisons.** Your LLM feature has three metrics: CTR on summaries, user satisfaction score (1–5), and time-to-first-action (seconds). Design and implement a test harness that simulates data for all three metrics across two variants, runs a z-test or t-test on each, and applies a Holm-Bonferroni correction across the three p-values to control family-wise error rate. Print the raw p-values, the corrected thresholds, and which metrics survive correction. Then write one paragraph to `outputs/skill-multi-metric-design.md` arguing whether multiple-comparison correction is overkill or essential when one of your metrics is an LLM-output-quality score — a metric known to be high-variance and poorly defined.
