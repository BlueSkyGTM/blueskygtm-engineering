## Exercises

1. **Implement a confidence-weighted voting aggregator.** Write a function that accepts a list of diagnostic agent outputs—each a dict with `agent_name`, `hypothesis`, and `confidence` (0.0–1.0)—and returns the hypothesis with the highest aggregated confidence, along with the summed score. Feed it three agent outputs where two agents agree on one root cause at moderate confidence (0.6, 0.55) and a third agent disagrees at high individual confidence (0.9). Print the winner and its aggregated score. Verify that the two-agent coalition wins despite the lone dissenter's higher individual score.

2. **Implement a sliding-window anomaly detector for correlated telemetry.** Write a detector that takes two aligned time-series lists—API latency (ms) and error rate (%)—along with a window size. Compute the rolling mean and standard deviation for each series, then flag any timestamp where both series exceed 2 standard deviations from their rolling mean simultaneously. Feed it a synthetic 30-point dataset where latency ramps gradually from 80ms to 450ms and error rate stays flat until point 22, then spikes. Print the flagged indices and confirm the detector fires only when both signals are anomalous, not on latency alone.

3. **Trace a degraded agent output through a three-stage pipeline.** Build a sequential pipeline: triage agent (classifies incident) → root-cause agent (queries logs) → mitigation agent (proposes fix). Each stage receives the previous stage's structured output. Run the pipeline twice with the same input incident: once with a correct root-cause hypothesis (confidence 0.92) and once with a poisoned root-cause hypothesis (confidence 0.31, wrong service blamed). Print both full pipeline traces side-by-side. Identify and print the exact stage where the mitigation recommendation diverges and explain why the downstream agent cannot recover.

4. **Compare a static runbook against a dynamically generated runbook graph for a Clay enrichment timeout.** Build two representations for the same incident type "Clay webhook enrichment timeout." The static version is a hardcoded ordered list of remediation steps. The dynamic version is a graph (dict of nodes and conditional edges) generated from incident metadata fields such as `severity`, `affected_records`, and `retry_count`. For the dynamic graph, add a `guardrails` key on each edge listing the safety check required before traversal (e.g., "verify API quota remaining," "confirm no active campaigns depend on this enrichment"). Print both runbooks and output a summary count of how many guardrails the static runbook is missing that the dynamic one enforces.
