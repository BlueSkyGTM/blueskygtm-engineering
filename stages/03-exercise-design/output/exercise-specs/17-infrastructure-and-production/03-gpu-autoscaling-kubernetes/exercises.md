## Exercises

1. **Compare poll-based vs event-driven provisioning latency.** Implement a Python script that simulates both provisioning models for a burst of 6 GPU pods arriving simultaneously. Model Cluster Autoscaler as a 10-second poll interval plus sequential instance API calls (180s per node) plus image pull (300s per node, but only first pull per node type). Model Karpenter as event-triggered (0s delay) with parallel instance API calls (90s per node) and the same image pull cost. Print a side-by-side table showing per-pod ready time and the total wall-clock time until all 6 pods are running. Compute and print the latency gap in seconds and as a percentage.

2. **Trace DCGM false-positive saturation for a vLLM workload.** Write a script that simulates 60 seconds of vLLM serving telemetry. Generate synthetic `DCGM_FI_DEV_GPU_UTIL` readings that stay pinned at 98–100% (because CUDA kernels are always dispatching during inference loops), while simultaneously generating two replacement signals: HBM memory utilization (reflecting KV cache pressure) and `vllm:num_requests_waiting` (queue depth). Print all three metric streams in aligned columns. Then implement a threshold check: flag a scale-up recommendation when utilization exceeds 95% for 30+ consecutive seconds, and show that DCGM alone fires a false positive on every interval while the two replacement signals correctly suppress the recommendation until queue depth actually rises.

3. **Trace a gang scheduling admission decision and compute idle-cost penalty.** A training job submits a pod group requesting 8 GPUs across 4 nodes (2 GPUs per node). The cluster currently has 5 GPUs free across 3 nodes. KAI Scheduler's gang policy is set to `Wait` with a timeout of 600s. Implement a script that takes this allocation state, determines whether the pod group is admitted, partially allocated, or rejected, and prints the decision with reasoning. If partially allocated, compute the idle-cost penalty: the dollar cost of the 5 already-allocated GPUs sitting idle (use $2.50/GPU-hour for the instance type) for the full 600s timeout window, then print the total wasted spend if the remaining 3 GPUs never become available before timeout. Run the script with two additional scenarios of your choice and print results for all three.

Apply Karpenter's consolidation logic: standard consolidation considers all pods disruptable unless protected; `WhenEmptyOrUnderutilized` only touches nodes with no pods or low utilization.
