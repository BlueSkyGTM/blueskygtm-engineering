## Exercises

1. **Compute weight-memory budgets for six formats.** Implement a Python script that takes a model name and parameter count (e.g., `Llama-3-8B`, 8.03 B params) and prints a table to stdout with estimated weight memory (in GB) for FP16, INT8 per-tensor, GPTQ INT4 group-128, AWQ INT4 group-128, GGUF Q4_K_M, and MXFP4. Use the per-format overhead factors discussed in the lesson (group scales, zero-points, lookup tables). Run the script for Llama-3-8B and Mixtral-8x7B. Your terminal output should show each format's GB footprint side by side.

2. **Trace the quantization-dequantization round-trip and report error.** Implement a function that takes an FP16 weight tensor (use `torch.randn(128, 128)`), quantizes it to INT4 with per-group scales at group sizes 32, 64, and 128, then dequantizes back to FP16. For each group size, compute and print the mean squared error between the original and reconstructed tensor. Also implement a per-tensor variant (single scale for the entire matrix) and print its MSE alongside. Your terminal output should be four MSE values that decrease as granularity increases.

3. **Build a format selector given deployment constraints.** Implement a Python function `select_format(gpu, engine, workload)` that accepts a GPU name (e.g., `"A100-80GB"`, `"RTX-4090-24GB"`, `"Mac-M2-16GB"`), a serving engine (`"vLLM"`, `"llama.cpp"`, `"TGI"`, `"TensorRT-LLM"`), and a workload type (`"classification"`, `"chat"`, `"reasoning"`, `"multi-LoRA"`). Return the recommended quantization format and a one-line justification. Encode at least these rules: vLLM prefers AWQ or GPTQ W4A16; llama.cpp prefers GGUF K-quants; TensorRT-LLM on H100 prefers FP8 or NVFP4; multi-LoRA workloads should avoid INT4 weight quantization due to accumulated adapter error. Run your selector across four distinct (gpu, engine, workload) combinations and print each result.

4. **Apply VRAM-fit analysis to a multi-tenant CRM enrichment scenario.** You are deploying a model to classify Apollo-exported leads into 12 ICP tiers for a Clay enrichment workflow. The target GPU is a single A10G (24 GB). The model is Mistral-7B (7.24 B params). Assume a KV cache budget of 4 GB for concurrent batch inference. Implement a script that computes whether each of the six formats from Exercise 1 fits within the 24 GB envelope (weights + 4 GB KV cache + 2 GB overhead for activation buffers). Print a table with format, weight GB, total GB, and a `FIT` or `NO FIT` verdict. Your terminal output should make clear which formats are viable for this deployment.

5. **Diagnose calibration-dataset drift on domain-specific text.** Download a GPTQ-quantized model and its FP16 counterpart from Hugging Face (e.g., `TheBloke/Mistral-7B-Instruct-v0.2-GPTQ` and the original). Compute log-perplexity for both models on two datasets: (a) general English text from a small WikiText sample, and (b) a domain-specific sample of 50 B2B SaaS product descriptions (scrape these from public company landing pages or use a local file of CRM account notes). Print a table showing FP16 perplexity, GPTQ perplexity, and the relative degradation percentage for each dataset. If the degradation on domain-specific text exceeds 15% relative while general text stays under 5%, print a `CALIBRATION DRIFT DETECTED` warning. Write a one-page analysis to `outputs/skill-quant-calibration-audit.md` documenting your methodology, numbers, and recommendation (re-calibrate with domain data, switch format, or accept the loss).

6. **Design an end-to-end quantization strategy for a multi-LoRA GTM pipeline.** You serve a 14B reasoning model on a single H100 (80 GB) running vLLM. Three LoRA adapters are hot-swapped at runtime: one for sales-email generation, one for CRM note summarization, one for ICP classification. No single quantization format is obviously correct because LoRA adapters interact with base-model quantization error. Implement a benchmark harness at `handlers/quant-lora-benchmark.py` that loads the FP16 base model alongside AWQ-INT4 and FP8 versions, attaches each LoRA adapter, runs 20 inference prompts per adapter per format, and records tokens-per-second, peak VRAM (via `torch.cuda.max_memory_allocated`), and a quality proxy (ROUGE-L against FP16 outputs). Print a summary table to stdout. Write your deployment recommendation (which format, which trade-offs, and whether any adapter needs de-quantization) to `outputs/skill-quant-lora-recommendation.md`.
