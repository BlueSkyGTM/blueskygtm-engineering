## Exercises

1. **Compute the bandwidth ceiling for all four targets.** Build a Python script that prints a formatted comparison table of the four edge targets (Apple ANE, Qualcomm Hexagon, WebGPU, NVIDIA Jetson Orin Nano) with columns: peak TOPS, memory bandwidth (GB/s), model weights size at the target's optimal quantization (e.g., 4B params at INT4 for ANE, INT4 for Hexagon, FP16 for WebGPU, INT8 for Jetson), and the theoretical max decode tokens/sec if throughput is bandwidth-bound (`bandwidth / weights_size`). Run the script and confirm that the TOPS-to-tokens/sec ratio diverges wildly across targets — proving TOPS alone is not a decode predictor. Print the final table to stdout.

2. **Build a quantization decision function.** Implement a Python function `select_quantization(target: str, param_count: int) -> dict` that returns the recommended data type, bytes-per-parameter, total weight footprint in GB, and whether the model fits within the target's typical memory budget (ANE: 8–16 GB unified; Hexagon: 4–8 GB DSP-accessible; WebGPU: browser-managed, typically ≤2 GB; Jetson: 4–8 GB shared). Test it with a 3B and a 7B model across all four targets. Print a table showing which (target, model) combinations are feasible at which quantization level.

3. **Measure prefill vs decode latency on WebGPU.** Using `webllm` or `transformers.js` in headless Node.js mode, load a small model (e.g., Phi-3-mini or Qwen2-0.5B) via WebGPU. Write a script that sends a 64-token prompt, measures prefill time (first-token latency), then generates 32 tokens and computes average per-token decode latency. Print both numbers and the ratio (prefill / decode). This ratio reveals whether your bottleneck is compute-bound (prefill dominates) or memory-bound (decode dominates).

4. **Inspect operator-to-hardware partitioning for Core ML or TensorRT.** Convert a small PyTorch model (e.g., a 2-layer transformer or a BERT-tiny) to either Core ML `.mlmodelc` via `coremltools` or a TensorRT `.engine` via `trtexec`/TensorRT Python API. Print a summary table: operator → assigned hardware unit. Count how many ops landed on the accelerator (ANE/DLA) vs fallback (CPU/GPU).
