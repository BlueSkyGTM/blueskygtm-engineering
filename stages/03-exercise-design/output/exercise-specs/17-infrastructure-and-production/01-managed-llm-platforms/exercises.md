## Exercises

1. **Configure** an authenticated `InvokeModel` call to AWS Bedrock using boto3 with your current IAM role credentials. Send the prompt `"Summarize the value of enriched CRM data in one sentence."` to Claude 3 Haiku (`anthropic.claude-3-haiku-20240307-v1:0`). Print the raw response body and the response's `ResponseMetadata` (HTTP status code, request ID, retry count). Confirm the HTTP status is 200 and note the `RequestId` value.

2. **Compare** request body schemas by sending the same prompt (`"List three B2B lead scoring criteria."`) to both Azure OpenAI and Google Vertex AI. For Azure, use the `openai` Python SDK pointed at your Azure endpoint with `api_type="azure"`. For Vertex, use the `google-cloud-aiplatform` SDK. Print each provider's raw JSON response and extract: the completion text, the input token count, and the output token count. Observe how each provider nests these fields differently.

3. **Implement** a `route_completion(prompt, providers)` function that attempts inference across an ordered list of providers (e.g., `["bedrock", "azure", "vertex"]`) and falls back to the next provider on failure. Normalize every response into a common dictionary with keys `text`, `input_tokens`, `output_tokens`, `provider`, `latency_ms`, and `model`. If all providers fail, raise a `RuntimeError` with the last exception. Test by calling with a working provider list and then with an intentionally invalid provider first to confirm fallback works.

4. **Compute** the break-even monthly request volume at which Bedrock Provisioned Throughput becomes cheaper than on-demand pricing for Claude 3 Sonnet. Given: on-demand input price is $3.00 per 1M tokens, output is $15.00 per 1M tokens. Provisioned Throughput costs $22,000/month for a unit that sustains 500 output tokens per second. Assume a workload with an average of 200 input tokens and 400 output tokens per request. Print the break-even request count and the cost-per-request at that volume.

5. **Build** a diagnostic script that sends a single enrichment prompt (e.g., `"Given this company description, extract industry, headcount band, and tech stack: <sample Clay-exported company record>"`) to all three providers in sequence. Capture and print per-provider: time-to-first-token, total latency, token counts, rate-limit headers (or their absence), and any throttling or auth-error messages. Write a markdown summary table ranking the providers by latency and noting quota signal availability. Save the report to `outputs/skill-provider-diagnostics.md`.

6. **Design** a cost-aware routing policy for a GTM automation pipeline that processes 50,000 enrichment calls per day across mixed latency tiers. Implement a `CostAwareRouter` class in `handlers/cost_aware_router.py` that selects a provider per request based on: (a) prompt length thresholds that favor cheaper models for short prompts, (b) daily budget caps per provider, (c) latency-based promotion/demotion from a rolling window. Include a `simulate_day(calls)` method that prints the simulated daily spend, provider distribution, and average latency. Document your pricing assumptions and fallback rules in the class docstring.
