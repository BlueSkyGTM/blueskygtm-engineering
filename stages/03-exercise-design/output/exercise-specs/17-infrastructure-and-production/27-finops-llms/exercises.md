## Exercises

1. **Implement a token-level cost attribution ledger.** Create a script that reads a JSONL file of LLM request logs (each line containing `tenant_id`, `model`, `prompt_tokens`, `tool_tokens`, `memory_tokens`, `response_tokens`, and `timestamp`). Define a pricing table for at least three models (e.g., `gpt-4o`, `claude-3.5-sonnet`, `gpt-4o-mini`) with separate input and output rates per 1K tokens. Compute the direct cost of each request by summing all four token buckets at the input rate and adding response tokens at the output rate. Aggregate by tenant and print a table showing per-tenant spend, request count, and average cost per request for the billing period.

2. **Implement two overhead allocation strategies.** Using the tenant spend data from Exercise 1 (or new sample data with at least 5 tenants), define monthly overhead costs for four shared resources: vector database hosting ($450), embedding compute ($200), gateway proxy ($150), and observability stack ($180). Implement `allocate_equal(tenants, overhead)` that splits total overhead equally across all active tenants, and `allocate_proportional(tenants, overhead, token_volumes)` that splits overhead proportional to each tenant's total token consumption. Print a side-by-side comparison table showing each tenant's allocated overhead under both methods, along with the delta between the two.

3. **Compute per-tenant gross margin under both allocation strategies.** Given a JSON file mapping each tenant to their monthly subscription revenue (e.g., `{"acme": 4900, "globex": 1200, ...}`) and the cost outputs from Exercises 1 and 2, compute gross margin percentage for each tenant under equal-split and proportional allocation. Print a ranked table identifying which tenants are profitable (margin > 60%), break-even (30–60%), and subsidized (< 30%) under each method. Flag any tenant whose profitability classification flips between the two strategies and print a summary count of how many tenants change categories.

4. **Build a budget circuit-breaker with graceful degradation.** Implement a function `check_budget(tenant_id, current_month_spend, monthly_cap)` that returns a degradation level as a string: `"full_service"` when spend is under 80% of cap, `"reduced_context"` when between 80–95% (truncate `memory_tokens` to 500 and disable tool calls), `"cached_only"` when between 95–100% (serve only cached responses), and `"blocked"` when over cap. Write a test harness that simulates a tenant's spend increasing across 10 requests and prints the degradation level and the effective token budget at each step. Verify that the transitions fire at the correct thresholds.
