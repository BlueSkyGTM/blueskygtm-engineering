## Exercises

1. **Implement a mechanism comparison table.** Write a Python script that prints a formatted table comparing llama.cpp, Ollama, TGI, vLLM, and SGLang across four columns: batching strategy, KV cache management approach, quantization support, and prefix deduplication capability. Each cell should contain one concise phrase (e.g., "Static batching" or "PagedAttention / block-level"). Run the script and confirm the table renders with all five rows and four columns aligned in the terminal.

2. **Build a recommendation CLI.** Write a Python script that accepts three arguments via `argparse` — `--hardware` (cpu | amd | nvidia-hopper), `--concurrency` (integer), and `--workload` (chat | agentic | shared-prefix-batch) — and prints the recommended engine plus a one-sentence justification. Cover at least these cases: CPU-only with concurrency 1 → llama.cpp; shared-prefix-batch with high concurrency → SGLang; NVIDIA Hopper with 10,000 concurrent users → vLLM. Run it against all six combinations and confirm output each time.

3. **Benchmark vLLM throughput across concurrency levels.** Launch a vLLM server with memory-constrained flags (`--gpu-memory-utilization 0.5`, `--max-model-len 2048`) on any available model (e.g., `Qwen/Qwen2.5-0.5B-Instruct` or a local GGUF). Write a benchmark script that sends prompts at concurrency levels 1, 4, 8, and 16 using `asyncio` and `httpx`, measures wall-clock time for each batch, and prints a table of concurrency vs. tokens-per-second. Confirm throughput increases sub-linearly with concurrency before plateauing. If no GPU is available, run vLLM in CPU mode and document the scaling curve instead.

4. **Measure SGLang KV cache reuse.** Launch an SGLang server with prefix caching enabled (`--enable-radix-cache`). Construct a workload of 50 prompts that all share an identical 200-token system prompt (e.g., a GTM ICP definition) but have unique user suffixes. Send them sequentially, capture the server logs, and compute the percentage of input tokens served from the KV cache vs. computed fresh. Print the reuse ratio. Then send the same 50 prompts a second time and compare the reuse ratio — it should approach 100% on the second pass.

5. **Build an Apollo enrichment classifier on a self-hosted engine.** Pull 100 company records from the Apollo People Search API (or use a provided CSV fallback with company descriptions). Write a handler that sends each company description to a local vLLM or SGLang endpoint with a shared system prompt that classifies the company into one of five ICP tiers. Measure total wall-clock time for the batch, then re-run the same classification using llama.cpp's server and compare. Write the results — engine, total time, prompts/sec, and per-prompt latency p50/p95 — to `handlers/apollo_icp_classifier.py` as a self-contained script with a `benchmark()` function that prints the comparison table when run directly.

6. **Author an engine-selection decision document for a GTM platform.** Write a markdown spec that analyzes the following scenario: a Clay enrichment workflow needs to classify 50,000 leads per day using a locally-served 8B model, with burst concurrency up to 200 simultaneous requests on a single NVIDIA H100. The document must (a) recommend exactly one engine with justification grounded in the five mechanisms, (b) explain why TGI is excluded given its December 2025 maintenance-mode status and what specific risk that creates for a production pipeline, (c) include a fallback recommendation if the H100 is replaced with an AMD MI300X, and (d) specify the launch command with memory and batching flags. Save the document to `outputs/skill-engine-selection.md`.
