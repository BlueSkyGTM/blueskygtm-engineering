## Exercises

1. **Compute GPU resource waste from measured phase timings.** Write a Python script that takes prompt token count (e.g., 1024), prefill time (e.g., 0.45 s), decode time per token (e.g., 0.025 s), and target output length (e.g., 128 tokens). Compute: (a) total inference wall-clock time, (b) GPU time spent on prefill, (c) GPU time spent on decode, (d) the fraction of wall-clock time the GPU spends doing "the wrong phase" — i.e., when a decode slot is blocked waiting for a prefill batch to finish (or vice versa). Print all four numbers. Re-run with prompt length 4096 and observe how the waste fraction changes.

2. **Compute KV cache size and transfer time across interconnect options.** Write a script that takes model parameters (`num_layers=80`, `num_kv_heads=8`, `head_dim=128`, `batch_size=32`, `seq_len=2048`, `bytes_per_element=2` for FP16) and computes the total KV cache size in bytes and gigabytes. Then compute transfer time over three interconnects: NVLink (900 GB/s), InfiniBand HDR (200 Gb/s), and TCP via 100GbE (100 Gb/s). Print a table with interconnect name, bandwidth, and transfer latency in milliseconds. State which interconnect keeps transfer overhead under 5% of a typical 2-second decode window.

3. **Implement a two-process KV cache transfer simulation and measure latency.** Create a `prefill_worker.py` and a `decode_worker.py`. The prefill worker constructs a mock KV cache as a NumPy array sized to match a 7B model at `seq_len=512` (pick reasonable layer/head/dim values). It serializes the array and sends it to the decode worker over a Unix domain socket (or `multiprocessing.connection`). The decode worker receives the array, verifies its shape matches the expected dimensions, and sends back an acknowledgment. Both processes print timestamps. Run both processes, capture the send timestamp (prefill side) and receive-complete timestamp (decode side), and print the one-way transfer latency. Run three times and print the median latency.

4. **Compare colocated vs. disaggregated cost per million tokens.** Write a script that takes: GPU hourly cost (`$2.50/hr` for an H100), target throughput (`200 req/s`), average prompt length (`512 tokens`), average output length (`128 tokens`), prefill throughput per GPU (`4000 prompt tokens/s`), decode throughput per GPU (`800 tokens/s`), and KV transfer overhead per request (`8 ms`, which costs nothing in colocated mode). Compute the number of GPUs needed for colocated mode (where prefill and decode share the same GPUs, so throughput is gated by whichever phase bottlenecks) vs. disaggregated mode (where you size prefill and decode pools independently). Print a cost-per-million-output-tokens figure for each architecture. Then vary the KV transfer overhead from 2 ms to 20 ms in 2 ms steps and print the disaggregated cost at each step so you can see where disaggregation stops being cheaper.

5. **Build a pool-ratio advisor that outputs a recommendation.** Create a script at `handlers/pool_ratio_advisor.py` that accepts a JSON config with `prompt_token_dist` (mean and stddev), `output_token_dist`, `request_rate`, `prefill_gpu_throughput`, `decode_gpu_throughput`, `transfer_overhead_ms`, and `total_gpus` budget. The script computes the optimal prefill:decode GPU split that maximizes aggregate throughput, then determines whether disaggregation is worthwhile by comparing against the colocated baseline. It prints: recommended ratio (e.g., `3:7`), projected throughput, projected cost/token, and a boolean `disaggregate_recommended` with a one-line justification. Run it with three different configs (short-prompt chatbot, long-document summarizer, and a balanced RAG assistant) and print all three recommendations side by side. Artifact: `handlers/pool_ratio_advisor.py`.

6. **Design a disaggregated inference plan for a GTM enrichment workload.** Pull 200 company records from the Apollo People Search API (or load a CSV export from Clay with company descriptions and ICP fit scores). Characterize the workload: measure the distribution of prompt lengths (company description + enrichment fields) and expected output lengths (a fit-score rationale paragraph). Feed these distributions into your `pool_ratio_advisor.py` from Exercise 5. Write a deployment recommendation to `outputs/skill-disaggregated-gtm-plan.md` that includes: the measured workload statistics, the recommended prefill:decode ratio, the cost comparison (colocated vs. disaggregated) at your measured token distributions, and a paragraph explaining whether your GTM workload benefits from disaggregation and why. Include the raw Apollo/Clay request or CSV path you used so the analysis is reproducible. Artifact: `outputs/skill-disaggregated-gtm-plan.md`.
