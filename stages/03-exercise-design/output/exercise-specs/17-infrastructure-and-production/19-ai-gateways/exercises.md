## Exercises

1. **Configure a LiteLLM proxy with a two-provider fallback chain.** Write a `config.yaml` that defines `gpt-4o` (OpenAI) as the primary model and `claude-3-5-sonnet` (Anthropic) as the fallback, with a per-minute rate limit of 10 RPM on the primary. Start the proxy with `litellm --config config.yaml` and verify it boots on port 4000 by hitting `http://localhost:4000/health/liveliness` in a second terminal. Kill the server with `Ctrl-C` and confirm you see the shutdown log line.

2. **Trace a single request through the five gateway functions.** With your proxy from Exercise 1 running, enable debug logging by adding `general_settings: { master_key: sk-1234 }` and setting `LITELLM_LOG=DEBUG`. Send a chat completion request via `curl` to `http://localhost:4000/v1/chat/completions` with model `gpt-4o`. Inspect the terminal output and write a one-line annotation for each of the five functions — normalization, routing, fallback, caching, observability — identifying the exact log line or response header that corresponds to it. Paste your five annotations into `outputs/gateway-trace.txt`.

3. **Implement a spend-tracking script that queries LiteLLM's transaction log.** LiteLLM stores usage data in a SQLite database (configured via `database_url`). Write a Python script that connects to this database, queries the spend table, and prints a per-provider cost breakdown: total tokens, total spend in USD, and number of requests for each provider in the last 24 hours. Run the script after sending at least five requests through the proxy and confirm the numbers reconcile against the response `x-litellm-response-cost` headers.

4. **Deploy LiteLLM in Docker with budget caps and retry policies.** Write a `Dockerfile` and `docker-compose.yml` that run LiteLLM with your `config.yaml` mounted as a volume. Configure a project-level budget cap of $5.00/day and a retry policy of 3 attempts with exponential backoff. Run `docker compose up --build`, send three requests through the containerized proxy on the published port, and verify the budget endpoint (`/budget/info`) reflects accumulated spend. Then intentionally send a request to a deliberately broken model name and confirm the retry log lines appear before the final error.

5. **Build a GTM enrichment handler that routes prospect data through the gateway.** Create `handlers/prospect_router.py` that accepts a list of Clay-exported prospect records (JSON with `company`, `title`, `linkedin_url`), sends each through your LiteLLM proxy for AI-powered icebreaker generation using `gpt-4o`, and falls back to `claude-3-5-sonnet` on failure or rate-limit hit. The handler must log per-prospect cost attribution (provider, tokens, USD) to stdout and write the final enriched list to `outputs/enriched_prospects.json`. Test with at least 10 sample prospects and verify the output file contains the generated icebreaker and cost data for each record. The script should read the proxy URL from an environment variable so it works against both local and Docker deployments.

6. **Benchmark LiteLLM overhead against direct API calls and document the four-gateway comparison.** Write a script that sends 50 identical chat completion requests two ways: directly to the OpenAI API and through your LiteLLM proxy. Measure p50 and p95 latency for each path. Then write `outputs/skill-ai-gateway-comparison.md` containing three sections: (a) your measured overhead numbers with a brief interpretation, (b) a feature-matrix table comparing LiteLLM, Portkey, Kong AI Gateway, and Bifrost across deployment model, scale ceiling, latency profile, and pricing model based on each vendor's public documentation, and (c) a recommendation for a 50-person GTM team running 10K enrichment calls per month. The markdown file must be well-structured and self-contained.
