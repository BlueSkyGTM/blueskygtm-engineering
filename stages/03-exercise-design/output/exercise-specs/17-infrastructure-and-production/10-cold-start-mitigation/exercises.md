# Exercises — Cold Start Mitigation for Serverless LLMs

## Exercises

1. **Implement** a Python script that prints each of the five cold-start mitigation layers alongside one concrete implementation at each layer (e.g., `provisioned-concurrency`, `weight-cache on local SSD`). The script must read the layer names from a dictionary and output them in a numbered table to the terminal. Run it and confirm all five layers appear.

2. **Compute** the total cold-start time for a 13B parameter model (FP16) given these inputs: node provision = 8.5 s, weight download speed = 200 MB/s, VRAM load bandwidth = 600 GB/s, engine init = 3.2 s. Write a function that accepts model size in billions of parameters and precision as arguments, calculates weight size, download time, and VRAM load time separately, then prints a breakdown table with a final total. Confirm the total is printed to two decimal places.

3. **Build** a keep-alive scheduler using Python's `sched` or `threading.Timer` that sends a lightweight HTTP HEAD request to a configurable serverless endpoint URL at a configurable interval (default: every 4 minutes). The scheduler must accept the platform's idle timeout as an argument, compute the ping interval as `timeout * 0.8`, and log each ping with a timestamp and HTTP status code. Run it against `httpbin.org/status/200` for at least three cycles and verify the log output shows three successful pings.

4. **Compare** the monthly cost of four mitigation strategies for a workload that receives 50,000 invocations per month with a 5-minute idle timeout. Use these unit costs: provisioned concurrency at $0.0000156/GB-second for 26 GB allocated, keep-alive pings via Lambda at $0.0000002/ invocation (ping every 4 minutes), weight caching on 50 GB EBS at $0.10/GB-month, and a warm pool of one `g5.xlarge` at $0.522/hour. Write a script that outputs a sorted cost comparison table from cheapest to most expensive, then prints which strategy offers the lowest cost-per-invocation and which offers the lowest P99 latency (assume warm pool and provisioned concurrency both eliminate cold starts, pings reduce cold-start rate by 90%, and weight caching reduces cold-start time by 60%).

5. **Trace** a cold-start event from a provided log format. Create a JSON log file at `outputs/cold-start-trace.json` containing a simulated cold-start event with timestamps for each phase (node_provision_start, node_provision_end, weight_download_start, weight_download_end, vram_load_start, vram_load_end, engine_init_start, engine_init_end, first_token). Write a parser at `handlers/cold-start-tracer.py` that reads the log, computes the duration of each phase, identifies the longest phase, and prints which mitigation layer (by name from Exercise 1) would eliminate that bottleneck. The script must output a phase-by-phase breakdown table and a final recommendation line like `MITIGATION RECOMMENDATION: <layer_name> — would save <N>s`.

6. **Design** a cold-start monitoring harness for a GTM enrichment pipeline. Assume your team runs a serverless LLM endpoint (via Modal, Replicate, or Lambda) that Clay webhooks call to score inbound leads in real time. Build a script at `signals/examples/clay-cold-start-guard.py` that simulates a webhook payload, fires it at a configurable endpoint, measures end-to-end latency, classifies the response as `WARM` (< 2 s), `COLD` (2–15 s), or `TIMEOUT` (> 15 s), and appends the result to `outputs/cold-start-guard-log.csv` with columns `timestamp, latency_s, classification, lead_id`. Run it against five simulated payloads (use `httpbin.org/delay/1` and `httpbin.org/delay/8` as stand-ins for warm and cold endpoints) and verify the CSV file contains five rows with correct classifications.
