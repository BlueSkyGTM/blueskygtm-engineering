# Exercises — Multi-Region LLM Serving and KV Cache Locality

## Exercises

1. **Compute the KV cache memory footprint** for a Llama-2-70B deployment (80 layers, 8 KV heads, head_dim 128, FP16). Write a Python script that takes `seq_len` and `batch_size` as command-line arguments, prints the KV cache size in GB, the model weight footprint (~140 GB), and the total HBM required. For an A100-80GB node, print `FITS` or `OOM` depending on whether the total stays under 80 GB. Verify by running with `--seq_len 4096 --batch_size 16` and `--seq_len 8192 --batch_size 32`.

2. **Trace the prefill vs decode compute cost** for a single request of 1024 prompt tokens generating 256 output tokens on a Llama-2-70B-class model. Write a script that computes approximate prefill FLOPs (all prompt tokens through all layers in parallel) and per-step decode FLOPs (one token, attending over growing KV cache), prints both, and then prints the simulated TTFT for a cold-cache node (full prefill) vs a warm-cache node (prefix already cached, prefill skipped for matched tokens). Use a notional GPU throughput of 312 TFLOP/s FP16 to convert FLOPs to milliseconds.

3. **Implement a prefix-cache-aware router simulator** in Python. Define three simulated GPU nodes, each with a dict of cached token-hash prefixes and a current utilization float (0.0–1.0). Feed the router a list of 50 incoming requests, each carrying a prompt-prefix hash and a token count. The router should: (a) check which nodes have the prefix cached, (b) among cache-hit nodes, pick the one with lowest utilization, (c) if no hit, route to the lowest-utilization node overall. After routing all 50 requests, print each node's final utilization, cache-hit count, and cache-miss count. Seed the initial caches so that roughly 60% of incoming prefixes have at least one hit.

4. **Diagnose cache miss rates across a simulated three-region deployment.** Create a workload of 200 requests using realistic GTM prompt prefixes — company descriptions pulled from Apollo or Clay enrichment data where multiple prospects from the same account share identical system prompts and company-context blocks (the shared prefix). Assign each request to one of three regions under three policies: round-robin, sticky-by-account-id, and hash-based. Write a script that replays the workload under each policy, prints the per-region cache-miss rate, and identifies which policy produces the lowest global miss rate and why. Output a short diagnostic table to the terminal.

5. **Build a multi-region KV cache strategy comparison tool** that accepts a workload trace (JSON array of `{request_id, prompt_tokens, output_tokens, region, prefix_hash}`) and computes total compute cost (FLOPs), peak memory cost (GB), and inter-region bandwidth cost (GB transferred for migration or cache sync) under three strategies: sticky routing, active migration, and stateless prefix caching (re-prefill on every cold node). Write the tool as a standalone Python script, run it against a provided trace file representing a GTM personalization workload (batched enrichment calls generating account-specific outreach), and write a markdown summary of the three strategies' costs and the recommended strategy to `outputs/skill-kv-cache-strategy.md`.

6. **Design and justify a routing policy** for a GTM team running real-time AI outreach personalization across three regions (US-East, EU-West, APAC-South). The workload consists of Clay-enriched prospect prompts that share long account-description prefixes (2K+ tokens) but have short unique suffixes (200 tokens). Write a design document as `outputs/skill-kv-cache-design.md` that specifies: which of the three multi-region strategies you would deploy, the expected cache-hit rate given the prefix-sharing pattern, the failure mode if a region goes down, and the bandwidth budget required for your chosen strategy. Include a Python snippet at the end of the document that computes the expected monthly bandwidth cost in dollars (assume $0.02/GB cross-region transfer) for a volume of 500K requests/month.
