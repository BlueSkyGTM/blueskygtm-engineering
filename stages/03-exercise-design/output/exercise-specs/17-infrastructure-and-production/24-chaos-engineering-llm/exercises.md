# Exercises — Chaos Engineering for LLM Production

## Exercises

1. **Implement a single-failure injector.** Write a Python function `chaotic_completion(prompt, failure_type)` that wraps an OpenAI (or mock) completion call and injects one of three failures based on `failure_type`: `"rate_limit"` (raise a 429-style exception), `"timeout"` (sleep beyond a configurable threshold then return), or `"malformed"` (return a response dict missing the `choices` key). Call the function three times — once per failure type — and print the exception type or the raw response for each. Your terminal output should clearly show three distinct failure signatures.

2. **Capture steady-state metrics.** Build a `measure_baseline(prompts)` function that calls a real or mock LLM API 20 times with a fixed list of prompts and records, for each call: latency (ms), success/failure boolean, and token count in the response. Print a summary table showing mean latency, p95 latency, success rate, and total tokens consumed. This establishes the steady-state baseline you will compare against in later exercises.

3. **Build a chaos test harness.** Implement a `ChaosHarness` class that accepts a list of failure injections (each specifying probability and type) and runs a batch of prompts through the LLM client while randomly applying those injections. Configure it with a 30% rate-limit injection rate and a 20% timeout injection rate. Run 50 prompted calls and output a structured report (printed as formatted text) showing: failure rate by type, observed latency distribution, and how many calls eventually succeeded. No code scaffold is provided — design the class interface yourself.

4. **Configure retry and fallback strategies.** Extend your harness so that rate-limit failures trigger an exponential-backoff retry (max 3 retries) and malformed responses trigger a fallback that re-prompts with a stricter instruction prefix. Run the same 50-call batch from Exercise 3 and compare the before/after success rates. Print both runs side-by-side in a comparison table. Then swap in a Clay enrichment API call (e.g., looking up company data by domain) as one of the prompt sources and re-run — report whether the failure profile changes when live external data is involved.

5. **Design a resilience experiment with pass/fail criteria.** Write a script `signals/examples/resilience_experiment.py` that defines explicit pass/fail thresholds (e.g., p95 latency < 5s, success rate > 90%, zero unhandled exceptions), runs a chaos experiment mixing all three failure types with retries/fallbacks enabled, and exits with code 0 (pass) or code 1 (fail) while printing a formatted experiment report. Include a `--no-chaos` CLI flag that runs the same batch without injections so you can verify steady state passes. The artifact must land at `signals/examples/resilience_experiment.py` and be executable via `python signals/examples/resilience_experiment.py`.

6. **Implement a degradation monitor.** Build `handlers/chaos_monitor.py` that loads your steady-state baseline from Exercise 2 (via JSON file or function import), runs a chaos experiment, computes the delta between baseline and chaos metrics across latency, success rate, and error-type distribution, and writes a Markdown report to `outputs/skill-chaos-monitor.md` summarizing whether each metric breached a degradation threshold you define. The monitor should print a one-line verdict to the terminal (`MONITOR: PASS` or `MONITOR: DEGRADATION DETECTED`) and the full report should include a recommendations section listing which failure modes need stronger mitigation.
