# Exercises — vLLM Production Stack with LMCache KV Offloading

## Exercises

1. **Implement the KV cache memory formula and find the crossover point.** Write a Python script that computes KV cache bytes given `layers`, `kv_heads`, `head_dim`, `seq_len`, and `precision_bytes`, then iterates over sequence lengths from 1,024 to 131,072 (doubling each step) for both Llama-2-7B (L=32, H=32, D=128, weights≈13 GB at FP16) and Mistral-7B (L=32, H=8, D=128, weights≈14 GB at FP16). Print a table showing seq_len, KV bytes, KV GB, and whether KV exceeds weights. Your script should exit with a nonzero code if no crossover is found within the range.

2. **Trace the lifecycle of a KV block through the production stack.** Write a Python class `KVBlockTracker` that simulates a single token's KV block moving through four stages: (1) GPU HBM during prefill, (2) LMCache connector save to CPU DRAM, (3) disk eviction when CPU pool exceeds a configurable threshold, and (4) reload from disk back to GPU HBM on cache hit. On each transition, print a timestamped log line like `[12.3ms] KV block 0x4F -> GPU_HBM (prefill complete)` and maintain a running total of bytes moved. Configure a scenario where 5 blocks are allocated, CPU capacity is 3 blocks, and blocks 1 and 3 are later requested again — print the final move count.

3. **Build a benchmark harness that measures prefill latency with and without LMCache prefix caching.** Launch vLLM locally with the LMCache KV offloading connector enabled on a model of your choice (e.g., `meta-llama/Llama-3.2-1B`). Construct 20 requests: 10 with an identical 2,048-token prefix followed by a unique 64-token suffix, and 10 fully unique prompts. Send the shared-prefix requests twice (first pass populates cache, second pass hits it) and record per-request prefill time. Print a summary table comparing mean/median prefill latency for cache-hit vs. cache-miss vs. uncached requests.

4. **Apply the storage-tier framework to a Clay enrichment workload.** Write a Python script that connects to the Apollo People Search API (or a local mock if API credits are unavailable) to fetch 50 prospect records, then constructs LLM prompts that classify each prospect into an ICP tier. Every prompt shares a 1,500-token system prompt defining the scoring rubric, followed by a ~200-token prospect-specific suffix. Run the classification through vLLM three times — once with GPU-only KV, once with CPU offload via LMCache, once with CPU + disk — and log total wall-clock time and estimated KV memory saved per request. Print which tier wins and by how much.

5. **Build a production KV offload monitor as a runnable handler.** Implement `handlers/kv_offload_monitor.py` that wraps the vLLM OpenAI-compatible server, intercepts every `/v1/completions` request, and tracks: prefix length, whether a cache hit occurred (parse vLLM response headers or LMCache logs), prefill time, and which storage tier served the hit. The handler should write a JSON-lines log to `outputs/kv-offload-metrics.jsonl` and, on exit or SIGINT, print an aggregate summary: cache hit rate, mean latency delta (hit vs. miss), and total KV bytes served from cache. Test it against at least 20 requests with a shared system prompt.

6. **Design and prove an optimal KV tier configuration for a multi-tenant GTM enrichment service.** Write a design document at `outputs/skill-kv-tier-design.md` that specifies the recommended LMCache configuration (GPU cache size, CPU offload pool size, disk path, eviction policy) for a service that runs 24/7 against a pool of 8 shared system prompts ranging from 500–4,000 tokens, with a peak batch size of 32 concurrent requests on an A100 80GB. Include: (a) computed worst-case KV memory at peak batch, (b) expected prefix-sharing hit rate given the prompt pool, (c) a back-of-envelope throughput estimate comparing no-cache vs. CPU-offload vs. CPU+disk, and (d) the specific vLLM launch flags and LMCache environment variables your design requires. Justify each choice quantitatively using the formulas from the lesson.
