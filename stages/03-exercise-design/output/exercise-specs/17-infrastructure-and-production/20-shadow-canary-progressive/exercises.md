# Exercises — Shadow Traffic, Canary Rollout, and Progressive Deployment for LLMs

## Exercises

1. **Implement a shadow traffic mirror.** Write a Python script that accepts a list of simulated production request payloads (strings), sends each to two model functions — `stable_model(prompt)` and `candidate_model(prompt)` — and computes an aggregate similarity metric across all response pairs (use any text similarity function: Jaccard, cosine on bag-of-words, or character-level ratio). Print a summary table showing per-request similarity scores, the mean similarity, and the percentage of requests where similarity falls below 0.7. Confirm your output changes when you modify `candidate_model` to produce degraded responses.

2. **Build a deterministic canary router.** Implement `route_request(request_id, canary_pct)` that returns either `"stable"` or `"canary"` based on a hash of `request_id`. Generate 10,000 requests with sequential IDs at 10% canary allocation and print the actual canary percentage. Then re-run with the same IDs and confirm identical assignments — determinism means the same request ID always lands on the same model. Print the first 20 route assignments for visual verification.

3. **Simulate a progressive deployment state machine.** Implement a state machine with stages `["5%", "25%", "50%", "100%"]`, each with a configurable dwell time (seconds) and a promotion gate that checks an error-rate input. Feed the machine a timeline of simulated error rates (some passing, some failing) and print every state transition with timestamp, stage, error rate observed, and decision (promote, hold, or rollback). Demonstrate both a successful full promotion and a mid-stage rollback by varying the input error rates.

4. **Apply guardrail detection to a GTM enrichment pipeline.** Build a guardrail checker that evaluates candidate-model responses for three LLM-specific failure modes: semantic drift (response embedding diverges from a reference set), cost explosion (token count exceeds a budget), and non-deterministic variance (running the same prompt 5 times yields divergent outputs). Use a simulated enrichment scenario: given a list of Clay prospect records, both models generate personalized icebreaker lines. Feed the outputs through your checker and print a per-record pass/fail report across all three guardrails, then print a summary showing which failure mode triggered most often.

5. **Design a policy-flag rollback controller.** Build a deployment controller that reads a JSON policy file specifying canary percentage, guardrail thresholds, dwell times, and a rollback target. The controller should run a simulated rollout loop: increment traffic, observe metrics (provided as a callable or data stream), evaluate guardrails, and execute rollback by flipping the policy flag to the stable model. Print the full decision log. On rollback, print the elapsed time between guardrail violation and flag flip in milliseconds. Save your controller as `handlers/rollback_controller.py` and the policy file as `handlers/deployment_policy.json`.

6. **Build a full progressive deployment simulator for an LLM-powered outbound workflow.** Combine shadow traffic, canary routing, guardrails, and policy-flag rollback into a single end-to-end simulation. Use Apollo or mock CRM prospect data as input — the LLM generates account summaries or enrichment fields. Your simulator should run a shadow phase (compare metrics), progress through canary stages with guardrail checks, and execute automatic rollback on failure. Produce a deployment report at `outputs/skill-progressive-deployment.md` containing the timeline, metrics at each stage, guardrail evaluations, and the final promotion or rollback decision with justification.
