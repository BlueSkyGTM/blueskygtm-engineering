# Exercises — TensorRT-LLM on Blackwell with FP8 and NVFP4

## Exercises

1. **Compute** the HBM footprint of Llama-3-70B under BF16, FP8, and NVFP4. Write a Python script that takes parameter count, bytes-per-parameter (including the 0.03125 byte/element NVFP4 scale overhead), and a KV-cache estimate, then prints a table to stdout showing total weight memory, KV-cache memory, and combined footprint for each precision. Run the script and verify that NVFP4 shows ~3.8× reduction versus BF16.

2. **Trace** the TensorRT-LLM compilation pipeline end-to-end. Invoke `trtllm-build` on a small HuggingFace checkpoint (e.g., `TinyLlama/TinyLlama-1.1B-Chat-v1.0`) with `--log_level verbose` and capture the terminal output to a file. Grep the log for each pipeline stage — checkpoint loading, calibration, quantization, kernel autotuning, engine serialization — and print the line numbers where each stage begins. Confirm that every stage appears in the expected order.

3. **Build** an NVFP4 calibration configuration from scratch for Mixtral-8x7B. Write a Python script that constructs a `QuantConfig` object with NVFP4 quantization mode, specifies a calibration dataset (a JSONL file of at least 512 samples), a max sequence length of 4096, and the appropriate sample count. Serialize the config to JSON and print it to the terminal. Then invoke the TRT-LLM quantization tool with your config and confirm it begins processing calibration samples without errors.

4. **Compare** FP8 and NVFP4 serving economics on a Blackwell B200. Run the TRT-LLM `gptManagerBenchmark` (or `mmlu` throughput harness) for both precision modes on the same model. Write a Python script that parses both benchmark JSON outputs and prints a comparison table: tokens/second, VRAM utilization, and cost-per-million-tokens assuming $2.50/hour B200 pricing. Identify which precision wins at batch size 1 versus batch size 256 and explain the crossover in a printed summary.

5. **Build** a batch enrichment handler that runs a TRT-LLM NVFP4 engine over Apollo-exported prospect data. Your handler should read a CSV of company descriptions, batch them into the engine at a configured concurrency level, extract a structured summary field from each response, and write enriched results to a new CSV. Include throughput logging (records processed per second) and VRAM monitoring via `pynvml`. Save the handler to `handlers/nvfp4-batch-enrichment.py` and run it on a sample of 100 Apollo records, printing final throughput and cost-per-record to stdout.

6. **Design** a decision framework for when TRT-LLM's NVIDIA lock-in is justified versus portable alternatives (vLLM, TGI) on Hopper H100. Write a script that benchmarks an identical model and workload on TRT-LLM FP8 and vLLM BF16, collects tokens/second, VRAM, time-to-first-token, and deployment complexity (measured as setup step count). Produce a markdown report at `outputs/skill-nvfp4-lockin-analysis.md` containing a recommendation table with a quantitative throughput-gap threshold (e.g., "TRT-LLM is justified when throughput exceeds vLLM by ≥1.8×") and at least three workload archetypes where the recommendation flips. Print the report path to the terminal.
