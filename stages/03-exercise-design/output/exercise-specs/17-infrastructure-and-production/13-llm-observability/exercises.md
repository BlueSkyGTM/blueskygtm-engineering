# Exercises — LLM Observability Stack Selection

## Exercises

1. **Configure a basic trace pipeline.** Build a Python decorator called `@trace_llm` that wraps any function making an OpenAI API call and logs a structured JSON record to stdout containing: the full prompt messages array, the model response, `usage.total_tokens`, and `latency_ms` (measured with `time.perf_counter`). Call it against a `gpt-4o-mini` completion with a system prompt of your choice. Verify by piping the output through `jq` to confirm every required field is present and non-null.

2. **Compute per-request cost from a trace record.** Write a standalone function that takes a JSON trace record (the kind produced by Exercise 1) and returns a cost figure using a pricing table you hardcode for three models (`gpt-4o`, `gpt-4o-mini`, `gpt-3.5-turbo`). The function must split input vs output token cost. Feed it three sample trace JSON objects — one per model — and print a table showing model, input tokens, output tokens, and total cost per call. Confirm the math against OpenAI's published pricing.

3. **Implement an evaluation hook for format compliance.** Build a function called `check_json_compliance(response_text: str) -> dict` that returns `{"passed": bool, "errors": list[str]}` by attempting to `json.loads` the response and validating required keys (`company`, `intent`, `score`). Feed it five sample LLM outputs — three valid, two subtly broken (trailing comma, missing key) — and print the pass/fail verdict for each. This is the same mechanism an observability loop would call on every traced response.

4. **Compare two tracing approaches against a Clay enrichment workflow.** Set up a mock enrichment function that takes a company domain, calls an LLM to classify intent (ICP fit vs. not), and returns a JSON verdict. Instrument it two ways: (a) with LangSmith tracing enabled, (b) with a hand-rolled OpenTelemetry span. For each approach, run 5 requests and capture: whether the full prompt-output pair is retrievable, whether token counts are visible, and whether you can filter by a `workflow_id` tag you attach. Print a comparison table showing which mechanism satisfies each requirement. No starter code — wire up both clients yourself.

5. **Build a production observability handler with regression detection.** Create a handler module that wraps an LLM call and runs three checks post-completion: cost attribution tagged to `campaign_id`, latency budget enforcement (alert if > 2s), and an evaluation hook that compares the JSON output's `score` field against a rolling baseline of the last 20 responses (flag if mean drops more than one standard deviation). Store traces as newline-delimited JSON in `outputs/traces.jsonl`. Seed the baseline with 20 historical responses at a known score distribution, then submit 5 new responses with a degraded score. Print which traces were flagged and why.
   **Artifact:** `handlers/observability_handler.py`

6. **Design and document a vendor-vs-self-hosted observability architecture.** Write a markdown spec that evaluates LangSmith, Helicone, Arize Phoenix, and a self-hosted OpenTelemetry + Postgres stack across five dimensions: prompt-level debug depth, evaluation hook integration, data retention controls, per-million-token pricing at 5M tokens/month, and deployment complexity for a 3-person GTM engineering team. Include a recommendation with justification and a sample trace schema the chosen tool would capture. Run a script that prints the comparison table to the terminal for quick scanning.
   **Artifact:** `outputs/skill-observability-stack-evaluation.md`
