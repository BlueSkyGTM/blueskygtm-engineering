## Exercises

1. **Decompose a vendor's pricing page into billing primitives.** Pick Fireworks AI's Llama 3 70B offering and write a Python script that prints a formatted table with four rows: per-token input rate ($/M tokens), per-token output rate ($/M tokens), per-GPU-second rate ($/sec, or "N/A"), and per-request surcharge ($/request, or "N/A"). Hard-code the rates from Fireworks' published pricing. Run the script and confirm the table prints cleanly.

2. **Compute a single-vendor monthly cost from a workload profile.** Write a script that takes a workload profile — 0.5 QPS, 1200 average input tokens, 300 average output tokens, 99.9% uptime over a 30-day month — and prints the projected monthly cost on Together AI (per-token) and on Modal (per-GPU-second, A100). Print both dollar amounts and the delta between them. Use published rates for a comparable model (e.g., Llama 3 8B).

3. **Build a multi-vendor monthly cost comparison.** Write a Python function `compare_vendors(profile)` that accepts a workload profile dictionary (QPS, avg_input_tokens, avg_output_tokens, concurrency, uptime_pct, model) and returns projected monthly costs for Fireworks, Together, Baseten, Modal, Replicate, and Anyscale. Print the results sorted cheapest to most expensive with dollar amounts and the pricing model each vendor uses.

4. **Find the crossover QPS where per-GPU-second beats per-token.** For a Llama 3 8B workload with 800 average input tokens and 150 average output tokens, sweep QPS from 0.1 to 50 in increments of 0.1. For each QPS value, compute the cost under Fireworks (per-token) and Baseten (per-GPU-second, A100). Print the QPS value at which Baseten becomes cheaper and the monthly cost at that crossover point. If no crossover exists in the range, print "No crossover found."

5. **Build a CLI cost calculator with sensitivity analysis.** Implement a command-line tool at `handlers/cost_calculator.py` that accepts a JSON workload profile file and a model name as arguments. The tool must compute monthly costs for all six platforms, rank them cheapest to most expensive, and then run sensitivity analysis by varying QPS, avg_input_tokens, and avg_output_tokens each ±50%. Print a formatted report to the terminal showing the base-case ranking and the best/worst-case rank for each vendor across the sensitivity sweep. Save a markdown summary of the report to `outputs/skill-cost-comparison.md`.

6. **Select inference platforms for two GTM workload archetypes.** Pull a sample Apollo company enrichment export (or generate 10,000 synthetic records matching Apollo's company schema with a `description` field averaging 400 tokens). Design two workload profiles: batch enrichment (all 10K records processed overnight, latency-tolerant, high throughput) and real-time scoring (single-record lookups triggered by a Clay webhook, p99 latency under 200ms, low sustained QPS). Run both profiles through your cost calculator and write a platform recommendation memo to `outputs/skill-platform-selection.md` that includes cost projections for each archetype, the selected platform with justification, and the crossover analysis showing why the cheaper batch platform fails for real-time.
