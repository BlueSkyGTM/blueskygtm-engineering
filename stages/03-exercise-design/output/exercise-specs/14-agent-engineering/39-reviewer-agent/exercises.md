# Exercises — Reviewer Agent: Separate Builder from Marker

## Exercises

1. **Implement** a two-pass builder-marker pipeline that generates and scores a cold email opening line. Create a builder function with its own system prompt that produces a JSON object `{"opening_line": "...", "intent": "..."}`. Create a separate marker function with a different system prompt that receives only the `opening_line` value and a 3-criteria rubric (relevance, specificity, tone — each scored 1–5), returning `{"scores": {"relevance": N, "specificity": N, "tone": N}, "total": N}`. Run both passes and print the builder output, then the marker output. Confirm the marker function never receives the `intent` field.

2. **Compare** single-actor vs. isolated-marker scoring on the same generated artifact. Build a third function where one LLM call both generates an opening line and self-scores it against the same rubric. Run the single-actor call and the two-pass pipeline on the same task prompt. Print both total scores and compute the delta. Run the comparison 5 times with different task prompts and print a summary table showing whether the single-actor consistently scores higher.

3. **Configure** a score threshold gate that reads a list of pre-generated artifacts and their marker scores from a JSON file, then routes each artifact to one of three paths: `approve` (total ≥ 13), `retry` (9–12), or `human_review` (≤ 8). Make the thresholds configurable via CLI arguments or environment variables. Print a routing report showing each artifact ID, its total score, and its assigned path, followed by a summary count per path.

4. **Apply** disagreement detection by running the same builder output through two different marker rubrics. Rubric A emphasizes persuasion and clarity. Rubric B emphasizes factual accuracy and compliance. Build both markers as separate functions with their own system prompts. Generate one artifact, score it with both rubrics, and print a criterion-by-criterion comparison. Flag any artifact where the two rubrics' total scores differ by more than 3 points with a `DISAGREEMENT` marker in the terminal output.

5. **Build** a complete GTM outreach review pipeline that pulls 5 prospect records from an enriched dataset (use exported Clay or Apollo data in CSV format with fields like `company`, `industry`, `headcount`, `recent_news`), generates a personalized first-touch email per prospect via the builder agent, scores each with an isolated marker using a 4-criteria rubric, and routes results through a threshold gate. Write the final approved emails and all routing decisions to `outputs/skill-outreach-reviewer.md` as a formatted markdown report with one section per prospect. Include the marker scores, routing decision, and approved/retry/human-review status for each.

6. **Design and implement** a multi-marker consensus system that evaluates marker reliability. Run a single builder output through 3 independent markers, each using a different rubric weighting (persuasion-heavy, accuracy-heavy, compliance-heavy). Compute a pairwise agreement matrix showing score deltas between every marker pair. Flag artifacts where any pair disagrees by more than 4 points. Test the system across 10 builder outputs and write a reliability report to `handlers/marker_consensus.py` containing the consensus engine and a summary of which rubric pairs produce the most disagreement. Print the agreement matrix to the terminal for visual verification.
