# Exercises — The Workbench on a Real Repo

## Exercises

1. **Clone and configure a real repository.** Clone the open-source project `https://github.com/leignercairn/signal-workbench` (a Python CLI for enrichment signal scoring). Install its dependencies in a virtual environment. Run the CLI's `--help` flag and then run the test suite with `pytest -q`. Capture the terminal output from both commands — the help text listing available subcommands and the test summary line showing pass/fail counts.

2. **Locate all entry points using grep.** Using `grep` and `find`, identify every entry point in the cloned repo from Exercise 1. Search for `argparse` subparser registrations, `click.command` decorators, and any symbols exported from `__init__.py`. Pipe the results to a text file called `entrypoints.txt` and print its contents to the terminal. Verify you have found at least three distinct entry-point functions.

3. **Trace a data flow from CLI invocation to final output.** Pick the `score-signal` subcommand in the repo. Starting from its argument parser registration, follow the import chain through every function that executes when the command runs. For each file in the chain, print the filename, function name, and a one-line summary of what it does — formatted as `file.py::function_name — summary`. Print the full chain to the terminal in order of execution.

4. **Apply the survey algorithm to a GTM enrichment codebase.** Clone `https://github.com/leignercairn/clay-enrichment-runner` (a script that pulls contacts from Apollo, enriches them via Clay, and writes results to a CSV). Map the module boundaries: identify the entry point, the data-fetching module, the enrichment module, and the output writer. Then trace the flow of a single contact record from Apollo API response through enrichment transformation to CSV row. Print a dependency diagram showing which module imports which, rendered as ASCII arrows (`module_a -> module_b -> module_c`).

5. **Execute a surgical modification with verification.** In the `clay-enrichment-runner` repo from Exercise 4, add a new field called `signal_grade` to the enrichment output. The field should assign a letter grade (A, B, or C) based on the existing `signal_score` numeric value (A ≥ 80, B ≥ 50, C otherwise). Implement the grading logic in the appropriate module — do not modify the API client or CSV writer directly; find the transformation layer where it belongs. Run the tool against the sample dataset bundled in the repo and confirm the new column appears in the output CSV with correct grades. Save your implementation to `handlers/signal_grader.py` and write a summary of the change to `outputs/skill-signal-grade.md` describing where you made the modification, why that module, and what terminal output confirms the change worked.

6. **Generate a structural summary for a teammate.** Choose any one of the repos from the previous exercises. Write a structural summary document to `outputs/skill-repo-map.md` that includes: a numbered list of every module and its single-sentence responsibility, a dependency diagram (ASCII arrows), the three most important entry points with their trigger conditions, and a labeled recommendation for where a future contributor should add a new enrichment provider integration. The summary must be specific enough that another engineer could open the repo and navigate directly to the correct file without further exploration.
