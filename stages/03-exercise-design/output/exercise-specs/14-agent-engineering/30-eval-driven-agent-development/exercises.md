# Exercises — Eval-Driven Agent Development

## Exercises

1. **Implement an exact-match grading function for a lead-classification eval suite.** Create a Python script that defines five test cases (input: company description string, expected: one of `["SMB", "MidMarket", "Enterprise"]`) and a `grade_exact(output, expected)` function that returns `1.0` or `0.0`. Run the suite against a mock classifier function you also write that returns predictions. Print each case's pass/fail status and the aggregate score. Verify the aggregate score prints as a float between 0.0 and 1.0.

2. **Implement an LLM-as-judge grading function for outreach email quality.** Write a `grade_llm_judge(email_text, criteria)` function that prompts an LLM with a scoring rubric (e.g., "Does the email mention a specific pain point? Is it under 150 words? Does it avoid generic openers?") and parses the returned numeric score. Run it against three sample emails you write — one clearly good, one clearly bad, one borderline. Print each email's score and confirm the good email scores highest. Use the API client and prompt patterns from the lesson.

3. **Build an eval suite that exposes a failure mode in a Clay enrichment output parser.** Write a mock enrichment function that takes a raw company payload (name, employee_count, industry, website) and returns a normalized record. Create eight test cases with edge-case inputs: missing fields, string where int is expected, industry as free-text vs enum, international domains. Write grading functions using structured extraction (regex or JSON parsing) that verify each output field's type and value. Print per-case results and identify which inputs cause the parser to fail. All test data should reflect realistic Apollo or Clearbit enrichment payloads.

4. **Run an eval-driven iteration cycle across two agent versions and measure improvement.** Implement two versions of a research-summarization agent (v1 and v2) — v1 can be naive (raw LLM call), v2 adds a prompt refinement (e.g., few-shot examples or output constraints). Build a shared eval suite with six test cases and a combination of exact-match and LLM-as-judge graders. Run both versions, print a side-by-side comparison table showing per-case scores and aggregate scores, and report the delta. Confirm v2 scores higher overall or identify which cases regressed.

5. **Diagnose a regression using eval trace output, then fix it and re-run.** Create an eval suite for a CRM-data-normalization agent with ten test cases. Introduce a deliberate bug in the agent (e.g., a regex that drops the `+` from international phone numbers, or a field-mapping error that swaps `city` and `state`). Run the suite, capture per-case trace output including the raw agent response and the grade, and print a diagnostic report that isolates which cases failed and what the agent returned vs. what was expected. Then fix the bug and re-run to confirm all cases pass. Write the final suite as `handlers/eval_regression_diagnostic.py` — it must be executable as `python handlers/eval_regression_diagnostic.py` and print a summary line `ALL CASES PASSED` or `N CASES FAILED`.

6. **Design and build a CI-runnable eval gate that blocks deployment on regression.** Create `signals/examples/eval_ci_gate.py` that loads a JSON file of test cases, runs them against a configurable agent endpoint (defaulting to a local mock function), grades results using at least two grading strategies from the lesson, computes an aggregate score, and exits with code `1` if the score drops below a configurable threshold (default `0.85`). Include a `--baseline` flag that loads a prior run's scores from a JSON file and fails the gate if any individual case regresses by more than `0.1` from baseline. The script must be runnable from a CI step with no manual intervention and must print a human-readable pass/fail summary to stdout. Verify by running it twice — once with a passing mock agent (exit 0), once with a degraded mock (exit 1).
