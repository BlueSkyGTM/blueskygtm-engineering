# Exercises — Repo Memory and Durable State

## Exercises

1. **Implement** a `StateManager` class backed by a JSON file at `state/pipeline.json`. The class must expose `load()`, `validate()` (enforce that `status` is one of `idle`, `running`, `completed` and that `processed_count` is a non-negative integer), `mutate(key, value)`, and `save()` methods. Write the save operation using a temporary file plus atomic rename so a crash mid-write cannot corrupt the file. Instantiate the class, mutate `status` from `idle` to `running`, call `save()`, and print the file contents to the terminal to confirm persistence.

2. **Configure** a `CLAUDE.md` file in a fresh Git repository. Include at least five conventions relevant to a Python GTM tooling project (e.g., "use `uv` for dependency management," "all HTTP calls must set a 10-second timeout," "never commit API keys"). Then invoke Claude Code with a prompt that would naturally violate one of the conventions — for example, "write a script that fetches data from the Apollo API" without mentioning timeouts. Print the generated code to the terminal and verify that Claude Code applied the convention without being reminded.

3. **Compare** read-order precedence by engineering a conflict. Create a `CLAUDE.md` that declares "use `unittest` for all tests." Then create a `.claude/settings.json` (or a second memory source documented in the lesson) that declares "use `pytest`." Finally, provide an inline instruction in your Claude Code prompt that says nothing about test framework. Ask Claude Code to generate a test file. Print the output, identify which framework was used, and write a one-paragraph explanation to `outputs/precedence-notes.md` documenting the observed precedence order and why it matters for team workflows.

4. **Build** a crash-safe enrichment pipeline state machine. The pipeline reads prospect rows from a CSV exported from Apollo, enriches each row by calling a mock enrichment function (a simple `time.sleep` + dictionary return is fine), and writes results to `outputs/enriched.csv`. The JSON state file at `state/enrichment.json` must track the index of the last successfully processed row. On startup, the pipeline reads the state file and resumes from the next row. Run the pipeline, kill the process with `Ctrl+C` after three rows, then restart it. Print the final row count of `outputs/enriched.csv` and confirm zero duplicate rows.

5. **Design and implement** a durable prospect-scoring handler that demonstrates correct separation of state across all three layers. Place project conventions (coding standards, API timeout rules, schema-field naming) in `CLAUDE.md`. Place pipeline progress and scored-prospect checkpoints in a JSON state file. Leave exploratory queries (e.g., "why did this prospect score low?") for chat history only. The handler must read prospect data from a Clay-exported CSV, apply a scoring formula, persist results idempotently to `state/scores.json`, and survive a mid-run restart without re-scoring any prospect. Produce the handler at `handlers/prospect_scorer.py` and confirm correct operation by printing the final state file contents after two interrupted runs.

6. **Design** a complete memory architecture for a GTM automation repository that orchestrates Clay table lookups, Apollo prospecting, and CRM sync. Produce a skill document at `outputs/skill-state-architecture.md` that specifies: what belongs in `CLAUDE.md` (with token-budget reasoning), what belongs in JSON durable state, what belongs in external stores like the CRM or Clay itself, and what should live only in chat history. Include a conflict-resolution protocol for when two developers update conventions simultaneously. Reference at least one concrete failure mode (e.g., context-window overflow from an oversized `CLAUDE.md`) and how your architecture prevents it.
