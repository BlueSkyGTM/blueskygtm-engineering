# Exercises — Memory Blocks and Sleep-Time Compute (Letta)

## Exercises

1. **Implement three core memory blocks for a GTM prospecting agent and print their state.** Create a Letta `AgentState` with three named memory blocks: `persona` (who the agent is), `icp` (ideal customer profile description, 500 char limit), and `active_deals` (key deal context, 1000 char limit). Seed each block with initial content. Write a script that instantiates the agent, sends a single user message (`"We just closed Acme Corp for $50k ARR"`), and prints the full contents of all three blocks after the agent responds. Verify that the agent used a memory tool to update at least one block — the printed state should differ from the seed values.

2. **Trace a single user message through the memory tool-call pipeline.** Using the agent from Exercise 1 (or a fresh one), enable Letta's step-level logging or inspect the `agent.steps` list after sending a message. Send the input: `"Update my ICP: we're now targeting Series A fintechs in Europe."` Then iterate over the returned steps and print every tool call name, its arguments, and the tool return value. Identify which step corresponds to a core-memory write and which steps are pure inference. Print a final summary line: `MEMORY WRITES: <count>`.

3. **Implement a sleep-time consolidation pass that migrates stale core-memory content to archival storage.** Build a background function that runs outside the primary agent's inference loop. It should: (a) read the current core memory blocks, (b) identify any `active_deals` entries older than a timestamp you embed in the content, (c) insert those entries into archival storage via `archival_memory_insert`, and (d) delete them from core memory using `core_memory_replace`. Trigger the pass manually after appending three fake deal records with varying timestamps. Print core memory before and after, and print the archival search results for one of the migrated deals to confirm it landed.

4. **Compare core-memory token cost against archival recall latency with a benchmarking harness.** Write a script that creates two agent configurations: one with a 2000-character `active_deals` block loaded in core memory, and one with that same data stored only in archival (core block empty, agent must call `archival_memory_search` to access it). Send the same query to both: `"What's the status of the Globex deal?"` Measure and print: (a) input token count for the first inference call in each config, and (b) wall-clock time from message send to final response. Print a comparison table with a recommendation line: which config is better if you expect >50 queries per session.

5. **Build a two-agent loop: a primary SDR agent that serves responses and a background memory manager that reorganizes during idle periods.** Create two Letta agents. The primary agent handles user messages about prospects (use a small sample of Apollo-exported leads as JSON input — at least 5 records with name, company, title, and LinkedIn URL). The background agent periodically reads the primary agent's recall memory, extracts firmographic patterns (e.g., "70% of engaged prospects are VP-level at companies under 200 employees"), writes the summary into the primary agent's `icp` core memory block, and archives the raw interactions. Run 5 user turns against the primary agent, trigger the background pass once, then run 2 more turns. Save the background agent's logic as a runnable module at `handlers/sleep_consolidator.py`. Print the `icp` block contents before and after the background pass to show the update.

6. **Simulate memory corruption and verify that checkpointing restores a known-good state.** Design an experiment where you deliberately corrupt a core memory block mid-conversation — for example, directly overwriting the `persona` block with malformed content (invalid JSON the agent expects to parse) after turn 3 of a 6-turn conversation. Show that the agent's behavior degrades on turns 4–6. Then implement a checkpoint-and-restore mechanism: snapshot the full `AgentState` (all memory blocks) after each successful turn. After corruption, restore from the turn-3 checkpoint and replay turns 4–6. Print a side-by-side comparison of responses with and without restoration. Write a short analysis (300–500 words) of where corruption entered the pipeline and how checkpointing bounds the blast radius. Save the analysis as `outputs/skill-memory-checkpointing.md`.
