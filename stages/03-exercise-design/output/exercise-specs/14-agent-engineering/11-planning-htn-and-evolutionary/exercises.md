# Exercises — Planning with HTN and Evolutionary Search

## Exercises

1. **Implement** a minimal HTN planner in a new file. Define the compound task `engage_prospect` with three methods — `warm_intro` (requires `has_mutual_connection`), `cold_outreach` (requires `not has_mutual_connection`), and `trigger_event` (requires `trigger_event_detected`). Each method decomposes into a chain of at least three primitive operators (e.g., `find_mutual_connection`, `request_intro`, `send_followup`). Given a world-state dictionary, your planner must match preconditions, select the applicable method, recursively decompose until only primitives remain, and print the resulting action list to the terminal.

2. **Trace** your planner across five prospect world-states of your own design — one that triggers warm intro, one cold, one trigger-event, one with two methods applicable (document which wins and why), and one where no method applies (print a clear fallback). Verify the output by confirming that every printed action sequence is fully primitive (no compound tasks remain) and that the chosen method's preconditions are satisfiable against the input state. Print each prospect's state, selected method name, and primitive chain.

3. **Compare** HTN decomposition against STRIPS-style forward state-space search on the same domain. Implement a second planner that treats each primitive as a STRIPS operator with preconditions and effects, then forward-searches from an initial state to a goal condition. Instrument both planners to record nodes expanded, maximum branching factor at any depth, and wall-clock time. Run both over the same portfolio of 10 prospects and print a side-by-side comparison table. Confirm that HTN explores strictly fewer nodes.

4. **Build** an evolutionary search loop that optimizes method selection across a portfolio of 20 prospects. Represent each individual as a chromosome mapping prospect index to method choice. Initialize a population of 50 random assignments, then run tournament selection (k=3), one-point crossover, and bit-flip mutation (rate=0.1) for 50 generations. Fitness is the number of prospects successfully reached (an assignment is valid only if the chosen method's preconditions hold for that prospect's state). Print the best fitness at each generation and the final best chromosome.

5. **Extend** the evolutionary loop to encode GTM cost constraints. Assign each method a credit cost (e.g., warm intro costs 5 enrichment credits, trigger-event costs 15) and an average latency (e.g., 300ms, 1200ms, 800ms). Configure a fitness function that penalizes any solution exceeding a daily credit budget of 500 and a per-prospect latency ceiling of 2000ms — invalid solutions receive fitness zero. Run the constrained evolutionary loop twice: once with free method selection and once constrained to only HTN-decomposable assignments. Print a comparison of credit spend, mean latency, and prospects reached for each run. Persist the combined planner to `handlers/gtm_htn_planner.py` and the comparison summary to `outputs/skill-htn-evolutionary.md`.

6. **Design and implement** a complete GTM prospect sequencing pipeline. Load 10 prospects from an Apollo CSV export or a Clay table export (you may use a provided sample). For each prospect, call at least one enrichment API — Hunter.io email verification, a LinkedIn mutual-connection check, or a company news trigger endpoint — to populate the world state. Run HTN decomposition to produce candidate engagement plans, then apply evolutionary search to optimize the full portfolio plan subject to a credit budget of 300 and a latency ceiling of 1500ms per prospect. Print the final sequence for each prospect. Persist the executable pipeline to `signals/examples/prospect_engagement_plan.py`.
