# Exercises — Production Runtimes: Queue, Event, Cron

## Exercises

1. **Implement** a pure Node.js in-memory queue with retry and exponential backoff. Your queue must accept jobs via an `enqueue(job)` function, process them sequentially with a single async worker, and retry failed jobs up to 3 times with backoff delays of `100ms * 2^attempt`. After max retries, move the job to a `deadLetters` array. Log every attempt, retry, and dead-letter to the console with timestamps. Submit a job that always fails and verify it lands in the dead-letter array after 3 attempts with increasing delays visible in the log.

2. **Extend** your queue from Exercise 1 to support idempotency keys. Each job carries a `key` field. If a job with the same key is already queued, in-flight, or completed within the last 60 seconds, reject the duplicate and log `DUPLICATE_REJECTED`. Enqueue the same key three times rapidly and confirm only one processes while the other two are rejected. Print a summary showing total enqueued, total processed, total rejected.

3. **Build** a cron-to-event-to-queue pipeline in a single Node.js script. Define a cron expression that fires every 2 seconds. On each tick, emit an event to a simple in-memory event bus (you build the bus — `emit`, `on`, `off`). At least two listeners should respond: one enqueues a "sync to CRM" job and another enqueues an "enrich account" job. Both feed into the same queue worker from Exercise 1. Run for 10 seconds, then print a structured summary: total cron ticks, total events emitted, total jobs processed per listener, any dead-letters. Confirm the counts reconcile.

4. **Diagnose** the correct runtime primitive for each of these five GTM scenarios and print a formatted table to the terminal: (a) hourly sync of 5,000 contacts from Apollo to HubSpot, (b) instant Slack ping when a Clay enrichment row completes, (c) nightly cleanup of stale CRM records older than 90 days, (d) real-time webhook handler for email reply events from SendGrid that must never drop, (e) weekly competitive intel scrape that aggregates 12 sources into a digest. For each scenario, output the runtime choice (queue, event, or cron), the delivery guarantee required, and a one-line justification grounded in the failure tolerance. No code scaffold — structure the output as a CLI tool that reads scenarios from a JSON file and prints the diagnosis table.

5. **Trace** a job end-to-end through all three runtimes and persist the trace as an artifact. Build a system where a cron job fires every 5 seconds, emits an event, and the event triggers a queue job that simulates an Apollo person-search API call (mock the HTTP with a 20% random failure rate). Assign each job a UUID and log a structured trace entry at every hop: `SCHEDULED → EVENT_EMITTED → QUEUED → PROCESSING → RETRYING → ACK` or `DEAD_LETTERED`. After 30 seconds of runtime, write the full trace log to `outputs/skill-runtime-trace.md` as a markdown table sorted by job UUID. Include a summary section counting jobs per terminal state. Verify the markdown renders cleanly and every failed job shows the full retry chain before reaching dead-letter or ack.

6. **Design** and implement a multi-worker queue with configurable concurrency and per-tenant fairness. Accept jobs tagged with a `tenantId`. Enforce that no single tenant can consume more than 50% of active worker slots when the queue is saturated with multiple tenants. Run a load test: enqueue 200 jobs across 5 tenants (one tenant submits 120 jobs, the others submit 20 each) and process with 4 concurrent workers. Log every job start and completion with tenant ID and worker slot. Write a summary report to `handlers/tenant-fairness-report.md` showing per-tenant throughput, average latency, and whether the fairness constraint held during the run. Include the raw counts and the fairness ratio (max-tenant-throughput / total-throughput) as evidence.
