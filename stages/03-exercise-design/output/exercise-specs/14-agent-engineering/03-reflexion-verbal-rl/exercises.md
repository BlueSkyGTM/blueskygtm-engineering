# Exercises — Reflexion: Verbal Reinforcement Learning

## Exercises

1. **Implement** a minimal Reflexion runner with three clearly labeled components. Write a Python script that defines three functions — `actor(task, reflections)`, `evaluator(response)`, `self_reflector(task, response, evaluator_signal)` — and a `run_reflexion` loop that calls them in order for up to 3 cycles. Use any simple task (e.g., "generate a string that contains the word 'hello' and is exactly 20 characters"). Print a labeled trace on each cycle showing which component is executing and what it produced. The script must terminate with either a success message or "max cycles reached."

2. **Trace** the episodic memory buffer through a 3-cycle Reflexion loop by hand and verify it programmatically. Start with an empty `reflections` list. Hard-code a task where the Actor always returns a slightly-wrong answer on cycles 1 and 2, and the correct answer on cycle 3. Hard-code the Self-Reflector to return distinct critique strings each cycle. After the loop completes, print the full `reflections` list and assert that its length equals the number of failed cycles. Confirm that each Actor call received the accumulated reflections from prior cycles by printing the reflection count at each call.

3. **Apply** Reflexion to a GTM cold-email drafting task using enrichment data. Fetch 3 prospect records from Apollo's People Search API (or load them from `signals/examples/apollo_sample.json` if you don't have an API key). Build a Reflexion loop where the Actor drafts a personalized cold email, the Evaluator is a heuristic function that checks: (a) prospect's company name appears, (b) email is under 150 words, and (c) no generic openers like "I hope this finds you well," and the Self-Reflector writes a critique for any failed draft. Run until all three heuristics pass or 4 cycles elapse. Print each draft, its evaluation breakdown, and the final reflection buffer.

4. **Compare** three evaluator types on the same generated output. Given a set of 5 sample sales-lead-qualification responses (hard-code them), implement: (a) a **scalar evaluator** that returns a 0–1 score based on number of required fields present, (b) a **heuristic evaluator** that checks whether the response contains a budget figure, timeline, and decision-maker name using regex, and (c) a **self-evaluator** that prompts an LLM to judge correctness against a rubric. Run all three on each response and print a comparison table showing where they agree and disagree. State in a printed summary which evaluator you would choose for a lead-scoring pipeline and why.

5. **Build** a complete multi-cycle Reflexion handler that qualifies inbound leads from CRM export data. Load lead records from `outputs/leads_export.csv` (create a 5-row sample file with columns: `company, industry, employees, revenue, source, notes`). The Actor proposes a qualification decision (SQL/MQL/UQL) with reasoning. The Evaluator cross-checks the decision against a rules engine (e.g., revenue > $5M and employees > 50 → SQL). The Self-Reflector critiques mismatches. Run up to 5 cycles per lead. Write the final qualification decisions and the accumulated reflection log to `outputs/skill-lead-qualification-reflexion.md`. Print a summary showing how many leads required more than one cycle and which reflections appeared most frequently.

6. **Design** and **Verify** a Reflexion system for a domain of your choice where you must justify your evaluator choice. Write a design document (saved to `outputs/skill-reflexion-design.md`) that: (a) names the task and why verbal reinforcement is better than fine-tuning for it, (b) specifies which evaluator type you chose and defends it against the alternatives, (c) identifies the context-window ceiling — how many reflections can accumulate before the Actor's prompt degrades — and proposes a mitigation strategy (e.g., summarization, sliding window, priority queue). Implement a working prototype in `handlers/reflexion_custom.py` that demonstrates the design on at least 3 test inputs with observable terminal output. Print the final design rationale and prototype results.
