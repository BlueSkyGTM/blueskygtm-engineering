# Exercises — Orchestration Patterns: Supervisor, Swarm, Hierarchical

## Exercises

1. **Implement a supervisor-worker loop for ICP qualification.** Write a Python script where a supervisor agent receives a company name, decides whether to call a `firmographic_worker` (checks company size, industry) or a `technographic_worker` (checks tech stack), collects the result, and either dispatches the other worker or returns a final qualification verdict. Print `[SUPERVISOR] routing to: <worker>` and `[WORKER: <name>] result: <value>` at each step so the control flow is visible in the terminal. Run the script on three test companies and capture the full trace.

2. **Implement a swarm pattern for the same ICP qualification task.** Remove the supervisor. Each worker agent receives the task, performs its check, and directly calls the next worker it thinks should run — or returns the final verdict. Print `[AGENT: <name>] handoff to: <next_agent>` at each step. Run the script on the same three companies from Exercise 1 and compare the trace output side-by-side with the supervisor trace.

3. **Implement a hierarchical topology for a multi-source enrichment pipeline.** Create two teams: an Enrichment Team (sub-supervisor + a `clay_worker` and a `apollo_worker`) and a Scoring Team (sub-supervisor + a `fit_scorer` and a `intent_scorer`). A top-level supervisor delegates to each team's sub-supervisor. Feed one Clay export row (company name, domain, employee count) through the full hierarchy. Print the complete call tree with indentation showing depth, e.g. `TOP → EnrichmentTeam → clay_worker`. Verify that no worker in one team ever calls a worker in the other team directly.

4. **Compute and compare token-cost estimates across all three patterns.** Load a 20-row Clay export CSV (`inputs/clay_export.csv`). For each row, simulate all three patterns by logging every "LLM call" as a token estimate: supervisor calls cost `task_tokens + state_tokens + worker_descriptions`, swarm calls cost `task_tokens + accumulated_context`, and hierarchical calls cost the sum of top-supervisor plus sub-supervisor overhead per team. Print a summary table showing total estimated tokens, estimated cost at $0.005/1K tokens, and estimated Clay credits consumed (assign 1 credit per enrichment worker call) for each pattern.

5. **Build a cost-aware orchestrator that selects the cheapest sufficient pattern.** Implement a function `select_pattern(task, clay_budget, token_budget)` that inspects the task description (number of distinct steps, whether steps share intermediate context, whether parallelism is needed) and returns `"single-agent"`, `"supervisor"`, `"swarm"`, or `"hierarchical"` with a printed cost justification. Then run it against five GTM task descriptions (e.g., "enrich 50 contacts from Apollo and sync to HubSpot", "score a single lead against ICP") and print the selected pattern plus estimated cost for each. Save the implementation to `handlers/cost_aware_orchestrator.py`. Run it and confirm terminal output shows all five selections.

6. **Design and prove the optimal topology for an end-to-end outbound pipeline.** The pipeline has five stages: Clay enrichment, Apollo contact lookup, ICP scoring, personalized email drafting, and CRM sync to HubSpot. Write a design document to `outputs/skill-orchestration-design.md` that includes: (a) which topology you chose, (b) an ASCII or Mermaid diagram of the agent structure, (c) a trace of one execution showing control flow, (d) a cost table estimating tokens and Clay credits for 100 prospects under your chosen topology versus the two alternatives you rejected, and (e) a one-paragraph justification for why a single agent with workflow patterns is or is not sufficient for this task. The document must be valid Markdown with headers and the table rendered as a GitHub-flavored Markdown table.
