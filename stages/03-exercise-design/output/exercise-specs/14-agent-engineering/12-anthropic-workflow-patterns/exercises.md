# Exercises — Anthropic's Workflow Patterns: Simple Over Complex

## Exercises

1. **Implement a prompt chaining workflow** with a programmatic gate between two steps. Write a function `chain_lead_extraction(raw_text)` that calls a mock LLM function (you define it — it returns a hardcoded response) to extract a company name and website from unstructured lead text, then validates the website is non-empty before passing it to a second mock LLM call that formats the result as JSON. Print a trace at each step showing the step name, input, gate result, and output. If the gate fails, exit early and print the reason. Run the function with three test inputs — one that passes the gate, one that fails it, and one edge case of your choosing.

2. **Implement a routing workflow** that classifies inbound leads by company size and dispatches to different handler functions. Write a function `route_lead(lead_dict)` that inspects the `employee_count` field and routes to `handle_smb`, `handle_mid_market`, or `handle_enterprise` based on thresholds you define. Each handler should print its route name and a mock enrichment strategy (e.g., "enriching via Apollo API" for SMB, "routing to AE for manual research" for enterprise). Print the routing decision and confidence before dispatching. Feed it at least four test leads spanning all three tiers plus a lead with a missing `employee_count` field, and confirm the trace shows each lead landing in the correct lane.

3. **Implement a parallelization workflow** that enriches a batch of five company domains concurrently using mock enrichment functions (e.g., `fetch_employee_count`, `fetch_tech_stack`, `fetch_funding`). Write `parallel_enrich(domains)` that fans out all calls using `concurrent.futures.ThreadPoolExecutor`, collects the results, and aggregates them into a single list of enriched dicts. Print start and end timestamps so you can observe the wall-clock time. Then write a sequential version that processes the same domains one at a time, and print both timings so the speedup is visible in your terminal output.

4. **Implement an evaluator-optimizer workflow** for cold email subject lines. Write a function `optimize_subject_line(company_name, product)` that generates a subject line (mock the generation call), scores it against three criteria (length under 50 chars, contains the company name, avoids spam trigger words), and loops — regenerating with the critique appended to context — until all three criteria pass or a max iteration count of 4 is reached. Print each iteration number, the generated subject line, the per-criterion pass/fail, and the final result. Run it with two company/product pairs and confirm the loop terminates correctly in your terminal.

5. **Build an orchestrator-workers enrichment waterfall** that maps the enrichment-waterfall GTM workflow to Anthropic's orchestrator-workers pattern. The orchestrator receives a company domain and a list of available enrichment sources (e.g., Apollo API, Clearbit, website scraping, LinkedIn). It selects which sources to query based on what data is already present, dispatches workers (as `concurrent.futures` calls) to query each selected source, then aggregates the results and decides whether another round is needed. Implement mock API calls that return realistic-looking data with varying completeness. Include at least one case where a worker returns a partial result and the orchestrator dispatches a follow-up worker to fill the gap. Print the full execution trace showing orchestrator decisions, worker dispatch, and aggregation. Save your implementation as a reusable module at `handlers/enrichment_waterfall.py` and run it from the command line with a domain argument to verify the trace output.

6. **Design a pattern-selection diagnostic** and implement it as a runnable script. Write a function `recommend_pattern(problem_description: str)` that takes a plain-text description of a GTM workflow (e.g., "enrichment waterfall for 5000 contacts," "route inbound leads to the right AE by territory," "generate and refine a multi-step outbound sequence") and returns a recommendation for which of the five Anthropic patterns to use — or whether a single augmented LLM call suffices — with structural justification referencing the coupling cost, failure mode that triggers the pattern, and the specific decomposition the pattern enables. Encode your reasoning as explicit rules (not an LLM call) so the diagnostic is deterministic and auditable. Include a test suite of at least six diverse GTM scenarios that exercise all five patterns plus the single-call case. Run the suite and print each scenario, the recommended pattern, and the justification. Write your structural analysis and decision rules to `outputs/skill-pattern-selection.md` so the artifact persists as a reference document.
