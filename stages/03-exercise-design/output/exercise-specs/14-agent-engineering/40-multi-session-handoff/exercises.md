# Exercises — Multi-Session Handoff

## Exercises

1. **Implement** a `serialize_session` function that takes a Python dict containing raw conversation messages, a list of entities, a list of decisions, and a list of pending actions, and returns a four-compartment state capsule as a JSON-serializable dict. Include a `schema_version` field set to `1`. Write the capsule to `outputs/capsule-v1.json` and print each compartment's key count to the terminal. Verify the file is valid JSON by loading it back in the same script.

2. **Implement** a `rehydrate` function that reads a capsule JSON file from disk, checks that `schema_version == 1`, and returns a formatted context string that could be injected as a system message into a new session. The context string must list each compartment as a labeled section. Run it against the capsule you produced in Exercise 1 and print the resulting context string. Then feed it a capsule with `schema_version: 0` and confirm your function raises or returns a clear error.

3. **Build** a `compress_transcript` function that accepts a list of raw message dicts (each with `role` and `content`) and reduces them into a structured payload of entities, decisions, and pending actions — no raw message text survives compression. Use any combination of regex, keyword extraction, or an LLM call. Your function must enforce a target token budget (passed as an argument) and drop or merge low-priority items when the budget is exceeded. Print a before-and-after comparison: original token count vs. compressed token count, computed via `tiktoken` or equivalent. Test against a 20-message synthetic conversation about a Clay enrichment workflow for a prospect.

4. **Apply** the migration pattern to a legacy capsule. Given a capsule at `schema_version: 1` that stores entities as a flat list of strings (e.g., `["Acme Corp", "Jane Doe", "Q3 budget cycle"]`), write a `migrate_v1_to_v2` function that upgrades it to `schema_version: 2` where each entity is a typed dict with `name` and `type` fields (company, person, or event). Load a sample v1 capsule, migrate it, write the result to `outputs/capsule-v2.json`, and print a diff showing what changed. Then write a `rehydrate` function that accepts both v1 and v2 capsules, auto-migrating v1 on load, and confirm both versions rehydrate successfully.

5. **Build** a capsule validator and rehydration pipeline at `handlers/capsule_pipeline.py`. The pipeline must accept a capsule file path, perform the following checks in order, and return either a rehydrated context string or a structured rejection: (a) file exists and parses as JSON, (b) `schema_version` is present and supported, (c) all four compartments exist and are non-empty, (d) no compartment exceeds a max-item threshold. On rejection, return a fallback context string that contains only the capsule's identifier and a `"state lost — cold start"` notice so downstream code always receives a string. Test with four files: a valid capsule, a corrupted JSON file, a schema-stale capsule, and a compartment-missing capsule. Print the validation result and output length for each. Include at least one test capsule whose payload mimics an Apollo prospect enrichment record (company domain, contact info, sequence stage) to demonstrate the pattern with realistic GTM data.

6. **Design and compute** a cost model that connects capsule size to enrichment economics. Build a script that loads the capsule from Exercise 1 (or any valid capsule you have produced), estimates its token cost using `tiktoken`, and calculates the projected monthly API spend for a workload that rehydrates 10,000 capsules per day at current GPT-4o pricing. Then compute the same figure for an un-compressed raw-transcript handoff of equivalent session length (assume 8,000 tokens per raw transcript). Print a comparison table showing: strategy, tokens-per-handoff, daily-cost, monthly-cost, and cost-as-percentage-of-a-$0.10-enrichment-credit. Write a one-paragraph analysis to `outputs/skill-capsule-cost-model.md` documenting whether compressed capsules are economically viable as a default handoff strategy for a GTM automation workflow running Clay + LLM enrichment at scale.
