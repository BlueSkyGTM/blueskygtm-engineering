# Exercises — AutoGen v0.4: Actor Model and Agent Framework

## Exercises

1. **Implement a two-actor message-passing system** using `asyncio` where each actor has its own `asyncio.Queue` as a private inbox. Actor A receives a string message on its inbox, transforms it (e.g., uppercases it), and sends the result to Actor B's inbox. Actor B receives the transformed message and prints it to the terminal. No actor may access the other's internal state directly — all communication goes through the queues. Run the system with `asyncio.run()` and verify that a message sent to Actor A appears transformed at Actor B's stdout.

2. **Trace a three-actor pipeline** where Actor 1 (Parser) receives a raw JSON string, Actor 2 (Enricher) adds a `"status": "processed"` key, and Actor 3 (Logger) prints the final payload. Add a print statement at the start of each actor's message handler that logs the actor name, the message type received, and the message content. Run the pipeline with a sample input and capture the full trace from the terminal. Verify that the output shows exactly three distinct actor boundaries and that the message type changes at each hop.

3. **Compare the actor model against shared-state concurrency** by implementing the same logic two ways: (a) two threads sharing a Python dictionary that accumulates results from simulated LLM calls, and (b) two actors communicating through message passing producing the same result. Introduce a deliberate `time.sleep` in the threaded version to expose a race condition. Run both implementations, capture the terminal output from each, and write a short analysis (3–5 sentences) identifying at least three failure modes that shared state introduces — such as lost updates, inconsistent reads, or deadlock — that the actor version avoids. Write the analysis as inline comments or a printout at the end of your script.

4. **Map a GTM enrichment waterfall onto an actor pipeline.** Build four actors chained together: (1) **SeedActor** receives a company domain string and emits `{"domain": domain}`; (2) **ApolloActor** receives the domain, simulates an Apollo API enrichment call (you may use a mock function that returns a company name and employee count), and emits the enriched payload; (3) **TechStackActor** receives that payload, simulates a tech-stack lookup (mock or real API call to a tool like Wappalyzer), and emits the further-enriched payload; (4) **CRMWriterActor** receives the fully enriched record and prints it as a formatted JSON object to the terminal. Each actor must own its own inbox and communicate only through messages. If you have Apollo or Clearbit API keys, use real calls; otherwise, mock the responses with hardcoded dicts. Verify by running the pipeline with `asyncio.run()` and confirming the terminal shows a progressively enriched record at each stage.

5. **Build a supervisor actor that restarts a failed child actor using only message passing.** Implement a `SupervisorActor` that spawns a `WorkerActor`, sends it a task, and monitors the worker's response. The worker should be designed to randomly fail (e.g., raise an exception with 50% probability during processing). When the worker fails, it must send an `ErrorMessage` back to the supervisor (not crash the entire program). The supervisor must respond to the `ErrorMessage` by recreating the worker's inbox, spawning a fresh `WorkerActor` with a new task, and logging `"Restarted worker after failure"` to the terminal. No shared variables, no cross-actor exception propagation — the supervisor learns about failure only because it receives a message. Run the system and verify that the terminal output shows at least one restart followed by a successful task completion. Save your implementation to `handlers/supervisor_actor.py` and confirm it runs with `python handlers/supervisor_actor.py`.
