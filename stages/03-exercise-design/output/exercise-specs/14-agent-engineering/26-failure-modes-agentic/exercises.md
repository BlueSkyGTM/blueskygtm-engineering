# Exercises — Failure Modes: Why Agents Break

## Exercises

1. **Classify** six pre-built agent trace files stored in `data/traces/` by writing a script that reads each JSON trace and prints the failure mode name (loop exhaustion, tool parameter drift, context window overflow, hallucination cascade, state loss, tool selection error) alongside the trace filename. Run the script from the terminal and verify your output matches the expected six classifications printed by `python data/traces/answer_key.py`.

2. **Trace** the step-by-step execution of a simulated agent loop by running `python simulate/loop_stall_demo.py` and capturing its log output to a file. Write a diagnostic script that parses the log, identifies the specific step where the stall begins, and prints the root cause (missing stopping condition, unresolvable tool dependency, or repeated identical calls). Your terminal output should end with a single line: `ROOT CAUSE: <your diagnosis>`.

3. **Implement** a drift-detection wrapper that sits between an agent loop and its tool registry. Configure it to intercept each outbound tool call, compare the call's parameters against the tool's declared schema, and emit a warning to stdout when any required field is missing or any parameter type is wrong. Test it against the provided `simulate/enrichment_agent.py` which makes ten sequential Apollo person-enrichment API calls — your detector should flag at least three calls as drifted and print a summary like `DRIFT DETECTED at step 7: missing 'email' field`.

4. **Compare** two trace files — `data/traces/scenario_a.json` and `data/traces/scenario_b.json` — and write a script that determines which trace exhibits a hallucination cascade (confident outputs that reference fabricated data compounding across steps) versus state loss (agent forgets earlier context mid-task). Print a structured comparison table to the terminal showing: trace name, classified mode, key symptom evidence, and step number where the failure first becomes observable.

5. **Build** a context window overflow detector that consumes a live message list from a running agent loop, estimates token count per message using the tokenizer from the lesson, and raises an alert when projected total tokens cross 80% of the configured model limit. Wire it into the existing `simulate/clay_enrichment_loop.py` agent that enriches leads pulled from a Clay export CSV, and confirm your detector fires before the loop crashes. Your detection logic must be saved as a reusable import at `handlers/overflow_detector.py` and produce terminal output like `OVERFLOW WARNING: 7,842 / 8,192 tokens projected at step 11`.

6. **Design** and implement a unified failure-mode classifier function that accepts any agent execution trace (JSON array of steps with role, content, tool calls, and tool results) and returns the single most likely failure mode plus a confidence score between 0 and 1. Your classifier must cover all six taxonomy categories. Test it against every file in `data/traces/` and print a results table with columns: filename, predicted mode, confidence, actual mode (from the answer key). Save your classifier at `signals/examples/failure_classifier.py` and save your results table as `outputs/skill-failure-classifier.md`.
