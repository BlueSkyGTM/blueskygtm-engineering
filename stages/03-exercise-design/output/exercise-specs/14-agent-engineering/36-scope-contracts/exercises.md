# Exercises — Scope Contracts and Task Boundaries

## Exercises

1. **Write a scope contract and load it at runtime.** Create a Python dict literal named `SCOPE_CONTRACT` with `IN_SCOPE`, `OUT_OF_SCOPE`, and `ESCALATION` keys for a "company research" handler. `IN_SCOPE` should cover company firmographics, tech stack detection, and industry classification. `OUT_OF_SCOPE` should cover contact-level data, email verification, and CRM writes. `ESCALATION` should route out-of-scope work to a `contact_enrichment` handler. Write a loader function `load_contract(path)` that reads a JSON file containing the contract and returns the dict. Save your contract to `contracts/company_research.json`, load it, and print all three fields. Verify the output shows each field with its full keyword list.

2. **Implement boundary enforcement.** Write `enforce_scope(task_description: str, contract: dict) -> str` that checks whether the task description contains any keyword from `IN_SCOPE` (return `"EXECUTE"`), any keyword from `OUT_OF_SCOPE` (return `"ESCALATE"`), or neither (return `"DEAD_LETTER"`). Test it against these five task strings and print the routing decision for each: `"Classify this company's industry"`, `"Verify the CEO's email address"`, `"Detect their CMS and analytics tools"`, `"Update the opportunity stage in Salesforce"`, `"Translate the homepage into French"`. Confirm that industry classification and tech detection route to EXECUTE, email verification and CRM writes route to ESCALATE, and translation routes to DEAD_LETTER.

3. **Build a multi-agent orchestrator with trace output.** Create three scope contracts (saved as JSON files): one for company research, one for contact enrichment, one for CRM operations. Write `route_task(task_description: str) -> dict` that iterates over all three contracts, calls your `enforce_scope` function against each, and returns a dict containing `task`, `matched_agent`, `routing_decision`, and `trace` — where `trace` is a list of every contract checked, the keyword that triggered the match, and the decision string. Feed it the task `"Find the VP of Marketing's LinkedIn profile and log it to HubSpot"` and print the full trace. The output must show which contract matched, which keyword fired, and what the routing decision was.

4. **Compare contract-based enforcement against prompt-based guidance.** Write a prompt-only scope description for the same company-research handler as Exercise 1 — a paragraph of natural-language instructions telling a model what to do and what to refuse. Then write a diagnostic function `compare_enforcement(test_cases: list[str]) -> None` that runs each test case through both the prompt-based approach (using a simple substring match against the prompt text) and the contract-based `enforce_scope` function. Print a side-by-side table showing agreement or disagreement for each case. Include at least one adversarial case where the prompt-based approach produces an ambiguous or wrong routing decision but the contract produces a clear one. Print which approach is testable, auditable, and versionable based on your results.

5. **Diagnose scope violations in a GTM enrichment pipeline.** You are given a log of 20 enrichment tasks that a Clay-style enrichment waterfall attempted to process (simulate these as a list of task description strings covering company lookups via Apollo's firmographics API, contact enrichment via Clearbit, email validation via MillionVerifer, and CRM field updates — some tasks are misrouted). Write a diagnostic script that loads two contracts (`company_research.json` and `contact_enrichment.json`), runs every logged task through `enforce_scope`, and writes a violation report to `outputs/scope-violation-report.md`. The report must contain: a table of every task with its expected handler versus its contract-determined handler, a count of misrouted tasks, and the specific contract keyword that produced each routing decision. Run the script and confirm the markdown file is written and contains at least one flagged misroute with the triggering keyword cited.
