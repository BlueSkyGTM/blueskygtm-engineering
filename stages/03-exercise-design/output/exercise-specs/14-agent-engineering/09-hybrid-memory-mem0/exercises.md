# Exercises — Hybrid Memory: Vector + Graph + KV (Mem0)

## Exercises

1. **Implement a three-store `ingest` function.** Write a Python function `ingest(entity: str, relation: str, value: str)` that writes the same fact to all three stores from the lesson's code: (a) append the text `"entity relation value"` to a list that you then embed with the lesson's `cosine_search` index, (b) add `(entity, relation, value)` as a typed edge in the adjacency-list graph, and (c) store `value` under the key `f"{entity}:{relation}"` in a dict. Call `ingest("Acme", "uses", "Snowflake")`, `ingest("Acme", "located_in", "Austin")`, and `ingest("Alice", "works_at", "Acme")`, then print the contents of each store to confirm all three hold the data. Your terminal output should show three distinct data structures populated with the same underlying facts.

2. **Compare isolated failure modes.** Using the stores you populated in Exercise 1, run three queries and print results side by side: (a) query the vector store with `"Where is Acme based?"` and observe whether it surfaces the `located_in` fact, (b) query the graph store with `"who works at Acme?"` and observe the traversal result, and (c) query the KV store with the key `"Acme:uses"` and observe the exact lookup. Then run a fourth query that each store handles poorly in isolation — query the vector store with `"What tool does the Austin company use?"` (a two-hop relationship) and confirm it misses. Print a formatted table showing each query, which store was queried, and whether it returned a result or missed.

3. **Implement reciprocal rank fusion from scratch.** Write a function `rrf(rank_lists: list[list[str]], k: int = 60) -> list[tuple[str, float]]` that accepts multiple ranked result lists (one per store) and returns a single fused ranking using the RRF formula `score(doc) = Σ 1/(k + rank_i(doc))`. Populate all three stores with at least eight facts (mix of companies, people, tools, and locations). Run a query that partially matches across stores — e.g., `"Acme tools"` — retrieve top-5 from each store independently, pass the three ranked lists to `rrf`, and print the fused ranking. Confirm that documents appearing in multiple stores rank higher than single-store hits.

4. **Apply hybrid retrieval to GTM enriched-lead data.** Create a dataset of at least 12 enriched leads (company name, employee count, tech stack, contact name, contact title — you may hardcode realistic data or pull from a CSV). Ingest each lead's facts across all three stores. Then run five different query types: a semantic query (`"companies using Salesforce"`), a graph-traversal query (`"contacts at companies with 500+ employees"`), an exact-lookup query (`"Acme:tech_stack"`), and two multi-hop queries that require fusion (`"what tools are used by companies in Austin?"` and `"who works at companies using Snowflake?"`). Print the results of each query and annotate which store(s) contributed to each answer.

5. **Build a fan-out vs. single-store benchmark.** Implement a `handlers/hybrid_memory.py` module that exposes a `HybridMemory` class with `add(text)` and `query(text, top_k=5)` methods. The `add` method must extract entities and relations from raw text (you may use a simple regex or rule-based extractor), write to all three stores in parallel, and return a write-receipt dict. The `query` method must fan out to all three stores, apply RRF, and return the fused top-k. Write a benchmark script that loads 50 sentences of mixed GTM content (company descriptions, role changes, tool adoptions, funding events), measures per-query latency for (a) vector-only, (b) graph-only, (c) KV-only, and (d) hybrid fan-out across all 50 documents, and prints a summary table: mean latency, p95 latency, and recall@5 for a set of 10 labeled test queries. The module must land at `handlers/hybrid_memory.py` and be importable.

6. **Evaluate when hybrid retrieval is not worth the cost.** Design and run an experiment that varies two dimensions: dataset size (50, 200, 500 facts) and query complexity (single-hop exact, single-hop semantic, multi-hop relational). For each combination, measure hybrid fan-out latency and recall@5 against the best single-store baseline. Write a 1–2 page analysis to `outputs/skill-hybrid-memory.md` that includes: (a) a markdown table of all measurements, (b) a recommendation for when to use hybrid vs. single-store, expressed as concrete thresholds (e.g., "use hybrid when query type distribution includes >30% multi-hop"), and (c) one example where adding the graph store actually *hurt* results (e.g., noisy extraction producing false edges). Include the raw numbers so a reviewer can verify your conclusions.
