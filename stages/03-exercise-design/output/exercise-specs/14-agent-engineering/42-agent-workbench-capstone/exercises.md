# Exercises — Capstone: Ship a Reusable Agent Workbench Pack

## Exercises

1. **Implement a versioned manifest file** for a workbench pack containing the seven surfaces (agent config, tool schemas, prompt templates, handlers, signals, outputs, installer). Create a `pack.yaml` or `pack.json` manifest at the root of a `workbench-pack/` directory. Declare a `pack_version` field (start at `0.1.0`), a `surfaces` list with each surface name and its entry-point file, and a `dependencies` section pinning any required runtime versions. Run a parser (e.g., `python -c "import yaml; print(yaml.safe_load(open('workbench-pack/pack.yaml')))`) in your terminal and confirm it prints a valid dict with all seven surface names visible in the output.

2. **Pin schemas and prompt templates** inside the pack so they are fixed at a specific version. Create a `workbench-pack/schemas/` directory containing at least two JSON Schema files (one for an enrichment signal, one for an outbound message template) and a `workbench-pack/templates/` directory with two `.md` prompt templates. Reference each file by relative path in the manifest. Write a short script that reads the manifest and prints the SHA-256 hash of each pinned file, then run it and confirm every file resolves and prints a hash.

3. **Build an idempotent installer** (`workbench-pack/install.sh` or `install.py`) that copies the pack into a target repository path passed as an argument. The installer must: skip any file that already exists at the destination and log `[skipped] <filename>`, copy files that do not exist and log `[installed] <filename>`, and exit non-zero if the target path does not exist. Run it twice against the same target directory — the first run should show all files as `[installed]`, the second run should show all files as `[skipped]`.

4. **Implement a pack integrity validator** that reads the manifest, checks every declared file path exists on disk, validates every JSON Schema file is parseable, and validates every template file is non-empty. The validator must print a summary line `[OK] <count> files validated` on success or `[FAIL] <filename>: <reason>` for each problem found, then exit with a non-zero status code if any validation error exists. Test it by temporarily deleting one file and adding a syntax error to a schema file — confirm the validator catches both and exits with code 1.

5. **Assemble the complete workbench pack and install it into a mock GTM repo.** Create a target directory simulating a sales-ops repository with a `clay-tables/` export folder and a stub `crm-sync.py` script. Build the full pack — manifest, schemas (including a Clay enrichment schema with fields like `company_domain`, `linkedin_url`, `employee_count`), prompt templates (including a cold-outreach template that accepts Clay-enriched variables), and handlers (including one that reads a Clay CSV export and normalizes rows against the enrichment schema). Run your installer, then run your validator against the installed target. Write a handler at `handlers/clay_enrichment_loader.py` that imports the installed pack schemas, reads a sample Clay export CSV, and prints the count of valid vs. invalid rows. Run it and confirm the counts appear.

6. **Design and document the pack-target boundary.** Write a boundary specification at `outputs/skill-pack-boundary.md` that lists every artifact that ships inside the pack versus every artifact that must live in the target repo. Include a decision rule (one sentence) for whether a new file belongs inside the pack or in the target repo. Then implement an automated check — a script that scans the installed target and flags any file that violates the boundary rule — and run it against your installed mock repo from Exercise 5. The artifact must land at `outputs/skill-pack-boundary.md` and the checker output must be visible in the terminal.
