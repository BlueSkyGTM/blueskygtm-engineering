# Exercises — Computer Use: Claude, OpenAI CUA, Gemini

## Exercises

1. **Implement a single-iteration perception-action loop.** Write a script that captures a screenshot via `pyautogui`, base64-encodes it, sends it to Claude's Messages API with the `computer_20241022` tool enabled, and prints the parsed action (type, coordinate, text) to the terminal. Run it against your own desktop and confirm you see a structured action like `{"type": "click", "coordinate": [x, y]}` in stdout.

2. **Compare the three provider schemas side by side.** Write a script that constructs the tool-definition JSON for Claude's `computer_20241022`, OpenAI's `computer_use_preview`, and Gemini's browser-action tool. Print each definition with a header label and a summary line listing the action types each schema supports (e.g., `click`, `type`, `scroll`, `wait`, `screenshot`). Your terminal output should show all three blocks so you can visually diff the parameter names and coordinate conventions.

3. **Implement coordinate-drift detection and retry.** Build a loop that executes an action returned by a vision model, then re-captures the screen and compares the post-action screenshot to the pre-action screenshot using pixel-diff percentage. If the diff is below a threshold (meaning nothing visibly changed), retry the action with a ±10-pixel offset on both axes, up to three attempts. Print each attempt's diff score and whether it triggered a retry. Run the loop targeting a known button or link and confirm the terminal log shows the drift correction firing or correctly passing through.

4. **Compute the break-even point for computer use vs. an API workflow.** Given a hypothetical enrichment task where an API-based flow costs $0.02 per profile (Apollo API call) and a computer-use flow costs $0.15 per profile (average 4 screenshots × $0.037 per vision call), write a script that takes total monthly profile volume as a CLI argument and prints: (a) total cost for each approach, (b) the cross-over volume at which one becomes cheaper, and (c) which approach wins at 5,000 profiles/month vs. 50,000 profiles/month. Add a second scenario where the API flow requires a $500/month minimum platform plan and recalculate the break-even.

5. **Build a GTM enrichment handler that uses computer use against a no-API system.** Implement a computer-use agent that navigates to a public LinkedIn company page (or a Clay table rendered in-browser), captures the company headcount and industry, and writes the extracted data as structured JSON to `outputs/company_enrichment_cua.json`. The agent must loop through the full perception-action cycle — screenshot, action, settle-wait, repeat — until it locates the target data or hits a 10-step limit. Include timing and cost logging in stdout so you can observe the per-company cost. Save the runnable script to `handlers/linkedin_enrich_cua.py`.

6. **Design and implement a production-grade computer-use loop with multi-modal failure recovery.** Build a loop that combines: (a) coordinate-drift retry with pixel-diff detection, (b) a dynamic settle-wait that polls the screen for stability before sending the next screenshot, (c) a step-limit circuit breaker, and (d) a cost accumulator that aborts if estimated spend exceeds a configurable threshold. Run the loop against a multi-step task (e.g., open a browser, navigate to a URL, fill a two-field form, submit). Save the script and a markdown run-report containing step count, total cost, retry events, and final status to `outputs/skill-computer-use-run-report.md`.
