# Exercises — Agno and Mastra: Production Runtimes

## Exercises

1. **Instantiate an Agno agent with a tool and inspect response metadata.** Create an Agno agent equipped with a simple tool (e.g., a function that formats a company name and returns its domain). Execute the agent with a user prompt that triggers the tool call. Print the full structured response object — specifically the token usage breakdown (input tokens, output tokens, total), the model identifier, and the finish reason. Run the agent a second time with a prompt that does *not* trigger the tool and confirm the token usage differs. Your terminal output must show both runs with their metadata side by side.

2. **Build a two-step Mastra workflow with intermediate state.** Define a typed state object with at least two fields (e.g., `raw_input` and `summary`). Create a workflow with two sequential nodes: the first accepts a string and writes it to `raw_input`, the second reads `raw_input`, transforms it, and writes the result to `summary`. After each node executes, print the full state object to the terminal. Run the workflow and confirm from the printed output that node 1 completed and wrote state before node 2 read and transformed it.

3. **Implement the same research task in both runtimes and compare structure.** Build a "company quick-look" agent that takes a company name and returns a one-line description plus a list of three likely competitors. Implement this once as an Agno agent and once as a Mastra workflow (with separate nodes for description generation and competitor enumeration). Run both with the same input and print both outputs. Then write a brief comparison — which runtime required more boilerplate, which made the two-step logic more explicit, and where state was managed in each case. Print this comparison to the terminal.

4. **Build an enrichment agent using a real GTM data source in both runtimes.** Using the Apollo People Search API (or Clay's enrichment API if you have an API key), create an Agno agent whose tool calls the enrichment endpoint with a company domain and returns a JSON payload of contacts. Then build the equivalent as a Mastra workflow with a fetch node (calls the API) and a format node (extracts names and titles into a clean list). Execute both with the same domain input. Print the contact list from each and the token usage from the Agno run. Confirm both produce the same contacts but through different execution paths.

5. **Run a head-to-head benchmark of the same agent across both runtimes and persist the results.** Implement a research agent that takes a company name, calls an LLM for a summary, validates the output against a Pydantic schema (company name, summary, competitor list, confidence score), and returns the structured result. Implement this in both Agno and Mastra. Then write a benchmark script that runs each implementation 5 times, measures wall-clock latency per run, records token counts from the response metadata, forces one retry in each runtime (simulate by injecting an invalid response on the first attempt), and confirms schema validation catches the malformed output. Write the full comparison table — runtime, mean latency, p100 latency, total tokens, retry count, schema-validation pass/fail — to `outputs/skill-runtime-comparison.md`.

6. **Deploy an agent behind an HTTP endpoint designed for CRM webhook ingestion.** Build a FastAPI application with a POST `/enrich` endpoint that receives a HubSpot-style contact-created webhook payload (contact properties including company domain), passes the domain to an Agno or Mastra research agent, and returns a structured enrichment result. The endpoint must: validate the incoming payload against a Pydantic schema, assign a UUID correlation ID per request, emit structured JSON logs (timestamp, level, correlation ID, agent runtime, token count, latency) to stdout, and expose a GET `/health` endpoint returning agent runtime status and model availability. Use `curl` to POST a sample webhook payload and to hit the health check; your terminal must show both responses. Save the full server implementation to `handlers/crm_webhook_agent.py`.
