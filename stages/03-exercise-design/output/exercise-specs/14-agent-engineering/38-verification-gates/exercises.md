# Exercises — Verification Gates

## Exercises

1. **Implement three format gates as pure boolean functions.** Write `is_valid_email(value)`, `is_valid_domain(value)`, and `is_valid_phone(value)` — each takes a string and returns `True` or `False` based on a regex pattern. Run them against a list of 10 test inputs (mix of valid and invalid) and print each input alongside its three verdicts. Your terminal output should show 10 lines, each with the value and three boolean results.

2. **Implement a plausibility gate for employee count.** Write `is_plausible_headcount(value)` that accepts either an int or a string and returns `True` only if the value is between 1 and 10,000,000 inclusive. Run it against the list `[-5, 0, 1, 42, 50000, 10000000, 50000000, "3300", "twelve", None]` and print each input with its verdict. Confirm that non-numeric strings and `None` return `False` without raising an exception.

3. **Build a composite verification gate and a rejection tracer.** Create a dict representing an enrichment result with keys `email`, `domain`, `headcount`, and `revenue`. Write a function `verify_record(record)` that runs at least four gates (email format, domain format, headcount plausibility, revenue plausibility) and returns a tuple `(bool, list[str])` where the bool is the AND of all gates and the list contains the names of every gate that returned `False`. Process a batch of 20 records (generate them yourself — include at least 5 that should fail) and print a summary table showing each record's pass/fail status and, when failed, which gates triggered the rejection.

4. **Apply verification gates to Apollo-enriched company data.** Create a list of 15 company records as you would receive them from Apollo's Companies API (include fields like `name`, `website`, `email`, `employees`, `industry`). Write gates that check: website domain resolves to a valid TLD, employee count is plausible for the stated industry, and email domain matches the website domain (consistency gate). Run the full gate set over all 15 records, print each company name with its verdict and failure reasons, and report the total count of accepted vs. rejected records at the bottom.

5. **Compute the credit-cost savings of inserting gates into a three-step enrichment waterfall.** Define a scenario: 1,000 records enter the waterfall. Provider A (cost: 1 credit each) returns results for all 1,000 but 30% are garbage. Provider B (cost: 2 credits each) catches 60% of the remaining. Provider C (cost: 3 credits each) is the last resort. Without gates, all 1,000 flow through each provider sequentially when earlier results are invalid. With gates, rejected results short-circuit to the next provider instead of being silently accepted. Write a script that computes total credits spent in both scenarios (gates vs. no gates) and prints a comparison table showing credits saved. Save the script and its printed output to `outputs/skill-credit-impact.md`.

6. **Design and document a Clay conditional-run-column mapping for a four-gate verification stack.** Using any company-enrichment scenario (Apollo, Ocean.io, or similar), write a spec that maps each of your four Python gates to its equivalent Clay column formula or conditional-run configuration. For each gate, document: the Python function signature, the Clay formula or AI-prompt condition, which waterfall step it guards, and what happens on `False`. Save the complete spec to `outputs/skill-clay-gate-mapping.md`. The artifact must include at least one Clay formula using conditional run logic (e.g., a regex match or a numeric range check expressed as a Clay column formula).
