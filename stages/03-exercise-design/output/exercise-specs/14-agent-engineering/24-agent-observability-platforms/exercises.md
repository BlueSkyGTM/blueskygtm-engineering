# Exercises — Agent Observability: Langfuse, Phoenix, Opik

## Exercises

1. **Implement a decorator-traced two-step agent with Langfuse.** Create a Python script that defines two functions: `enrich_company(domain)` which calls an LLM to summarize a company, and `draft_outreach(company_summary)` which generates a cold email. Decorate both with `@observe` from the Langfuse SDK and call them in sequence from a top-level `run_pipeline` function (also decorated). Run the script against a local Langfuse instance (or `LANGFUSE_HOST=http://localhost:3000`). After execution, query the Langfuse API (or open the UI) and print the trace tree showing the parent span and its two children with their names, durations, and token usage. Your terminal must display the nested span hierarchy.

2. **Configure Phoenix embedded tracing for an OpenAI call.** Start an embedded Phoenix server (`phoenix.run()` on port 6006), enable OpenInference auto-instrumentation for OpenAI, and make a single `client.chat.completions.create` call with a GTM-flavored prompt (e.g., "Classify this lead as SQL or not: Acme Corp, 50 employees, Series A"). Then fetch the trace via Phoenix's client API and print the span name, latency in milliseconds, input token count, and output token count to your terminal.

3. **Compare trace-capture structure across Langfuse and Phoenix.** Write a single script that runs the identical 3-function agent (prospect lookup → LLM scoring → email draft) twice: once wrapped in Langfuse `@observe` decorators and once inside a Phoenix session with `OpenInferenceTracer`. After both runs, export each trace as JSON and write a terminal-printed comparison table: columns for `mechanism`, `auto_captures_parent_child`, `span_count`, `cost_field_present`, and `eval_attachment_support`. Print the table to stdout.

4. **Compute cost-per-action and p95 latency from trace data.** Given a Langfuse project that contains at least 20 traces from a lead-enrichment agent (seed it by running your agent in a loop over 20 company domains), write a script that pulls all traces via the Langfuse Python client, extracts per-trace total cost and end-to-end latency, then computes: (a) mean and p95 latency, (b) total cost across all runs, (c) cost-per-email-sent. If your agent sends one email per run, cost-per-email-sent equals total cost divided by trace count. Print all five metrics. Assume a pricing table of `$0.005 / 1K input tokens` and `$0.015 / 1K output tokens` for gpt-4o-mini if the model doesn't self-report cost.

5. **Build a diagnostic handler for a failing GTM enrichment pipeline and pinpoint the broken span.** Create `handlers/gtm_enrichment_diagnostic.py` that implements a 3-step agent: (1) call Apollo's People Search API to fetch a prospect, (2) pass the prospect through an LLM call to score fit, (3) push the scored lead to a mock CRM endpoint. Instrument every step with Langfuse `@observe`. Inject a deliberate failure in step 2 — a malformed prompt that exceeds context limits or triggers an API error 60% of the time. Run the pipeline 10 times. Then write a `diagnose(trace_id)` function that fetches a failed trace from Langfuse, walks the span tree, and prints the error span's name, its error message, its token count (if any), its latency, and its parent span name. The script must end by printing: `FAILURE ROOT CAUSE: <span_name> — <error_message>`. Persist the fully working handler at `handlers/gtm_enrichment_diagnostic.py`.

6. **Design a local-to-production observability routing strategy and document it.** Write a markdown playbook at `outputs/skill-observability-routing.md` that specifies: when to use Phoenix embedded (development, single-machine, no external deps), when to switch to self-hosted Langfuse (team-shared tracing, eval workflows), and how to route traces to a production backend via environment-variable-based backend selection (`OBSERVABILITY_BACKEND=phoenix|langfuse|opik`). Include a concrete Python snippet (no more than 20 lines) that reads the env var and initializes the correct tracer. Include a decision table with at least four rows covering scenarios (local dev, CI testing, staging, production) and the recommended backend plus rationale. The artifact must be a valid markdown file that renders cleanly.
