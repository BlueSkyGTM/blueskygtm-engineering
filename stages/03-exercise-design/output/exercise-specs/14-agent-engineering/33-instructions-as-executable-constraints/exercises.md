# Exercises — Agent Instructions as Executable Constraints

## Exercises

1. **Implement a constraint stack for a lead-enrichment agent.** Write a Python file that defines a constraint stack as a list of dictionaries, each with `category` (one of `input_contract`, `output_schema`, `behavioral_guardrail`, `fallback_directive`), `rule` (a natural-language constraint string), and `enforcement` (`soft` or `hard`). Your stack must cover all four categories with at least two rules each. The agent's job: given a company domain, return `company_name`, `employee_count`, and `confidence_score`. Print the full stack to the terminal in a readable enumerated format.

2. **Compare soft vs. hard enforcement by provoking a failure.** Write a script that constructs two prompt variants for the same lead-enrichment task: one using only natural-language constraints ("respond in JSON with keys…") and one using a JSON schema dict passed as a structured-output specification. Send both prompts to an LLM with a deliberately ambiguous input (e.g., domain `""`). Print both raw responses side by side, then print a one-line classification of the failure mode each exhibits (`schema_violation`, `hallucinated_data`, `polite_refusal`, `valid`).

3. **Build a rule checker that scores agent outputs against a constraint stack.** Given a constraint stack (same structure as Exercise 1) and a JSON agent output, implement a function `score_output(stack, output)` that returns a dict mapping each category to a pass/fail count and a list of violated rule descriptions. Use at least these checks: output parses as JSON, required keys exist, values have correct types, `confidence_score` is between 0 and 1, and no key outside the allowed set appears. Run your checker against three sample outputs (one clean pass, one with wrong keys, one with out-of-range confidence) and print the per-category score report for each.

4. **Diagnose polite compliance and prompt drift from a log file.** Create a file `outputs/constraint-diagnosis.md`. You are given a log of 10 agent responses in a JSON array (write these yourself — include examples of polite refusals like "I'd be happy to help with that, however…", schema drift where keys rename across calls, and at least one case where the agent follows a conflicting instruction from the user prompt instead of the system constraint). For each response, classify the failure type, cite the token span that reveals the failure, and state which constraint category was violated. Write the full diagnosis into the markdown file.

5. **Package a tested constraint stack into a versioned config with embedded regression tests.** Create `signals/examples/enrichment-constraints.yaml` containing: a `version` field, the full constraint stack, and a `regression_tests` block with at least five test cases — each pairing a sample input with an expected pass/fail verdict and expected violated categories. Write a companion runner script that loads the YAML, runs each regression test through your rule checker from Exercise 3, and prints a summary table showing test name, expected verdict, actual verdict, and pass/fail status. All tests must pass.

6. **Design a constraint monitor for a Clay enrichment workflow.** You have a Clay table exporting enriched rows via webhook to your service. Design and implement a constraint validation layer that receives each row, checks it against a constraint stack (company must be non-empty string, employee count must be integer or null, confidence must be float in [0, 1], no extra keys beyond a defined allow-list), and writes rejected rows to `outputs/rejected-enrichments.jsonl` with a `rejection_reason` field. Simulate the webhook payload with a local fixture containing 8 rows (6 valid, 2 violating constraints). Run the script and verify the JSONL file contains exactly 2 entries.
