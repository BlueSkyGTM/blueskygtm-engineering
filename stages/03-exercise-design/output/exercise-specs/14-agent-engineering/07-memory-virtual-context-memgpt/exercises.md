# Exercises — Memory: Virtual Context and MemGPT

## Exercises

1. **Implement a `CoreMemory` class** with two editable blocks: `persona` (who the agent is) and `human` (facts about the user). Each block must support `append(text)` and `replace(old, new)` operations. Write a driver that creates an instance, seeds it with two persona facts and two human facts, performs one append and one replace, then prints the full formatted core memory block with clear section headers to the terminal.

2. **Implement an `ArchivalMemory` class** backed by a JSON file on disk. It must support `insert(text)` and `search(query)` where `search` performs case-insensitive substring matching and returns all matching entries ranked by match count. Seed it with 10 sentences about B2B sales workflows. Run two searches — one that matches 3+ entries and one that matches zero — and print both result sets so the difference is observable.

3. **Implement the paging algorithm.** Given a token budget (simulate token count as `len(text) // 4`) and a growing list of conversation turns, write a function that prints a trace of paging decisions: at each turn, report current context usage, whether a page fault is triggered, which core memory block gets evicted to archival, and what gets reloaded. Feed it 15 turns of dialogue where each turn is 50–80 words and a core memory block of 200 words. Your terminal output must show at least two page faults occurring.

4. **Apply the MemGPT vs. RAG distinction to a GTM enrichment workflow.** Load 20 mock company records (name, industry, employee count, tech stack) from a JSON file as your corpus. Implement two retrieval strategies on the same data: (a) a RAG pipeline that embeds and retrieves fixed chunks via keyword match, and (b) a self-editing core memory where the agent stores the top 3 most relevant company facts and can overwrite them mid-conversation. Feed both systems a 5-turn sequence of account research queries (e.g., "Find SaaS companies under 500 employees using Salesforce"). Print which facts each system surfaces at each turn and where they diverge. Demonstrate this using Clay-formatted export columns or Apollo-style company JSON as your input schema.

5. **Build a minimal virtual context manager** that persists agent state across simulated multi-step tool calls. Your manager must: maintain core memory in an always-injected system prompt, write evicted content to archival on disk, accept tool-call messages that trigger memory edits, and reload from archival when a tool requests a search. Simulate a 10-turn SDR research session where the agent researches a prospect, edits core memory with new findings, triggers page faults, and retrieves from archival. Each turn must print the current context window contents, memory tier status, and any paging actions taken. Persist final state to disk and reload it on a second run to prove continuity. Produce your implementation at `handlers/virtual_context_manager.py`.

6. **Design a written evaluation comparing model-controlled memory adaptation against deterministic memory management** for a GTM use case of your choice (e.g., meeting-notes-to-CRM pipeline, multi-touch outbound sequencing, or churn signal monitoring). For each paradigm, provide a concrete architecture sketch, identify three failure modes specific to GTM data (noisy enrichment fields, stale CRM records, conflicting intent signals), and argue which paradigm handles each failure better. Include a decision rule for when to choose one over the other. Produce your evaluation at `outputs/skill-memory-paradigm-comparison.md`.
