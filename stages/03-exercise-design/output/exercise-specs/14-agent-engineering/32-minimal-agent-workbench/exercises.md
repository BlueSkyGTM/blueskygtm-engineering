# Exercises — The Minimal Agent Workbench

## Exercises

1. **Implement a two-tool agent loop.** Build a minimal agent in under 80 lines of Python that implements the observe-plan-act cycle. Give it two tools: `search_companies(domain: str) -> dict` (returns a mock company record) and `lookup_contact(company_name: str) -> dict` (returns a mock contact). Run the agent with the prompt "Find a contact at Stripe and tell me their role." Your program must print each loop iteration's raw LLM output and the tool call it dispatched, then the final answer. Terminal output should show at least two iterations before the agent stops.

2. **Configure typed tool schemas and dispatch table.** Write a `TOOL_SCHEMAS` list (JSON-style dicts with `name`, `description`, and typed `parameters`) for four tools and a `DISPATCH` dict mapping each name to a real Python function. At least one tool must take a parameter typed as `enum` (e.g., a `data_provider` field restricted to `["apollo", "clay", "clearbit"]`). Write a `validate_and_dispatch(tool_name, arguments)` function that returns a helpful error string if the tool name is unknown or a required argument is missing. Run a test script that calls `validate_and_dispatch` with three valid and two invalid inputs and prints the results.

3. **Trace a ReAct loop from saved output.** Take the terminal output from Exercise 1 (or generate a fresh run) and write a `trace_loop(log_file: str)` function that parses the saved log and classifies each iteration as `REASONING_ONLY`, `TOOL_CALL`, or `FINAL_ANSWER`. Print a summary table showing iteration number, classification, and the tool name if applicable. This exercise requires you to identify where reasoning yields tool calls and where tool output triggers the next reasoning step — your parser must distinguish these states from the raw text.

4. **Compare agent vs. hardcoded enrichment.** Implement two versions of a 3-step GTM enrichment flow: (a) a hardcoded pipeline that always calls `find_domain -> enrich_company -> find_decision_maker` in sequence, and (b) an agent loop with those same three tools available but free to choose its own path. Run both with five test company names where two are missing domain data (the enrichment tool returns empty). Print the tool-call sequence and total tool invocations for each approach. Identify which companies benefited from conditional reasoning (the agent skipping or retrying a step) and print a one-line comparison.

5. **Build an agent failure-mode harness.** Extend your agent loop with a configurable `max_iterations` limit and a `cost_tracker` that estimates token cost per iteration (assume 1 token per 4 characters of messages). Run the agent against three adversarial prompts designed to trigger different failure modes: (a) a prompt that causes an infinite loop (ambiguous stop condition), (b) a schema mismatch where the model passes a list instead of a string, and (c) a prompt with no available tool that can answer the question. Write results to `outputs/skill-agent-failure-modes.md` as a markdown table showing prompt, failure mode detected, iterations consumed, estimated token cost, and whether `max_iterations` safely terminated the run.

6. **Design a GTM enrichment agent with Clay/Apollo data.** Build an agent that takes a list of company domains, uses the Apollo person search API (or a mock that returns realistic JSON) to find contacts, applies conditional reasoning to decide whether a lead is worth enriching further (e.g., skip companies with fewer than 50 employees), and writes qualified leads to `outputs/skill-gtm-enrichment-agent.md`. The agent must have at least four tools, print its reasoning at each step, and demonstrate at least one case where it chose *not* to call an available tool because prior output changed its plan. Save your full agent code to `signals/examples/gtm_enrichment_agent.py`.
