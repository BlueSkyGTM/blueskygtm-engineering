# Exercises — Initialization Scripts for Agents

## Exercises

1. **Build a deterministic agent-state loader.** Create a Python script that reads a JSON config file containing `agent_name`, `model`, `temperature`, and `max_tokens`, then prints a canonical JSON representation of the resulting agent state to stdout. Run the script twice against the same config file and diff the two outputs — they must be byte-identical. Insert a deliberate non-deterministic element (e.g., a timestamp) into a second version of the script and confirm that the diff now shows differences, proving the original was deterministic by construction.

2. **Implement environment validation with specific exit codes.** Write a function `validate_env(required_vars)` that checks each variable in the provided list against `os.environ`. When a variable is missing, print a message formatted as `MISSING_ENV: <var_name> — set it via: export <var_name>=<value>` and exit with code `2`. When a variable is present but empty, exit with code `3` and a distinct message. Test by invoking the script with all variables set, with one unset, and with one set to an empty string — confirm the exit codes differ across all three runs using `echo $?` in your terminal.

3. **Construct a modular system prompt at boot time.** Create a `prompts/` directory containing four separate text files: `role.txt`, `constraints.txt`, `tools_description.txt`, and `output_format.txt`. Write a function `assemble_system_prompt(prompts_dir)` that reads each file, concatenates them with `---` separators and a section header derived from each filename, and returns the full prompt string. Print the assembled prompt and its total character count. Modify one component file and re-run to confirm only that section changes while the others remain stable.

4. **Compare token cost: implicit defaults vs explicit initialization.** Write a script that defines two agent boot paths. Path A uses implicit defaults — a bare system prompt with no structured config, simulating a "just send it" approach. Path B uses your explicit initialization from Exercise 3. Use the `tiktoken` library to count tokens in each system prompt. Then simulate a 50-turn conversation (append a 20-token user message per turn) and compute the cumulative system-prompt token cost for both paths across all 50 turns, since the system prompt is resent with every call. Print a comparison table showing per-turn cost and total cost difference. Use a GTM-flavored config: Path B's prompt modules should include a section that instructs the agent to query Apollo's people-search API and format enrichment data into Clay-compatible column mappings, so the token cost reflects real GTM prompt complexity.

5. **Package a standalone CLI boot-report tool.** Build `handlers/boot_report.py` as a CLI module invoked via `python -m handlers.boot_report --config agent.json --output outputs/skill-boot-report.md`. The module must run environment validation (Exercise 2 logic), load config (Exercise 1 logic), assemble the system prompt (Exercise 3 logic), and write a Markdown boot report to `outputs/skill-boot-report.md` containing: validation status (pass/fail per variable), resolved config values, assembled system prompt with token count, and a SHA-256 hash of the final agent state for reproducibility verification. Run the tool, then re-run it against the same config and confirm the SHA-256 hash in both reports matches. Modify the config, re-run, and confirm the hash changes. The artifact lands at `outputs/skill-boot-report.md` and the executable module at `handlers/boot_report.py`.
