# Exercises — Runtime Feedback Loops

## Exercises

1. **Run the lesson's feedback loop and trace the retry sequence.** Execute the example feedback loop from the lesson against a prompt that intentionally produces a missing field on the first attempt (e.g., ask for a JSON object with `company_name`, `domain`, and `employee_count` but craft the prompt so the model omits `employee_count`). Print the error history at each iteration so the terminal shows which fields failed validation and how the feedback message evolved across attempts. Verify you can see at least two retry iterations in stdout before the loop succeeds or terminates.

2. **Detect and classify the three failure modes.** Write a validator function that takes a raw LLM output string, attempts to parse it as JSON, and checks against a schema with at least four fields of mixed types (string, int, list, enum). Produce three separate test inputs — one that triggers a missing field error, one that triggers an invalid type error, and one that triggers a constraint violation (e.g., `employee_count` must be positive, or `stage` must be one of `["seed", "series_a", "series_b"]`). Run all three through your validator and print a labeled report showing which failure mode each input triggered.

3. **Build a self-correcting enrichment loop for Apollo-style company data.** Without using any scaffold, implement a complete feedback loop that calls an LLM (use a mock function that returns canned JSON if you don't have API access) to extract `company_name`, `website`, `linkedin_url`, `estimated_arr`, and `industry` from a raw company description string. Validate each field, inject structured error feedback on failure, cap retries at 3, and return either a valid enriched record or a structured error dict with `status: "failed"`, `errors: [...]`, and `attempts_used: N`. Run it against three different input descriptions and print the final result for each.

4. **Implement dual-mode enrichment with a runtime toggle.** Build a single enrichment function that accepts a `mode` parameter: `"fire_and_forget"` makes one LLM call and returns whatever comes back with no validation; `"self_correcting"` runs the full feedback loop with schema validation and retries. Run both modes against the same set of 5 messy company descriptions and print a side-by-side comparison showing: success rate, total LLM calls made, and number of fields that passed validation. The terminal output should make the tradeoff between the two modes visible.

5. **Build a production feedback loop with cost tracking for Clay enrichment workflows.** Create a module at `handlers/enrichment_loop.py` that wraps a feedback loop around company enrichment. Accept a list of raw lead records (each containing a company name and a short description). For each record, run the self-correcting loop with a Pydantic schema covering at least 6 enrichment fields including `icp_fit_score` (float 0–1), `buying_signals` (list of strings), and `recommended_channel` (enum: `email`, `linkedin`, `phone`, `direct_mail`). Track token estimates or API calls per record. After processing all records, write a summary report to `outputs/skill-enrichment-loop.md` containing: per-record outcome (success/failed), attempts used, estimated cost, aggregate success rate, and total retries consumed. Include a section analyzing whether the retry budget was justified for records that failed all attempts.

6. **Design and benchmark a tiered retry strategy.** Many production systems use escalating feedback: the first retry sends a generic "your output had errors" message, the second sends the full validation error with field-level detail, and the third sends a worked example of a correct output. Implement this tiered feedback injector, run it against a dataset of 20 challenging enrichment inputs (you can generate these synthetically — ambiguous company descriptions, missing domains, non-standard industry labels), and compare its success rate and cost against a flat-feedback strategy that always sends the same error message. Write your benchmark results and analysis to `outputs/skill-tiered-feedback-benchmark.md`, including a recommendation for when the added complexity of tiered feedback pays off.
