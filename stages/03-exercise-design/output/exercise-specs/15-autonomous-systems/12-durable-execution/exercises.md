# Exercises — Long-Running Background Agents: Durable Execution

## Exercises

1. **Build a minimal checkpointing state machine.** Implement a workflow class that writes a JSON checkpoint file after each named step completes. The workflow should have three steps — `fetch`, `transform`, `send` — each printing `[step] executing <name>` to the terminal when it actually runs. Run the workflow once and confirm all three steps execute. Then simulate a crash by deleting the final checkpoint entry for `send` from the JSON file (or truncating it manually). Re-run the workflow and confirm that `fetch` and `transform` are skipped (no terminal output) while `send` re-executes. The terminal output after the second run should show only `send`.

2. **Compare three recovery strategies by tracing execution.** Implement the same three-step workflow three separate ways: (a) naive retry — no checkpoints, all steps re-run on every invocation; (b) queue-based retry — the entire task is re-queued, but a shared in-memory set tracks completed step names within a single task run (lost on crash); (c) replay-based — checkpoints persisted to disk, completed steps skipped on resume. Simulate a crash after step 2 by inserting a `sys.exit(1)` call. Comment out the crash line and re-run each strategy. Print the execution trace for each (which steps ran vs were skipped) and verify the replay-based strategy is the only one that skips steps 1 and 2 on the second run.

3. **Implement an idempotent activity wrapper.** Write a decorator called `idempotent_activity` that takes an `idempotency_key` argument and stores the activity's return value in a SQLite table keyed by that key. If the same key is seen again, return the stored result without re-executing the function body. Apply it to a simulated `enrich_company(domain)` function that calls a mock API with a random delay and returns a dict. Call it twice with the same key, measure both call durations, and print them. Verify the second call is near-instant and prints `[CACHE HIT] key=<key>` while the first prints `[EXECUTING] key=<key>`.

4. **Benchmark checkpointing overhead.** Implement the same five-step workflow twice: once with JSON-file checkpointing after each step and once with no checkpointing. Each step should sleep for 50ms to simulate real work. Run both versions 10 times, measure total wall-clock time for each run using `time.perf_counter()`, and print a summary table showing average time per run, average per-step overhead, and the percentage cost of checkpointing. Determine the break-even threshold: at what step duration does the checkpoint overhead drop below 5% of total execution time? Print the answer.

5. **Build a durable GTM lead enrichment pipeline.** Create a workflow that enriches a batch of 5 leads from `signals/examples/leads_sample.json` through four steps: `apollo_lookup` (mock — returns person data keyed by email), `company_enrich` (mock — returns firmographics keyed by domain), `score_lead` (computes a fit score from the enriched data), and `push_to_crm` (mock — returns a CRM record ID). Checkpoint each step's output to a SQLite database keyed by `(lead_email, step_name)`. Use idempotency keys derived from the lead email for each step. Simulate a crash after step 2 by raising an exception on the third lead during `score_lead`. Re-run the workflow and verify that leads 1 and 2 skip all four steps, lead 3 resumes from `score_lead`, and leads 4 and 5 run the full pipeline. Print a final status table showing each lead, which steps executed vs were replayed, and the final CRM record ID. Persist the workflow handler to `handlers/durable_enrichment.py`.

6. **Design a durability decision framework.** Write a script in `outputs/skill-durability-framework.md` that takes three inputs — estimated workflow duration, number of steps, and per-step failure probability — and outputs a recommendation: use durable execution, use simple retry, or run without retry. The recommendation should be based on the expected cost of rework (steps × failure probability × per-step duration) compared against checkpointing overhead. Include a worked example with real numbers for a 12-step Apollo-to-HubSpot sync workflow running on 500 leads, and justify the recommendation with the computed expected values.
