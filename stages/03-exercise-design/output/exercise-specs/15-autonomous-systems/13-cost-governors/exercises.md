# Exercises — Action Budgets, Iteration Caps, and Cost Governors

## Exercises

1. **Implement an action budget class** that accepts an integer budget, exposes a `consume(amount=1)` method that decrements the counter and returns `False` when the budget is exhausted, and an `exhausted` property. Write a simulation loop that calls `consume()` on each iteration and prints the remaining budget until it hits zero. Run the script and confirm the terminal shows the counter reaching exactly 0 with no further consumption attempts.

2. **Build an iteration cap as a context manager or decorator** that counts how many times a function (simulating one ReAct cycle) is invoked and raises a custom `IterationLimitReached` exception when the cap is exceeded. Write a dummy `reason_act_observe()` function that returns a random observation, then call it in a `while True` loop wrapped by your cap logic. Print each cycle number and the exception message when the cap triggers. Verify the terminal output shows exactly N successful cycles before the exception fires.

3. **Construct a cost governor** that maintains a running total of simulated token spend. Feed it a list of mock API responses (Python dicts with `usage.total_tokens` keys representing responses from an enrichment API such as Apollo's person search). Configure a dollar threshold (e.g., $0.50 at a rate of $0.002 per 1K tokens). Iterate through the responses, accumulate cost, and print the running total after each response. The governor must raise a `BudgetExceeded` exception when cumulative spend breaches the threshold. Confirm the terminal shows the exact response index where the breach occurred and the final dollar amount.

4. **Compose all three controls into a single agent runner.** Write a function `run_agent(tasks, action_budget, iteration_cap, cost_threshold)` that simulates processing a list of GTM lead-enrichment tasks (e.g., looking up companies via Clay or Apollo). Each task consumes one action, one iteration, and a random token cost between 500–2000 tokens. On each cycle, check all three ceilings in order: iteration cap, action budget, cost governor. When any ceiling is hit, return a partial results dict containing completed tasks, remaining tasks, and which control triggered the shutdown. Print the final results dict to the terminal and confirm it identifies the triggering control correctly.

5. **Build a budget consumption diagnostic tool** that accepts a JSON log file of agent runs (each entry containing `actions_consumed`, `actions_budget`, `iterations_completed`, `iteration_cap`, `tokens_spent`, `cost_threshold`, `tasks_completed`, `tasks_total`). Compute three diagnostic ratios per run: action utilization, iteration utilization, and cost utilization. Classify each run as `too_tight` (any utilization above 0.95 with incomplete tasks), `too_loose` (all utilizations below 0.3), or `well_tuned`. Write the full diagnostic report with per-run classifications and aggregate recommendations to `outputs/skill-budget-diagnostics.md`. Run the tool against a sample log of at least 10 simulated runs and confirm the markdown file is written with the classification table.

6. **Design and implement a production-grade governor orchestrator** that reads configuration from a YAML file specifying per-agent budgets (e.g., a Clay enrichment agent gets 50 actions / 15 iterations / $2.00, a CRM sync agent gets 200 actions / 50 iterations / $10.00). The orchestrator instantiates the correct controls for each agent type, runs a simulated multi-agent workflow where agents hand off partial results to each other, and produces a shutdown report when any agent hits a ceiling. Persist the orchestrator logic to `handlers/budget_governor.py` and write the run report to `outputs/skill-governor-run.md`. Verify both artifacts exist after execution and the report identifies which agent type triggered the workflow halt.
