# Exercises — Constitutional AI and Rule Overrides

## Exercises

1. **Implement** a single generate-critique-revise cycle in Python. Use any LLM API (OpenAI, Anthropic, or a local model) to generate a one-paragraph outbound email to a prospect. Then pass the email back to the same model with a prompt asking it to identify any policy violations from a hardcoded list of three rules (e.g., "must include company name," "must not mention pricing," "must be under 100 words"). Print the original response, the critique, and the revised response to the terminal as three labeled sections.

2. **Configure** a constitutional rule as a Python dataclass or Pydantic model with fields for `name`, `condition` (a natural-language description of what triggers evaluation), `constraint` (what the rule enforces), and `priority` (an integer where 1 is highest). Instantiate five rules relevant to outbound email compliance. Serialize the list to JSON and print it. Verify your terminal output shows priority-ordered rules with all four fields populated.

3. **Build** a priority-based conflict resolver. Given two or more constitutional rules that produce contradictory critiques on the same output (e.g., "make the email shorter" vs. "include all three value propositions"), implement a function that selects which critique wins based on the `priority` field. Feed the resolver a mocked critique list where two rules conflict and print which rule's critique is applied. Then call the LLM again with only the winning critique as the revision instruction.

4. **Trace** the full self-correction loop across multiple iterations. Wrap your generate-critique-revise logic in a `while` loop that continues until either (a) the critique returns zero violations or (b) a configurable `max_iterations` threshold (default 5) is reached. After each iteration, append a JSON line to a trace file at `outputs/cai-trace.jsonl` containing `iteration`, `violations_found`, `rules_triggered`, and the revised text. Run the loop on a seed prompt that deliberately violates two rules and confirm the trace file grows by one line per iteration. Print the final violation count to the terminal.

5. **Apply** the CAI pattern to a GTM enrichment pipeline. Build a script that loads a sample of 10 enriched company records from Apollo's API (or a local JSON fixture at `signals/examples/apollo_sample.json` if you lack an API key) and scores each against an ICP constitution containing five rules with priorities (e.g., "employee_count between 50-500" at priority 1, "uses Salesforce" at priority 2). For each company, run the generate-critique-revise loop where the model writes a one-sentence fit assessment, critiques it against the ICP rules, and revises. Write the final scored and compliance-checked output as a markdown table to `outputs/skill-icp-scoring.md`. Include a CAN-SPAM compliance rule (priority 1) that flags any company missing a contact email as non-compliant. The artifact must show per-company scores, triggered rules, and iteration counts.

6. **Design** a constitutional rule set for a multi-channel GTM workflow (email, LinkedIn, and phone) where CAN-SPAM rules apply to email, GDPR consent rules apply to EU prospects across all channels, and an internal frequency-cap rule limits touches to three per 14-day window. Encode at least eight rules with priorities, implement conflict resolution between GDPR and frequency rules, and run the loop against a dataset that includes both EU and non-EU prospects. Produce a compliance report at `handlers/multichannel_compliance.py` that, when executed, reads prospect records and outputs a per-prospect compliance verdict with the specific rules evaluated and their resolution order.
