# Exercises — Human-in-the-Loop: Propose-Then-Commit

## Exercises

1. **Implement a `ProposalStore` class** that persists proposals to a JSON file on disk (e.g., `proposals.json`). Each proposal must carry: a UUID, an `intent` string, `status` (initialized to `proposed`), `blast_radius` (integer record count), and an `idempotency_key` derived from a hash of the intent and target record IDs. Implement `create()`, `commit(id)`, and `reject(id, reason)` methods that transition status and write back to disk. Attempting to commit a proposal whose idempotency key matches an already-committed proposal must raise a `DuplicateCommitError`. Write a script that creates three proposals, commits one, rejects one, and then prints the full store contents as a formatted table. Verify: the duplicate commit attempt prints the error and the store shows three distinct statuses.

2. **Add lifecycle tracing** to your `ProposalStore` from Exercise 1. Every state transition (`proposed → committed`, `proposed → rejected`, `proposed → expired`, `committed → verified`) must append a timestamped event to a `lifecycle_log` list stored alongside the proposal record. Implement a `trace(proposal_id)` method that prints the ordered event log for a single proposal. Create a proposal, commit it, mark it verified, then call `trace()` and confirm the printed output shows each transition in chronological order with timestamps.

3. **Implement a confidence-based routing function** called `route_proposals(proposals, threshold)` where each proposal in the input list includes a `confidence` float between 0 and 1. Proposals at or above the threshold are auto-committed (status set to `committed`, tagged `auto_committed=True`); the rest are held at `proposed` with `auto_committed=False` for human review. Return both groups. Then write a comparison function `summarize_routing(proposals, threshold)` that prints: count auto-committed, count held for review, average confidence of each group, and the blast-radius sum for each group. Run it against a synthetic dataset of 20 proposals with varying confidence scores and two different thresholds (0.7 and 0.9). Verify: the printed counts change meaningfully between thresholds.

4. **Build a stale-proposal cleanup job** for a CRM enrichment workflow. You have a directory of proposal JSON files (simulated: write a script that generates 15 proposal files with `created_at` timestamps spread across the last 72 hours and a `ttl_seconds` field of 3600). Implement `cleanup_stale(directory)` that scans all files, transitions any `proposed`-status proposal whose `created_at + ttl_seconds` has elapsed to `expired`, and returns a summary of how many were expired vs. how many remain active. Then implement `batch_review(directory, decision_map)` that applies a mapping of `{proposal_id: "commit"|"reject"}` to all non-expired proposals and prints the final status distribution. Run both functions and verify the terminal output shows expired proposals excluded from the review batch.

5. **Implement an approval-pattern auditor** that detects rubber-stamp behavior. Given a JSON log file of historical proposal decisions (write a generator that produces 200 decisions across 5 reviewers with realistic skew — one reviewer approves 98%, others range 60–80%, each rejection includes a `reason` field), compute per-reviewer: approval rate, rejection-reason distribution, and average blast radius of approved proposals. Flag any reviewer whose approval rate exceeds 90% as a `RUBBER_STAMP` risk. Write the full audit report to `outputs/skill-approval-audit.md` in markdown with a table per reviewer and a flagged-risks section. Verify: the file exists at the specified path and at least one reviewer is flagged.

6. **Design and implement a propose-then-commit pipeline for Apollo contact enrichment updates.** Build a script that: (a) takes a CSV of 10 contacts with stale or missing data, (b) generates proposals for field updates (e.g., title, company, LinkedIn URL) with each proposal carrying the old value, proposed new value, source provenance, confidence score, and idempotency key, (c) applies a confidence threshold of 0.85 for auto-commit, routing the rest to a pending-review queue, (d) writes the committed updates as an output CSV and the pending proposals to `handlers/apollo_enrichment_proposals.json`. Run the script against a sample contacts CSV (you may generate synthetic data). Verify: both output files exist, the committed CSV contains only high-confidence updates, and the proposals JSON contains the deferred ones with full metadata.
