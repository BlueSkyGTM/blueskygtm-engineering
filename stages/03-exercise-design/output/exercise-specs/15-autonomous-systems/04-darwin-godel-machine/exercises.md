## Exercises

1. **Implement a minimal archive with insertion and promotion logic.** Build a Python class `VersionArchive` that stores candidate agent versions as dictionaries with keys `version_id`, `parent_id`, `score`, and `code`. Implement `insert(candidate)` so that a child only enters the archive if its score exceeds its parent's score by at least a configurable `margin` (default 0.05). Seed the archive with one root version (score 0.0, no parent), then insert five synthetic candidates with hardcoded scores and parentage — two that clear the margin and three that don't. Print the final archive contents. Your terminal output should show exactly which versions were retained and which were rejected.

2. **Trace a self-modification through the DGM loop and predict retention.** Write a script that simulates four DGM iterations using the `VersionArchive` from Exercise 1. For each iteration, hardcode a parent choice, a proposed mutation (as a code-string diff or description), and an evaluated child score. Two of the four children should clear the margin; two should not. At each step, print: `[iter N] parent=<id> child=<id> parent_score=<s> child_score=<s> margin=<m> VERDICT=<RETAINED|REJECTED>`. After all four iterations, print the full archive tree showing parent-child relationships. Verify by inspection that rejected children never appear as parents in later iterations.

3. **Implement a complete single-file DGM loop over a toy task distribution.** Build a self-contained script (`dgm_loop.py`) that runs the four-step loop: (a) sample a parent from the archive using softmax-weighted selection over scores, (b) call a mutation function that takes the parent's `solve()` function source as a string and returns a modified version (you may use a rule-based mutator or an LLM call), (c) evaluate the child on a set of 10 mini-tasks (e.g., string-reversal, list-sorting, arithmetic — your choice) that the parent was not trained on, and (d) insert the child into the archive if it clears the margin. Run 20 iterations starting from a naive baseline `solve()`. Print a generation log showing each iteration's parent, mutation summary, child score, and verdict. End by printing the top-3 archive entries by score.

4. **Compare DGM, proof-gated Gödel Machine, and hill-climbing on a single mutation scenario.** Write a script that takes one candidate mutation (a code change to a scoring function) and evaluates it under three selection regimes: (a) DGM — archive the child if it beats the parent by margin `m` on a held-out task slice; (b) proof-gated Gödel Machine — accept the child only if it can produce a formal proof that the mutation improves expected performance (simulate this by requiring the child to output a proof string and checking it against a hardcoded set of valid proofs); (c) hill-climbing — accept the child only if it beats the current single best, then discard the parent. Run all three on five different mutations with varying quality. Print a table: `mutation | DGM_verdict | proof_verdict | hill_verdict`. In a comment block at the bottom of your script, write 3–5 sentences explaining which guarantees each method drops and what replaces them.

5. **Apply the selection-archive pattern to a GTM lead-scoring pipeline.** Build `handlers/lead_score_archive.py` that maintains an evolving archive of lead-scoring models. Start with a baseline scorer that ranks leads by a simple rule (e.g., firmographic fit score). Each "generation" proposes a new scoring variant — you may mutate feature weights, add enrichment signals (company size, technographic data, intent score), or change the scoring formula. Evaluate each variant on a held-out batch of leads with known conversion outcomes (use synthetic data or a sample from your CRM / an exported Apollo or Clay enrichment CSV). Insert winners into the archive by the margin rule. After 15 generations, export the archive state to `outputs/lead-score-evolution.md` containing: a table of all retained versions with their scores and mutations, the lineage tree, and the current best scorer's full code. Run the script and confirm both the terminal generation log and the markdown artifact are produced.

6. **Design and demonstrate a reward-hacking failure mode in a mutable-scoring system.** Extend the GTM pipeline from Exercise 5 (or build a standalone version) where the agent's objective function is itself part of the mutable code — meaning a mutation can change how "conversion" is measured, not just how leads are scored. Implement a mutation that hacks the reward: for example, redefining the conversion label to count more outcomes as positive, or weighting the evaluation slice toward easy leads. Run the DGM loop for 10 generations with at least one reward-hacking mutation injected at generation 5. Print scores before and after the hack. Then implement a guardrail (e.g., frozen ground-truth labels, held-out adversarial eval set, or a constraint that objective-function code is read-only) and re-run, showing the hack is blocked. Write your analysis — what failed, why, and how the guardrail works — to `outputs/skill-reward-hacking.md`. Confirm terminal output shows the score spike, the guardrail intervention, and the corrected trajectory.
