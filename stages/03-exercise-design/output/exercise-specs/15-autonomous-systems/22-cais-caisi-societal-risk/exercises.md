# Exercises — CAIS, CAISI, and Societal-Scale Risk

## Exercises

1. **Trace failure propagation in a monolith vs. a CAIS pipeline.** Implement two Python functions: `monolith_failure(model_output, noise)` that applies a single corruption to one model output, and `cais_failure(service_chain, noise)` that takes a list of callable services (each taking a dict and returning a dict) and applies the same `noise` at stage 0, letting it propagate through every subsequent stage. Feed both a starting record like `{"lead_score": 0.82, "segment": "enterprise", "tone": "neutral"}` and print the final output of each path. Confirm visually that the CAIS output diverges more from the clean baseline than the monolith output.

2. **Quantify differential outcomes from a single biased stage.** Build a list of 200 synthetic lead records with a `region` field uniformly sampled from `["NA", "EU", "APAC", "LATAM"]`. Write a `score_leads()` function that assigns scores but silently depresses scores for `"APAC"` by 20%. Pass the scored records through a downstream `prioritize_outreach()` function that flags leads scoring above 0.75 for immediate contact. Print the percentage of each region flagged for outreach. Your terminal should show the disparity clearly.

3. **Implement an audit checkpoint that blocks distributional shift.** Write a function `audit_checkpoint(incoming_records, reference_distribution, threshold=0.05)` that computes the Kolmogorov-Smirnov statistic (use `scipy.stats.ks_2samp`) between the score distribution of `incoming_records` and `reference_distribution`. If the statistic exceeds `threshold`, raise a `DistributionalShiftError` with a message showing the KS value and threshold. Wire it between two stages in a 3-service pipeline (score → audit → prioritize) and demonstrate that injecting biased records triggers the checkpoint while clean records pass through. Print confirmation messages for both cases.

4. **Evaluate CAIS societal-scale risk categories for a real GTM stack.** Review the four risk categories discussed in the lesson (capability concentration, cascading bias, opaque composition, inter-service coupling). Pick a real GTM AI deployment — e.g., an Apollo prospecting flow feeding a Clay enrichment table feeding an AI email writer feeding a CRM sequence. Write a Python script that prints a structured assessment: for each of the four categories, print the category name, a 1–2 sentence description of how it manifests in that deployment, and a severity rating of `low`, `medium`, or `high`. Run the script and verify the output table is complete.

5. **Build an inter-service handoff logger with rollback.** Create a module at `handlers/handoff_logger.py` containing a `HandoffLogger` class that wraps any callable pipeline stage. Each wrapped call must: (a) write a JSON line to `outputs/handoff_log.jsonl` containing `timestamp`, `stage_name`, `input_hash` (SHA-256 of the JSON-serialized input), `output_hash`, and `status`; (b) store the input payload in a `rollback_store` dict keyed by `stage_name`; and (c) expose a `.rollback(stage_name)` method that returns the last input payload for that stage so a caller can re-run from that point. Demonstrate it by wrapping three stages (`enrich`, `score`, `message`), running a record through, calling `.rollback("score")`, and printing the recovered payload. Confirm `outputs/handoff_log.jsonl` contains three entries.

6. **Design a full audit-and-rollback protocol for a multi-service GTM pipeline.** Using real CRM-exported data (export 50 contacts from HubSpot, Salesforce, or a CSV with equivalent columns), build an end-to-end pipeline with at least three stages — enrichment (call a real API such as Apollo's people search or mock it with a local lookup), scoring, and message generation. Insert an `audit_checkpoint` (from Exercise 3) between each stage. At the scoring stage, inject a bias that depresses scores for one segment. Your pipeline must: detect the shift at the checkpoint, log the handoff state using your `HandoffLogger`, trigger a rollback, and halt with a printed incident report showing which stage failed, the KS statistic, and the affected segment count. Write the full incident report to `outputs/skill-cais-incident-report.md`. Run the pipeline end-to-end and confirm the report file is generated with all fields populated.
