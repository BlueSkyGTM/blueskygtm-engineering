# Exercises — The Shift from Chatbots to Long-Horizon Agents

## Exercises

1. Implement a single-turn chatbot function that takes a user prompt string, calls the model once, and prints the response. Then implement a second function that calls the same model three times in sequence, feeding each output into the next prompt. Run both against the prompt `"Summarize the GTM strategy of a B2B SaaS startup."` and print the number of model calls and total tokens consumed for each. Your terminal output should clearly show the architectural difference: one call vs. three, no shared state vs. accumulated context.

2. Build a planning-execution loop with a hardcoded plan. Define a Python list of three step dictionaries, each with a `name`, `prompt`, and `status` field initialized to `"pending"`. Initialize a `state` dictionary with a `goal` key and an empty `results` dict. Write a loop that iterates over each step, calls a model (or a stub function returning a placeholder string), stores the result in `state["results"]` under the step name, and sets `status` to `"completed"`. After the loop, print the full `state` dictionary as formatted JSON. Verify every step shows `"completed"` and all results are populated.

3. Apply the planner-executor-state manager pattern to an enrichment waterfall. Define a hardcoded plan with four steps that a real GTM enrichment flow would run: (1) look up a company domain via an API stub, (2) find decision-maker names from that domain, (3) look up each person's LinkedIn profile URL, (4) synthesize a personalized outreach snippet using all accumulated data. Each step's executor function should read from the shared state dictionary (results of prior steps) and write its output back. No scaffold is provided — design the state schema and step function signatures yourself. Run the loop against the company `"Anthropic"` and print the final state. Your output should show each enrichment layer building on the previous one.

4. Build an observability wrapper that logs every planning-execution cycle as structured JSON lines. Write a decorator or context manager called `@trace_step` that wraps each executor call, captures the step name, input arguments, output result, timestamp (ISO 8601), and token count (use a counter or a mock), then appends it as a JSON object to a file at `outputs/agent-trace.jsonl`. Run the enrichment waterfall from Exercise 3 (or the hardcoded plan from Exercise 2) with the wrapper active. After execution, print the contents of `outputs/agent-trace.jsonl` and verify each line is valid, parseable JSON with all five fields present.

5. Extend the planning-execution loop into a complete agent module at `handlers/enrichment_agent.py`. Your agent must: (a) accept a company name and contact name as input, (b) execute a five-step enrichment plan (company lookup, contact lookup, technographic data, intent signals, outreach synthesis), (c) use the observability wrapper from Exercise 4 to log every step to `outputs/enrichment-trace.jsonl`, (d) support a simple retry: if any step fails (simulate a failure by checking for a `"FAIL"` token in the input), retry once before marking it `"failed"` in state, and (e) write the final synthesized outreach snippet and full state to `outputs/enrichment-result.json`. Run the agent against two inputs: `("Anthropic", "Dario Amodei")` and `("OpenAI", "Sam Altman")`. Print the path to both output files and the final status of every step for each run.
