# Exercises — Checkpoints and Rollback

## Exercises

1. **Implement a checkpoint-save-and-resume loop for a step counter.** Write a Python script that simulates a 1,000-step processing loop, writing a checkpoint to `outputs/checkpoint.json` every 50 steps containing the current step number, a running total, and a timestamp. Hard-code a simulated crash (e.g., `raise RuntimeError` at step 327) on the first run. On the second run, detect the existing checkpoint, load it, and resume from the saved step. Run the script twice in your terminal and observe that the second run starts at step 300 (the last checkpoint), not step 0. Print the final total and confirm it matches what a uninterrupted run would produce.

2. **Detect a corrupted checkpoint and roll back.** Manually corrupt your checkpoint file from Exercise 1 by truncating its JSON (e.g., remove the closing brace) or injecting an invalid field type. Modify your script to validate the checkpoint on load using a schema check (required keys present, types correct, step is a positive integer). On detection of corruption, print `CHECKPOINT CORRUPT — rolling back` and fall back to a secondary backup file `outputs/checkpoint.bak.json` that you write alongside every primary checkpoint. Run the script and observe the rollback message and successful recovery from the backup.

3. **Compare full versus incremental checkpointing on an enrichment workload.** Build a script that processes a list of 500 simulated company records (generate mock dicts with `company_id`, `name`, `website`, `employee_count`). Implement two checkpoint strategies: (a) **full** — serialize the entire list of completed records to `outputs/full_ckpt.json` every 25 records, and (b) **incremental** — append only the 25 new records to `outputs/incr_ckpt.jsonl` each interval. Use `time.perf_counter` to measure cumulative write time across all checkpoints for each strategy. Print a summary table showing total write time, final file size on disk (in bytes via `os.path.getsize`), and number of write operations. Confirm that incremental produces smaller individual writes but a larger or equal total file, and that full produces fewer but larger writes.

4. **Wire checkpoint-rollback into an Apollo-style enrichment pipeline.** Write a pipeline that takes a list of 200 prospect dictionaries (each with `email`, `company_domain`) and calls a mock enrichment function `enrich_prospect(prospect)` that returns a dict with `title`, `seniority`, and `linkedin_url`. The function should randomly fail ~5% of the time (raise an exception). Implement checkpointing so that every 20 successfully enriched prospects are persisted to `outputs/enrichment_checkpoint.jsonl`. When a failure occurs, catch the exception, log the failed prospect ID to `outputs/enrichment_failures.jsonl`, and continue. Run the pipeline to completion, then re-run it and verify via terminal output that zero prospects were re-enriched (the mock function should print its call count). This simulates an Apollo or Clay enrichment workflow where you must not re-call a paid API after recovery.

5. **Build a checkpoint manager with idempotency keys and precondition checks.** Design a `CheckpointManager` class in `handlers/checkpoint_manager.py` that wraps any sequential data pipeline. Each work item receives an idempotency key (e.g., `f"{pipeline_name}:{step}:{item_id}"`). On resume, the manager checks whether an item's key already exists in the checkpoint store (`outputs/idempotency_store.jsonl`) before processing it. Implement a precondition check that rejects items already present in the store even if called directly (not via resume). Include a method `recover()` that loads the last checkpoint, verifies all idempotency keys in the store are not duplicated, and returns the set of completed item IDs. Write a test script that processes 100 items, crashes at item 50, resumes, and prints a verification report: total items processed, items skipped due to idempotency, and confirmation that no item was processed twice. The artifact lands at `handlers/checkpoint_manager.py`.

6. **Design and prove a checkpoint strategy for a multi-stage enrichment pipeline with mixed failure modes.** Build a pipeline in `signals/examples/enrichment_resilience.py` with three sequential stages: (1) company domain lookup from a seed list of 300 names, (2) contact discovery via a mock Apollo API call per domain, (3) CRM write via a mock Salesforce insert per contact. Each stage has a different failure profile: stage 1 never fails, stage 2 fails ~3% of the time, stage 3 fails ~8% of the time. Implement per-stage checkpoints so a failure in stage 3 does not re-run stage 2 for that record. Produce a report at `outputs/skill-checkpoint-strategy.md` documenting your checkpoint frequency per stage, the rationale (cost of re-computation vs. cost of the checkpoint write), measured re-computation savings versus a naive no-checkpoint baseline, and the idempotency strategy for the CRM write stage. The report must include a table comparing the naive baseline (total API calls, total CRM writes, wall-clock time) against your checkpointed version.
