# Exercises — Bounded Self-Improvement Designs

## Exercises

1. **Implement** a minimal generate-and-test loop from scratch. Define an improvement operator that mutates a short string by inserting, deleting, or swapping a single character. Define a scoring function that rewards the presence of the substring `"GTM"`. Define a gatekeeper with one hard invariant: the mutated string must be between 5 and 30 characters long. Run the loop for 50 iterations from a seed string of your choice, printing each iteration's proposal, gatekeeper verdict, score, and whether it was accepted as the new best. Terminal output should show the current best evolving toward the target substring.

2. **Trace** accept-and-reject decisions through a pre-built loop and diagnose the failure mode. Hardcode an array of 15 mutation records — each with a proposed score, a boolean `invariant_passed` field, and the current best score at that step. Write a function that replays these records through the gatekeeper logic (invariant check first, then score comparison) and prints a line per record showing the decision and reason. After replaying, print a diagnosis: `"STAGNATION"` if no improvement occurred in the last 8 accepted candidates, `"DRIFT"` if scores oscillate by more than 20% across consecutive accepted candidates, or `"HEALTHY"` otherwise. Terminal output must show the full trace and the final diagnosis label.

3. **Build** a scoring function and at least two hard invariant bounds for a self-improving outbound email subject line. The scoring function should reward the count of personalization tokens (e.g., `{{company}}`, `{{first_name}}`) and penalize excessive punctuation. The two invariants: (a) character length must fall between 20 and 60 characters inclusive, (b) the subject must not contain any word from a blacklist of spam-trigger terms (define at least 5 terms). Write the improvement operator as a function that calls an LLM to propose a variant of the current best subject line. Run the loop for 25 iterations and print the final winning subject line, its score, and how many candidates were rejected by each invariant across the run.

4. **Apply** the four bounding primitives to a GTM prospecting workflow. Load a CSV of prospect records (or fetch 25 prospects from the Apollo API using a job-title filter). Build a bounded self-improvement loop that optimizes a qualification scoring formula — a weighted sum of seniority, company size, and intent signal. Implement at least two of the four bounding primitives: a **formal invariant** (score must stay within `[0, 100]`), an **alignment anchor** (the weight vector's cosine similarity to a reference vector must exceed 0.85), and **regression detection** (reject any weight change that causes more than 3 previously-qualified prospects to drop below threshold). Print the accepted/rejected trace, the final weight vector, and which bounding primitives you chose to implement with a one-line justification each.

5. **Design and implement** a complete bounded self-improvement system for a GTM outreach message that incorporates all four bounding primitives. The system takes prospect data (from Apollo, Clay, or a local CSV of at least 20 prospects), generates message variants via an LLM, and enforces: formal invariants (word count between 40 and 120), an alignment anchor (cosine similarity to a brand-voice reference message above 0.75), multi-objective constraints (a personalization sub-score and a clarity sub-score must each be at least 90% of the current best for the variant to be accepted), and regression detection (no variant may cause a previously-passing prospect to receive a message scoring below 70% of the prior best). After the loop completes, write a diagnostic report to `outputs/skill-bounded-self-improvement.md` containing: the final winning message, the full accept/reject trace table, a stagnation-or-drift diagnosis, and a written section (150–300 words) explaining why this loop cannot be formally proven safe by its own logic — referencing the information-theoretic limit that the gatekeeper and the artifact share the same computational substrate.
