# Exercises — Kill Switches, Circuit Breakers, and Canary Tokens

## Exercises

1. Implement a `CircuitBreaker` class with `Closed`, `Open`, and `HalfOpen` states. Track consecutive failures against a configurable threshold (default 3) and a cooldown period in seconds. Write a test script that calls `breaker.on_success()` and `breaker.on_failure()` in a sequence that forces all three state transitions, printing the state and failure count after each call so the transitions are visible in terminal output.

2. Build a kill switch that reads an environment variable `KILL_SWITCH` before each iteration of a loop that simulates sending 20 outreach emails (print "Sending email N to recipient@company.com" for each). Start the script with the variable unset, then mid-run open a second terminal and run `export KILL_SWITCH=1` (or write the value to a file the script reads). Verify that the loop halts within one iteration of the flag being set.

3. Deploy a canary token: create a Python module named `credentials.py` that, upon import or upon calling its `get_api_key()` function, logs the caller's process ID, working directory, `sys.argv[0]`, timestamp (ISO 8601), and machine hostname to a file at `outputs/canary_alert.log` before returning a fake key string. Then write a separate script that imports it to simulate an attacker stumbling on the file. Verify the log captures accurate caller metadata by inspecting the output file.

4. Integrate circuit breakers into an enrichment waterfall. Write a script that attempts to enrich a list of 10 company domains (e.g., `["stripe.com", "notion.so", "figma.com", ...]`) by trying three mock providers in sequence: `apollo_lookup`, `clearbit_lookup`, `hunter_lookup`. Each provider is a function that randomly succeeds or raises an exception 50% of the time. Wrap each provider in its own `CircuitBreaker` instance. When a provider's breaker is `Open`, skip it without calling the function and print `[SKIP] provider breaker is open`. Print the final enrichment result or failure for each domain. Run the script multiple times and confirm that providers with repeated failures are skipped while healthy ones continue serving requests.

5. Build a combined safety controller for an automated outbound campaign agent. The controller must run a loop that processes a list of leads from a CSV file (create one with 30 rows of name, email, company). Each iteration must: (a) check the kill switch from an environment variable, (b) route CRM-write calls through a circuit breaker that opens after 5 consecutive failures, and (c) emit a canary token in the form of a fake enrichment API endpoint string planted in the agent's config so that if an unauthorized process calls it, the attempt is logged with caller metadata. Write the controller to `handlers/campaign_safety.py` and a runner script that executes one campaign pass, printing state transitions, kill-switch checks, and any canary alerts to stdout. The handler must be importable and the campaign runnable via `python -m handlers.campaign_safety`.

6. Design a decision framework that maps failure modes to safety mechanisms. Write a markdown document to `outputs/skill-safety-mechanisms.md` that includes: a table comparing kill switch, circuit breaker, and canary token across dimensions (trigger condition, response time, reversibility, detection vs. prevention, operational cost). Then write a companion Python script `signals/examples/safety_decision.py` that, given a failure scenario description string (e.g., "provider API returning 500s", "attacker exfiltrating credentials", "operator needs emergency stop"), prints which mechanism(s) apply and why, drawing from the framework in the markdown file. Run the script against at least four scenarios and verify the recommendations differ by failure mode.
