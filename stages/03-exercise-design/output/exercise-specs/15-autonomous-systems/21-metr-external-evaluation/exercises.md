# Exercises — METR Time Horizons and External Capability Evaluation

## Exercises

1. **Compute** a time horizon half-life from synthetic task-completion data. Generate 200 synthetic `(duration_minutes, success)` observations where success probability decays exponentially with a true half-life of 45 minutes — use `numpy` with durations sampled log-uniformly from 1 to 480 minutes. Fit `P(success) = 0.5^(t / half_life)` using `scipy.optimize.curve_fit`. Print the fitted half-life and the absolute error from the true value of 45 minutes.

2. **Compare** two agents' time horizons side by side. You are given two CSVs in `data/agent_a_tasks.csv` and `data/agent_b_tasks.csv`, each containing columns `duration_minutes` and `success`. Fit a decay curve for each agent, print both half-lives, and state which agent can sustain longer autonomous runs. Also print the duration at which Agent A's success probability drops below Agent B's, if such a crossover exists within the 1–480 minute range.

3. **Simulate** all three failure modes that collapse agent time horizons. Build a single simulation that accepts parameters for context saturation (`context_limit_tokens`), tool-error compounding (`per_tool_error_prob`), and goal drift (`drift_rate_per_hour`). For each failure mode independently, run 500 simulated tasks at durations of 5, 15, 30, 60, 120, and 240 minutes, fit the resulting half-life, and print a table showing which failure mode causes the most aggressive half-life collapse at each duration tier.

4. **Apply** time horizon analysis to internal-vs-external metric disagreement. You are given `data/model_evaluations.csv` containing five models with columns `mmlu_score`, `humaneval_score`, `perplexity`, and a JSON-encoded `task_results` column of `(duration, success)` pairs. Compute the external time-horizon half-life for each model. Rank models by MMLU score and separately by fitted half-life. Print both rankings and identify any model that scores highly on internal benchmarks but has a short external time horizon — this is your deployment risk signal.

5. **Build** a GTM outbound agent time-horizon monitor that processes real prospect data. Load a sample of 100 prospects from an Apollo export (`data/apollo_prospects.csv`) and simulate an autonomous outbound workflow: for each prospect, the agent performs enrichment, personalization draft, and send — with stochastic failures injected for each step (API timeout, rate limit, personalization rejection). Log the wall-clock duration and success/failure for each completed outreach sequence. Fit a half-life from the resulting data and produce a deployment policy artifact at `outputs/skill-gtm-time-horizon-policy.md` that states the recommended maximum autonomous run length before human review, the fitted half-life, and which failure mode dominates the collapse.

6. **Design** a deployment decision framework that treats published METR time horizons as upper bounds, not predictions. Write a script that reads `data/model_time_horizons.json` (a mapping of model names to published METR half-lives) and a `data/deployment_contexts.json` file describing GTM use cases with their tool-error rates, context window requirements, and goal complexity scores. For each model–context pair, derate the published half-life based on the context's failure-mode risk profile and print a deployment-readiness matrix. Persist the full matrix and reasoning to `outputs/skill-deployment-readiness-matrix.md`.
