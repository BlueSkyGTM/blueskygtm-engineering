# Exercises — OpenAI Preparedness Framework and DeepMind Frontier Safety Framework

## Exercises

1. Implement a function `classify_risk(category: str, score: float) -> str` that accepts one of the four shared risk categories (`cbrn`, `cybersecurity`, `persuasion`, `autonomy`) and a normalized evaluation score from 0.0 to 1.0, then returns a risk level from the four-level taxonomy: `low`, `medium`, `high`, or `critical`. Use thresholds of 0.25, 0.50, and 0.75 as the boundaries. Run the function across all four categories with a score of 0.82 and print each result to the terminal.

2. Implement a function `framework_category_lookup(category: str) -> dict` that returns the label each framework assigns to a given capability category. For example, the key `cbrn` should return a dictionary with the OpenAI Preparedness label and the DeepMind Frontier Safety label for that same category. Populate the lookup for all four shared categories. Call the function on every category and print the full mapping. Verify that the terminal output shows both frameworks tracking all four categories even though the labels differ.

3. Build a composite risk scorecard. Given a dictionary of evaluation results such as `{"cbrn": 0.30, "cybersecurity": 0.72, "persuasion": 0.15, "autonomy": 0.88}`, compute the composite risk level using the conservative aggregation rule both frameworks share: the highest individual category level determines the composite. Print each category name, its score, its individual risk level, and the final composite risk level. Test with at least two different evaluation dictionaries and confirm the composite changes accordingly.

4. Implement `map_to_ccl(eval_results: dict) -> tuple` that takes a dictionary of per-category evaluation scores and returns a tuple of `(critical_capability_level, safety_governance_tier)`. Map composite risk levels to DeepMind's Critical Capability Levels 1 through 4, with each level corresponding to a named governance tier (Standard, Enhanced, High, Critical). For each governance tier, include a list of at least two required governance actions drawn from the framework's escalation protocol. Run the function on three sample evaluation sets representing low, high, and critical risk profiles, and print the CCL, governance tier, and required actions for each.

5. Build a vendor risk questionnaire mapper that reads a CSV file of enterprise AI vendor security questions — the kind exported from a Clay enrichment workflow or a CRM vendor-intake form — and maps each question to the corresponding framework risk category. The CSV will contain columns `question_id`, `question_text`, and `vendor_response`. Your mapper must classify each question into one of the four shared risk categories (or `unmapped` if no fit exists), determine which framework's evaluation criteria apply, and flag any vendor response that would trigger a governance escalation. Write the mapper to `handlers/vendor_risk_mapper.py`. Run it on the provided sample CSV at `signals/examples/vendor_questionnaire_sample.csv` and print a summary table showing each question, its mapped category, governing framework, and escalation flag.

6. Design and implement a disagreement detector that takes a set of evaluation results and identifies every case where the OpenAI Preparedness Framework v2 and the DeepMind Frontier Safety Framework v3 would mandate different governance actions for identical scores. For each disagreement, document the category, the evaluation score, the action each framework requires, and the reason for the divergence (e.g., category tracked by one framework but not the other, different threshold boundaries, different escalation binding strength). Write the full analysis to `outputs/skill-framework-disagreements.md`. The document must include at least three concrete disagreement scenarios with the supporting evidence for each.
