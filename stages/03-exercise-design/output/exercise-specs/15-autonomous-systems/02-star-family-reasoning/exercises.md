## Exercises

1. **Implement a minimal STaR loop.** Write a Python script that takes a list of 10 math word problems (you define the problems and their correct answers), calls an LLM API to generate a rationale + final answer for each, checks whether the extracted answer matches the label, and prints a summary: `Problem | Generated Answer | Correct? | Rationale (truncated)`. At the bottom, print the retention rate: `Retained X/10 rationales`. Run it and capture the terminal output.

2. **Compare STaR vs V-STaR retention sets.** Extend your STaR loop so it also retains *incorrect* (problem, rationale, answer) triples in a separate list — this is the V-STaR candidate pool. For each problem, sample 3 rationales. Print two tables side by side: (a) the STaR training set (correct-only) and (b) the V-STaR training set (correct + incorrect). Report the total dataset size for each. Your terminal output should make the retention difference immediately visible.

3. **Implement rationalization with hint.** Using the problems from Exercise 1 where your STaR loop produced zero correct rationales, construct a second-chance prompt that injects the correct answer as a hint (e.g., `"The correct answer is {answer}. Explain step by step why this is correct."`). Generate a new rationale for each failed problem. Print: which problems were rescued by the hint, which were still unsolvable, and the new overall retention rate after hint-based rationalization. No scaffold — design the prompt template and extraction logic yourself.

4. **Detect shortcut reasoning by multi-chain sampling.** For 5 problems, sample 5 independent rationales per problem (temperature ≥ 0.7). For each problem where ≥3 of 5 chains produce the correct answer, manually inspect the rationales and classify each as either `genuine_reasoning` (the chain of logic is sound and complete) or `shortcut` (the answer is right but the reasoning is circular, pattern-matched, or hand-waved). Print a per-problem report showing the shortcut ratio. At the end, print the overall shortcut rate across all sampled chains and state whether the dataset is safe to fine-tune on.

5. **Build a GTM content iteration pipeline modeled on STaR.** Create a script that treats cold-email variants as "rationales" and open/reply rates as the "correctness signal." Define 5 campaign segments with known high-performing messaging angles (your answer key). For each segment, generate 3 email variants via an LLM, score each against the known angle using a similarity check or keyword overlap, and retain only variants scoring above a threshold. Write the retained (segment, variant, score) triples to `outputs/skill-star-gtm-iteration.md` as a structured table. At the top of the file, document the STaR loop mapping: what maps to "generate," what maps to "score," what maps to "retain," and what maps to "fine-tune." For the GTM tooling requirement: pull at least 5 target company names from a Clay export CSV or Apollo-saved list and use them as personalization tokens in the generated variants.

6. **Prove Quiet-STaR cannot run at the API level.** Write a short analysis script that attempts to reproduce the *spirit* of Quiet-STaR using a standard chat-completions API call. Specifically: (a) call the API on a reasoning problem with `temperature=0`, (b) inspect what the API returns — note that you receive only the final visible tokens, never the internal forward-pass activations or hidden-state reasoning, (c) call the same prompt again and compare. Write your findings to `outputs/skill-quiet-star-analysis.md` structured as three sections: **What Quiet-STaR modifies** (the model's internal forward pass during generation), **What the API exposes** (only output tokens), and **Implication for reasoning placement** (reasoning lives inside the transformer's residual stream, not in a separable API artifact). Include the actual API inputs and outputs from your test calls as evidence.
