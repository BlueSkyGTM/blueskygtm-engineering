## Exercises

1. **Implement** a state machine that traces the seven-stage AI Scientist v2 pipeline from idea generation through peer review. Define an enum or set of string constants for each stage, and a `transition(current_stage, result)` function that returns the next stage. Run a simulated forward-only pass (no failures) and print each state transition to the terminal in the format `[stage_03_experimental_design] → [stage_04_code_execution]`. Your output should show all seven stages visited in order with no backward transitions.

2. **Extend** your state machine from Exercise 1 to handle the code-execution failure transition. When the code execution stage receives a failure result, route execution back to `experimental_design` instead of forward to `result_analysis`. Inject a simulated stderr string into a `context` list that the upstream stage can read. Add a bounded `max_attempts` counter (default 3); when attempts are exhausted, transition to a `TERMINATE` state. Run the simulation with a mock experiment that fails twice then succeeds — your terminal output should show the backward transitions, the injected error messages, the attempt count incrementing, and final success on attempt 3. Then run it again with an experiment that always fails and confirm it terminates with attempts exhausted.

3. **Implement** a complete failure-aware re-planning loop that handles **both** failure transitions: code execution failure (routes back to experimental design with stderr) and peer review sub-threshold score (routes back to manuscript drafting with reviewer feedback). No scaffold — design the loop from scratch. Your loop should accept a list of research ideas, process each through all seven stages, and print a structured log entry for every state transition including the stage name, attempt number, decision (advance/retry/terminate), and any injected context. Feed it a dataset of three simulated ideas: one that passes cleanly, one that hits a code execution failure and recovers, and one that fails peer review and recovers on revision. Verify the log shows all three completing the full pipeline with different retry patterns.

4. **Apply** the autonomous re-planning pattern to a GTM enrichment workflow. Simulate a Clay or Apollo enrichment call as the "code execution" stage of a pipeline that tests ICP hypotheses (e.g., "companies with <50 employees in fintech convert 2x better"). Write a loop that issues enrichment API calls against a CSV of 10 target companies, handles rate-limit or timeout responses by injecting the error context back into the planner, adjusts the query parameters, and retries within a bounded count. If you have live API access, use it; otherwise mock the API responses with a fixture that returns errors for specific parameter combinations. Print a final summary showing which companies were successfully enriched, how many retries each required, and which were abandoned.

5. **Build** a complete simulated research loop that executes the full AI Scientist v2 cycle: idea generation through automated peer review, with both failure transitions active. Write it as a standalone module that takes a research topic string as input, simulates each of the seven stages with placeholder LLM responses (hardcoded or random), and produces a structured JSON artifact containing the idea, experimental plan, execution log, results summary, manuscript draft text, and peer review score with feedback. The loop must retry on failures with bounded attempts and must not produce a manuscript if any stage exceeds its retry limit. Place the module at `handlers/research_loop.py` and write the JSON output for at least two topics to `outputs/`. Run the module from the terminal with `python -m handlers.research_loop --topic "..."` and confirm the JSON artifacts are created and contain all expected fields.

6. **Design** and document a security model for executing LLM-generated code in a production GTM stack, mapping directly from AI Scientist v2's sandboxing requirements. Write a specification that covers: sandbox isolation strategy (container, process, or WASM), network egress policies for LLM-generated code, resource limits (CPU, memory, wall-clock timeout), secret injection boundaries (what the sandboxed code can and cannot access), webhook authentication patterns for calling back into the loop, and data isolation between concurrent research runs. For each requirement, state the threat it mitigates and give a concrete implementation approach (library, tool, or configuration). Include a comparison table between the AI Scientist v2 research-sandbox threat model and a GTM enrichment-automation threat model, noting where the risks converge and diverge. Save the document to `outputs/skill-code-sandbox-isolation.md`.
