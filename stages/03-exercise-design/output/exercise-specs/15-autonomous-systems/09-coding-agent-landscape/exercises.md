# Exercises — The Autonomous Coding Agent Landscape (2026)

## Exercises

1. **Classify ten commercial coding tools into their architectural tier.** Write a Python script that defines a list of tools (e.g., `"GitHub Copilot ghost-text"`, `"Cursor Chat"`, `"Aider"`, `"Claude Code"`, `"Codex CLI"`, `"Devin"`, `"Copilot Chat"`, `"Cursor Agent Mode"`, `"SWE-agent"`, `"Cline"`) and maps each to its tier number 1–5 using a dictionary. Print a formatted table to the terminal with columns `Tool | Tier | Loop Type | Can Execute?`. Run the script and confirm output.

2. **Implement the three primitive tools a Tier 3 agent needs.** Write a Python module with three functions — `read_file(path)`, `write_file(path, content)`, and `run_bash(command)` — that each return a string result. Call each tool manually in a `__main__` block: read a small file you create, write a new file to `/tmp/`, and run `echo "agent tool test"`. Print the return value of each call. Confirm all three produce the expected output in your terminal.

3. **Build a trace analyzer that detects agent failure modes.** Create a Python script that reads a JSON-formatted agent execution log (an array of step objects with `action`, `tool_result`, and `tokens_used` fields). Implement three detection functions: one that flags consecutive duplicate actions as a potential infinite loop, one that flags any step where cumulative tokens exceed 100,000 as context exhaustion risk, and one that flags write operations to test files as silent test corruption risk. Generate a synthetic trace with at least 15 steps that triggers all three detectors. Run the analyzer and print which steps are flagged for each failure mode.

4. **Implement path traversal prevention for a file-write tool.** Write a Python function `safe_write_file(base_dir, requested_path, content)` that rejects any `requested_path` containing `..`, absolute paths, or symlinks, and only writes within `base_dir`. Test it with four cases: a valid relative path, a `../../etc/passwd` attempt, an absolute path attempt, and a path with a nested symlink. Print `[ALLOWED]` or `[BLOCKED]` with the reason for each test case. Then add a token budget guardrail: a function `check_budget(estimated_tokens, limit=8000)` that returns a boolean and prints a warning when usage exceeds 80% of the limit.

5. **Build a GTM code-validation gate for an enrichment pipeline.** Agents in your enrichment system generate Python snippets that transform Apollo or Clay export data. Implement a validator in `handlers/gtm_code_gate.py` that takes agent-generated Python source code as a string and checks it against a GTM schema before execution: (a) the code must define a function named `enrich(record: dict) -> dict`, (b) it must not import `subprocess`, `os.system`, or `eval`, (c) it must reference at least one expected field from the schema (e.g., `email`, `company_domain`, `linkedin_url`), and (d) it must pass `ast.parse` without syntax errors. Create three test snippets — one valid, one with a forbidden import, one missing the `enrich` function — and run them through the gate. Print `[ACCEPTED]` or `[REJECTED: reason]` for each. The artifact must land at `handlers/gtm_code_gate.py`.

6. **Design and implement a complete guardrailed Tier 3 agent loop for GTM data enrichment.** Build a standalone agent in `handlers/guardrailed_enrichment_agent.py` that accepts a natural-language task description (e.g., "Write a script that normalizes company domains from a Clay export"), calls an LLM to generate code, runs the generated code through your `gtm_code_gate.py` validator from Exercise 5, executes the validated code against a sample Clay export record (a dict with `company_name`, `website`, `linkedin_url`), and prints the enriched output. The loop must enforce: a maximum of 5 iterations (infinite loop prevention), a 8,000-token budget (context exhaustion prevention), and the validation gate before any code execution (silent corruption prevention). Run the agent against two tasks and capture the terminal output showing iteration count, token usage, gate decision, and final enriched record. The artifact must land at `handlers/guardrailed_enrichment_agent.py`.
