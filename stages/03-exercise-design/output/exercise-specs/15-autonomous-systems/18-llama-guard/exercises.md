# Exercises — Llama Guard and Input/Output Classification

## Exercises

1. Implement a single-call Llama Guard classifier that takes a role (`User` or `Agent`) and a text string, formats the MLCommons taxonomy into the model's prompt template, and calls the model. Parse the raw generation into a structured dict with keys `verdict` (`safe`/`unsafe`), `violations` (list of category codes like `S2`, `S8`), and `raw_output`. Run your classifier against three test strings: a benign greeting, a prompt containing explicit violent content, and a message with subtle harassment. Print the structured dict for all three so the verdict and violation list are visible in the terminal.

2. Trace the two pipeline paths through a mock agent system. Implement a `SafetyGuardrail` class with `check_input(user_text)` and `check_output(agent_text)` methods—each calling your Exercise 1 classifier with the appropriate role. Feed a conversation where the user asks for something benign and the agent responds normally, then feed a conversation where the agent's drafted response contains unsafe content. Print a trace log showing which check fired, the verdict at each stage, and whether the pipeline allowed or blocked the exchange.

3. Build a two-pass classification pipeline for outbound sales email screening. Your pipeline receives a rep's draft email plus an AI-generated prospecting email, runs Llama Guard's input check on the rep draft and output check on the AI email, and applies a configurable rejection policy: `BLOCK` (halt and log), `REWRITE` (strip flagged phrases and re-run), or `DRY_RUN` (log violations but send anyway). Include structured logging that writes each classification result to `outputs/guardrail-log.jsonl`. Test against three scenarios: a clean email, an AI email that makes unsubstantiated competitive claims (flag under S7 if your taxonomy includes economic harm), and a rep draft containing aggressive language. Print the final disposition and the log path for each run. Use a real prospect record from Apollo or a Clay export CSV as the email context input.

4. Compute per-category precision and recall for your classifier against a labeled test set. Build a JSONL test set of 20 examples covering at least 4 harm categories, each with a ground-truth label. Run all 20 through your classifier and emit a metrics table to the terminal showing, for each category: true positives, false positives, false negatives, precision, recall, and F1. Identify which category has the worst recall and print its false-negative examples so you can inspect what the classifier missed.

5. Diagnose false positives from your Exercise 4 run and implement a mitigation layer. Design a `MitigationHandler` that takes each false-positive verdict, classifies the root cause (prompt ambiguity, category overlap, overcautious threshold), and emits one of three actions: `EXEMPT_CATEGORY` for that input's domain, `ESCALATE_HUMAN_REVIEW` with context, or `REWRITE_PROMPT` with a suggested taxonomy-prompt fix. Persist the handler as `handlers/false_positive_mitigation.py` and write a human-readable diagnosis of all false positives with proposed fixes to `outputs/skill-false-positive-diagnosis.md`. Run the handler against your Exercise 4 false positives and print the action assigned to each one.
