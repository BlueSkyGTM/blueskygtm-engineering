# Exercises — Browser Agents and Long-Horizon Web Tasks

## Exercises

1. **Implement the core observe-act loop.** Build a `BrowserAgent` class with an `__init__` that accepts a list of action primitives (click, type, scroll, navigate, extract), and a `run(goal, max_steps)` method that calls an LLM to select an action from the current page state, executes it, and loops until the goal is met or `max_steps` is reached. Use Playwright's `page.accessibility.snapshot()` as your observation source. Print the step number, selected action, and result on each iteration so terminal output shows the full trace. Verify by running the agent against `https://books.toscrape.com` with the goal "navigate to the Science category and report the first book title."

2. **Compare observation representations on the same page.** Write a script that loads `https://books.toscrape.com` in Playwright and produces three observation artifacts from the identical DOM state: (a) the accessibility tree JSON, (b) a PNG screenshot saved to disk, (c) the raw `page.content()` HTML. Measure the byte size of each and the number of tokens each produces when passed through `tiktoken.encoding_for_model("gpt-4o")`. Print a comparison table to the terminal showing bytes, tokens, and the number of interactive elements each representation surfaces. Re-run the script on a second page with a large table (e.g., a Wikipedia comparison page) and print both tables so you can see how the representations scale.

3. **Implement failed-action detection and recovery.** Extend the `BrowserAgent` from Exercise 1 so that after each action execution, it validates the post-action state. Define at least three failure modes: element-not-found after click, navigation timeout, and extract returning empty string. For each failure, implement a recovery strategy — retry with a re-queried selector on the first failure, backtrack to the previous URL on the second consecutive failure of the same type, and terminate with a diagnostic message on the third. Log each failure, recovery attempt, and its outcome to the terminal. Test by pointing the agent at a page where the target selector has been intentionally renamed (use a local HTML file you control) so you can observe all three recovery tiers firing in sequence.

4. **Build an indirect prompt injection detector for browser-collected data.** Write a Python function `scan_for_injection(extracted_text: str) -> InjectionReport` that takes text a browser agent scraped from a web page and flags potential injection attempts: hidden instruction patterns ("ignore previous instructions", "system:", role-play prompts, base64-encoded payloads, and zero-width character sequences). Apply it to the scraped `textContent` of prospect LinkedIn profiles or company "About" pages fetched via Playwright. Print each page's URL, number of injection signals found, and the matched patterns. Run it against at least five real prospect pages and report which pages contain flags. This exercise evaluates the attack surface from Learning Objective 4 using live web data.

5. **Build a browser agent that fills a GTM enrichment gap no API covers.** Apollo and Clay expose company and contact data through APIs, but many niche data points — e.g., a company's current job openings with role counts, a prospect's recent conference speaking engagements, or a startup's investor list from a cap table page — have no structured API. Choose one such gap, then implement a complete browser agent in `signals/examples/enrichment_agent.py` that: navigates to a target site (e.g., a company's careers page or an angel investor directory), uses the observe-act loop with accessibility-tree observations, paginates or filters as needed, extracts the data into a structured list of dicts, and writes the result to `outputs/enrichment-agent-results.json`. Include error recovery from Exercise 3's patterns and run the injection scanner from Exercise 4 on all extracted text before writing output. The artifact must be terminal-executable via `python -m signals.examples.enrichment_agent --url <target>` and produce observable JSON output.

6. **Design a browser-agent-vs-API capability map for a GTM stack.** Produce a markdown document at `outputs/skill-browser-agent-gtm-map.md` that inventories ten common enrichment targets (company headcount growth, tech stack, funding history, key hires, job openings, press mentions, social proof, pricing pages, case studies, leadership bios). For each target, identify whether an API exists (name it — Apollo, Clay, Crunchbase, etc.), whether the API covers the data fully or partially, and whether a browser agent could fill the gap. For cases where a browser agent is the only option, specify the observation representation you would use, the primary action primitives required, the top two failure modes, and the injection risk level (low/medium/high) with justification. Conclude with a decision tree: given a new enrichment requirement, when do you reach for an API vs. a browser agent? This document is a persistent reference artifact — it should be specific enough that an engineer reading it could begin implementation.
