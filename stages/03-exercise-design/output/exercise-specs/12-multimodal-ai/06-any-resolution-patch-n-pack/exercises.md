# Exercises — Any-Resolution Vision: Patch-n'-Pack and NaFlex

## Exercises

1. **Compute** native-resolution patch counts for a list of image dimensions. Given a patch size of 16×16 and image dimensions `[(224, 224), (480, 1920), (1024, 768), (256, 256)]`, implement a function that returns `(grid_h, grid_w, num_patches)` for each image, then concatenates the patch counts and prints the total packed sequence length. Verify that the 480×1920 receipt yields 3,600 patches — the kind of resolution a GTM team needs for full OCR fidelity on scanned business cards.

2. **Build** a block-diagonal attention mask for a packed sequence of three images whose patch counts are `[196, 60, 400]`. Your function should accept a list of per-image patch counts and return a boolean mask of shape `(total_patches, total_patches)` where `True` means "allowed to attend." Print the mask sparsity ratio and confirm there are zero `True` entries in any off-diagonal block.

3. **Implement** factorized 2D positional embeddings for a packed batch. For each image in the batch, generate `(row, col)` grid coordinates at that image's native grid resolution, flatten in row-major order, and assign each patch a pair `(row_id, col_id)` relative to its own image origin — not the global sequence. Print the max row and max column index per image and confirm that a 30×120 receipt grid and a 14×14 thumbnail coexist without index collision beyond their own sub-grid.

4. **Build** a first-fit-decreasing bin-packing scheduler. Accept a list of `(image_id, height, width)` tuples and a token budget (e.g., 8192 patches per batch). Sort images by patch count in descending order, then greedily assign each image to the first batch with remaining capacity. Print the batch assignments, per-batch utilization percentage, and the number of batches needed. Test with a mix of logo crops (256×256), website screenshots (1920×1080), and chart images (800×600).

5. **Compute** token budgets for a GTM enrichment pipeline using real CRM data. Export 30 company logos and 20 website screenshots from an Apollo or Clay enrichment table (or use `pillow` to generate representative images at those aspect ratios if API access is unavailable). Run your patch-count function from Exercise 1, then run your bin-packing scheduler from Exercise 4 with a budget of 4096 tokens. Compare the total batches and estimated inference passes against a fixed 224×224 resize baseline. Save the full comparison table — image dimensions, native patch counts, batches needed, utilization per batch, and cost delta — to `outputs/skill-naflex-token-budget.md`.

6. **Implement and compare** all three variable-resolution strategies. Write three independent functions — AnyRes tiling (fixed tile grid, e.g., 2×2), NaFlex patch-n'-pack (native patches + block-diagonal mask), and M-RoPE (rotary position encoding with separate frequency bands for row and column) — that each accept the same list of 20 variable-resolution images and return `(token_count, positional_encoding_shape, attention_mask_complexity)`. Run all three on the test set and write a structured comparison to `outputs/skill-vision-packing-comparison.md` covering: (a) token efficiency ratio (tokens used / tokens a fixed 384×384 baseline would consume), (b) whether each approach supports arbitrary aspect ratios at inference time without retraining, and (c) lines of code and external dependencies for each implementation.
