# Exercises — MIO and Any-to-Any Streaming Multimodal Models

## Exercises

1. **Implement a unified vocabulary allocator** that assigns non-overlapping integer ranges for text (32K BPE tokens), image (16K VQ-VAE codebook entries), speech (2K EnCodec entries across 4 codebooks), and music (8K codec entries). Write `vocab.py` that exports a `Vocab` class with methods `text_range()`, `image_range()`, `speech_range()`, `music_range()`, and `modality_of(token_id: int) -> str`. Run a validation loop that prints every range boundary and asserts zero overlap, then print `modality_of()` for tokens 0, 31999, 32000, 47999, 48000, and 59999 to confirm correct partitioning.

2. **Build a token-stream inspector** that reads a list of `(token_id, modality_hint)` tuples from a JSON file and prints the stream as a color-tagged sequence, inserting a visible marker (e.g., `[IMG→]`, `[SPCH→]`) every time the modality changes. Feed it a synthetic 40-token stream that cycles text→image→text→speech→text and confirm the switch markers appear at the correct positions. Terminal output should show the full annotated sequence.

3. **Compare cascade vs. any-to-any latency** using a simulated workload. Write `latency_sim.py` that models a cascade pipeline (ASR: 180ms, LLM: 350ms first-token, TTS: 120ms) and an any-to-any single model (unified first-token: 420ms, streaming at 75 tokens/sec). Compute and print total wall-clock time to deliver the first multimodal output chunk and the full 10-second response for both architectures. Print which architecture wins each metric and by how much.

4. **Trace the streaming decode path** for a single inference step. Implement `decode_trace.py` that takes a unified token ID (e.g., 48342), resolves its modality via your `Vocab` class, routes it to the correct per-modality decoder stub (text detokenizer, VQ-VAE image decoder, EnCodec audio decoder), and prints a step-by-step trace: `[token 48342] → modality: speech → codebook 1, entry 342 → SpeechDecoder → waveform chunk (240 samples)`. Include at least one token from each modality in your trace run.

5. **Build a production token-stream quality monitor** for an any-to-any model that serves GTM content (sales call transcripts enriched with speaker voice and product screenshots). Create `handlers/gtm_stream_monitor.py` that ingests a token log from `outputs/gtm_token_stream.jsonl` (each line: `{"token_id": int, "timestamp_ms": int, "modality": str}`), computes three metrics — modality-switch frequency per minute, speech-codebook utilization (unique entries / codebook size), and inter-modality gap latency (ms between last text token and first speech token in a switch). Output a summary report to `outputs/skill-gtm-stream-report.md` with a table of all three metrics and a pass/fail threshold for each. Populate the JSONL with at least 200 synthetic tokens simulating a 30-second GTM pitch clip.

6. **Design and prototype a unified vocabulary schema** for a GTM-specific any-to-any model that adds a fifth modality: CRM-structured-data tokens (deal stage, ICP fit score, sequence step) using 1,024 entries. Write `vocab_design.py` that defines all five modality ranges, validates non-overlap, and generates `outputs/skill-vocab-spec.md` documenting the schema with a rationale for each range size, expected tokens-per-second per modality, and a worked example of a 5-modality interleaved token sequence representing: text pitch → product image → CRM enrichment data → voice CTA → background music sting. Print the full schema table to terminal and confirm the spec file is written.
