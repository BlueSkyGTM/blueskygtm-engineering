# Exercises — Open-Weight VLM Recipes: What Actually Matters

## Exercises

1. **Implement a five-axis ranking printer.** Write a standalone Python script that prints each axis of the VLM design space (image encoder, connector, LLM backbone, data mixture, resolution schedule) alongside its ablation-determined impact rank. Use the Cambrian-1 encoder-swing data (8+ DocVQA points), MM1 connector findings, and Molmo caption-quality results as the basis for your ordering. Run the script; the terminal must print all five axes, their ranks (1–5), and a one-line justification string for each. Your output is self-checking: if any rank justification contradicts the ablation evidence cited in the lesson, revise.

2. **Apply ablation tables to benchmark prediction.** Write a script that takes a JSON file of ablation deltas (axis, old_value, new_value, delta_MMMU, delta_DocVQA, delta_TextVQA) and, given a target benchmark name as a CLI argument, prints which single axis change produced the largest gain on that benchmark. Create two sample JSON inputs modeled on MM1 and Cambrian-1 tables. Run the script against both inputs for each of the three benchmarks. Terminal output must show the winning axis per benchmark per table.

3. **Build a screenshot task-classifier with routing output.** Implement a Python classifier that accepts a local image path, computes lightweight features (dimensions, aspect ratio, estimated text density via pixel variance or edge detection), and routes to one of four presets: `(model, quantization, resolution)`. Examples: a dense dashboard screenshot routes to Qwen2-VL at FP16 at 1568×1568; a simple logo routes to INT4 at 384×384. Your script must print the chosen preset and the feature values that drove the decision. Test against at least three screenshots of different types and capture the terminal output.

4. **Deploy a quantized Qwen2-VL FastAPI service.** Stand up a FastAPI application with three endpoints: `GET /health` (returns model load status and GPU memory free), `POST /predict` (accepts base64 image + text prompt, returns model output), and `POST /batch` (accepts a list of image-prompt pairs, processes them sequentially with OOM fallback that drops to INT4 or reduces resolution on `torch.cuda.OutOfMemoryError`). Use Qwen2-VL with AWQ or GPTQ quantization. Start the server with `uvicorn`, curl the health endpoint, and submit two test predictions. Terminal must show the server startup log, the health-check JSON response, and both prediction outputs.

5. **Build an OCR-extraction evaluation harness over Apollo-enriched company data.** Pull 20 company domain screenshots (or use Apollo's company enrichment API to fetch company descriptions and pair them with homepage screenshots). For each company, extract the company description text using Qwen2-VL at three quantization levels (FP16, INT8, INT4). Compare the extracted text against the Apollo ground-truth description using a token-level F1 score. Print a per-quantization mean F1 and failure-rate table (failure = F1 below 0.6). Write the full harness to `handlers/quant_ocr_eval.py` and write a markdown summary of results to `outputs/skill-quant-ocr-eval.md`. Run the harness end-to-end; terminal output must show the three-row comparison table.

6. **Design a production VLM routing policy document for a GTM enrichment pipeline.** Produce a markdown spec at `outputs/skill-vlm-routing-policy.md` that defines routing rules mapping input image characteristics (resolution, text density, document type) to model + quantization + resolution presets, with a decision-tree diagram in text form. Include a cost-latency-quality tradeoff table justifying each routing decision, and a fallback chain for OOM and timeout scenarios. The document must reference all five VLM design axes and explain how each axis factored into at least one routing rule. Validate your policy by running the classifier from Exercise 3 against a held-out set of 10 screenshots and confirming that the policy's predicted routes match the classifier's outputs for at least 8 of 10 cases; print the match rate to the terminal.
