# Exercises — Chameleon and Early-Fusion Token-Only Multimodal Models

## Exercises

1. **Implement a VQ codebook lookup.** Build a function that accepts a tensor of continuous patch embeddings (shape `[num_patches, embed_dim]`) and a learned codebook matrix (shape `[codebook_size, embed_dim]`), and returns the integer token index of the nearest codebook entry for each patch via Euclidean distance. Print the resulting token IDs for a fixed random seed so output is reproducible. Verify that two patches with identical embeddings map to the same token.

2. **Implement a sentinel-based token interleaver.** Write a function that takes a list of text token IDs and a list of image token IDs (produced by your Exercise 1 codebook), then merges them into a single flat sequence using sentinel tokens `<IMG_START>` and `<IMG_END>` to delimit image regions. Feed it a 4-word text sequence and a 2×2 grid of image tokens. Print the final interleaved sequence and confirm the sentinels appear at the correct positions.

3. **Trace attention over a mixed sequence.** Build a minimal single-head self-attention layer (no training — random fixed weights). Feed it two sequences of the same length: one all-text tokens and one interleaved text+image tokens using the same shared embedding table. Compute the attention weight matrix for each. Print both matrices and verify that the attention layer applies identical operations regardless of token origin — no branching, no separate projections, no modality-specific code path.

4. **Compare early-fusion vs. late-fusion forward passes.** Implement two minimal forward functions for a multimodal model: (a) early-fusion, where text and image tokens share one embedding table and one transformer, and (b) late-fusion, where a separate vision encoder processes image features and a projector maps them into the LLM's embedding space before concatenation. Feed the same input through both. Print the parameter counts, the output shapes, and the number of loss terms each architecture computes. Explain in a docstring why early-fusion applies one uniform next-token loss while late-fusion splits its objective.

5. **Build a complete early-fusion prototype.** Combine your VQ tokenizer, sentinel interleaver, and a 2-layer transformer into a single script that accepts a text string and a synthetic image (random pixel tensor), tokenizes both modalities into a shared vocabulary, interleaves them, runs a forward pass, and prints the next-token logits at every position. Save the script to `handlers/early_fusion_demo.py`. Run it and confirm that logits are produced at both text and image positions, demonstrating uniform autoregressive capability across modalities.

6. **Evaluate early-fusion suitability for a GTM enrichment pipeline.** Write a script that pulls company data from Apollo's API (or reads a sample JSON export) containing both text fields (description, industry, tech stack) and image data (company logo URLs). Tokenize the text fields into BPE token IDs. Fetch each logo, resize to a fixed grid, and run it through your VQ codebook to produce image token IDs. Interleave both into early-fusion sequences. Then write a design document to `outputs/skill-early-fusion-gtm-eval.md` that: (a) shows three sample interleaved sequences with printed token counts for each modality, (b) identifies whether the logo images carry enough signal density to justify VQ tokenization over a late-fusion CLIP encoder, (c) recommends early or late fusion for this specific GTM task with concrete reasoning about representational quality, vocabulary contention, and downstream generation needs (e.g., does the model need to *generate* logos, or only *read* them?).
