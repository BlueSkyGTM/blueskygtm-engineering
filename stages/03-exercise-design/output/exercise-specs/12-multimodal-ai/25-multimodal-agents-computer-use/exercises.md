# Exercises — Multimodal Agents and Computer-Use (Capstone)

## Exercises

1. **Implement a minimal observe-reason-act agent loop.** Create a Python script that reads a sequence of pre-captured screenshots from `fixtures/screenshots/` (or generate simple placeholder images), passes each one to a mock VLM function that returns a canned action, prints the action, and loops until the mock returns `{"action": "done"}` or a step limit of 10 is reached. Each iteration must print the step number, the observation filename, and the chosen action as structured JSON. Run the script and confirm terminal output shows the full cycle.

2. **Implement and validate a GUI grounding action schema.** Write a schema validator that accepts raw JSON strings representing VLM emissions and parses them into one of four action types: `click` (with x, y coordinates), `type` (with text payload), `scroll` (with direction and amount), and `done`. Reject malformed or out-of-range payloads. Feed the validator a file containing ten sample emissions—seven well-formed, three broken—and print which pass, which fail, and the parse reason for each.

3. **Compare screenshot-coordinate grounding against DOM-tree grounding.** Build a small benchmark harness that loads five saved HTML fixtures, extracts a DOM element list for the DOM-tree approach and a rendered screenshot for the coordinate approach, and sends each to a mock grounding function with latency timers. Compute average latency and a simple error metric (whether the predicted element or coordinate matches a labeled ground-truth target) for both approaches. Print a comparison table to the terminal.

4. **Build a simulated computer-use agent that extracts structured data from a multi-page web flow.** Using Playwright, launch a local mock CRM dashboard (serve the provided HTML fixtures or build your own) that paginates across three pages of contact records. Implement an agent that takes a screenshot of each page, sends it to a mock VLM that returns click and extract actions, navigates page to page, and accumulates discovered contacts into a list. Write the final extracted records to `outputs/extracted-contacts.json` and print a summary count.

5. **Build an execution tracer for a computer-use agent.** Instrument an agent loop (your own or the one from Exercise 1) with a tracing layer that logs every step to a structured event log: the observation hash, the predicted action, the action result, and the resulting page state. Add heuristic detectors that flag three failure classes—grounding error (action coordinates land on no interactive element), action mismatch (action type incompatible with current state), and state drift (page content changed unexpectedly between steps). Run the tracer against a fixture set with injected errors and print a diagnostic report identifying each flagged failure, its step number, and its category. Persist the tracer as `handlers/agent_tracer.py` and the diagnostic report as `outputs/trace-report.json`.

6. **Design and implement a multimodal GTM enrichment agent.** Build an agent that navigates a web-based prospecting interface (Apollo search results, a Clay table view, or a local mock of either) using screenshot-based computer-use, discovers contact fields at runtime by reading what it sees, extracts structured prospect data, and writes enriched records to `signals/examples/gtm_enrichment_agent.py` as a self-contained, runnable module. The agent must handle at least one failure recovery path (wrong page loaded, element not found) and log each recovery decision. Run the module end to end and confirm it produces terminal output showing the enrichment steps and a final JSON record set.
