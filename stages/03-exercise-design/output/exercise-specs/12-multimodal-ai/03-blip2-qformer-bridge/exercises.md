## Exercises

1. **Implement a single cross-attention block with learnable query tokens.** Using PyTorch, create a `CrossAttentionBlock` module that takes a tensor of learnable queries `(N_q, d_model)` and a tensor of visual patch embeddings `(N_kv, d_model)` and produces attended output `(N_q, d_model)`. Initialize 32 query tokens of dimension 768 with `nn.Parameter(torch.randn(32, 768))`. Feed a dummy `(257, 768)` patch tensor through the block and print the output shape. Confirm it equals `torch.Size([32, 768])`.

2. **Trace dimensional changes through the full BLIP-2 pipeline.** Write a script that constructs dummy tensors representing each stage — ViT input image `(1, 3, 224, 224)`, patch embeddings `(1, 257, 1408)`, Q-Former queries `(1, 32, 768)`, Q-Former output `(1, 32, 768)`, projected LLM input `(1, 32, 2560)` — and prints the shape at each stage with a descriptive label. Use a `nn.Linear(1408, 768)` for the patch-to-Q-Former dimension match and `nn.Linear(768, 2560)` for the projection to LLM space. Verify all six stages print correctly.

3. **Build a multi-layer Q-Former and verify output-token independence from input resolution.** Implement a `QFormer` module containing 6 cross-attention layers (each reusing your `CrossAttentionBlock` from Exercise 1) with shared 32 query tokens. Run three random image-embedding inputs through it: one simulating `224×224` (257 patches), one simulating `512×512` (1025 patches), and one simulating `1024×1024` (4097 patches), all at dimension 1408. Print the input patch count and output token count for each. Confirm all three produce exactly 32 output tokens.

4. **Compute and compare parameter counts for a Q-Former bridge versus a linear MLP projector.** Implement both: (a) your 6-layer Q-Former from Exercise 3 with `d_model=768` and (b) a two-layer MLP projector `nn.Sequential(nn.Linear(1408, 768), nn.GELU(), nn.Linear(768, 2560))` that maps every patch token directly. Count trainable parameters for each and print the totals. Then compute the LLM token cost: print how many tokens the LLM receives under each design assuming a `512×512` input image. State which design adds more FLOPs to the LLM forward pass and by what factor.

5. **Build a Q-Former-powered screenshot analyzer that enriches company data from Apollo.** Fetch 5 company records from Apollo's API (use a search query like `"AI infrastructure"`). For each company, download the homepage screenshot using a headless tool such as Playwright or a screenshot API. Pass each screenshot through a frozen CLIP ViT encoder to get patch embeddings, then through your `QFormer` module from Exercise 3 to get 32 bottlenecked tokens. Save the 32-token embeddings as a JSON file and print each company name alongside its embedding tensor shape. Write the full pipeline to `handlers/screenshot_qformer_enricher.py` and the output embeddings to `outputs/screenshot-embeddings.json`. Confirm all 5 companies produce identically shaped `(32, 768)` embeddings despite different page dimensions.

6. **Design a decision framework for Q-Former versus linear projection in a GTM enrichment pipeline.** Write a standalone module at `handlers/projector_selector.py` that takes a config specifying: average image resolution, target LLM context budget (max input tokens), latency target (ms), and whether downstream generation or classification is required. The module should output a recommendation (`"qformer"` or `"linear_projection"`) with a one-paragraph justification computed from the inputs. Include a `__main__` block that runs three configurations: a logo-classification task at `128×128` with a 512-token budget; a full-page-screenshot VQA task at `1024×1024` with a 64-token budget; and a product-catalog-description task at `512×512` with a 256-token budget. Print the recommendation and justification for each. Save a markdown summary of all three recommendations and the reasoning to `outputs/projector-selection-guide.md`.
