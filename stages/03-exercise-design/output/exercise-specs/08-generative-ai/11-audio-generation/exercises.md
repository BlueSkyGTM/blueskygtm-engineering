# Exercises — Audio Generation

## Exercises

1. **Implement a TTS API call that writes audio to disk.** Choose any TTS API available to you (OpenAI Audio, ElevenLabs, Google Cloud TTS, or Azure Speech). Synthesize the sentence "This is a test of our audio generation pipeline." and save the result as a WAV file at `outputs/tts_test.wav`. Print the file path, the file size in bytes, and the audio duration in seconds to the terminal. Verify by playing the file or checking that the duration matches what the API reported.

2. **Compare three TTS architectures by generating the same text through at least two different engines or model presets.** If you have access to multiple providers, use them. If not, use two different voices or model tiers from a single provider (e.g., OpenAI `tts-1` vs. `tts-1-hd`, or ElevenLabs `eleven_turbo_v2` vs. `eleven_multilingual_v2`). Time each generation using `time.perf_counter()`. Save each output as a separate WAV file. Print a comparison table to the terminal showing: model name, wall-clock latency (ms), file size (bytes), and sample rate. Write a one-line comment in your script noting which tradeoff (latency vs. quality) you observe.

3. **Implement a speaker-configuration function and a consent-checking guard.** Write a Python function `configure_speaker(voice_id, stability, similarity_boost, clone_reference_path=None, consent_verified=False)` that returns a dict of speaker parameters suitable for your chosen TTS API. If `clone_reference_path` is set but `consent_verified` is `False`, raise a `ValueError` with a message explaining that voice cloning requires explicit consent. Call the function three times: once with a stock voice (no cloning), once with a clone reference and `consent_verified=True`, and once with a clone reference and `consent_verified=False`. Print the returned dict for the first two calls and the caught exception message for the third. All output must be visible in the terminal.

4. **Compute and compare mel-spectrograms of two audio files you generated in Exercise 2.** Load both WAV files using `librosa` or `scipy.io.wavfile`. Compute a mel-spectrogram for each (128 mel bins, hop length 512). Calculate a spectral distance metric between them — mean squared error between log-mel matrices is sufficient. Print the per-file mean mel energy, the MSE distance, and the sample rate of each file. The terminal output should let you confirm whether the two architectures produced spectrally similar or divergent output for the same input text.

5. **Build a batch personalized-audio pipeline that reads account data from a CSV and generates one audio file per prospect.** Export a CSV from Apollo, Clay, or your CRM containing at minimum: `first_name`, `company`, and `headline` columns (use a 5-row sample if your account is live). Write the pipeline at `handlers/batch_personalized_audio.py`. For each row, synthesize a personalized opener such as "Hi {first_name}, saw what your team is doing at {company} — {headline}. Wanted to share something relevant." Save each audio file to `outputs/personalized_audio/{company}_{first_name}.wav`. Log a progress line per row to the terminal showing the row index, the generated filename, and the audio duration. At the end, print the total number of files generated and the average generation time per row.

6. **Design and document an evaluation protocol for selecting a TTS architecture for a cold-outreach use case.** Write the protocol at `outputs/skill-tts-eval.md`. It must cover: (a) which three metrics you will measure (at least one latency-based, one quality-based, one cost-based), (b) how you will collect ground-truth quality judgments from sales reps or prospects, (c) how spectral analysis fits into your automated screening step, and (d) the decision rule for promoting a model from test to production. Include a table template with placeholder rows for three candidate models. Conclude with a recommendation for which architecture family (autoregressive, diffusion, or vocoder pipeline) you expect to win for short-form cold-outreach audio and why.
