# Exercises — Optimization

## Exercises

1. Implement vanilla gradient descent from scratch on the quadratic function `f(x, y) = x² + 4y²`. Start from `(-3.0, -3.0)`, use a learning rate of `0.1`, and run for 50 iterations. Print the current `(x, y)` and function value at each iteration so you can watch convergence in the terminal. Verify that both coordinates are moving toward `(0, 0)`.

2. Extend your gradient descent implementation to SGD with momentum on the same quadratic function. Keep the same starting point and learning rate, add a momentum coefficient of `0.9`, and run for 50 iterations. Print `(x, y, f(x,y))` at each step alongside the vanilla GD trace from Exercise 1 so you can see momentum accelerating along the steep `y`-axis. Report which optimizer reaches `f < 0.01` in fewer iterations.

3. Implement Adam from scratch — including bias-corrected first and second moment estimates — and run it alongside vanilla GD and SGD-with-momentum on the Rosenbrock function `f(x, y) = (1 - x)² + 100(y - x²)²` starting from `(-1.5, 1.5)`. Use `lr=0.001, beta1=0.9, beta2=0.999, eps=1e-8` for Adam and `lr=0.001, momentum=0.9` for SGD. Print a table at each iteration showing all three optimizers' current positions and loss values. Report which optimizer reaches the valley `(1, 1)` first and explain in a printed summary why Adam's per-parameter learning rates matter for this function's curved ravine.

4. Build a waterfall enrichment pipeline simulator that calls three providers in sequence for a list of 100 synthetic company records: `provider_A` (cost $0.02, hit rate 0.60), `provider_B` (cost $0.05, hit rate 0.35 on misses from A), `provider_C` (cost $0.10, hit rate 0.20 on misses from B). Add a deduplication gate that skips a provider if the record was already enriched. Run the pipeline and print total cost, number of enriched records, average cost per successful enrichment, and marginal cost of each successive provider tier (cost added by that tier divided by additional records enriched by that tier).

5. Extend your waterfall pipeline with a TTL-based cache layer: store enriched results in a dictionary keyed by domain with a configurable TTL (in seconds). Simulate a second run of the same 100 records after 30 seconds, 120 seconds, and 300 seconds with a TTL of 60 seconds. For each re-run, print cache hit rate, total cost, and average cost per enriched record. Add a `kill_depth` parameter (1, 2, or 3) that caps how many providers a single record can touch, and print a comparison table showing yield, total cost, and cost-per-success for each kill-depth setting. Save the full pipeline implementation to `handlers/waterfall_enrichment.py`.

6. Using Apollo's People Search API (or a mock that returns realistic JSON with `email`, `phone`, and `linkedin` fields for a given company domain), configure a selective enrichment strategy: query Apollo first, and only fall back to a second provider (e.g., Hunter.io or a mock) for records where Apollo returned no email. Run the pipeline across 50 companies sampled from a CRM export CSV, log every provider call with timestamp and cost, and compute the marginal value of the fallback provider. Write a one-page configuration document specifying your recommended kill depth, cache TTL, and selective-enrichment threshold, with the cost/yield/latency tradeoff rationale grounded in your measured numbers. Save the document to `outputs/skill-enrichment-config.md`.
