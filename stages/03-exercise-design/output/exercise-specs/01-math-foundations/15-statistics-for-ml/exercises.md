# Exercises — Statistics for Machine Learning

## Exercises

1. **Implement descriptive statistics from scratch.** Write a Python script that computes mean, median, variance, and skew for a hardcoded list of B2B deal sizes (use at least 30 values with a clear right tail). Use only the `math` and `statistics` modules — no numpy, no scipy. Print each statistic on its own line, then print a one-line diagnosis: `"RIGHT-SKEWED"`, `"LEFT-SKEWED"`, or `"SYMMETRIC"` based on the relationship between mean and median. Confirm your script runs with `python stats_diagnose.py` and prints all five lines.

2. **Implement a bootstrap confidence interval.** Given the same deal-size list from Exercise 1 (or a new one with at least 50 values), write a function that resamples the data with replacement 10,000 times and computes the mean each time. Use only `random.choices` from the standard library. Print the 2.5th and 97.5th percentile bounds of the resampled means. Run it twice and confirm the interval is stable to within a few percentage points across runs.

3. **Apply Bayes' theorem to a lead-scoring problem.** Implement a script that starts with a prior conversion probability of 0.03 for a cold lead list from Apollo. Update the prior sequentially given three pieces of evidence: the lead opened two emails (likelihood ratio: 2.1), visited the pricing page (likelihood ratio: 4.3), and attended a demo (likelihood ratio: 12.0). After each update, print the posterior probability and the absolute change from the previous posterior. Identify which evidence step drove the largest jump and print its name as the final line.

4. **Run a permutation test on two CRM export groups.** You have two lists of conversion rates — one from a Q3 outbound campaign and one from a Q4 outbound campaign (hardcode at least 40 values per group, or read from CSVs in `signals/`). Implement a permutation test from scratch: pool both groups, shuffle and split 10,000 times, count how often the shuffled group-mean difference exceeds the observed difference. Print the observed difference, the p-value, and a deployment decision: `"GO"` if p < 0.05 and the observed difference is practically meaningful (at least 1 percentage point), otherwise `"NO-GO"`.

5. **Build a significance-vs-effect-size evaluator.** Create `handlers/ship_evaluator.py` that accepts two numeric lists via command-line arguments (or reads from a JSON file path) and outputs four things: the p-value from a permutation test you implement yourself, Cohen's *d* as the effect size, a boolean `is_significant` (p < 0.05), and a final `SHOULD_SHIP` decision that is `True` only when both p < 0.05 **and** |d| > 0.2. Include a hardcoded example comparing two model-variant prediction-error arrays and run it to demonstrate a case where p is significant but effect size says no-go. The artifact must persist at `handlers/ship_evaluator.py` and be executable via `python handlers/ship_evaluator.py`.

6. **Design an experiment spec for a Clay enrichment rollout.** Write a markdown document to `outputs/skill-experiment-spec.md` that defines a go/no-go statistical evaluation for switching from one enrichment vendor to another. Specify: the metric you will compare, minimum sample size needed to detect a 0.3-standard-deviation effect at alpha 0.05 and power 0.80, the permutation-test procedure you will run, and the decision rule combining significance and effect size. Include a short justification for why a p-value alone would be insufficient. This document should be specific enough that a teammate could implement the test in Python without further instruction.
