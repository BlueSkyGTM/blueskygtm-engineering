# Exercises — Open Models: Architecture Walkthroughs

## Exercises

1. **Implement a config parser that extracts architectural identity from a Hugging Face model card.** Fetch `config.json` from the `mistralai/Mistral-7B-Instruct-v0.3` repository using the HTTP API at `https://huggingface.co/{model}/raw/main/config.json`. Write a script that prints the following derived facts: attention type (MHA, MQA, or GQA — infer from `num_attention_heads` vs `num_key_value_heads`), position encoding scheme (`rope_theta` presence means RoPE, `max_position_embeddings` gives the trained limit), MLP variant (check for `intermediate_size` and any SwiGLU indicators), and whether embeddings are tied. Run the script against three models — Mistral-7B-Instruct-v0.3, `meta-llama/Meta-Llama-3-8B`, and `Qwen/Qwen2.5-7B-Instruct` — and print a comparison table to the terminal.

2. **Compute the KV cache size for a single forward pass at maximum context length.** Write a Python function `kv_cache_bytes(config: dict, batch_size: int, seq_len: int) -> int` that reads `num_key_value_heads`, `head_dim` (compute as `hidden_size // num_attention_heads` if absent), `num_hidden_layers`, and `torch_dtype` (default fp16 = 2 bytes) from a config dict. Load the config for `mistralai/Mistral-7B-Instruct-v0.3`, compute the cache size for batch_size=1 at seq_len=4096, then at seq_len=32768, and print both values in gigabytes. Confirm that the KV cache — not the model weights — dominates memory at long context.

3. **Estimate total GPU memory footprint from config alone, without loading weights.** Write a function `estimate_vram_gb(config: dict, seq_len: int, batch_size: int = 1) -> float` that sums: parameter memory (compute total params from embedding size, layer count, hidden size, intermediate size, and KV head count — account for tied vs untied embeddings), KV cache memory (reuse your function from Exercise 2), and a 20% activation overhead multiplier on the combined total. Run it against both `meta-llama/Meta-Llama-3-8B` and `Qwen/Qwen2.5-14B` configs at seq_len=8192. Print a breakdown table showing params (B), param memory (GB), KV cache (GB), activations (GB), and total VRAM (GB). Verify your parameter estimate against the published count on each model card.

4. **Convert Mistral-7B's GQA config to a hypothetical MHA config and compute the delta.** Take the Mistral-7B config and produce a modified version where `num_key_value_heads` equals `num_attention_heads` (32) instead of 8. Compute: the additional KV projection parameters introduced, the new KV cache size at seq_len=8192 batch_size=1, and the additional VRAM required. Print a before/after comparison. Then reverse the exercise: starting from `meta-llama/Meta-Llama-3-8B` (which already uses GQA), convert it to MQA by setting `num_key_value_heads` to 1 and compute the same deltas. Report which conversion yields larger memory savings and by what factor.

5. **Build a model selector for a GTM enrichment pipeline.** You are processing Apollo-exported company CSVs (5,000 rows, each with a `company_description` field averaging 400 tokens) and extracting structured firmographic signals (ICP fit score, tech stack indicators, growth signals). Write a script in `signals/examples/model_selector.py` that takes a directory of candidate model configs (download `config.json` from at least four models: Mistral-7B-Instruct-v0.3, Llama-3-8B, Qwen2.5-7B, and Phi-3-medium-14k), then computes for each: estimated throughput (assume compute-bound at `1 / (num_hidden_layers × hidden_size²)` as a proxy), VRAM at batch_size=8 seq_len=1024, and whether it fits the full company description in its context window. Rank the models and print a recommendation table with columns: model, fits_context, est_vram_gb, throughput_rank, recommendation_reason. The script must be runnable with `python signals/examples/model_selector.py` and produce the table as terminal output.

6. **Design and prove an optimal batching strategy for multi-model GTM inference.** Scenario: you have two GPU tiers available — a single A100 80GB and a single L4 24GB. Your GTM pipeline runs two tasks: (a) high-throughput ICP scoring on 50,000 Apollo leads (short prompts, ~200 tokens, needs >1000 tokens/sec aggregate), and (b) deep enrichment on 500 high-value accounts (long prompts, ~6000 tokens, lower throughput acceptable). Write a brief in `outputs/skill-model-routing.md` that specifies which model goes on which GPU, the batch size and KV cache budget for each, the expected throughput, and a proof (backed by the formulas from this lesson) that the configuration fits in memory and meets the latency targets. Include a table of the computed numbers. The artifact must be self-contained — a reader should be able to verify every claim by re-running the calculations from the config values you cite.
