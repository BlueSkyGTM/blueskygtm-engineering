# Exercises — Building a Tokenizer from Scratch

## Exercises

1. **Implement BPE training from scratch.** Using only the Python standard library, write a script that reads a plaintext corpus from `data/corpus.txt` (create one with 20–30 sentences of B2B outreach copy), converts text to byte-level character sequences, and iteratively merges the most frequent adjacent pair for `N` merge rounds. After each merge round, print the round number, the pair selected, its frequency, and the resulting vocabulary size. Run with `N=20` and verify the merge table grows by one entry per round.

2. **Implement encode and decode using your trained merge table.** Extend the previous script with an `encode(text)` function that applies merges in priority order to convert raw text into a sequence of token IDs, and a `decode(ids)` function that reconstructs the original text. Encode five sample sentences, print each sentence alongside its token ID sequence and token count, then decode the IDs back and assert the round-trip output matches the input exactly. Run the script and confirm zero assertion failures.

3. **Compare tokenization strategies on the same corpus.** Implement three tokenizers — character-level, word-level (split on whitespace), and your BPE subword tokenizer — and run all three on every line of `data/corpus.txt`. For each strategy, print the total token count, average sequence length per line, vocabulary size, and compression ratio (characters per token). Output a summary table to the terminal showing all four metrics side-by-side. Verify that subword tokenization achieves a lower token count than character-level while maintaining a smaller vocabulary than word-level.

4. **Build a token cost estimator for enrichment records.** Write a script that loads a JSON array of 50 Apollo-style lead records (each containing `first_name`, `last_name`, `company`, `title`, `email`, `company_description`, and `headline` fields — generate synthetic data if no file exists). Serialize each record into the prompt format your enrichment pipeline would send to an LLM, tokenize each serialized prompt with your BPE tokenizer, and flag any record whose token count exceeds a configurable threshold (default 4000). Print the record index, token count, and the specific fields contributing the most tokens for every flagged record. Verify the script exits with a non-zero code if any record is flagged, so it can be used as a pre-flight gate in a pipeline.

5. **Sweep vocabulary size and frequency thresholds, then report findings.** Train BPE tokenizers with merge counts of `[10, 30, 50, 100, 200]` and minimum pair frequency thresholds of `[2, 5, 10]` on a combined corpus of B2B outreach emails and CRM company descriptions (at least 500 lines). For each configuration, compute the compression ratio, vocabulary size, and percentage of tokens that are single characters (a proxy for under-merging). Write the full results table with observations about which configurations produce diminishing returns to `outputs/skill-tokenizer-eval.md`. Confirm the file contains all 15 configurations and a written recommendation for which setting to use for a GTM enrichment pipeline with a 4096-token context window.

6. **Design and implement a token budget router for heterogeneous CRM payloads.** Build a handler that accepts a list of enrichment records with varying field sizes (some with long company descriptions, some with URLs, some with multi-line notes). The router should assign each record to one of three processing tiers based on estimated token cost: `lite` (< 500 tokens, batch with other records), `standard` (500–2000 tokens, solo API call), or `heavy` (> 2000 tokens, trigger truncation or field selection logic). Print a routing summary showing how many records landed in each tier, which fields were most responsible for `heavy` classification, and what percentage of total API cost the tiering saves versus sending every record as a solo call. Save the router and its tier thresholds to `handlers/token_budget_router.py` and verify it runs end-to-end on 100 synthetic CRM records.
