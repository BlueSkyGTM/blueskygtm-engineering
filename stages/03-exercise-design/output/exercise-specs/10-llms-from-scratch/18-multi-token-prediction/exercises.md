# Exercises — Multi-Token Prediction (MTP)

## Exercises

1. **Implement** a parallel MTP forward pass from scratch using PyTorch. Define a toy transformer backbone with `nn.TransformerEncoder` (2 layers, `d_model=64`, `vocab_size=1000`). Instantiate K=4 linear projection heads of shape `(d_model, vocab_size)`. For a random input batch of token IDs `(batch=2, seq_len=32)`, compute per-head cross-entropy loss against targets shifted by offsets 1 through 4. Print each head's individual loss and the summed total loss to the terminal. All four losses should be finite and in a plausible range (roughly 1–10 for a random-init model on this vocab).

2. **Compute** the parameter overhead of adding K MTP heads to models of three configurations: `(d_model=512, vocab=32000, K=4)`, `(d_model=2048, vocab=128000, K=4)`, and `(d_model=4096, vocab=256000, K=8)`. For each configuration, print the total backbone parameter count (treating the backbone as 12 layers of `4*d_model^2` feedforward parameters plus `d_model*vocab` embedding parameters), the total MTP head parameter count (`K * d_model * vocab`), and the percentage overhead the heads add to the model. Run the script and confirm the overhead percentages appear in the terminal.

3. **Build** both a parallel MTP module and a sequential MTP module for the same toy backbone from Exercise 1 (`d_model=64`, `vocab=1000`, K=3). The parallel module attaches K independent heads to the shared hidden state. The sequential module feeds `h_t` through head 1 to produce logits for `t+1`, then feeds the predicted embedding (argmax → embedding lookup during a forward demo) concatenated with `h_t` through a small projection layer into head 2 for `t+2`, and so on. Run a forward pass on the same random input through both modules. Print per-head losses for both designs side by side. Write a docstring or printed statement (3–5 sentences) explaining why the sequential design preserves the causal chain — specifically, what information each head conditions on that the parallel design ignores. The printed comparison must be visible in terminal output.

4. **Apply** the MTP concept to a structured GTM extraction task. Write a Python script that loads a JSON array of 10 raw company descriptions (you may hardcode them or fetch from a public API like `https://api.clearbit.com/v2/companies/find?domain=stripe.com` if you have a key). Define a target schema with fields: `company_name`, `industry`, `employee_range`, `headquarters_city`. Simulate how an MTP-trained model could decode all four fields in a single forward pass (K=4 parallel predictions from one hidden state) versus a next-token-only model that must autoregress through each field character by character. Compute and print the estimated token-decoding step count for each approach assuming average field length of 8 tokens. Print which approach requires fewer forward passes and by what factor.

5. **Build** an MTP evaluation harness that tests whether multi-token prediction training improves structured-output fidelity on GTM enrichment data. The harness should: (a) accept a path to a JSONL file of Clay or Apollo export records with fields like `name`, `title`, `company`, `email`, `linkedin_url`, (b) construct K=5 synthetic prediction targets from those fields (tokenized as offsets from each anchor position), (c) compute per-head cross-entropy for a randomly initialized head layer as a baseline, and (d) print a summary table showing per-field prediction difficulty ranked by loss. The script must be importable and runnable as `python -m handlers.mtp_eval <path_to_jsonl>`. Save the module to `handlers/mtp_eval.py`.

6. **Design** a model-selection rubric for a GTM engineering team evaluating candidate LLMs for multi-field extraction and enrichment schema generation. The rubric must explicitly address: whether MTP training (parallel or sequential) is a relevant selection criterion, how it interacts with JSON schema adherence, latency requirements for batch enrichment, and cost-per-token tradeoffs. Include a scoring matrix (weighted criteria, 1–5 scale) and a worked example comparing two hypothetical models (one MTP-trained, one next-token-only) on a Clay enrichment workflow targeting 10,000 contacts. Save the completed rubric to `outputs/skill-mtp-gtm-model-selection.md`.
