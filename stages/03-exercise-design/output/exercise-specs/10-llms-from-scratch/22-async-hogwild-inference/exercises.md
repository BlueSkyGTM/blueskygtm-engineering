# Exercises — Async and Hogwild! Inference

## Exercises

1. **Implement** an async inference pipeline using `asyncio.Semaphore` to bound concurrency to 4 workers. Simulate 20 inference calls against a mock LLM endpoint that sleeps a random 0.1–0.5 seconds per call. Then implement the same 20 calls serially. Print both wall-times and the speedup ratio. Run the script three times and confirm the async version is consistently faster.

2. **Compute** the expected collision count for lock-free updates under three configurations: (parameter_space=1,000, sparsity=0.01, workers=4), (parameter_space=10,000, sparsity=0.05, workers=8), and (parameter_space=500, sparsity=0.10, workers=16). Use the birthday-problem approximation from the lesson. Print a table showing configuration, per-worker expected updates, and expected collisions per epoch.

3. **Build** a two-worker Hogwild! shared-cache simulator. Create a shared dictionary representing a company-enrichment cache with 200 keys. Each worker iterates through 50 randomly selected keys, performs a simulated enrichment (short sleep), and writes the result back to the shared dict with no lock. Run both workers concurrently using threads. After completion, verify cache integrity by checking that all 100 expected keys are present and no values are corrupted. Print the number of keys each worker wrote and the final cache size.

4. **Apply** the locked-vs-lock-free comparison to a shared embedding matrix. Initialize a 500×64 matrix of random floats. Spawn 4 threads that each apply 100 sparse rank-1 updates (random row index, random gradient vector) under two modes: (a) guarded by a `threading.Lock`, and (b) lock-free. Run each mode 10 times, compute the L2 norm of the difference between the locked-final and lock-free-final matrices for each trial, and print the mean and standard deviation of divergence. Report whether divergence correlates with your collision-rate estimate from Exercise 2.

5. **Design and build** a GTM enrichment waterfall evaluator that tests whether the Hogwild! assumption holds under realistic conditions. Fetch 50 company domains (use the provided `data/sample_companies.csv` or pull from Apollo's Companies API if you have an API key). Run three enrichment stages concurrently across 4 async workers: (1) Clearbit logo lookup, (2) a mock tech-stack detector, (3) employee-count estimation. In lock-free mode, all workers write results to a shared company record dict. In serial mode, each company completes all three stages before the next begins. Measure wall-time for both modes, count any record inconsistencies (missing fields, overwritten values), and write a verdict file to `outputs/skill-hogwild-gtm-eval.md` stating the collision rate, divergence count, and whether lock-free is safe for this workload. Include a one-paragraph recommendation on when to switch from Hogwild! to locked updates in a production enrichment pipeline.
