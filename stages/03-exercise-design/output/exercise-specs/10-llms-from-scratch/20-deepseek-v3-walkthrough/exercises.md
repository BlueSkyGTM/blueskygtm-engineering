# Exercises — DeepSeek-V3 Architecture Walkthrough

## Exercises

1. **Compute the MLA KV-cache footprint at 128k context.** Write a Python script that calculates the total KV-cache memory for (a) standard multi-head attention with 128 heads × 128 dims per head and (b) MLA with a 512-dimensional latent vector. Use a batch size of 1 and bf16 precision. Print both values in GB and the compression ratio. Vary the context length across `[4096, 32768, 131072]` and print a table showing how the gap widens.

2. **Calculate the active-parameter ratio and expert firing economics.** Write a script that computes: the fraction of parameters active per forward pass (37B / 671B), the fraction of experts firing per token (8 / 256), and the expected number of tokens each expert processes per 1000-token batch under uniform routing. Print all three values. Then add a brief in-code comment block (printed to stdout) defending why top-8 is chosen over top-1 or top-32 — cite the throughput/quality tradeoff.

3. **Implement a minimal MoE router with auxiliary-loss-free load balancing.** Build a router that takes a batch of token embeddings (shape `[batch, d_model]`), produces logits for 256 experts, applies top-8 selection, and adjusts expert bias terms based on rolling expert-load statistics (no auxiliary loss). Generate a random batch of 10,000 tokens through your router and print the per-expert firing counts. Compare against a uniform-random top-8 baseline by printing the standard deviation of expert counts for both. Your balanced router should show measurably lower variance.

4. **Measure n-gram repetition in model output to evaluate MTP impact.** Write a script that accepts a list of text samples (e.g., 20 paragraphs of ~200 tokens each — use any open-weight model's output or sample text files). Compute the repetition rate for unigrams, bigrams, and trigrams (ratio of repeated n-grams to total n-grams). Print a summary table. Then write a one-paragraph interpretation (printed to stdout) of whether low repetition rates are consistent with MTP's dense training signal hypothesis.

5. **Build a GTM inference pipeline evaluator that benchmarks DeepSeek-V3 against alternatives.** Create a script that calls the Apollo or People Data Labs enrichment API (or reads a local JSON file of 50 enriched company records) and sends each record's firmographic data through DeepSeek-V3's API (or an equivalent open-weight endpoint) to generate a one-line ICP-fit justification. Measure: average tokens generated, latency per call, and estimated cost at DeepSeek-V3's published FP8 pricing versus a dense model of your choice. Write a recommendation to `outputs/skill-foundation-model-eval.md` that defends or rejects DeepSeek-V3 for this GTM use case, citing the active-parameter ratio, MoE throughput characteristics, and FP8 training economics.

6. **Design a signal module that adapts MLA-style compression to a GTM feature pipeline.** Build a Python module in `signals/examples/mla_feature_compression.py` that takes a high-dimensional enrichment feature vector (e.g., 4096 dims from stacked Clay/Apollo/Waterfalls data across 500 companies), applies a low-rank down-projection to 512 dims (mirroring MLA's latent compression), reconstructs via up-projection, and measures the reconstruction error (cosine distance). Compare memory footprint and downstream classification accuracy (on any labeled ICP column) between the compressed and uncompressed representations. Print a final report showing the compression ratio, reconstruction fidelity, and accuracy delta — then conclude whether MLA-style compression is viable as a feature-store optimization for GTM data.
