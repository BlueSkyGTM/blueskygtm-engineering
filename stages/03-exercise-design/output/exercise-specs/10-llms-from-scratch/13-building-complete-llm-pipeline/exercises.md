# Exercises — Building a Complete LLM Pipeline

## Exercises

1. **Implement the input validation stage.** Build a function `validate_input(raw: str, max_chars: int = 5000) -> dict` that returns `{"valid": bool, "errors": [str], "cleaned": str}`. Reject empty strings, inputs exceeding `max_chars`, and inputs containing the literal substring `<script>`. Emit a structured log line (JSON to stderr) showing the validation decision and the input length. Call it on three test inputs — a valid company description, an empty string, and an oversized blob — and print each result. Verify the three outputs differ and the log lines appear in the terminal.

2. **Compute token accounting and prompt assembly.** Write a function `assemble_prompt(input_text: str, system_prompt: str, schema_hint: str) -> dict` that concatenates the three components, estimates tokens using the heuristic `len(text) // 4`, and returns `{"prompt": str, "estimated_tokens": int, "within_budget": bool}` where `within_budget` is `False` above 4000 tokens. Run it with a 10-token system prompt, a 50-character company description, and a schema hint that specifies two fields (`company_name`, `icp_fit_score`). Print the assembled prompt and the token estimate.

3. **Trace execution through a multi-stage pipeline.** Build a function `run_pipeline(input_text: str) -> dict` that chains: `validate_input` → `assemble_prompt` → `simulate_api_call` → `parse_json` → `validate_schema`. For `simulate_api_call`, return a hardcoded JSON string `{"company_name": "Acme", "icp_fit_score": 7}` after a 0.5s sleep. For `parse_json`, strip markdown code fences if present before `json.loads`. At each stage emit a structured trace event to stderr with the stage name, timestamp, and a `status` field (`entered`, `passed`, `failed`). Run the pipeline with a valid input and an input containing code fences in the simulated response. Verify you see at least 10 trace lines in the terminal covering all five stages.

4. **Apply fallback routing with schema validation.** Implement a fallback chain: call a `primary_model()` function that randomly raises `ConnectionError` 70% of the time and otherwise returns `{"company_name": "Globex", "icp_fit_score": 8, "summary": "Strong ICP match."}`. When the primary fails after 3 retries with exponential backoff (base delay 0.2s, factor 2x), route to `fallback_model()` which always returns `{"company_name": "Globex", "icp_fit_score": 6}`. Validate both responses against a schema requiring `company_name` (str) and `icp_fit_score` (int 1–10). Run the pipeline 10 times in a loop and print which model handled each call and whether the output passed validation. Verify you see both primary and fallback outcomes across the 10 runs.

5. **Build a GTM enrichment pipeline that processes CRM export data.** Implement a pipeline in `handlers/enrichment_pipeline.py` that reads a CSV file (`data/companies.csv` with columns `domain`, `description`) and enriches each row by passing the description through a full pipeline — validation, prompt assembly, simulated LLM call returning `{"company_name": str, "industry": str, "employee_band": str, "icp_score": int}`, JSON parsing, schema validation, and fallback. For any row whose enrichment fails both primary and fallback, write the row to `outputs/enrichment_failures.csv`. Write successful enrichments to `outputs/enrichment_results.csv`. Run the pipeline on a 20-row CSV you generate (include 2 deliberately malformed rows) and verify the two output files contain 18 and 2 rows respectively.

6. **Design a mapping between the LLM pipeline stages and a Clay-style enrichment waterfall.** Write a document to `outputs/skill-pipeline-vs-waterfall.md` that maps each pipeline stage (input validation, token accounting, retry/backoff, JSON parsing, schema validation, fallback routing) to its equivalent in a multi-provider data enrichment workflow (e.g., Apollo → Clearbit → Hunter email-finding waterfall). For each stage, state what silent failure looks like in the enrichment context and how the trace events would surface it. Include a concrete example: a company where Apollo returns no phone number, Clearbit provides one that fails format validation, and Hunter provides the fallback. Verify the file exists and contains at least one mapping table with all six stages.
