# Exercises — Quantization: Making Models Fit

## Exercises

1. **Implement INT8 symmetric quantization from scratch.** Given a random FP32 PyTorch tensor of shape `(1000,)` drawn from a normal distribution, compute the symmetric scale factor, quantize to INT8 (round + clamp to [-127, 127]), dequantize back to FP32, and print the original tensor's first 5 values alongside the reconstructed values. Then compute and print the MSE between original and reconstructed tensors.

2. **Implement INT4 symmetric quantization from scratch on the same tensor and compare.** Repeat the quantize → dequantize → MSE pipeline for INT4 (16 levels, range [-7, 7]) using the same tensor from Exercise 1. Print both MSE values side by side and the ratio of INT4_MSE / INT8_MSE so you can see how much error INT4 introduces relative to INT8 for this distribution.

3. **Compute VRAM budgets for three model sizes across four precision levels.** Build a Python script that takes a model's parameter count (e.g., 7B, 13B, 70B) and a target precision (FP32 = 4 bytes, FP16/BF16 = 2 bytes, INT8 = 1 byte, INT4 = 0.5 bytes) and prints the required VRAM in GB. For each model size, print which precision levels would fit on an NVIDIA A10G (24 GB), an A100 40GB, and an A100 80GB. Run the script and verify that a 70B model fits in INT4 on an A100 80GB but does not fit in FP16 on an A100 40GB.

4. **Apply bitsandbytes NF4 quantization to a real model and measure the difference.** Load `EleutherAI/pythia-410m` in both standard FP32 and 4-bit NF4 using `BitsAndBytesConfig`. For each variant, print the GPU memory footprint in MB. Generate a completion for the prompt `"The best way to qualify a B2B lead is"` with each model (temperature 0, max_new_tokens 50) and print the outputs side by side so you can visually inspect quality degradation, if any.

5. **Design a cost-per-inference comparison for a GTM lead-scoring workflow that produces a persistent artifact.** You process 50,000 inbound leads per month, each requiring a single LLM classification call (ICP fit: yes/no/uncertain). Build a Python script at `outputs/skill-quantized-vs-api-cost.md` that outputs a markdown table comparing three deployment options: (a) GPT-4o-mini API at $0.15/1M input tokens with an average 200-token prompt, (b) a locally quantized INT4 model on a rented GPU instance ($0.75/hr A10G spot, 20ms per inference, amortized over 24/7 uptime), and (c) the same local model in INT8 on an A100 80GB ($2.50/hr, 12ms per inference). Compute monthly cost for each, cost-per-1K-inferences, and the break-even volume where local becomes cheaper than the API. Print the full markdown table to the terminal and write it to the artifact path.

6. **Build a PTQ-vs-QAT comparison report that produces a persistent artifact.** Write a Python module at `handlers/quant_compare.py` that does the following: loads the MNIST or CIFAR-10 dataset, trains a small CNN for 1 epoch in FP32 as a baseline, then applies dynamic post-training quantization via `torch.quantization.quantize_dynamic` and measures test accuracy. Document (as a printed summary and in a docstring) what additional steps QAT would require (fake-quantize modules inserted, observer calibration during forward passes, the `torch.quantization.prepare_qat` call) and why QAT typically recovers 1-3% accuracy over PTQ. Print baseline accuracy, PTQ accuracy, model size for each, and the accuracy delta. Note in comments which approach you would choose for a production GTM classifier retrained weekly on fresh CRM data.
