# Exercises — Evaluation: Benchmarks, Evals, LM Harness

## Exercises

1. **Implement the three scoring metrics from scratch.** Write three functions — `exact_match(predictions, references)`, `token_f1(predictions, references)`, and `bleu_score(predictions, references)` — that each take a list of prediction strings and reference strings and return a float score between 0 and 1. For BLEU, compute the 4-gram precision with brevity penalty. Run all three on this input pair and print the results:

   ```python
   predictions = ["Schedule a demo call for next Tuesday at 2pm"]
   references  = ["Can we schedule a demo call next Tuesday at 2:00pm?"]
   ```

   Your terminal should print three distinct float values. Confirm that `exact_match` returns `0.0` (the strings differ), `token_f1` returns something between `0.3` and `0.7` (partial overlap), and `bleu_score` returns a low but nonzero value.

2. **Apply your metrics to a benchmark-style comparison.** Create a JSON file containing 10 question-answer pairs (you can invent them — e.g., trivia-style questions with short factual answers). Write a script that generates "model outputs" by applying deterministic string mutations to the reference answers (e.g., dropping punctuation, swapping synonyms, truncating). Score the mutated outputs using all three metrics from Exercise 1. Print a formatted table showing per-example scores and aggregate averages. Verify that exact match is the harshest metric and BLEU is the most lenient on your dataset.

3. **Implement per-class F1 scoring for a multi-label GTM intent classifier.** You are given a list of inbound support emails, each labeled with one or more intents from the set `{billing, technical, sales, churn_risk, onboarding}`. Build a dataset of 20 examples (you may write these yourself or use a simple template). Write a classifier that assigns intent labels using keyword matching (e.g., "invoice" → billing, "cancel" → churn_risk). Compute per-class precision, recall, and F1, then print a table showing all classes even if F1 is 0 for some. Your terminal output must show which intent class has the worst F1 score.

4. **Configure a regression testing pipeline that compares two model versions.** Create a dataset of 15 GTM-style examples (e.g., company name extraction from raw LinkedIn bios, or lead-scoring rationale generation). Produce two sets of outputs: one labeled `v1` (lower quality — apply aggressive truncation) and one labeled `v2` (higher quality — apply only minor punctuation changes to the reference). Write a script that runs both versions through your token F1 scorer and prints a side-by-side comparison table showing per-example delta and overall average delta. Confirm that `v2` outperforms `v1` on aggregate. Print a clear `REGRESSION DETECTED` or `NO REGRESSION` verdict based on whether `v2_avg >= v1_avg`.

5. **Build a failure-tracing harness that produces a per-example score breakdown.** Combine everything from the previous exercises into a single Python script that: (a) loads an evaluation dataset from a JSON file, (b) runs predictions through a callable (you may use a mock callable that applies string transformations), (c) scores each example with exact match, token F1, and BLEU, (d) identifies the bottom 3 examples by token F1, and (e) writes a Markdown report showing each failing example's input, prediction, reference, all three scores, and a brief failure-hypothesis string (e.g., "numeric formatting mismatch," "truncated output"). The report must be written to `outputs/skill-eval-harness-report.md`. Run the script and confirm the file exists and contains at least 3 failure breakdown entries.

6. **Design a complete evaluation harness for a Clay-enriched lead-routing pipeline.** Build a script that loads a JSON dataset of 25 enriched company records (include fields like `company_name`, `industry`, `employee_count`, `tech_stack`, and a human-assigned `routing_label`). Define a prompt template that asks an LLM (or a mock callable) to classify each lead into one of `{enterprise, mid_market, smb, unsupported}`. Run the classifier over the full dataset, compute per-class F1 and overall accuracy, and compare your task-specific eval score against the model's published MMLU or benchmark score (hardcode the benchmark number as a constant). Print a summary table showing the benchmark score, your eval score, and the gap between them. Write the full results — including per-example predictions, scores, and a section analyzing where domain-specific reliability diverges from general capability — to `outputs/skill-clay-routing-eval.md`. Confirm the Markdown file is created and contains both the numeric comparison and at least one paragraph of divergence analysis.
