# Exercises — The Direct Preference Optimization Family

## Exercises

1. **Implement the DPO loss function from scratch in PyTorch.** Define `dpo_loss(policy_chosen_logps, policy_rejected_logps, ref_chosen_logps, ref_rejected_logps, beta=0.1)` that computes `−log σ(β · [log(π(y_w)/π_ref(y_w)) − log(π(y_l)/π_ref(y_l))])`. Create a toy batch of 8 preference pairs using random log-probabilities. Print the loss value, then manually increase the policy's chosen log-probs by 0.5 and reprint — confirm the loss decreases.

2. **Implement IPO and KTO losses alongside your DPO function and compare all three on the same toy batch.** IPO replaces the logistic with a squared hinge: `(log_ratio_diff − 1/2)²`. KTO uses binary good/bad labels with a KL penalty instead of pairing. Generate a toy batch where one "chosen" response has only marginally higher log-prob than the rejected one. Print DPO, IPO, and KTO loss values side-by-side and state in a comment which loss is most sensitive to the small margin and why.

3. **Fine-tune a small language model with `DPOTrainer`.** Load `trl`'s DPOTrainer with a small model (e.g., `HuggingFaceTB/SmolLM-135M`) and the `trl-lib/ultrafeedback_binarized` dataset (or any HF preference dataset). Configure `beta=0.1`, train for 50 steps, and log the reward margins (`rewards/accuracies` and `rewards/margins`) at every logging interval. Print the final reward margin and accuracy. The reward margin should increase over training steps.

4. **Configure `ORPOTrainer` for the same model and dataset and compare training dynamics against your DPO run from Exercise 3.** ORPO fuses SFT and preference optimization into one stage with no reference model — set `beta=0.1` (the odds-ratio λ). Train for the same number of steps. Print the logged metrics for both runs (DPO vs ORPO) in a comparison table showing: final loss, reward/accuracy, and training time. State which method converges faster and which produces a larger reward margin.

5. **Build a GTM outreach preference dataset from CRM outcome data and fine-tune with the appropriate DPO-family method.** Pull email engagement data from your CRM or Apollo export — each record should contain the email body, send context (sequence step, persona), and outcome (replied vs. no-reply, or meeting booked vs. not). Convert this into preference pairs: for matched segments (same ICP, same sequence step), pair a replied email as `chosen` and a no-reply email as `rejected`. If you cannot construct reliable pairs, use KTO with binary labels instead. Write the dataset conversion logic, load it into `DPOTrainer` or `KTOTrainer`, fine-tune a small model, and log reward margins. Save your dataset builder and training script to `handlers/gtm_preference_trainer.py`. Print the dataset size, pair count (or label distribution for KTO), and the final reward margin.
