# Exercises — Data Provenance and Training-Data Governance

## Exercises

1. **Build a hash-based provenance tracker.** Implement a Python script that accepts JSON records via stdin (one per line), computes a SHA-256 content hash for each record, and appends a provenance entry to a JSONL log file at `outputs/provenance.jsonl`. Each entry must contain four fields: `source` (read from an environment variable `PROV_SOURCE`), `timestamp` (ISO 8601 UTC), `content_hash`, and `record_id` (extracted from the record's `id` field). Feed the script five sample records and verify the log contains five entries with distinct hashes. Print the log to stdout as a final step.

2. **Implement a schema validator for field-level drift.** Write a Python program that loads a JSON schema definition (field names + expected types) from a file path given as a command-line argument, then reads records from stdin. For each record, validate that every required field is present and that each value matches the declared type. Output a drift report to stdout: for each rejected record, print the record ID, the offending field, the expected type, and the actual type. Exit with code 1 if any record fails validation, 0 otherwise. Test with a schema that expects `{"email": str, "company": str, "revenue": int}` and feed it three records — one valid, one missing a field, one with `revenue` as a string.

3. **Detect dataset contamination with MinHash signatures.** Implement MinHash from scratch (no `datasketch` library): generate 128 hash functions using `hashlib.md5` with salted prefixes, compute MinHash signatures for two sets of text records loaded from JSONL files passed as command-line arguments, and estimate Jaccard similarity between the two sets. Run the program against a training file and an evaluation file. Print the estimated overlap percentage and a list of any records whose estimated membership probability exceeds 0.5, flagging them as contamination candidates.

4. **Construct a lineage DAG and answer traversal queries.** Build a Python program that maintains an in-memory lineage graph where nodes are named dataset versions (e.g., `raw_leads_v1`, `deduped_v2`, `enriched_v1`) and edges are transformation records containing `operation`, `timestamp`, and `commit_hash`. Load the graph from a JSON file given as a command-line argument. Implement a `trace-upstream` subcommand that takes a node name and prints the full chain of ancestors from source to that node, one transform per line. Implement a `trace-downstream` subcommand that prints every artifact that depends on a given source node. Test with a Clay enrichment pipeline: `apollo_export` → `clay_enrich` → `crm_sync` → `model_features_v3`, and confirm `trace-upstream model_features_v3` prints all three transforms in order.

5. **Map California AB 2013's twelve transparency fields to a machine-readable dataset summary.** Research the twelve fields mandated by AB 2013 (e.g., data source, collection method, intended use, data volume, demographic breakdowns, opt-out mechanism, retention period, and others). Write a Python program that takes a dataset name and a metadata JSON file as arguments, validates that all twelve fields are present and non-empty, and emits a formatted Markdown summary. Save the artifact to `outputs/skill-dataset-summary.md`. The program must exit non-zero with a clear error listing missing fields if any of the twelve are absent. Run it against a sample CRM enrichment dataset and confirm the Markdown renders all twelve sections.

6. **Build an end-to-end provenance governance handler.** Implement a Python module that combines all four mechanisms into a single ingest pipeline stage. The handler must: (a) compute and log a provenance hash for each incoming record, (b) validate records against a configurable schema before they proceed, (c) record the operation as a node and edge in a persistent lineage DAG stored as JSON, and (d) emit an AB 2013 compliance summary stub for the resulting dataset version. Accept records from a real GTM export — fetch 20 company records from the Apollo API (or use a local Apollo export CSV if no API key is available) and run them through the handler. Save the module to `handlers/provenance_governance.py` and run it as a script, printing: the count of records that passed validation, the lineage path of the output artifact, and the compliance summary path.
