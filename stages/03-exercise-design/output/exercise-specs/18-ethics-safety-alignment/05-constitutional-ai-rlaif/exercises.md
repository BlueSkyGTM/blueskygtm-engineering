# Exercises — Constitutional AI and RLAIF

## Exercises

1. **Implement a constitutional self-critique loop for a single prompt.** Define a constitution as a Python list of at least four principle strings (e.g., "Do not make unverifiable claims," "Acknowledge uncertainty when relevant," "Prefer specific, actionable guidance over generic statements," "Maintain a neutral, non-salesy tone"). Write a script that makes three sequential LLM calls: (a) generate a response to a prompt of your choice, (b) critique that response against each principle, and (c) revise the response based on the critique. Print all three outputs to the terminal with clear section headers: `=== ORIGINAL ===`, `=== CRITIQUE ===`, `=== REVISED ===`.

2. **Compare RLAIF against RLHF on four dimensions.** Write a Python script that prints a formatted comparison table (using plain string formatting or a library like `tabulate`) with rows for feedback source, marginal cost per labeled example, scalability bottleneck, and dominant failure mode. Fill each cell with a one-phrase summary for RLAIF and RLHF. Run the script and confirm the table renders correctly in your terminal. Then add a fifth row labeled "Best suited for" and complete it for both approaches.

3. **Construct a preference dataset from AI-generated critiques and revisions.** Build a script that takes a list of five prompts and, for each one, generates two candidate responses (you may use different temperature settings or different system prompts to produce variation). Send both responses to an AI judge prompt that includes a constitution of your choice, and ask it to output a JSON object: `{"preferred": "A" or "B", "reason": "...", "principle_cited": "..."}`. Save all five JSON judgments to `outputs/preference_pairs.jsonl` (one JSON object per line, including the prompt and both responses). Print a summary line to the terminal showing how many times A was preferred vs. B.

4. **Compute constitutional evaluation metrics across a batch.** Extend or reuse your self-critique loop to process a batch of ten prompts. For each prompt, count the number of principles the critique identifies as violated, and compute the character-length delta between the original and revised response. After processing all ten, print a table with columns: prompt index, violation count, original length, revised length, and revision delta (revised − original). At the bottom, print the mean violation count and the mean revision delta across the batch. Confirm the script runs end-to-end and produces numeric output you can inspect.

5. **Build a constitutional review handler for GTM cold outbound emails.** Define a constitution tailored to outbound sales quality — for example, "Do not claim ROI percentages without a cited source," "Reference a specific trigger event or signal, not a generic pain point," "Limit to one CTA," "Do not use manipulative urgency language." Write a handler that ingests a batch of email drafts from a CSV file (columns: `company`, `contact`, `email_body`), runs each draft through the critique-and-revise loop, and writes results to `outputs/reviewed_emails.csv` with columns: `company`, `original_email`, `critique`, `revised_email`, `violations_found`. If you have access to enrichment data from Clay or Apollo, use real prospect rows as input; otherwise generate ten synthetic prospects. The CSV artifact must persist after the script exits.

6. **Diagnose sycophancy and reward-hacking failure modes in AI-generated feedback.** Design a deliberately flawed constitution that omits guardrails against sycophancy (e.g., no principle like "Do not flatter the recipient") and includes a principle that is easy to game (e.g., "Always be as helpful as possible" with no bound on claim specificity). Run your preference-judging pipeline from Exercise 3 using this flawed constitution on a set of five prompts designed to elicit each failure mode — for instance, prompts where a sycophantic or over-claiming response would be tempting. Collect the AI judge's preferences and write a short analysis to `outputs/skill-rlaif-failure-diagnosis.md` that includes: (a) at least two concrete examples where the judge preferred the sycophantic or hacked response, (b) the quoted constitution principle that allowed it, and (c) a proposed revised principle that would close the gap. The markdown artifact must include verbatim model outputs as evidence.
