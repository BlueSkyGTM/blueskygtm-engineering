# Exercises — Scalable Oversight and Weak-to-Strong Generalization

## Exercises

1. **Compute PGR from fixed scores.** Given a weak supervisor at 45% accuracy, a human ceiling at 92%, and a strong model fine-tuned on weak labels at 71%, implement a script that computes the Performance Gap Recovered using PGR = (strong − weak) / (ceiling − weak). Print the raw PGR value and a boolean indicating whether the strong model surpassed the weak ceiling. Verify: PGR ≈ 0.491 and the boolean is `True`.

2. **Map oversight mechanisms to failure modes.** Create a Python script that defines a dictionary mapping each of the three mechanisms (debate, recursive_reward_modeling, task_decomposition) to two fields: its decomposition strategy and its primary failure mode. Iterate over the dictionary and print each mechanism as a formatted table row with both fields. Run the script and confirm all three mechanisms print with non-empty strategy and failure mode strings.

3. **Implement a weak-to-strong generalization test.** Load 50 product records from `signals/data/products.jsonl` — each record has a `weak_label` produced by a keyword classifier, a `gold_label` from human review, and a `text` field. Train a simple classifier (TF-IDF + logistic regression or equivalent) using only `weak_label` as the target, then evaluate it against `gold_label`. Print weak supervisor accuracy, strong model accuracy, computed PGR, and a classification label: `"generalizes"` if PGR > 0.5, `"collapses"` if PGR < 0.1, otherwise `"partial"`.

4. **Apply task decomposition to Apollo enrichment validation.** Load 20 company records from `signals/data/apollo_companies.jsonl`. Implement an oversight pipeline that decomposes the question "Is this enriched record trustworthy?" into three independent sub-checks: domain resolves via DNS lookup, employee count is within a plausible range (10–50000), and industry tag matches the company's website meta description or title keywords. Aggregate the three boolean sub-checks into a single pass/fail verdict per record. Print a table of company name, each sub-check result, and final verdict. At least one record must fail at least one sub-check.

5. **Build a production quality gate for GTM email generation.** Implement `handlers/wtm_quality_gate.py` that accepts generated cold-email drafts as input, runs a weak supervisor (rule-based checks: spam-word list, length bounds, personalization token presence) and a strong-model judge (via API or local model call) that scores each email 0–1 on relevance and tone. Only emails where the strong score exceeds the weak score by a configurable margin (default 0.15) pass the gate. Write a Markdown report to `outputs/skill-quality-gate-report.md` containing pass rate, mean weak score, mean strong score, and a table of failed emails with reasons. The handler must be CLI-callable: `python -m handlers.wtm_quality_gate --input emails.jsonl --margin 0.15` and must produce non-empty terminal output confirming the report path.

6. **Trace supervision signal degradation across a multi-model pipeline.** Design and implement a diagnostic script that simulates a four-stage GTM pipeline: lead scoring → email drafting → follow-up scheduling → CRM sync. For each stage, run a weak supervisor judgment and compare it against a gold label to compute an agreement delta (1 − |supervisor − gold|). Print a table showing each stage's agreement delta and assign the oversight mechanism (debate, recursive reward modeling, or task decomposition) that best addresses the degradation at that stage, with a one-sentence justification. Write the full analysis — table, assignments, and the highest-priority intervention target — to `outputs/skill-oversight-trace.md`. Verify the file exists and is non-empty by printing its line count to the terminal.
