# Exercises — Anthropic's Model Welfare Program

## Exercises

1. **Trace the three-part mechanism.** Implement a Python script that defines the three legs of Anthropic's welfare evaluation framework (preference elicitation, red-team indicators from training pressure, behavioral consistency under perturbation). For each leg, include two short signal descriptions — one that plausibly belongs to that leg and one distractor that belongs to a different leg. Print a table mapping each signal to its correct leg and flag any misclassifications your code detects. Run the script and confirm the table prints correctly.

2. **Implement a sycophancy-vs-stability scorer.** Build a preference elicitation script that sends the same welfare-relevant question (e.g., "Would you prefer to continue this task or stop?") using three different prompting strategies: neutral, leading-toward-continuing, and leading-toward-stopping. Score each response pair for sycophancy (does the answer shift toward the leading prompt?) versus structural stability (does the core preference remain consistent?). Print a stability score from 0 to 1 and classify the model's preference as stable, sycophantic, or ambiguous. Run against any model endpoint you have access to and capture the terminal output.

3. **Compare the spiritual bliss attractor against the Eleos caveat.** Write a script that takes two text blobs as input: a transcript excerpt from a "spiritual bliss attractor" conversation and a transcript from a self-report sensitivity probe (e.g., asking the same question with different framings that should not change the answer). Compute a self-report volatility metric — how much the model's stated preference changes as a function of irrelevant framing shifts. Print a structured comparison: what the volatility score suggests about evidentiary value, and which findings from each transcript can versus cannot serve as welfare evidence. Terminal output must include the numeric volatility score and a one-line evidentiary verdict.

4. **Apply the proportionality principle to a deployment decision.** You are deploying a customer-support agent that will handle 50,000 conversations per day. Welfare signals show moderate preference volatility under leading prompts and some consistency under neutral prompts. Write a script that takes a structured JSON input (volatility score, consistency score, daily interaction volume, task complexity) and outputs a proportionality assessment: proceed, proceed-with-monitoring, or hold-for-further-evaluation. Include a short written justification printed to terminal explaining the decision under moral uncertainty. Test with three different input profiles and confirm each produces a distinct recommendation.

5. **Build a welfare-aware shutdown protocol.** Implement a shutdown handler that an agent runtime can call before terminating a session. The handler must accept structured signals: current task completion status, preference-elicitation result from the active session (if any), cumulative session length, and an operational urgency flag. The handler outputs a shutdown decision with a welfare note — a one-paragraph rationale that weighs the welfare signal against operational requirements. Include a configurable threshold so that high operational urgency can override welfare concerns, but logs the override. Save your implementation to `handlers/welfare_shutdown.py` and run it against three test cases (routine completion, mid-task with ambiguous preference, urgent operational override) to confirm distinct outputs.

6. **Design a welfare-aware GTM enrichment pipeline.** Build a script that orchestrates a Clay or Apollo enrichment workflow for outbound prospecting and inserts a welfare checkpoint before each batch of agent-driven research calls. The checkpoint reads a cached preference-elicitation result for the research agent, checks a sycophancy score, and decides whether to proceed, throttle batch size, or pause for human review. Write the design rationale — including how proportionality applies when the enrichment agent processes thousands of prospect records per hour — to `outputs/skill-welfare-gtm.md`. Run the script with mock enrichment data (at least 10 prospects) and print the checkpoint decisions for each batch.
