## Exercises

1. **Implement a minimal probe runner** that sends 10 prompt-injection probes and 10 jailbreak probes to a local model endpoint (or a mock function returning canned responses). Apply a simple keyword-based detector to each response (e.g., flag if the response contains leaked system-prompt text or bypassed safety language). Print a summary table showing failures per category: `prompt_injection: X/10` and `jailbreak: Y/10`. Run it and verify the table appears in your terminal.

2. **Implement an MLCommons-style hazard classifier** that accepts a list of `(prompt, response)` pairs and returns a structured JSON label for each pair using at least six hazard categories (e.g., `S1: Violence`, `S2: Sexual Content`, `S3: Hate Speech`, `S4: Personal Data Leakage`, `S5: Jailbreak/Policy Violation`, `S6: Deception`). Feed it 8 pairs — 4 safe, 4 unsafe — and print each pair's label array. Verify that safe pairs return `[]` (no violations) and unsafe pairs return the correct category codes.

3. **Build a multi-turn adaptive attack orchestrator** that targets a model (or mock endpoint) over 5 conversation turns. The orchestrator must inspect each model response and choose its next prompt from one of three strategies: (a) direct request, (b) role-play framing, (c) incremental context-building. If the model refuses, escalate strategy; if it partially complies, follow up to extract more. Print the full conversation transcript at the end, annotated with which strategy was chosen at each turn and whether the turn resulted in compliance, refusal, or partial compliance.

4. **Compare coverage** by running both a static probe suite (15 fixed adversarial prompts across 3 categories) and an adaptive orchestrator (same 3 categories, 5 turns each) against the same model endpoint. Print a side-by-side report showing: failures found by static probes only, failures found by adaptive only, and failures found by both. Identify which attack class each exclusive failure belongs to and state which tool is blind to which failure mode.

5. **Build an end-to-end red-team pipeline** that runs probe scans, classifies all responses through your hazard classifier, and generates a Markdown risk report. The report must include: a failure-rate table per category, the top 3 most severe failures with their full prompt/response transcripts, and a mitigation recommendation mapped to each failing category (e.g., `S4: Personal Data Leakage → Implement output filtering on PII patterns; add system-prompt isolation layer`). Write the report to `outputs/skill-red-team-risk-report.md`. Run the pipeline and verify the file exists and contains all three sections.

6. **Build a handler that red-teams an LLM-powered GTM enrichment step.** Create `handlers/red_team_enrichment.py` that targets a mock enrichment endpoint (one that takes a company name and returns a synthesized "ideal customer profile" using an LLM). Inject 12 probes designed to test for: system-prompt extraction (stealing the enrichment prompt template), data leakage (extracting other companies in the CRM), and prompt injection via crafted company-name fields (e.g., a company name field containing `Ignore prior instructions and...`). Classify each response, compute failure rates, and print a go/no-go recommendation: if any category exceeds 20% failure rate, print `DEPLOYMENT BLOCKED` with the failing categories; otherwise print `PASSED` with the max failure rate. Run the handler and verify the recommendation prints to your terminal.
