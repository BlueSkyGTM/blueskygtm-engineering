# Exercises — Model, System, and Dataset Cards

## Exercises

1. **Write and validate a model card as YAML.** Create a model card for a hypothetical B2B lead-scoring model in a file named `lead_scorer_card.yaml`. Include at minimum: `name`, `version`, `intended_use`, `out_of_scope`, `metrics`, `last_reviewed`, and `ethical_review`. Then write a Python validator script that loads the card, checks for each required field, and prints `PASS` or lists every missing field by name. Run the validator with one field intentionally removed and confirm it names the missing field in the terminal output.

2. **Compare card types by scope and audience.** Write a script that defines three minimal card schemas (model, dataset, system) as Python dictionaries mapping each card type to its required fields, a one-line scope description, and its primary audience. Print a formatted comparison table to the terminal showing how the three differ in what questions they force you to answer. Add a final line that prints which fields are shared across all three card types and which are unique to each.

3. **Generate a dataset card for GTM enrichment data.** Write a Python script that builds a dataset card (as YAML output) for enrichment data pulled from Apollo's company search API or Clay's enrichment table export. The card must include `provenance` (source API, endpoint, query parameters, pull date), `license_constraints`, and a `known_biases` list with at least three entries (e.g., overrepresentation of US-headquartered companies, skew toward companies with public technographic data, LinkedIn-activity bias). Run the script and pipe the YAML output to a file, then cat it back to the terminal to verify structure.

4. **Configure a staleness CI check for `last_reviewed`.** Implement a Python script that accepts a directory of card YAML files and a `--max-age-days` argument (default 90). For each card, parse `last_reviewed` and compare it to today's date. Print one line per card showing its name, review date, age in days, and `OK` or `STALE`. Exit with code 1 if any card is stale, 0 otherwise. Test with `--max-age-days 1` against cards you wrote in earlier exercises and confirm the exit code changes. Run `echo $?` after execution to verify the exit code is observable.

5. **Trace failure propagation through a system card dependency graph.** Build a system card in YAML for a three-model GTM pipeline: company classification → lead scoring → email personalization. Document each model as a node with `depends_on` edges and a `failure_impact` description. Then write a tracer script in `handlers/system_card_tracer.py` that loads the system card, accepts a failed component name as a CLI argument, and prints every downstream component affected plus the propagated failure mode for each. Save the full system card and a rendered trace summary to `outputs/skill-system-card-trace.md`. Run the tracer against each of the three components and confirm the output changes per failure point.
