# Exercises — Real-Time Audio Processing

## Exercises

1. **Compute and verify per-chunk processing latency.** Write a Python script that loads a WAV file, splits it into 30 ms chunks, and measures the wall-clock time to process each chunk (reading bytes into a NumPy array and computing `np.abs()` on the samples). Print each chunk's latency in milliseconds alongside the chunk duration, and print `PASS` or `FAIL` depending on whether latency stays below chunk duration. Run it on any WAV file you generate with `python -c "import wave, struct, math; w=wave.open('test.wav','w'); w.setnchannels(1); w.setsampwidth(2); w.setframerate(16000); w.writeframes(struct.pack('<'+'h'*48000,*[int(10000*math.sin(2*math.pi*440*i/16000)) for i in range(48000)])); w.close()"` and confirm every chunk prints `PASS`.

2. **Implement an energy-based VAD gate.** Build a function that accepts a 30 ms mono audio frame (NumPy int16 array) and a threshold in dBFS, computes the root-mean-square energy of the frame, converts it to dBFS, and returns `"speech"` or `"silence"`. Write a second function that reads a WAV file in 30 ms increments, runs the gate on each frame, and prints a line per frame: `frame 0: speech (RMS=-23.1 dBFS)`. Generate a test file with 1 second of 440 Hz tone followed by 1 second of silence and confirm the first ~33 frames read `speech` and the remaining ~33 read `silence`.

3. **Extract dominant frequencies via FFT from streaming chunks.** Write a script that reads a WAV file in 30 ms chunks, applies `numpy.fft.rfft` to each chunk, identifies the top three frequency peaks by magnitude, and prints them as: `chunk 0: 440.0 Hz, 880.0 Hz, 1320.0 Hz`. Generate test audio containing a known mixture (e.g., 300 Hz + 800 Hz tones) and verify the printed peaks include both frequencies within ±20 Hz tolerance.

4. **Build a real-time streaming CLI tool with VAD.** Create a command-line program that accepts a WAV file path and an output path, streams the audio at real-time speed (using `time.sleep` keyed to chunk duration), applies the energy-based VAD from Exercise 2 to each frame, and writes contiguous runs of `speech` frames as separate WAV segments to disk. The program should print a progress line per detected segment: `[00:01.23 – 00:03.45] speech segment written → segment_002.wav`. Test it on the 1-second-tone-then-silence file and confirm it produces one segment file for the speech portion.

5. **Compute talk-to-listen ratio and detect monologues from a sales call.** Download a stereo call recording where channel 0 is the rep and channel 1 is the prospect (you can use a synthetic one: generate two channels of alternating tone bursts and silence). Write a script that runs VAD per channel, computes the talk-to-listen ratio (rep talk time ÷ prospect talk time), flags any contiguous rep speech exceeding 60 seconds as a monologue, and emits a JSON report to `outputs/skill-call-analysis.md` containing the ratio, total talk time per speaker, and a list of monologue segments with timestamps. Print a summary table to the terminal. This mirrors the core analysis pipeline that tools like Gong or Chorus run on recorded sales calls to coach reps on talk ratio.

6. **Design and implement a real-time call-quality monitor.** Extend the CLI tool from Exercise 4 into a live monitor that ingests a stereo audio stream, applies VAD to both channels independently, computes a rolling 30-second talk-to-listen ratio, and emits a desktop notification or terminal beep whenever the ratio exceeds 70/30 (rep monologue risk) or drops below 30/70 (rep is too quiet). The monitor must process each chunk fast enough to keep up with real-time playback — include the latency assertion from Exercise 1 as a runtime guard that prints `LATENCY WARNING` if any chunk exceeds the budget. Write the full implementation to `handlers/call_monitor.py`, and produce a usage guide at `outputs/skill-call-monitor.md` documenting CLI flags, expected output format, and how to connect it to a live call source (virtual audio cable or file stream).
