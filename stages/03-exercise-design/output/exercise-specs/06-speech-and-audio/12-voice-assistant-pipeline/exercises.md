## Exercises

1. **Implement a minimal end-to-end voice assistant.** Write a single Python script that records 5 seconds of microphone audio (16kHz mono), sends it to your STT provider, passes the transcript to an LLM with a fixed system prompt, and plays back the LLM response through TTS. Run the script, speak a short sentence, and confirm the assistant responds with synthesized audio. Print the final transcript and the LLM response text to the terminal so you can verify both stages fired.

2. **Add energy-based voice activity detection to gate the pipeline.** Replace the fixed 5-second recording window with a continuous mic loop that computes RMS energy per frame and only triggers STT when energy exceeds a threshold (e.g., 0.02). When energy drops below the threshold for 1.5 seconds, treat the utterance as complete. Run the script, speak a phrase, pause, and confirm the terminal prints "VAD: speech detected" followed by the transcript and response — without any response when you stay silent.

3. **Compute and trace per-stage latency across the full pipeline.** Instrument your assistant from Exercises 1–2 with `time.perf_counter()` timestamps at five checkpoints: audio capture end, STT complete, LLM first token, LLM complete, and TTS complete. Print a latency breakdown table showing milliseconds for each stage (capture→STT, STT→LLM-first-token, LLM-stream-duration, LLM-complete→TTS-complete). Define a 2-second total latency budget and print `BUDGET: PASS` or `BUDGET: FAIL` based on time-to-first-audio. Run the script three times and paste all three tables into your terminal output.

4. **Compare batch versus streaming processing at the STT→LLM seam.** Implement two modes in the same script: (a) batch mode — wait for VAD speech-end, send the full audio to STT, then send the full transcript to the LLM; (b) streaming mode — send audio chunks to a streaming STT endpoint and forward partial transcripts to the LLM as they arrive. Run both modes on the same spoken input. Print a side-by-side comparison of time-to-first-LLM-token and total transcript accuracy (word count) for each mode. State which mode wins on latency and which wins on accuracy in a printed summary line.

5. **Build a GTM sales-qualifying voice agent with failure-mode guards.** Extend your assistant into a sales qualification bot that asks discovery questions (company size, current toolstack, budget timeline) and logs structured notes. Pull a list of 5 company names and domains from a Clay or Apollo export CSV (or use a mock enrichment payload). When the user speaks a company name, enrich it via an API (Clay, Apollo, or Clearbit) and weave the firmographic data into your next response. Implement three guards: (a) suppress LLM responses when STT returns empty or whitespace-only transcripts (silence hallucination), (b) detect user interruption (new VAD speech event while TTS is playing) and cancel the current TTS playback, (c) prepend a non-instructable system preamble that resists prompt injection from transcribed user speech. Save the full conversation log with enrichment data and qualification scores to `outputs/voice-gtm-session.json`. Print a summary table showing each company, its qualification score, and the per-call latency breakdown.

6. **Design and benchmark a streaming-first architecture that minimizes time-to-first-audio.** Architect a pipeline where VAD triggers streaming STT, partial transcripts stream token-by-token into the LLM, and LLM output tokens stream into chunked TTS — all connected via `asyncio` queues with bounded buffer sizes. Write a design document (max 400 words) specifying buffer sizes, backpressure strategy, and cancellation propagation at each seam. Implement the design and benchmark it against your Exercise 5 batch pipeline on the same 5-company qualification script. Save the design document and benchmark results to `outputs/voice-streaming-design.md` and print a comparison table showing time-to-first-audio, total session time, and word-error-rate for both architectures.
