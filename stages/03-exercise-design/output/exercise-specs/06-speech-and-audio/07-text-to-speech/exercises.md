## Exercises

1. **Generate and verify a mel spectrogram.** Using a TTS library of your choice (Coqui TTS, ESPnet, or Hugging Face `transformers`), synthesize a mel spectrogram from the text `"The quick brown fox jumps over the lazy dog."` Print the mel tensor's shape, the sample rate, and the hop length the model expects. Compute the expected number of mel frames analytically from the audio duration and hop length, and assert that it matches the actual tensor's frame dimension. Play the resynthesized audio back and confirm you hear the sentence.

2. **Quantify speaker-embedding difference with cosine similarity.** Generate mel spectrograms for the same sentence using two distinct speaker embeddings (or two different `speaker_wav` reference clips if using a zero-shot model). Flatten each mel spectrogram and compute cosine similarity between the two vectors. Repeat for a second sentence and print both similarity scores. State whether the inter-speaker difference is consistent across utterances and by how much.

3. **Implement a minimal duration predictor.** Using `g2p_en` or `phonemizer`, convert a sentence to its phoneme sequence. Then use `aeneas` or `Montreal Forced Aligner` to obtain forced-alignment boundaries between phonemes and a reference audio clip. Implement a function that reads the alignment output and maps each phoneme to its mel-frame count (duration in frames = round(boundary_end_sec − boundary_start_sec) × (sample_rate / hop_length)). Print the phoneme-to-frame table and verify that the frame counts sum to the total mel-frame count of the reference audio.

4. **Trace architectural latency differences.** Run the same input text through two TTS models from different generations — for example, an autoregressive model (Tacotron 2 via `espnet` or a compatible checkpoint) and a non-autoregressive or flow-matching model (VITS or F5-TTS). Measure wall-clock inference time for each using `time.perf_counter()`, print the real-time factor (audio_seconds / inference_seconds) for both, and write a two-paragraph summary explaining where the speedup comes from architecturally.

5. **Build a personalized voicemail pipeline for outbound prospecting.** Pull 5 prospect records from Apollo's People Search API (or export a CSV from Clay with name, company, and a personalization field). Write a script that templates a short voicemail script per prospect (e.g., `"Hi {first_name}, this is {rep_name} from {company}. I saw your team is scaling {topic}…"`) and synthesizes an audio file for each using a zero-shot TTS model with your own cloned voice from a 10-second reference clip. Save the generation logic as a reusable module at `handlers/voicemail_generator.py` and write a summary of latency, cost per voicemail, and quality observations to `outputs/skill-tts-voicemail.md`.

6. **Benchmark vocoder tradeoffs and defend a production choice.** Select three vocoders representing different generations (e.g., WaveNet, HiFi-GAN, BigVGAN). For a fixed mel spectrogram, synthesize waveforms with each and record: inference time, audio length, mean opinion score estimated via a neural audio quality metric (e.g., NISQA or DNSMOS), and file size. Output a comparison table to the terminal and write a one-page recommendation document to `outputs/skill-vocoder-benchmark.md` arguing which vocoder a 50-rep SDR team should use for real-time personalized voicemail drops, justifying the choice on speed, quality, and infrastructural cost grounds.
