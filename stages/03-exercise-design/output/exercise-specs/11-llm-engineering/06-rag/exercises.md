# Exercises — RAG (Retrieval-Augmented Generation)

## Exercises

1. **Implement cosine similarity retrieval from a pre-built embedding index.** Write a Python script that loads a JSON file of pre-computed chunk embeddings (each chunk is a dict with `text` and `vector` keys), accepts a query string, embeds it using the same model, computes cosine similarity against every chunk, and prints the top-5 chunks ranked by similarity score. Verify your output by confirming that the query "how does data enrichment work?" surfaces the chunk containing "Clay waterfall enriches leads by sequencing data providers" in position 1 or 2.

2. **Compare fixed-size chunking against semantic chunking on the same source document.** Take a 3-paragraph product description (write your own or use one from the lesson) and implement two chunking functions: one that splits every 100 characters with a 20-character overlap, and one that splits on sentence boundaries. Print the resulting chunk list from each strategy side-by-side. Count how many chunks each produces and identify at least one case where fixed-size chunking breaks a sentence mid-thought.

3. **Build a complete RAG pipeline from scratch with no pre-built index.** Load any 5 text files from a directory of your choice (markdown docs, plain text notes, etc.), chunk them using semantic boundaries, embed all chunks using a sentence-transformer model, store the embeddings in a JSON file as your vector index, then accept a user query, retrieve the top-3 chunks, and print them as assembled context. Confirm the pipeline works by querying something present in your documents and verifying the retrieved chunks are semantically relevant.

4. **Trace and fix a retrieval failure.** Create a scenario with a deliberate bug: embed 10 chunks using one embedding model, then query using a different embedding model (different dimensionality or different provider). Run your retrieval and observe that the results are garbage or the script errors out. Add diagnostic logging that prints the embedding dimensions for both the stored vectors and the query vector, then fix the pipeline by ensuring the same embedder is used for both indexing and querying. Print the before-and-after top-3 results to confirm the fix.

5. **Build a GTM research RAG agent that retrieves internal playbooks and competitive intel before generating account briefs.** Create a set of 5–10 mock playbook documents covering topics like outbound sequencing, ICP definition, competitive positioning against a rival, pricing tiers, and enrichment workflows. Index them with your RAG pipeline. Then implement an agent function that takes a company name and a research question (e.g., "What playbook should we apply for Acme Corp, a mid-market SaaS company?"), retrieves the relevant chunks, and generates a grounded account brief using an LLM call. For real GTM tooling integration, pull firmographic data on the target company from the Apollo API (or a mock Apollo response if no API key) and include it in the brief alongside the retrieved context. Save the agent as `handlers/gtm_rag_agent.py` and produce at least one sample brief at `outputs/account-brief-acme.md`.

6. **Design and implement a RAG evaluation harness that scores your GTM agent on retrieval and generation quality.** Build a test set of 10 queries with known-relevant playbook chunks (manually labeled as ground truth). For each query, compute precision@3 and recall@5 over your retrieval results. Then for the generated answers, implement a faithfulness check (does the answer only use facts present in the retrieved context?) and a relevance check (does the answer address the query?) using an LLM-as-judge prompt. Run the full evaluation and write a summary report including per-query scores and aggregate averages to `outputs/skill-rag-evaluation.md`. Identify the worst-performing query and propose one concrete improvement to chunking or retrieval that would raise its score.
