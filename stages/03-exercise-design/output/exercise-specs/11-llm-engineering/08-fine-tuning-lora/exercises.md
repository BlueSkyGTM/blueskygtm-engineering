# Exercises — Fine-Tuning with LoRA & QLoRA

## Exercises

1. **Configure a LoRA adapter** for `Qwen/Qwen2.5-0.5B` targeting only the `q_proj` and `v_proj` modules. Set rank `r=8` and `alpha=16`. Load the model in full precision (no quantization), attach the adapter via PEFT, and print the total parameter count alongside the trainable parameter count. Confirm that trainable parameters are strictly less than 1% of total. Output both numbers and the percentage to the terminal.

2. **Re-run the configuration from Exercise 1** three times with rank values `r=4`, `r=16`, and `r=32` (keep alpha at `2 × r` each time). Print a table to the terminal showing rank, trainable parameter count, and trainable-as-percentage-of-total for each setting. Verify that trainable parameter count scales roughly linearly with rank.

3. **Compare memory footprints of full fine-tuning vs. LoRA vs. QLoRA.** Load `Qwen/Qwen2.5-0.5B` three separate times: (a) with all parameters trainable, (b) with a LoRA adapter attached (`r=16`, `alpha=32`, targeting `q_proj`/`v_proj`), and (c) with 4-bit quantization via `BitsAndBytesConfig` plus the same LoRA adapter. After each load, measure GPU memory consumed using `torch.cuda.memory_allocated()` (or RSS via `psutil` if no GPU). Print a summary table showing approach, total params, trainable params, and peak memory in MB. Confirm QLoRA uses the least memory of the three.

4. **Compute the theoretical LoRA adapter parameter count by hand** for the `q_proj` layer of `Qwen/Qwen2.5-0.5B`: look up its `hidden_size` and number of attention heads, derive the matrix dimensions, and calculate the expected trainable parameter count for `r=8`. Then write a script that loads the model with that LoRA config and prints the actual trainable parameter count reported by PEFT. Confirm your hand-computed value matches the empirical value to within rounding.

5. **Build a QLoRA fine-tuning pipeline on a GTM-relevant task.** Using the provided dataset at `data/gtm_email_classifications.jsonl` (rows containing a sales email body and a label of `meeting_booking`, `follow_up`, `objection`, or `other`), configure a 4-bit quantized `Qwen/Qwen2.5-0.5B`, attach a LoRA adapter (`r=16`, `alpha=32`, targeting `q_proj`, `k_proj`, `v_proj`, `o_proj`), fine-tune for 3 epochs, then save the adapter to `handlers/email_classifier_lora/`. After training, run inference on the five examples in `data/gtm_email_examples.jsonl` and print the predicted label vs. the ground-truth label for each. Then load the un-finetuned base model and run the same five examples. Print both sets of predictions side by side so you can see the improvement. Verify that the adapter directory contains only adapter weights (`adapter_model.safetensors`), not a full model copy.

6. **Merge the LoRA adapter from Exercise 5 into the base model** using `merge_and_unload()`, then save the merged model to `handlers/email_classifier_merged/`. Write a benchmark script that loads three configurations — (a) base model, (b) base model with adapter attached via PEFT, (c) merged model — and runs each on the same 20 prompts, measuring mean generation latency per prompt in milliseconds. Print a comparison table. Confirm that the merged model produces identical outputs to the adapter-attached model (string equality on all 20 prompts) and that merged-model latency is lower than adapter-attached latency. Save the latency benchmark results to `outputs/skill-lora-merge-benchmark.md` as a markdown table.
