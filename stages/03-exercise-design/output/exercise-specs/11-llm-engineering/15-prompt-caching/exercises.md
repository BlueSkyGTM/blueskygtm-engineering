# Exercises — Prompt Caching and Context Caching

## Exercises

1. **Implement** a Python script that sends three Anthropic API requests in sequence: (a) an initial request with a large static system prompt (at least 1200 tokens of fixed instructions), (b) a second request whose system prompt is byte-for-byte identical to the first but whose user message differs, and (c) a third request identical to (b) except for a single changed character in the system prompt at around token position 30. Print `cache_creation_input_tokens`, `cache_read_input_tokens`, and `input_tokens` from each response. Confirm that request (b) shows a nonzero cache read while request (c) does not.

2. **Apply** Google's `cachedContents` API to cache a shared prefix (a 1500+ token rubric for evaluating sales outreach emails), then send two separate generation requests that reference the cached content. Print the `cachedContentTokenCount` from the cache creation response and the `usageMetadata` fields (`cachedContentTokenCount`, `promptTokenCount`) from each generation. Verify that the second generation reports the cached token count.

3. **Build** a bulk GTM enrichment pipeline that processes a list of 20 company profiles (fetch them from Apollo's API or load them from a JSON file at `data/companies.json`). Design a prompt with a static prefix containing your ICP scoring rubric and a dynamic suffix containing one company's profile data. Run all 20 companies through Anthropic with `cache_control` on the static block. Print the cumulative cache hit ratio (`cache_read_input_tokens / (cache_read_input_tokens + cache_creation_input_tokens + input_tokens)`) and the total cost using Anthropic's published pricing for cache writes (1.25× base) and cache reads (0.1× base).

4. **Compute** a cost-and-latency comparison table for two processing strategies on a batch of 500 enrichment calls where each call has a 2000-token static prefix and a 300-token dynamic suffix. Strategy A places no cache markers. Strategy B uses explicit prompt caching. Assume a 90% cache hit rate after the first call, a base input cost of $3/MTok, cache write at 1.25×, cache read at 0.1×, and a 50ms latency reduction per 1000 cached tokens. Print a table showing total input cost, total latency (based on tokens processed in forward pass), and dollar savings for Strategy B vs Strategy A.

5. **Implement** a cache-miss diagnostic tool at `handlers/diagnose_cache.py` that takes a directory of saved Anthropic API response JSONs (from a prior bulk run) and classifies each response's cache state as `HIT`, `MISS_PREFIX_MUTATION`, `MISS_TTL_EXPIRY`, or `MISS_INSUFFICIENT_PREFIX`. Use heuristics such as: `cache_read_input_tokens == 0` and `cache_creation_input_tokens > 0` on later requests suggests prefix mutation or TTL expiry (check timestamps against the 5-minute TTL window); `input_tokens` large with zero cache activity suggests prefix below the minimum cacheable length (1024 tokens for Anthropic). Output a summary report to `outputs/cache-diagnosis.md` with counts per category and a list of the top 5 most likely prefix-mutation positions (diff the system prompt strings between consecutive calls to find where they diverge).

6. **Design** and implement a production-ready enrichment pipeline at `handlers/enrichment_pipeline.py` that ingests Clay-exported company and contact data from `data/clay_export.csv`, constructs cache-optimized prompts (shared system prompt > 1024 tokens containing your enrichment schema, scoring logic, and output format; dynamic per-company suffix), batches requests to stay within TTL windows (add a `--batch-delay` flag), and handles TTL expiry gracefully by re-warming the cache. The pipeline must write per-call metrics (cache hit ratio, tokens, latency) to `outputs/enrichment-metrics.jsonl` and print a final summary showing overall cache hit rate, estimated cost vs uncached baseline, and p50/p90 latency. Include a `--dry-run` flag that validates prefix stability across all rows without making API calls.
