# Exercises — Guardrails, Safety & Content Filtering

## Exercises

1. **Implement a PII detection classifier.** Write a Python module that takes a string of LLM-generated text and scans it for email addresses, phone numbers (formats like `555-123-4567`, `(555) 123-4567`), and Social Security numbers (`XXX-XX-XXXX`). Your classifier should return a list of matches with the type of PII detected and the matched substring. Run it against three sample strings you create — one clean, one containing an email and phone number, one containing an SSN — and print the match results for each. Confirm the clean string produces zero matches.

2. **Extend the classifier with keyword rules for competitor mentions and prohibited claims.** Add two new detection categories to your module: (a) a configurable list of competitor company names that triggers when any name appears in the output, and (b) a list of prohibited marketing claims (e.g., "guaranteed ROI", "100% success rate", "#1 platform", "risk-free"). Each rule should return a severity level (`low` for competitor mentions, `high` for prohibited claims). Test against a sample personalized email output that contains one competitor name and one prohibited claim. Print the full classification report showing every violation, its category, and its severity.

3. **Build the three-layer guardrail stack.** Implement a `GuardrailStack` class (or set of functions) with three sequential layers: (1) input validation that rejects inputs containing prompt injection patterns (e.g., "ignore your instructions", "disregard the above", "you are now") and inputs missing required fields, (2) output classification using your PII and keyword detectors from Exercises 1–2, and (3) a fallback router that returns `deliver`, `rewrite`, `reject`, or `escalate` based on the highest-severity violation found. Feed it four test cases — a clean input/output pair, a prompt injection in the input, PII in the output, and a prohibited claim in the output — and print the routing decision for each.

4. **Compute false positive and false negative rates against a labeled test set.** Create a JSON or CSV file with 20 labeled examples of LLM output text — each marked `safe` or `unsafe` with the expected violation category. Run your classifier from Exercises 1–2 against every example and compare predictions to labels. Calculate and print the false positive rate (safe examples flagged as unsafe) and false negative rate (unsafe examples missed). Identify which violation category has the worst recall and print a one-line hypothesis for why.

5. **Deploy guardrails into a GTM content pipeline.** Build a pipeline script that reads enriched prospect records from a CSV (columns: `name`, `company`, `linkedin_bio`, `generated_email`), runs each `generated_email` through your full guardrail stack, and writes results to an output CSV with columns `prospect_id`, `routing_decision`, `violations`, and `severity`. Add a logging hook that appends every rejected or escalated output to a JSONL log file at `outputs/guardrail_log.jsonl`. Use a sample enriched dataset of 10 prospects (include at least 2 with planted PII, 1 with a competitor mention, and 1 prompt injection in the bio field). Print a summary report showing how many emails were delivered, rewritten, rejected, and escalated. **Artifact: `handlers/gtm_guardrail_pipeline.py`**

6. **Design a confidence-based escalation policy and prove it reduces review burden.** Extend your pipeline so that outputs flagged by the keyword classifier carry a confidence score (e.g., exact match = 1.0, fuzzy/partial match = 0.6–0.9). Outputs below a configurable confidence threshold for the `unsafe` label are routed to a human review queue instead of being auto-rejected. Run the pipeline twice on your test dataset — once with the threshold set to 0.0 (everything auto-decided) and once at 0.8 — and print a comparison table showing how many items hit human review under each setting. Write a one-paragraph justification for your chosen threshold based on the false positive/negative data from Exercise 4. **Artifact: `outputs/skill-guardrail-policy.md`**
