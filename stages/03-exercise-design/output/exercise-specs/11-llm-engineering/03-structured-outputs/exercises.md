# Exercises — Structured Outputs: JSON, Schema Validation, Constrained Decoding

## Exercises

1. **Implement all three enforcement layers against the same prompt.** Write a single script that calls the OpenAI Python SDK three times with the identical extraction task — pulling `company_name`, `employee_count`, and `funding_stage` from a short company description — but using (a) prompt-only instructions, (b) `response_format={"type": "json_object"}`, and (c) a Pydantic schema passed via structured outputs. For each response, print the raw output string, then attempt `json.loads` and access each expected key, printing `[PASS]` or `[FAIL: <error>]` per field. Run the script five times per layer and tabulate which layers produce failures.

2. **Build a Pydantic validation gate that catches what JSON mode misses.** Define a Pydantic model with fields `company_name: str`, `employee_count: int`, and `funding_stage: Literal["seed", "series_a", "series_b", "series_c", "growth"]`. Feed it five hand-crafted JSON strings — one valid, one with a string where an int belongs, one with an enum value outside the allowed set, one with a missing key, and one containing an extra unexpected key (with `model_config = ConfigDict(extra="forbid")`). Print the validation result for each and the specific error message Pydantic raises. Confirm that JSON mode alone would have accepted all five as syntactically valid JSON.

3. **Trace a finite-state machine that enforces JSON grammar at the token level.** Implement a minimal FSM (no library needed — a dictionary mapping states to sets of legal next-token prefixes) that accepts only the exact string pattern `{"name": "<letters>", "age": <digits>}`. Feed it three candidate token sequences: a valid one, one with an unquoted key, and one with a trailing comma before the closing brace. For each sequence, print the FSM state at every step, flag the transition where an invalid token is rejected, and print `[ACCEPTED]` or `[REJECTED at state X, token Y]`. Explain in a comment block why constrained decoding based on this FSM never produces the invalid sequences in the first place.

4. **Design a Clay waterfall enrichment schema with fallback behavior.** Define a Pydantic model for extracting LinkedIn-derived prospect data from raw HTML: `full_name`, `title`, `company`, `location`, and `confidence_score (0.0–1.0)`. Mark `title` and `location` as `Optional` with a default of `"unknown"`. Add a custom validator on `confidence_score` that raises a clear error if the value falls outside `[0, 1]`. Write a retry wrapper that catches validation errors, appends the error message to the system prompt, re-calls the model, and gives up after three attempts — returning a dict with all fields set to `None` and `error` populated on final failure. Test it against two raw HTML snippets: one clean LinkedIn profile and one with missing location. Print the final validated object for each.

5. **Build a production enrichment handler and compute effective retry cost.** Create `handlers/enrichment_handler.py` that (a) takes a list of 10 company descriptions (mix of clean text and adversarial inputs with missing or contradictory fields), (b) extracts structured data using a Pydantic schema with `company_name`, `industry`, `headcount_range`, `tech_stack: list[str]`, and `enrichment_confidence: float`, (c) runs up to 3 validation retries with error feedback as in Exercise 4, (d) logs each attempt's token usage and whether validation passed or failed, and (e) after processing all 10 inputs, prints a cost summary: total input tokens, total output tokens, total API calls, number of successful extractions on first attempt vs. after retry vs. failed, and the **effective cost per successful extraction** (total estimated cost / successful extractions). Use `$0.15` per 1M input tokens and `$0.60` per 1M output tokens as pricing constants. Run the handler and paste the cost summary into a comment at the bottom of the file.
