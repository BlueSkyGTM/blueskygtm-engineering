## Exercises

1. **Build a minimal typed state graph with three nodes and fixed edges.** Define a `TypedDict` called `PipelineState` with fields `company_name` (str), `website` (str), and `status` (str). Implement three node functions: `fetch_company` (sets `company_name` and `status="fetched"`), `validate_url` (sets `website` and `status="validated"`), and `mark_ready` (sets `status="ready"`). Wire them with fixed edges in a `StateGraph`, compile, and invoke with an initial state containing only `company_name="Stripe"`. Print the final state dict and confirm all three fields are populated.

2. **Trace state mutations through a four-node graph and predict the output before running.** Build a graph with state schema holding `messages` (list[str]) and `turn_count` (int). Nodes `entry`, `classify`, `respond`, and `log` each append a string to `messages` and increment `turn_count`. Wire them as entry → classify → respond → log. Before executing, write down what you expect `messages` and `turn_count` to contain. Then invoke the graph with `messages=[]`, `turn_count=0` and compare.

3. **Implement conditional routing based on a state value.** Extend the graph from Exercise 1: after `validate_url`, add a conditional edge that routes to `mark_ready` if the website string starts with `https://`, otherwise routes to a new `fix_url` node that prepends `https://` to the website and loops back to `validate_url`. Test with two inputs — one where the initial website is already `https://stripe.com` and one where it is `stripe.com` — and confirm the second triggers the cycle.

4. **Map an enrichment waterfall onto a state machine with branching logic.** Define state with fields `email` (str), `source` (str), and `confidence` (float). Implement nodes `try_apollo`, `try_clearbit`, and `mark_unverified`. The Apollo node sets `email` and `confidence=0.9` if the lookup succeeds (simulate success by checking whether a pre-populated `apollo_result` field is non-empty). The Clearbit node does the same at `confidence=0.6`. Build conditional edges so the graph tries Apollo first, falls back to Clearbit on failure, and routes to `mark_unverified` if both fail. Run three test cases — Apollo hit, Clearbit-only hit, both miss — and print the final `source` and `confidence` for each.

5. **Build a production-ready enrichment waterfall state machine and export it as a runnable module.** Extend Exercise 4 to include a retry cycle: if Apollo returns a result with `confidence < 0.5`, loop back to try Clearbit before giving up. Add a `max_retries` guard so the cycle cannot exceed three iterations. Write the complete graph to `handlers/enrichment_waterfall.py` with a `run_enrichment(prospect: dict) -> dict` entry point. Invoke it against a sample of five prospects (at least one should trigger the retry path) and print a summary table showing each prospect's final `email`, `source`, `confidence`, and `turns_taken`.
