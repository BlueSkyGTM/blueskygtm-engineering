# Exercises — Building a Production LLM Application

## Exercises

1. **Implement** a production LLM client class that sends a prompt to the live Anthropic API (`claude-sonnet-4-20250514`) and prints the completion. Configure the client with exponential backoff retry: maximum 4 attempts, base delay of 1 second, multiplier of 2, and jitter. Run the client with a simple prompt such as `"Summarize Salesforce in one sentence."` and verify that the terminal prints the response text, the HTTP status, and the number of retry attempts that occurred.

2. **Build** a failure classification function that accepts an HTTP status code (or an exception type such as `TimeoutError`) and returns a dictionary containing `{"category": "transient" | "semantic", "action": "retry" | "abort", "reason": "<short string>"}`. Cover at minimum: 429, 500, 502, 503, 401, 400, 404, and `TimeoutError`. Write a small test harness that iterates over all these cases and prints the classification for each. Confirm that transient codes map to `retry` and semantic codes map to `abort`.

3. **Extend** your client from Exercise 1 to capture and accumulate token-level cost across multiple calls. Use the `usage` field from the API response to extract input tokens and output tokens. Maintain a running total of accumulated cost using pricing of $3 per million input tokens and $15 per million output tokens. Make three sequential calls with different prompts, then print a final summary table showing per-call token counts, per-call cost, and total accumulated cost. Verify the total is the sum of individual calls.

4. **Apply** your client to a GTM enrichment task. Load the CSV at `data/companies.csv` (create it with 5 rows containing `company_name` and `description` columns if it does not exist). Send each company description to the LLM with a prompt that asks it to return a JSON object with fields `icp_fit_score` (1–10), `estimated_employee_range`, and `likely_tech_stack` (list). Validate each response against a JSON schema using `jsonschema.validate`. Print each enriched record or a validation error if the response does not conform. Confirm that all 5 records print valid JSON or a clear schema-violation message.

5. **Build** a health check and graceful degradation handler for a GTM enrichment pipeline. The handler should expose a `health_check()` method that makes a lightweight API call (a 10-token prompt) and returns `{"healthy": bool, "latency_ms": float, "error": str | None}`. The handler's `enrich_company(company_name, description)` method should first check health; if the backend is unhealthy, it should fall back to a rule-based enrichment that uses keyword matching on the description to estimate ICP fit score and return a cached or heuristic result flagged with `"source": "degraded"`. Write the handler to `handlers/enrichment_handler.py`. Run it against 3 companies with the API key set and then run it again with an intentionally invalid key to demonstrate degradation. Verify that healthy runs print `"source": "llm"` and degraded runs print `"source": "degraded"`.

6. **Design and implement** a complete request-response lifecycle tracer that wraps every LLM call with structured logging. The tracer must log: prompt assembly time, token budget check result, API call latency, response status classification, retry events with attempt number and delay, parsed output, validation result, usage tokens, and accumulated session cost. Write each event as a JSON line to stdout and to a log file at `outputs/llm-lifecycle.log`. Then run the tracer on a batch enrichment of 10 companies (use `data/companies.csv` or generate synthetic data) and confirm that the log file contains every lifecycle stage for each call. Finally, compute and print the aggregate session report: total calls, total retries, average latency, total tokens, total cost, and degradation count if you integrated the handler from Exercise 5.
