# Exercises — Context Engineering: Windows, Budgets, Memory, and Retrieval

## Exercises

1. **Compute a token budget breakdown for a near-limit configuration.** Write a Python script that defines a `budget_report` function accepting `context_window`, `system_prompt_tokens`, `tool_tokens`, `history_tokens`, `retrieved_tokens`, and `output_reserve`. Print a formatted table showing each segment's token cost, total input tokens, total with output reservation, and remaining headroom. Test with a context window of 8192 tokens and a configuration where `system_prompt_tokens=400`, `tool_tokens=1200`, `history_tokens=3500`, `retrieved_tokens=2500`, and `output_reserve=1000`. Confirm the report correctly flags whether the configuration fits.

2. **Compare memory architectures on the same conversation.** Implement three functions — `full_history(messages)`, `sliding_window(messages, n=4)`, and `summarization(messages, max_summary_tokens=200)` — that each return the token count they would consume for a given conversation. Use the sample conversation from the lesson (or build a 12-turn conversation about qualifying a lead). Print a comparison table showing token cost for each strategy and the percentage savings relative to full history.

3. **Build a context assembler with dynamic allocation.** Implement a `ContextAssembler` class that accepts a total token budget and a list of segments, where each segment has a `name`, `priority` (integer, 1=highest), and `min_tokens`. The assembler allocates remaining budget to segments in priority order after satisfying all minimums, then prints a pre-call headroom report showing allocated tokens per segment, total used, and remaining headroom. Instantiate it with five segments (system prompt, tool definitions, conversation history, retrieved context, output reserve) and print the report. Raise a `BudgetExceededError` if minimums alone exceed the budget.

4. **Apply RAG budget allocation to Apollo prospect data.** Fetch 10 company records from the Apollo API (or use mock data structured like Apollo's `people.search` response with fields: name, title, company, linkedin_url, company_size, industry). Each record averages ~150 tokens. Given a retrieved-context budget of 2000 tokens alongside a system prompt (300 tokens) and 6 turns of conversation history (1800 tokens) within an 8192-token window, implement a function that selects which records fit and prints the cutoff: which records made it, which were dropped, and the headroom remaining after insertion.

5. **Build an overflow diagnostic handler.** Implement a diagnostic tool at `handlers/overflow_diagnostic.py` that accepts a prompt configuration (all segment token counts + provider name from `openai`, `anthropic`, or a generic `truncating` runtime). The tool predicts whether the configuration will overflow, what the provider will do (hard error with exact message pattern, 400 status, or silent front-truncation), and outputs a recommended fix (reduce history, compress tools, shrink retrieval, switch model tier). Run the handler against three test configurations — one that fits, one that overflows by 500 tokens on OpenAI, and one that overflows by 2000 tokens on a truncating runtime. Print the diagnostic report for each.

6. **Design a context strategy for a multi-step Clay enrichment workflow.** Document a complete context management strategy at `outputs/skill-context-strategy.md` for a Clay table
