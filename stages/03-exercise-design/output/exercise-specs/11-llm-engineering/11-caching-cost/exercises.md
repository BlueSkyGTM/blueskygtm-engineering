# Exercises — Caching, Rate Limiting & Cost Optimization

## Exercises

1. **Implement a token bucket rate limiter** that accepts `capacity`, `refill_rate`, and a `try_acquire()` method returning `True` or `False`. Feed it 25 requests in a tight loop against a bucket with capacity=5 and refill_rate=10/min. Print each request index alongside its `ALLOWED` or `BLOCKED` status. Confirm that the first 5 are allowed and the remainder are blocked until the bucket refills.

2. **Build a hash-keyed cache with TTL expiration.** Use `hashlib.sha256` on the input string to derive keys, store entries with a timestamp, and implement `get(key)` / `set(key, value, ttl_seconds)`. Populate the cache with three entries using a 2-second TTL, then immediately retrieve one (should hit), sleep 3 seconds, and retrieve again (should miss). Print `[HIT]` or `[MISS]` next to each lookup so the TTL boundary is observable in the terminal.

3. **Compute per-request LLM costs** across three model tiers — a flagship model ($15/M input, $75/M output), a mid-tier model ($3/M input, $15/M output), and a budget model ($0.15/M input, $0.60/M output). Given a batch of 100 enrichment prompts averaging 1,200 input tokens and 300 output tokens each, print the total dollar cost for each tier and the savings (in both dollars and percentage) of routing the full batch through the budget model instead of the flagship.

4. **Apply rate limiting and caching to an Apollo-style company-enrichment loop.** Start with a list of 50 company domains where 18 are duplicates. Deduplicate the list, then process each unique domain through a rate limiter (capacity=10, refill_rate=60/min) before calling a mock `enrich_company(domain)` function that returns a placeholder dict. Print the final dedup count, how many calls were allowed vs. blocked, and the total elapsed time — then re-run the same list and confirm zero new API calls thanks to your cache.

5. **Build a composite enrichment pipeline** as `handlers/enrichment_pipeline.py` that ingests a CSV of 200 prospect rows (name, email, company_domain), deduplicates by email, checks a TTL cache for prior enrichment results, rate-limits uncached calls to 30 requests/minute against a mock enrichment provider, and writes a JSON report containing: total rows processed, unique rows, cache hits, cache misses, API calls made, 429s encountered, estimated cost at a chosen model tier, and final enriched records. Run it twice on the same input file and confirm the second run shows 100% cache hits with $0.00 API spend.

6. **Diagnose a simulated cache-and-rate-limit failure scenario.** Construct a setup where 150 requests hit a cache with a 5-second TTL but arrive in a staggered pattern (some at t=0, some at t=6, some at t=12), all feeding through a token bucket with capacity=10 and refill_rate=20/min. Instrument every request with its cache status, rate-limiter status, and timestamp. Write your findings to `outputs/skill-cache-rate-limit-diagnostics.md` including: observed cache hit rate over time, exact points where 429-equivalent blocks occurred, a root-cause explanation of why TTL expiration caused a burst that overwhelmed the rate limiter, and a recommended fix (e.g., jittered TTL, pre-warming, or bucket resizing) with a brief justification.
