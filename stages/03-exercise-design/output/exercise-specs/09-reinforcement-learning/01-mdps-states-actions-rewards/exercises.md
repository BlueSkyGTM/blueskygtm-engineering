# Exercises — MDPs, States, Actions & Rewards

## Exercises

1. **Implement** a function `describe_mdp()` that hardcodes the five components of a simple 4-cell gridworld MDP: states `S = {s0, s1, s2, s3}` in a 2×2 grid, actions `A = {left, right, up, down}`, a deterministic transition dict `T[(s, a)] -> s'`, a reward dict `R[(s, a)] -> float`, and a discount `γ = 0.9`. The function must print each component in a labeled section (states, actions, transitions, rewards, discount) and then print whether the MDP satisfies the Markov property — specifically, check that every transition entry is keyed only by `(s, a)` and contains no reference to prior states. Run the script and confirm the labeled output appears in the terminal.

2. **Compute** the discounted return for the following fixed episode: `[(s0, right, 0), (s1, right, 0), (s2, down, 10)]`. Write a function `compute_return(episode, gamma)` that takes a list of `(state, action, reward)` tuples and a discount factor. Call it three times — with `γ = 0.5`, `γ = 0.9`, and `γ = 0.99` — and print all three returns side by side. Verify that higher discount factors produce higher returns for this reward-at-the-end episode.

3. **Compare** two policies on the 2×2 gridworld from Exercise 1. Policy A always moves right (then down at the edge). Policy B always moves down (then right at the bottom). Both terminate after reaching `s3`. Implement a `simulate(policy, mdp, start_state)` function that returns the total discounted return for one episode. Run 1,000 simulations under each policy (with stochastic transitions of your choosing — e.g., 80% intended direction, 20% perpendicular) and print the mean and standard deviation of returns for each. State in a printed summary line which policy has the higher expected return.

4. **Apply** the MDP formulation to a GTM enrichment waterfall. A lead starts with only an email address. Define four states based on data completeness: `S = {email_only, +name, +name+company, +name+company+phone}`. Actions are `A = {query_clearbit, query_apollo, query_hunter, terminate}`. Each provider populates different fields with a known probability and costs a fixed amount (Clearbit: $0.05, 60% chance to add company; Apollo: $0.03, 70% chance to add name; Hunter: $0.02, 50% chance to add phone). Reward is net data value minus cost. Implement this MDP as a Python class with `step(state, action)` returning `(next_state, reward)`. Simulate the waterfall policy "query Apollo → Clearbit → Hunter → terminate" for 500 leads and print the average net reward per lead.

5. **Build** a configurable enrichment waterfall MDP simulator that accepts a JSON config file specifying multiple providers, their costs, field fill rates, and the fields they populate. Load enrichment provider data from a real source — query the Apollo People Search API (or a local CSV export from Clay containing at least 50 leads) to seed your lead records. Implement three different provider-ordering policies, simulate each across all leads, and write a Markdown report comparing average cost-per-lead, average fields-populated, and net reward per policy. Save the artifact to `outputs/skill-enrichment-mdp.md`.

6. **Design** an optimal ordering policy for the enrichment MDP from Exercise 5 using Monte Carlo Q-value estimation. For each `(state, action)` pair, simulate 200 rollouts with random subsequent actions and compute the average discounted return. Derive a greedy policy from the estimated Q-table. Implement this policy as a callable handler that takes a lead's current data state and returns the next provider to query. Save the artifact to `handlers/enrichment_policy.py`. Print the Q-table and the derived policy to the terminal, and verify the greedy policy differs from a naive fixed-order policy in at least one state.
