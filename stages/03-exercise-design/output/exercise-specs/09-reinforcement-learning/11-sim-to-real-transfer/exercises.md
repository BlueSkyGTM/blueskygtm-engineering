# Exercises — Sim-to-Real Transfer

## Exercises

1. **Implement domain randomization on a cartpole simulator and compare it against a fixed-parameter policy.** Write a training loop that samples pole mass and friction from uniform distributions on each episode (mass ∈ [0.5, 2.0] kg, friction ∈ [0.01, 0.3]). Train a simple policy (linear controller, small MLP, or PID) under randomization. Separately train an identical policy on fixed parameters (mass = 1.0, friction = 0.1). Evaluate both on a test sweep of 20 perturbed parameter pairs and print a table showing mean reward, min reward, and worst-case degradation percentage for each policy. Your terminal output must clearly show whether the randomized policy is more robust.

2. **Classify sim-to-real techniques by their resource profile.** Write a Python script that defines a dataclass or dictionary for each of the five techniques covered in the lesson (domain randomization, domain adaptation, system identification, and any two others discussed). Each entry must encode: compute cost (low/medium/high), real-world data requirement (samples or none), robustness guarantee (qualitative), and a one-sentence use-case. Print a formatted comparison table to the terminal. Then add a function that takes a project constraint string (e.g., "no_real_data", "low_compute", "max_precision") and returns the recommended technique. Verify your recommender returns sensible answers for at least three different constraint inputs.

3. **Trace how perturbations in physical parameters propagate through a trained policy's reward signal.** Take any simple continuous-control environment (cartpole, mountain car, pendulum, or a custom 1D dynamics simulation). Freeze a single policy (you may use a random or hand-tuned controller if no trained policy exists). Sweep one physical parameter (friction, mass, gravity, or sensor noise std) across 30 evenly spaced values while holding all others fixed. For each setting, run 10 episodes and record the mean reward. Print the raw sweep results as a table and identify the parameter value at which mean reward drops below 50% of the nominal-parameter reward. State the sensitivity threshold you found.

4. **Build a system identification loop that calibrates a simulator to match a "real-world" trajectory log.** Create a target trajectory file by running a simulator with hidden parameters (e.g., true mass = 1.35, true friction = 0.18 — do not reveal these in your code). Then write an optimization loop (grid search, random search, or gradient-free optimizer like CMA-ES or scipy.optimize.minimize with Nelder-Mead) that adjusts the simulator's mass and friction parameters to minimize trajectory divergence (L2 distance on state vectors or MSE on observations). Print each iteration's candidate parameters and divergence score. Your loop must converge to within 10% of the hidden true parameters. Print the final identified parameters alongside the ground truth for verification.

5. **Apply domain randomization thinking to synthetic data generation for a GTM lead-enrichment model.** Build a synthetic lead generator that randomizes non-target fields (company size noise, industry label perturbations, technographics shuffling) across a wide distribution while preserving the core signal (e.g., ICP fit label). Generate 5,000 randomized synthetic leads. Train a simple classifier (logistic regression, random forest, or gradient boosting) on the synthetic data, then evaluate on a held-out slice of real CRM leads (you may use Apollo-enriched data, a Clay export, or any CRM extract you have access to). Compute precision, recall, and F1 on the real holdout. Compare against a second classifier trained only on non-randomized synthetic leads with fixed feature distributions. Save the generator as `signals/examples/synthetic_lead_randomizer.py` and write a summary of both models' real-world performance to `outputs/skill-sim-to-real-gtm-transfer.md`. State explicitly where the domain-randomization analogy held and where it broke down.

6. **Design a hybrid sim-to-real pipeline that combines all three technique families and prove its robustness.** Implement a three-stage pipeline for a control task of your choice: (1) system identification to calibrate base simulator parameters from a small real trajectory sample, (2) domain randomization layered on top of the calibrated simulator with parameter ranges derived from the sys-id uncertainty, (3) domain adaptation via a small fine-tuning pass on real data. Train a policy through the full pipeline. Compare its deployment reward distribution against: a domain-randomization-only policy, a sys-id-only policy, and a naive fixed-parameter policy. Print all four reward distributions (mean, std, 5th percentile) to the terminal. Save the pipeline orchestration code to `handlers/sim_to_real_pipeline.py`. Write a one-paragraph analysis to `outputs/skill-sim-to-real-gtm-transfer.md` (append, do not overwrite Exercise 5's content) explaining which stage contributed the most robustness gain and whether the added complexity was justified by the reward improvement.
