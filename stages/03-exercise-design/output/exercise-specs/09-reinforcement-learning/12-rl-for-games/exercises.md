## Exercises

1. **Implement UCB1 selection for a fixed game tree.** Create a `Node` class with `visits`, `total_value`, and `children` attributes. Hardcode a root node with four children that have differing visit counts and win rates (e.g., child A: 10 visits, 7 wins; child B: 3 visits, 2 wins; child C: 8 visits, 5 wins; child D: 1 visit, 1 win). Compute the UCB1 score for each child with exploration constant c=1.41 and print a ranked table showing each child's name, visits, average value, UCB1 score, and which child is selected. Re-run with c=0.5 and c=3.0 and compare which node gets selected at each setting. Print the full comparison table.

2. **Trace a single MCTS iteration on a Tic-Tac-Toe position.** Implement a complete MCTS search consisting of selection, expansion, simulation, and backpropagation for a mid-game Tic-Tac-Toe board (e.g., X has played center and top-left, O has played center-top). Run exactly one iteration from the root and print the board state at each phase transition: which node was selected and why, which action was expanded, what the random rollout returned, and how the values changed along the backpropagation path. The output should be a labeled trace readable as a step-by-step narrative.

3. **Build a self-play data generator for Tic-Tac-Toe.** Implement a `SelfPlay` class that runs N random-playout games, records every (board_state, action_taken, final_outcome) tuple from the perspective of the player who moved, and writes the results to a JSON file. Run 500 games and print: total games played, total training tuples generated, win/draw/loss distribution, and three sample tuples with their board states rendered as 3×3 grids. Verify the output file exists and contains the expected number of tuples.

4. **Compare AlphaZero and MuZero assumptions on a partially observable domain.** You are given a CSV file (`data/sales_transitions.csv`) containing logged CRM state transitions: each row has `current_state_features` (deal stage, days_in_stage, contact_count, last_activity_days), `action_taken` (email, call, meeting, demo), `next_state_features`, and `reward` (deal progressed or not). You have no simulator — you cannot query arbitrary state-action pairs, you only have historical logs. Implement two approaches: (a) an AlphaZero-style MCTS that attempts random rollouts using only the CSV data, and (b) a MuZero-style approach where you train a simple dynamics model (a nearest-neighbor or linear regressor on the logged transitions) to predict next-state and reward, then use that model inside MCTS. Print which approach can complete a full search tree and which fails, along with the predicted best action from the current deal state. Include a short printed summary explaining the failure mode.
