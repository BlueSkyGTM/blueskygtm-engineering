# Exercises — Handling Imbalanced Data

## Exercises

1. **Compute the accuracy floor.** Load the Credit Card Fraud dataset from `sklearn.datasets.fetch_openml(data_id=1597)`. Print the class distribution counts. Train a `DummyClassifier(strategy='most_frequent')` and print its accuracy on a holdout test set. Then train a `LogisticRegression(max_iter=1000)` on the same split and print accuracy, recall, and precision. Observe how both models report near-identical accuracy while the dummy model catches zero fraud cases.

2. **Wrap SMOTE in a pipeline.** Using the same fraud dataset, build a `Pipeline` that chains `SMOTE` (from `imbalanced-learn`) followed by `LogisticRegression(max_iter=1000)`. Print the class distribution of `y_train` before and after SMOTE by extracting the resampled data from the pipeline's first step. Evaluate on the untouched test set and print precision, recall, F1, and `average_precision_score`. Compare the four metrics against the plain logistic regression from Exercise 1.

3. **Compare `class_weight` against SMOTE on a new dataset.** Load the `winequality` dataset (`fetch_openml(data_id=40691)`, binarize quality into "high" ≥ 7 vs "low" < 7). Train three logistic regressions: one plain, one with `class_weight='balanced'`, and one preceded by SMOTE in a pipeline. Print a comparison table (use `tabulate` or manual formatting) showing precision, recall, F1, and PR-AUC for all three on the same test split. State in a comment at the top of your script which mechanism you would choose and why.

4. **Tune a decision threshold for a GTM lead-scoring constraint.** Download a sample of Apollo-style lead data (use `https://raw.githubusercontent.com/datasets-io/lead-scoring/main/sample_leads.csv`, or construct a synthetic DataFrame with 2000 rows, 3% conversion rate, and features like `company_size`, `industry_code`, `page_visits`, `email_opens`). Train a `GradientBoostingClassifier` with `class_weight` if supported, otherwise default. Sweep thresholds from 0.05 to 0.95 in steps of 0.05. For each threshold, print threshold, precision, recall, and number of predicted-positive leads. Identify the lowest threshold that still achieves precision ≥ 0.20. Print that threshold and the corresponding expected lead count for a 10,000-prospect batch.

5. **Build an end-to-end churn prediction handler with imbalance handling.** Using the Telco Churn dataset (load from a CSV of your choice or `fetch_openml(name='telco-customer-churn')`), implement a full training script that: (a) preprocesses features with a `ColumnTransformer`, (b) trains a pipeline comparing SMOTE vs `class_weight='balanced'`, (c) selects the approach with higher PR-AUC via cross-validation, (d) tunes the decision threshold to achieve recall ≥ 0.75, and (e) prints a final report with the chosen method, threshold, and all four metrics. Save the trained pipeline and threshold to disk using `joblib`. Write the script as a reusable handler at `handlers/churn_scorer.py` that exposes a `predict_churn(features_df)` function returning predicted probabilities and binary labels using the tuned threshold.

6. **Design a threshold policy for a sales development team.** A GTM team scores outbound prospects with a model that produces conversion probabilities. The team has 1 SDR who can manually review 300 prospects per week. The prospect pool contains 50,000 leads with a 2% historical conversion rate. Each converted lead is worth $12,000 in ARR. Each manual review costs $15 in SDR time. Write a script that loads a predictions file (generate synthetic predictions with a beta distribution calibrated to 2% base rate), then computes the expected revenue, total SDR cost, and net profit for threshold values from 0.01 to 0.50. Print a table and identify the revenue-maximizing threshold. Then write a 300-word analysis to `outputs/skill-threshold-policy.md` explaining your recommended threshold, the tradeoff between coverage and precision, and how the team should respond if the prospect pool grows to 200,000 but headcount stays the same.
