# Exercises ‚Äî Heritage of FIPA-ACL and Speech Acts

## Exercises

1. Build a performative classifier. Create a Python dictionary that maps each of these FIPA-ACL performatives to its Searlean category: `inform`, `request`, `agree`, `refuse`, `propose`, `accept-proposal`, `reject-proposal`, `cfp`, `confirm`, `disconfirm`, `subscribe`, `cancel`, `failure`, `not-understood`. Then write a function `classify(performative)` that returns the category string. Print the full classification table and run it against a test list of five performatives not in your dictionary (e.g., `query-if`, `propagate`, `proxy`, `request-whenever`, `call-for-proposal`) by assigning them based on their semantics. Verify your output shows all five Searlean categories represented at least once.

2. Write a FIPA-ACL message envelope validator. Define a function `validate_acl(msg: dict) -> tuple[bool, str]` that checks for mandatory fields (`sender`, `receiver`, `performative`, `content`) and reports the first missing field by name. If all mandatory fields are present, verify that `performative` is one of the known performatives from Exercise 1. Construct five test messages ‚Äî three valid, two invalid (one missing `receiver`, one with an unknown performative `shout`) ‚Äî and print the validation result for each. Each line of output should show the message index, the boolean, and the reason string.

3. Trace a Contract Net protocol sequence end-to-end. You are given a JSON file at `signals/inputs/contract-net-trace.json` containing a list of eight ACL messages (some in order, some not). Implement a function `trace_contract_net(messages) -> list[str]` that sorts the messages by their position in the canonical Contract Net flow (`cfp` ‚Üí `propose` / `refuse` ‚Üí `accept-proposal` / `reject-proposal` ‚Üí `inform-done` / `failure`) and returns the ordered list of performatives. Detect and print any message that cannot be placed in the sequence (e.g., an `inform` arriving before a `cfp` in the same `conversation-id`). Your terminal output should list the sorted sequence and any anomalies found.

4. Implement a conversation-state validator as a class `ConversationValidator` that enforces the `request ‚Üí agree ‚Üí inform-done` protocol. The validator tracks state per `conversation-id` and exposes a method `ingest(msg: dict) -> tuple[bool, str]`. Valid transitions: from `INIT` state, only `request` is accepted; from `AWAITING_AGREE`, only `agree` is accepted; from `AWAITING_INFORM`, only `inform-done` or `failure` is accepted. Any other performative at any state returns `False` with a descriptive reason. Feed the validator a hardcoded list of ten messages spanning three conversations (two that complete successfully, one that fails with an out-of-order `inform-done` before `agree`). Print each message, the validator's response, and the final state of every conversation.

5. Build a GTM enrichment coordinator that demonstrates the difference between raw JSON messaging and ACL-style performatives. Create two modules: `raw_agent.py` (agents exchange bare JSON like `{"action": "enrich", "company": "Acme"}`) and `acl_agent.py` (agents exchange FIPA-ACL envelopes). In the raw version, simulate the ambiguity problem: the same JSON payload could mean "please enrich this" (request) or "I already enriched this" (inform) ‚Äî show a case where the receiving agent misinterprets it. In the ACL version, the performative disambiguates and the correct behavior follows. Use a mock Apollo company-enrichment API call (you may stub the HTTP layer) to show an enrichment flow. Run both modules and print a side-by-side comparison showing where the raw version produces an incorrect action and the ACL version does not. Persist your ACL coordinator to `handlers/acl_enrichment_coordinator.py` and a summary of your findings to `outputs/skill-acl-vs-raw.md`.

6. Design and implement a new FIPA-ACL conversation protocol for a multi-step GTM workflow: a list-building pipeline where Agent A asks Agent B to find companies matching ICP criteria (`cfp`), Agent B proposes candidates (`propose`), Agent A accepts (`accept-proposal`), Agent B enriches via a real enrichment API of your choice (Apollo, Clay webhooks, or Clearbit ‚Äî use a stub if you lack API credits), and Agent B returns results (`inform-done` with the enriched list as content). Implement the full protocol with a state machine that rejects any deviation. Document the protocol specification (states, valid transitions, performatives used) in `outputs/skill-cfp-gtm-protocol.md`. Run a full conversation and print every message exchanged plus the final enriched company list. Then inject a deliberate protocol violation (a `propose` without a preceding `cfp`) and print the rejection.
