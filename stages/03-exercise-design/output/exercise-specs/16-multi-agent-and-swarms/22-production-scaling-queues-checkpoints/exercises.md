## Exercises

1. **Build** a SQLite-backed task queue from scratch. Create a database file `queue.db` with a `tasks` table containing columns for `id`, `payload` (JSON text), `status` (default `'pending'`), `created_at`, and `updated_at`. Insert 50 tasks with payloads representing lead enrichment jobs (`{"company": "...", "domain": "..."}`). Write a worker loop that claims the next `pending` task, sets it to `processing`, simulates work with `time.sleep(0.1)`, then sets it to `done`. Run the worker and print the count of tasks in each status bucket every 5 iterations. Confirm all 50 reach `done`.

2. **Implement** write-ahead durability on the queue from Exercise 1. Reopen `queue.db` with `PRAGMA journal_mode=WAL` and `PRAGMA synchronous=NORMAL`. Modify the worker so that the status transition from `pending` to `processing` and the transition from `processing` to `done` each happen in their own explicit `BEGIN`/`COMMIT` transaction. Before each commit, print the transaction intent (e.g., `TX: task 14 → processing`). Kill the process mid-run with `Ctrl+C`, then restart it. Verify the worker picks up only tasks still in `pending` or `processing` — no task is lost and no task is double-counted in the final `done` tally.

3. **Trace** the interaction between queue depth and rate limiting under a controlled crash. Seed `queue.db` with 200 tasks. Add a token-bucket rate limiter to your worker that allows no more than 5 claims per second. Halfway through processing (after roughly 100 `done` tasks), simulate a crash by raising `SystemExit` inside the worker loop. Restart the worker and confirm: (a) no `processing` task was orphaned after restart, (b) total wall-clock time is consistent with the rate limit across both runs, and (c) all 200 tasks eventually reach `done`. Print a summary line: `[recovery] resumed at task N, rate-limit honored, total elapsed: Xs`.

4. **Apply** at-least-once delivery semantics to a Clay-style enrichment waterfall. Seed the queue with 30 tasks where each payload contains a company domain. Write a worker that calls a mock enrichment function (returning a fabricated employee count and industry) and writes the result to an `enrichment_results` table. Introduce a 10% random failure rate in the enrichment function that simulates a transient API error — on failure, the task stays in `processing` and the worker retries it on the next loop. After all tasks complete, query for any domains that appear more than once in `enrichment_results` and print them as `DUPLICATE: <domain> (count=N)`. Then add idempotency: before inserting a result, check whether the domain already exists and skip if so. Re-run and confirm zero duplicates. Print the before-and-after duplicate counts.

5. **Evaluate** and **build** a production-grade durable queue handler for an Apollo lead-export webhook. Create `handlers/apollo_webhook_queue.py` that: (a) defines a `receive_webhook(payload: dict)` function that validates the payload contains `leads` (a list of dicts with `email`, `company`, `title`) and enqueues each lead as a task in a SQLite queue (`queue.db`), (b) defines a `process_queue()` worker that pulls tasks, calls a mock CRM-sync function, and handles both success and failure with proper checkpoint transitions, (c) includes a `--replay` CLI flag that resets any `processing` tasks back to `pending` for crash recovery, and (d) logs every state transition to `outputs/skill-queue-durability.md` in a human-readable audit format (timestamp, task ID, old status, new status, result). Seed it with a mock webhook payload of 25 leads, run the worker, interrupt it once, run `--replay`, run the worker again, and confirm the audit log in `outputs/skill-queue-durability.md` shows the full lifecycle including the recovery.
