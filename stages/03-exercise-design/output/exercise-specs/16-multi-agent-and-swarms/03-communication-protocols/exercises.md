# Exercises — Communication Protocols

## Exercises

1. **Implement a request/response exchange** between a "prospect agent" and a "lookup agent" using JSON over standard I/O. The prospect agent sends a JSON request containing a company domain; the lookup agent responds with a JSON object containing `company_name`, `employee_count`, and `industry`. Run both in the terminal and print the full round-trip so the request and response are visible in the output. Verify that malformed JSON on the request side produces a structured error response, not a crash.

2. **Build a JSONL streaming agent** that emits structured progress updates. Write a "research agent" that iterates over a list of five company domains and writes one JSONL line per company to stdout as it processes each one (fields: `domain`, `status`, `timestamp`). Pipe the output to a file and confirm each line is independently parseable JSON by reading the file back and parsing each line. The observable output is the streamed sequence of JSONL records appearing incrementally in the terminal.

3. **Implement an in-process pub/sub broker** with `subscribe(topic, callback)` and `publish(topic, message)` methods. Wire up three agents: a "lead-capture agent" that publishes to a `new_lead` topic, an "enrichment agent" that subscribes and publishes enriched data to `lead_enriched`, and a "notification agent" that subscribes to `lead_enriched` and prints a summary. Publish three leads through the system and observe the message flow in the terminal. All messages must be JSON-serialized at the publish boundary.

4. **Compare protocol fit for four integration scenarios.** Write a Python script that takes a list of four scenarios — (a) calling a tool exposed by an MCP server, (b) delegating a research task to another autonomous agent, (c) joining a shared agent session with turn-by-turn history, (d) discovering an agent's capabilities from a manifest — and prints a table to the terminal mapping each scenario to the appropriate protocol (MCP, A2A, ACP, or ANP) along with the coupling characteristic that drove the selection (schema coupling, temporal coupling, state coupling, or discovery coupling). The output must be a formatted table visible in the terminal.

5. **Build a Clay enrichment waterfall as a multi-agent pipeline** in `handlers/enrichment_waterfall.py`. Create three enrichment step-agents chained in sequence: (1) a domain-to-company agent, (2) a company-to-contacts agent, (3) a contacts-to-email agent. Each step receives the prior step's output, validates it against a schema you define using `pydantic` or manual `assert`-based checks, and either returns enriched data or passes an empty record to the next step. Map each step-chaining boundary to request/response semantics — each downstream agent acts as a responder to the upstream agent's request. Feed three test domains through the pipeline and print the final enriched record for each. The artifact must land at `handlers/enrichment_waterfall.py` and be runnable as `python handlers/enrichment_waterfall.py`.

6. **Design a communication architecture for a four-agent GTM system** (research, enrichment, scoring, outreach) and document it in `outputs/skill-comm-architecture.md`. For every inter-agent link, specify the protocol primitive (request/response, streaming, or pub/sub), the wire format (JSON or JSONL), the schema contract, and the coupling trade-off you accepted. Include a fallback strategy for each link (what happens when the responder is unavailable or returns invalid data). Conclude with a one-paragraph justification for why your overall architecture favors the coupling level you chose. The artifact must land at `outputs/skill-comm-architecture.md`.
