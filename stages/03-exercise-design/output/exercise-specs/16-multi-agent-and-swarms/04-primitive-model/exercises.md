# Exercises — The Multi-Agent Primitive Model

## Exercises

1. **Implement the four primitives in isolation.** Build a single Python script that defines each primitive — `Agent`, `handoff`, `SharedState`, and `Orchestrator` — as minimal classes or functions. Wire them together so a single orchestrator calls one agent, hands off to a second agent via a shared state object, and prints the final state. Run the script and confirm the terminal shows both agents' outputs and the mutated shared state dict.

2. **Trace control flow through a chain topology.** Using the agent + handoff code from the lesson, add a `trace` list to the shared state and have each agent append its name and a timestamp before returning. Run the chain of three agents and print the final trace. Then modify the same code into a fan-out topology where one source agent hands off to two independent target agents in sequence. Compare the trace lists side by side in your terminal output — note how many times each agent appears and in what order.

3. **Build an enrichment waterfall as a 3-agent chain.** Create three agents: `EmailAgent` (hardcoded email lookup), `PhoneAgent` (hardcoded phone lookup), and `LinkedInAgent` (hardcoded profile URL). Each agent receives a company dict, checks if its target field already exists and is non-null, and only enriches if the field is missing. Each agent must also write a provenance key (e.g., `email_source: "EmailAgent"`) into the dict. Feed in a company record with only a name field and print the fully enriched record plus all provenance keys to the terminal.

4. **Inject a failure and add supervisor rerouting.** Modify your enrichment chain from Exercise 3 so that `PhoneAgent` raises a `ValueError` on 50% of runs (use a random flag). Build a supervisor function that catches the exception, logs the routing decision (`"REROUTING PhoneAgent failure → LinkedInAgent for phone fallback"`), and invokes `LinkedInAgent` as the fallback. Run the pipeline 10 times in a loop, print each run's log, and confirm the supervisor reroute triggers and the final record is still fully enriched on every iteration.

5. **Build a GTM enrichment waterfall with real Clay/Apollo-style data.** Implement a 4-agent chain that enriches a lead: `CompanyDomainAgent` (takes company name, returns a mock domain), `ApolloAgent` (simulates an Apollo person-search API call using the domain, returns a name and title), `ClearbitAgent` (simulates a Clearbit enrichment on the domain, returns company size and industry), and `CRMWriteAgent` (formats the final record and writes it to `outputs/skill-enrichment-waterfall.md` as a markdown table). Each agent must implement null-check fallback: if Apollo returns no person, the record proceeds with `person_found: False` and Clearbit still runs. Include a provenance dict mapping every field to the agent that populated it. Run the pipeline against three test companies and inspect the markdown artifact.

6. **Design and prove a supervisor-based fallback graph.** Build an orchestrator that manages five agents — `DomainAgent`, `ApolloAgent`, `ClearbitAgent`, `PhoneAgent`, `LinkedInAgent` — where each agent can fail (simulate with a random 20% failure rate). The supervisor must log every routing decision to a structured log file and retry a failed agent once before rerouting to an alternative (e.g., if `ApolloAgent` fails, fall back to `LinkedInAgent` for person data). Run 20 enrichment cycles, then write a `handlers/supervisor-routing-handler.py` that reads the log and prints a summary table showing: total runs, success rate, which agents failed most, and which fallback paths were exercised. Confirm the summary table prints to terminal and the handler file is importable.
