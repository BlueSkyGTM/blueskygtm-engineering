## Exercises

1. **Implement** a minimal four-role pipeline (Planner → Critic → Executor → Verifier) in a single Python script. Each role receives a distinct system prompt that states its name, its one responsibility, and what it must *not* do. Run the pipeline on the goal `"Draft a cold-outreach email to a VP of Engineering at a Series B SaaS company"` and print each role's output to the terminal in the format `[ROLE NAME] → {output}`. The Critic must print `ACCEPT` or `REJECT: {reason}`. The Verifier must print `APPROVED` or `REJECTED: {reason}`. Confirm your terminal shows exactly four staged blocks before the final approval line.

2. **Trace** three pre-written failing pipeline logs (create them yourself as string variables in your script — each is 15–30 lines of staged output). One log has a Planner gap (missing step), one has a Critic gap (accepted a plan with an untestable success criterion), and one has a Verifier gap (approved output that does not meet the stated criteria). Write a function `diagnose_role_failure(log: str) -> str` that returns `"PLANNER"`, `"CRITIC"`, or `"VERIFIER"`. Print the diagnosis for all three logs. Verify your function correctly classifies each one.

3. **Compare** a full four-role pipeline against a Planner+Executor-only pipeline on the same task: `"Enrich a list of 5 company names with their tech stack and categorize each as 'enterprise-ready' or 'early-stage'."` Run both configurations, capture the final executor output from each, and print a side-by-side diff of results. Write a brief comment in your script identifying at least one failure mode the subset hits that the full pipeline avoids. Terminal must show both outputs and a `DIFF:` line.

4. **Configure** role-specific system prompts for a GTM enrichment workflow using the Apollo People Search API (or a mock function returning sample company JSON if no API key is available). The Planner must list enrichment steps; the Critic must check whether each step maps to a real Apollo endpoint field; the Executor must call the API and format results; the Verifier must confirm every requested field is populated or explicitly marked `null`. Run the pipeline on a target company name and print each stage. Confirm the Verifier catches at least one missing or null field.

5. **Build** a reusable four-role account-research pipeline at `handlers/account_research_pipeline.py`. The pipeline accepts a company domain, runs Planner (produce a 4–6 step research plan), Critic (validate steps are testable), Executor (perform research using any combination of public APIs, scraped data, or mock data functions), and Verifier (check all plan steps have outputs). The script must log each stage to `outputs/account-research-logs/{domain}.txt` and print a JSON summary to the terminal. Include a `if __name__ == "__main__"` block that runs on `stripe.com` as the default argument.

6. **Design** a role-assignment document at `outputs/skill-role-assignments.md` for a three-stage GTM workflow: (1) score 100 leads by ICP fit, (2) generate a personalized opener for the top 20, (3) verify openers don't reference incorrect facts. For each stage, specify which roles to activate, which to skip, and justify the tradeoff in terms of latency, cost per run, and risk of incorrect output. Include a table comparing two configurations: "maximum quality" (all four roles at every stage) vs. "balanced" (selective role activation). Write at least 50 lines of markdown. Print the file path to the terminal when complete.
