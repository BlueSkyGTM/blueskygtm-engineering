# Exercises — Consensus and Byzantine Fault Tolerance for Agents

## Exercises

1. **Trace the three PBFT phases with a Byzantine primary.** Run the lesson's PBFT simulation with node N0 designated as the Byzantine primary. Modify the Byzantine node so it sends `PRE-PREPARE(value=A)` to N1 and N2 but `PRE-PREPARE(value=B)` to N3. Print each node's received pre-prepare, the prepare messages it collects, and its final committed value. Verify that all honest nodes commit the same value. Your terminal output should show the full message trace and a final line like `COMMITTED: {N1: A, N2: A, N3: A}`.

2. **Compute fault-tolerance thresholds for PBFT vs Raft.** Write a Python script that takes `f` (number of faults to tolerate) as a command-line argument and prints the minimum cluster size required for PBFT (`n ≥ 3f + 1`) and for Raft (`n ≥ 2f + 1`). Run it for `f = 1, 2, 3, 5, 10`. Then extend the script to compute, for a fixed cluster size of `n = 7`, how many Byzantine faults PBFT tolerates vs how many crash faults Raft tolerates. Print both numbers.

3. **Detect a Byzantine agent by diffing commit logs.** You are given four enrichment agents that each independently process the same 10 leads. Three agents are honest; one (`Agent-2`) is Byzantine and intermittently writes conflicting firmographic data (e.g., employee_count of 500 for a lead that all other agents record as 5000). Each agent writes its commit log to a CSV at `outputs/agent-N-commit-log.csv` with columns `lead_id, company, employee_count, revenue, source`. Implement a Python function that reads all four CSVs, diffs the `employee_count` and `revenue` fields across agents for each `lead_id`, identifies which agent is the outlier on the majority of leads, and prints: `BYZANTINE AGENT: Agent-2, conflicts: 7/10`. Generate the four CSVs yourself using a script that injects deterministic conflicts.

4. **Apply quorum-based signal reconciliation to a GTM enrichment waterfall.** Build a Python module that queries three enrichment providers for a given domain — Apollo (company size, industry), Clearbit (revenue band, tech stack), and a custom Clay webhook (ICP fit score). For 20 companies in your target ICP, fetch data from all three sources. Where any single field disagrees across providers (e.g., Apollo says 200 employees, Clearbit says 500, Clay says 350), apply a quorum rule: if two or more providers agree, take that value; if all three disagree, take the median for numeric fields or flag for manual review for categorical fields. Output a reconciled CSV at `outputs/reconciled-enrichment.csv` with columns `domain, employee_count, revenue_band, industry, icp_fit_score, conflict_flag`. At least 3 of the 20 companies must have a conflict that triggers your quorum logic — inject test data to guarantee this.

5. **Implement a complete 4-node PBFT consensus protocol.** Build a standalone Python implementation of PBFT with four nodes (`N0`–`N3`) where one node is configurable as Byzantine. Your implementation must include: (a) a `Primary` election or static assignment, (b) all three phases (pre-prepare, prepare, commit) with proper message types and sequence numbers, (c) a quorum check of `2f + 1` matching prepare messages before entering commit, (d) a Byzantine node that sends conflicting pre-prepares to different replicas. Each node must maintain a local commit log. After running the protocol for 5 consensus rounds (each with a different client request value), print every node's commit log and assert that all honest nodes have identical logs. Save your implementation to `handlers/pbft_consensus.py`. Run it as `python handlers/pbft_consensus.py --byzantine N0` and verify honest agreement.

6. **Design a multi-agent enrichment committee with Byzantine detection and automatic fallback.** Design and implement a system where five enrichment agents (simulating Apollo, Clearbit, ZoomInfo, Clay, and a custom LLM-based agent) each vote on whether a batch of 50 target accounts fits the ICP criteria (employee count > 100, revenue > $10M, in target industry). One agent is Byzantine and votes randomly. Implement a PBFT-style agreement protocol among the five agents: they exchange prepare and commit messages, reach consensus on each account, and produce a single canonical ICP-fit list. After consensus, run a detection pass that diffs each agent's votes against the committed result and flags any agent whose disagreement rate exceeds 20%. Output three artifacts: the consensus results at `outputs/consensus-icp-fit.csv`, the per-agent disagreement report at `outputs/byzantine-detection-report.md`, and the protocol implementation at `handlers/enrichment_committee.py`. Verify that your detection report correctly identifies the Byzantine agent.
