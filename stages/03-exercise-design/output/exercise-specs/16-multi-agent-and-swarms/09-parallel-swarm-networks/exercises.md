# Exercises — Parallel / Swarm / Networked Architectures

## Exercises

1. **Implement a parallel fan-out/fan-in enrichment pipeline.** Given a list of 8 company domains, write a script that launches one concurrent research agent per domain using `asyncio.gather` or a thread pool. Each agent fetches the company's public website, extracts a one-sentence description and estimated employee count, and returns a dict. After all 8 complete, implement a fan-in step that merges results into a single JSON array sorted by employee count, deduplicating any domain that appears more than once. Run the script and print the final sorted array to the terminal. Measure and print wall-clock time so you can compare it against sequential execution in Exercise 5.

2. **Trace control flow through a swarm handoff.** Implement a triage agent that receives one of the following request types: `icp_fit_check`, `competitor_analysis`, or `outbound_draft`. The triage agent inspects the request, prints `[TRACE] triage → <specialist>`, and hands off to the appropriate specialist agent by serializing the request context into a JSON payload and calling the specialist's `handle(context)` method. Each specialist prints `[TRACE] <specialist> received: <payload_summary>` and returns a result string. Feed all three request types through the triage agent and print the full trace output plus final result for each. Hardcode the specialist routing logic — do not use an LLM call for triage.

3. **Build a networked pipeline using a shared event bus.** Implement an in-memory pub/sub event bus with `publish(event_type, payload)`, `subscribe(event_type, handler)`, and a `run()` method that processes the queue. Create three agents: (a) a **LeadIngestor** that publishes a `new_lead` event with a name and company, (b) an **EnrichmentAgent** that subscribes to `new_lead`, simulates an enrichment lookup, and publishes `lead_enriched` with appended data, and (c) a **QualificationAgent** that subscribes to `lead_enriched`, scores the lead, and publishes `lead_qualified` with a score. No central orchestrator — the event bus drives all execution. Publish 5 `new_lead` events and print every event that flows through the bus in order, showing the full chain reaction per lead.

4. **Diagnose a race condition on shared state.** Implement a parallel fan-out that distributes 10 lead-scoring tasks across concurrent agents, all writing their results into a single shared Python list using `threading.Thread`. Run it multiple times. Print the list length after each run. If the length is not consistently 10, you have reproduced a race condition. Add a synchronization mechanism (lock, queue, or thread-safe collection) and re-run 10 times to prove the fix produces deterministic results. Print a summary table showing the count for each of the 10 runs before and after the fix.

5. **Compare all three architectures on a real GTM enrichment task.** Implement the same enrichment workflow — pull 12 leads from the Apollo People API (or a local CSV of Apollo-exported leads if you don't have an API key), enrich each with a research agent, and score — using three separate scripts: one parallel fan-out/fan-in, one swarm with a triage agent handing off to an enrichment specialist and then a scoring specialist, and one networked event-bus pipeline. Run all three and record wall-clock time, number of LLM or API calls, and any failures observed across 3 runs each. Write the comparison results, including which architecture handled a deliberately injected failure (one lead with an unreachable domain) best, to `outputs/skill-architecture-comparison.md`. The markdown must include a results table, a prose section on tradeoffs, and a recommendation for which architecture to use at 100 leads vs. 10,000 leads.

6. **Design a multi-agent system for a complex GTM workflow and justify your architecture choice.** Scenario: your team needs to monitor 50 target companies for hiring signals, trigger competitive analysis when any target posts a job for a role matching your ICP, and draft a personalized outreach email to contacts at that company when the competitive analysis flags them as a strong fit. This requires persistent monitoring, conditional multi-step agent chains, and human-in-the-loop approval before sending. Write a design document to `outputs/skill-gtm-design.md` that specifies which architecture (parallel, swarm, networked, or a hybrid) you would use for each subsystem, includes an architecture diagram in ASCII, describes how you would handle the three failure modes from the lesson (race conditions, deadlocks, divergent parallel results), and explains where the human approval checkpoint fits in the control flow. Then implement a minimal proof-of-concept of the core routing logic — the part that decides whether a hiring signal should trigger competitive analysis — and print a trace demonstrating correct routing for 3 sample inputs: one that should trigger, one that should not, and one edge case of your choice.
