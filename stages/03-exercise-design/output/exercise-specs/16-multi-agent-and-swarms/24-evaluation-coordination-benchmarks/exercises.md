# Exercises — Evaluation and Coordination Benchmarks

## Exercises

1. Build a golden dataset of at least 12 labeled examples as a JSONL file — each record containing a text field (e.g., a support ticket subject line) and a `label` field with one of three classes (`billing`, `bug`, `feature_request`). Write a scorer function that takes a list of `(prediction, ground_truth)` pairs and prints a per-example score table (1 for match, 0 for miss) followed by aggregate accuracy. Run it against a set of mock predictions where you deliberately get 3–4 wrong and confirm the output shows both the per-example misses and the final accuracy number.

2. Implement precision, recall, and F1 from scratch — no `sklearn`, no `collections.Counter` shortcuts beyond what you write yourself. Create a small inline dataset of 20 predictions across three classes with intentionally imbalanced errors (e.g., the model over-predicts one class and under-predicts another). Compute per-class precision, recall, and F1, then print a formatted table. In the same run, print which class has the lowest F1 and by how much it trails the best class.

3. Export 30–50 lead records from a CRM (HubSpot, Salesforce, or an Apollo export) or construct a realistic CSV with columns like `company`, `employee_count`, `industry`, `title`, `snapshot_score`. Hand-label each record with a disposition (`qualified`, `nurture`, `disqualify`). Build an operational benchmark: load the data, run a simple rule-based classifier over it (e.g., employee_count > 50 and title contains "VP" → qualified), score predictions against your labels, and print per-class precision, recall, and F1. Report which disposition class the classifier fails on first and what aggregate accuracy alone would have hidden.

4. Simulate benchmark contamination: create a 100-example "static" benchmark set and a separate 50-example "operational" set drawn from a different distribution (different class balance, harder edge cases, noisier text). Run the same classifier over both. Print side-by-side accuracy, per-class F1, and a one-line summary of the accuracy delta between the two sets. Then print which specific failure modes appear in the operational set but are absent from the static set — demonstrate concretely why a static benchmark score alone is insufficient to justify a promotion decision.

5. Implement a coordination gate as a Python module at `handlers/promotion_gate.py`. The gate receives a benchmark result object (per-class precision, recall, F1, aggregate accuracy, sample count) and a threshold config specifying: minimum aggregate accuracy, minimum per-class F1 for every class, and minimum precision on a designated critical class. The gate returns a structured decision — `PROMOTE` or `REJECT` — along with a list of every threshold that was violated. Run the gate against three different benchmark result sets: one that passes cleanly, one that fails on a single per-class F1 threshold, and one that passes aggregate accuracy but fails on critical-class precision. Print the decision and violated-threshold list for each run so the output demonstrates multi-metric agreement in action.
