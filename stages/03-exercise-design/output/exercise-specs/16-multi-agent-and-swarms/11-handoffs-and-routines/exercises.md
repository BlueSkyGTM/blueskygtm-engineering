## Exercises

1. **Implement** a `run_routine(payload, agents)` function that accepts a dict payload and a list of `(name, fn)` tuples. Each `fn` takes the payload, returns a new dict (merged or modified), and the runner passes the accumulated payload to the next agent. After the loop, print the final payload as JSON. Define three dummy agents — `add_timestamp`, `add_status`, `add_owner` — each of which injects one key. Run the routine and confirm all three keys appear in the terminal output.

2. **Apply** an early-exit condition to the runner from Exercise 1. Add an optional `exit_when` parameter that accepts a callable `(payload) -> bool`. If any agent's output satisfies the predicate, stop iteration and return immediately. Create an agent called `fill_phone` that sets `payload["phone"]` to a hardcoded value, and run the routine with `exit_when=lambda p: p.get("phone") is not None`. Print which agents executed (include a `trace` list in the payload) and verify the runner stopped early.

3. **Build** a payload tracer without modifying the runner. Write a decorator `@traced` that wraps any agent function so that, before returning, it appends `{"agent": fn.__name__, "fields_added": [...], "ts": time.time()}` to `payload["_trace"]`. Wrap five dummy agents, run them through your routine, and then write a separate `print_trace_report(payload)` function that prints a readable table showing agent name, fields added, and wall-clock ordering. Confirm the report matches the expected execution order.

4. **Compare** stateless vs. stateful orchestration by implementing the same three-step enrichment logic two ways. First, implement it statelessly using your routine runner. Second, implement it using a class `StatefulRunner` that stores `self.state` and mutates it across method calls. Both should produce the same final payload. Add a `reset()` method to the stateful runner and demonstrate that calling `run()` twice without reset produces a polluted payload (stale keys from the prior run), while the stateless runner produces identical, clean output both times. Print both payloads side-by-side to the terminal as evidence.

5. **Build** a GTM enrichment waterfall in `handlers/enrichment_waterfall.py` that tries multiple data providers in sequence and exits on first success for each field. Model four agents: `apollo_lookup` (mock an Apollo API call that returns a phone number 60% of the time via `random.random()`), `clay_enrich` (mock a Clay webhook that returns an email 40% of the time), `clearbit_fallback` (mock Clearbit that returns both fields at a lower confidence), and `mark_unresolved` (tags remaining gaps as `"unresolved"`). Each agent receives a payload containing `{"company_domain": "acme.com"}` and adds whatever fields it can. Use exit conditions so that once `phone` is filled, no later phone-provider runs, and same for `email`. Run the waterfall 20 times in a loop, print each run's final payload to the terminal, and at the end print a summary: how many runs had complete data, how many had partial, how many were unresolved.
