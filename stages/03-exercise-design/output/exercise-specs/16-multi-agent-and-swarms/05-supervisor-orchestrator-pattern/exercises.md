# Exercises — Supervisor / Orchestrator-Worker Pattern

## Exercises

1. **Implement a minimal supervisor-worker system** with three hardcoded workers (e.g., `summarizer`, `translator`, `fact_checker`). Your orchestrator should accept a task string, decompose it into a JSON subtask plan (each entry naming a worker and its instruction), dispatch each subtask to the matching worker, and print the synthesized result. Run the system with the input `"Summarize the benefits of cold email outreach, translate the summary to Spanish, and fact-check any claims."` and print the decomposition plan, each worker's output, and the final synthesis to the terminal.

2. **Trace the full control flow** of your supervisor-worker system by adding structured logging at each of the five stages (task decomposition, worker selection, execution, aggregation, final synthesis). Each log line should include a stage label, a timestamp, and the payload entering or leaving that stage (subtask plan JSON, worker name, worker result, aggregated dict, final output). Re-run the system from Exercise 1 and verify that the terminal output shows all five stages in order with correctly nested payloads.

3. **Build a GTM lead-enrichment orchestrator** that processes a list of sample company records (hardcode 3 companies with `name`, `domain`, and `industry`). Decompose the task across specialized workers: a `firmographics_worker` that augments industry and company-size signals, an `icp_scorer` that assigns a 0–100 fit score based on industry and size, and a `personalizer` that drafts a one-line outreach hook. The orchestrator should aggregate all worker outputs into a single enriched record per company and print a formatted table to the terminal showing each company's score and personalized hook.

4. **Implement retry logic and error handling** within an orchestration loop. Create a `flaky_worker` that raises an exception 60% of the time (use a random check). Wrap each worker dispatch in a retry loop with a configurable `max_retries` of 3 and exponential backoff (e.g., `0.5 * 2^attempt` seconds). If a worker exhausts all retries, the orchestrator should log the failure, substitute a fallback result, and continue with remaining subtasks. Run the orchestrator 5 times and print each run's retry attempts, failures, and whether the final synthesis completed despite worker errors.

5. **Build a complete GTM enrichment pipeline** that fetches real company data from the Apollo People Search API (or a mock Apollo response if no API key is available) and runs it through a supervisor-worker system. Workers should handle: (1) Apollo data extraction and normalization, (2) ICP scoring against a configurable ideal profile, and (3) Clay-style enrichment augmentation (add tech-stack signals or recent-funding signals from hardcoded reference data). The orchestrator should synthesize a prioritized outreach list sorted by ICP score. Save the full implementation as a importable, runnable module at `handlers/gtm_enrichment_orchestrator.py`. Running `python -m handlers.gtm_enrichment_orchestrator` must print the prioritized list and write a JSON summary to `outputs/enrichment_results.json`.

6. **Design and justify an architecture choice** for a GTM workflow that takes a list of 500 target accounts, enriches each with firmographic data, scores ICP fit, detects buying signals from LinkedIn activity, and generates personalized multi-channel sequences. Write a design document at `outputs/skill-orchestration-design.md` that: (a) specifies the decomposition into subtasks, (b) evaluates supervisor-worker, swarm, and mesh patterns against this problem along at least four dimensions (coordination overhead, latency, context isolation, failure isolation), (c) identifies which subtasks are predictable enough for a deterministic hardcoded pipeline vs. which require an LLM orchestrator, and (d) gives a final architecture recommendation with a diagram in text or Mermaid. The document must be at least 400 words and include a concrete decision rule for when to swap an LLM orchestrator for a hardcoded pipeline.
