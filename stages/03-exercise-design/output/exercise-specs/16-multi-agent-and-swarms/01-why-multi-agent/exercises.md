# Exercises — Why Multi-Agent?

## Exercises

1. **Compare** single-agent versus multi-agent output on a dual-task problem. Write a script that takes five sample company descriptions and processes each through two paths: (a) one LLM call whose system prompt asks the model to both classify the company into a tier (A/B/C) AND write a positioning sentence, and (b) two sequential calls where the first returns only `{"tier": "A", "reason": "..."}` as JSON and the second consumes that JSON to write the positioning. Print both outputs side by side for all five companies. Identify at least one case where the single-agent output degrades — wrong classification, weaker positioning, or conflated reasoning — and note it in a comment.

2. **Trace** the state contract at each boundary of a three-agent enrichment chain. Given the pipeline: Agent 1 (enrich raw lead) → Agent 2 (score fit) → Agent 3 (draft outreach), write a script that defines and prints the exact handoff payload schema between each pair. Output should show: what keys Agent 1 must produce for Agent 2 to start, what keys Agent 2 must add or transform for Agent 3 to start, and which keys from Agent 1 are irrelevant to Agent 3. Use `print()` or `json.dumps()` to render each boundary contract to the terminal.

3. **Implement** a two-agent sequential chain using real lead data from Apollo's API (or a local JSON fixture of three Apollo lead records if no API key). Agent 1 receives a raw lead record and outputs structured JSON with keys: `persona_type`, `likely_pain_point`, `urgency_score` (1–5). Agent 2 consumes that JSON — no fields added, removed, or renamed between agents — and generates a 2-sentence cold email. Run all three leads through the chain and print the final emails along with the intermediate JSON from Agent 1. Verify that Agent 2 never receives the raw lead data directly, only Agent 1's structured output.

4. **Compute** a latency budget analysis for five GTM tasks and determine whether multi-agent architecture is justified for each. For each task, estimate: number of responsibility boundaries, expected latency per agent call (use 2–4 seconds per GPT-4 call as a baseline), cost of a single agent retry on failure versus a multi-agent retry, and whether a failure in step 3 invalidates steps 1–2. Print a table to the terminal with columns: task name, boundaries, total multi-agent latency, single-agent latency, retry cost ratio, and a final recommendation (`multi-agent` / `single-agent` / `conditional`).

5. **Build** a complete three-agent enrichment-to-outreach pipeline that processes a CSV of 10 target accounts. Agent 1 calls an enrichment API (Clearbit, Apollo, or a mock enrichment function) and normalizes the result into a structured company profile JSON. Agent 2 consumes that JSON and produces a scoring JSON with `fit_score`, `buying_signals`, and `recommended_angle`. Agent 3 consumes Agent 2's JSON and writes a personalized LinkedIn connection note. The pipeline must fail gracefully: if any agent's output is malformed, log the error and skip that account. Write the implementation to `handlers/outreach_pipeline.py` and run it so that processed results print to the terminal and a summary report (accounts processed, accounts failed, average fit score) is written to `outputs/skill-multi-agent-pipeline.md`.
