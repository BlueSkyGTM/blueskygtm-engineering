# Exercises — Case Studies and the 2026 State of the Art

## Exercises

1. **Build a metadata-structured case-study index.** Create a JSON file containing at least 8 case-study records, each with fields: `industry`, `employee_range`, `acv_range`, `use_case`, `outcome_label`, `outcome_value`, and `narrative`. Load the file in a Python script, embed each narrative using `sentence-transformers` (or any embedding library), and store all records in an in-memory list of dicts pairing the metadata with the embedding vector. Print each case study's index position, industry, outcome label, and outcome value to the terminal so you can visually confirm the metadata parsed correctly.

2. **Construct a prospect query and retrieve top-k case studies.** Define a prospect profile as a dict with `industry`, `employee_range`, `current_baseline` (e.g., `{"metric": "demo conversion rate", "value": 0.02}`), and `intent_signal` (e.g., "evaluating sales-engagement platforms"). Concatenate the prospect's firmographic and intent text into a single query string, embed it, compute cosine similarity against every case-study vector from Exercise 1, and print the top 3 results ranked by score. Include the similarity score, industry, and outcome metric in each printed line.

3. **Implement the outcome-baseline reranking filter.** Write a function `rerank_by_outcome(retrieved, prospect_baseline)` that takes the top-k results from Exercise 2 and rejects any case study whose `outcome_label` does not match the prospect's baseline metric or whose `outcome_value` is weaker than the prospect's current value. Call it on your Exercise 2 results and print a before/after summary: how many candidates entered the filter, how many survived, and which were rejected with a one-line reason for each rejection.

4. **Compare single-vector and multi-vector representations.** Re-index your case-study library two ways: (a) embedding the `narrative` field only (single-vector), and (b) embedding the `narrative` and a concatenated metadata string (`industry + use_case + employee_range`) as two separate vectors (multi-vector), combining scores with equal weights. Run the same prospect query from Exercise 2 against both indexes and print the top 3 results for each side-by-side. Manually score each result's relevance on a 0–2 scale and print the total relevance score for each approach so you can compare which representation surfaces more contextually appropriate social proof.

5. **Build an agentic decision loop that evaluates case-study impact on outbound copy.** Write a script that takes a draft cold email and a retrieved case study as inputs, sends both to an LLM with a prompt asking it to return a structured JSON verdict: `{"strengthens": true/false, "reason": "...", "suggested_edit": "..."}`. The loop should iterate over the top 3 retrieved case studies for a given prospect, print each verdict, and select the case study with the strongest strengthening signal. Save the final selected case study and the edited email as a Markdown report at `outputs/skill-case-study-rag.md`.

6. **Design an end-to-end pipeline that consumes enriched prospect data and produces case-study-augmented outbound messages.** Fetch 5 prospect records from Apollo's People Search API (or load a provided CSV export of Apollo results if no API key is available). For each prospect, extract firmographics and intent signals, query your case-study index, apply the reranking filter from Exercise 3, run the agentic evaluation loop from Exercise 5, and generate a personalized email that integrates the selected case study as social proof. Save the full pipeline — index construction, retrieval, filtering, evaluation, and email generation — as a runnable script at `signals/examples/case_study_pipeline.py`. Execute the script and print each prospect's name, the case study selected, the filter's accept/reject log, and the final generated email to the terminal.
