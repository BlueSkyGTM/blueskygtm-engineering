# Exercises — Capstone Lesson 38: Classifier Fine-Tuning by Head Swap

## Exercises

1. **Configure a new classification head for a custom label space.** Load `distilbert-base-uncased` via `AutoModelForSequenceClassification` with `num_labels=6`, targeting six GTM-relevant intent categories (e.g., `pricing_inquiry`, `demo_request`, `integration_question`, `complaint`, `churn_risk`, `upsell_opportunity`). Print the model's classification head module and verify that the output projection dimension matches `6`. Then pass a single tokenized example through the model and print the shape of the resulting logits tensor.

2. **Freeze the backbone and audit trainable parameters.** Starting from the model you built in Exercise 1, set `requires_grad = False` on every backbone parameter (embeddings + all encoder layers), leaving only the classification head trainable. Iterate over all named parameters and print two running totals: total parameter count and trainable parameter count. Confirm that trainable parameters account for less than 1% of the model. Print the names of every trainable parameter to verify that only head-layer weights and biases appear.

3. **Train a frozen-backbone head on a small labeled GTM dataset.** Construct a dataset of at least 40 short customer-support messages labeled with the six intent categories from Exercise 1 (you may generate synthetic examples or pull from a CSV). Train only the classification head for 5 epochs using a standard cross-entropy loss and AdamW with learning rate `2e-5`. After training, print the final training loss and the classification accuracy on the training set itself. All output must appear in the terminal.

4. **Compare frozen-backbone versus full fine-tuning on identical data and metrics.** Using the same dataset from Exercise 3, train a second model end-to-end (all parameters unfrozen) for the same number of epochs. Measure wall-clock training time for both runs using `time.time()`. Compute and print a comparison table showing: accuracy, macro-F1, and training time in seconds for each approach. Use `sklearn.metrics` for F1. Print a one-line conclusion stating which approach achieved higher F1 and by what margin.

5. **Export a fine-tuned model and build a batched inference handler for CRM ticket routing.** Save your better-performing model from Exercise 4 to `./models/intent-router` using `save_pretrained`. Then implement a standalone Python script at `handlers/intent_router.py` that loads the saved model, accepts a list of raw support-ticket strings from a JSON file passed as a command-line argument, runs batched inference (batch size ≥ 4), and prints each ticket's predicted label and confidence score. Include at least 5 unseen test messages in your JSON input. Run the handler and capture the terminal output showing predictions.

6. **Diagnose domain mismatch via training loss and validation accuracy curves.** Split your dataset from Exercise 3 into 80% train / 20% validation. Train a frozen-backbone head for 15 epochs, recording per-epoch training loss and validation accuracy. Generate two side-by-side line plots saved to `outputs/skill-domain-mismatch-diagnosis.md` as a Markdown report containing: the plotted curves as inline base64 images or a linked PNG, the epoch at which validation accuracy plateaued, the epoch at which training loss stopped decreasing, and a 3–5 sentence written diagnosis of whether domain mismatch is evident and what evidence supports your conclusion.
