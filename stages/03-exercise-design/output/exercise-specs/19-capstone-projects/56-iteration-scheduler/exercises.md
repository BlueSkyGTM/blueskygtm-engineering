# Exercises — Iteration Scheduler

## Exercises

1. **Implement** an `IterationScheduler` class that accepts `max_iterations`, a `delay_strategy` (a callable returning seconds to wait before the next attempt), and a `convergence_predicate` (a callable taking `(prev_result, curr_result)` and returning a bool). Give it a `.run(step_fn)` method that loops until the bound is hit, the convergence predicate returns `True`, or `step_fn` raises. For the delay, use `time.sleep`. Wire a simple `step_fn` that appends a character to a string each call so the loop visibly converges. Print the final result and the number of iterations executed. Run it with `max_iterations=10` and confirm the terminal shows fewer than 10 attempts when convergence triggers.

2. **Implement** a `similarity_ratio(a, b)` function that computes Levenshtein edit distance between two strings and converts it to a ratio in `[0, 1]` (use the same formula as Python's `difflib.SequenceMatcher`: `2 * matches / (len(a) + len(b))`, but derive it from edit distance). Write a `convergence_predicate` closure factory `make_edit_distance_predicate(threshold)` that returns `True` (converged) when the ratio between two outputs exceeds `threshold`. Wire it into the scheduler from Exercise 1 and print the ratio at every iteration so you can watch it climb toward 1.0.

3. **Compare** three scheduling strategies against the same step function: (a) fixed-bound only — run exactly `max_iterations` times with no convergence check, (b) exponential backoff — delay doubles each iteration starting at 0.1s, convergence active, and (c) pure convergence — no max bound, rely solely on the predicate. For each strategy, record total wall-clock time, number of iterations, and final result. Print a comparison table to the terminal. State which strategy gives the lowest cost-to-quality ratio and why.

4. **Extend** your `IterationScheduler` to emit a structured trace entry on every continue/stop decision. Each entry must be a dictionary with fields: `attempt`, `elapsed_ms`, `result_delta` (the similarity ratio, or `None` on the first attempt), and `reason` (`"first_attempt"`, `"converged"`, `"max_iterations_reached"`, or `"continuing"`). After the run completes, print the full trace as pretty-printed JSON. Verify that a converged run ends with a `"converged"` entry and an unconverged run ends with `"max_iterations_reached"`.

5. **Build** a GTM waterfall enrichment handler that uses the `IterationScheduler` to try data providers in sequence for a list of company domains. Each "iteration" calls a different provider (e.g., a mock Apollo lookup, a mock Clearbit lookup, a mock web-scrape fallback). The convergence predicate returns `True` when a provider returns a non-empty `employee_count` field (early exit). The step function takes the current domain and the provider index, returns a dict, and the scheduler feeds results forward. Process at least 10 domains, print which provider resolved each one, and write the enriched results to `outputs/skill-waterfall-enrichment.json`. Save the handler code to `handlers/waterfall_enrichment.py`. Confirm the output file contains a list of 10 dicts, each with a `resolved_by` field.

6. **Design** a proof-of-concept that demonstrates when convergence-based scheduling fails. Construct a synthetic step function whose output oscillates between two states (never converging) and show that without a max-iterations bound the loop runs indefinitely. Then add the bound and show the scheduler exits gracefully. Write a short analysis to `outputs/skill-convergence-failure.md` explaining two real-world GTM scenarios where oscillation can occur (e.g., an LLM that alternates between two summary phrasings, or an enrichment API whose response flips between cached and fresh data). Include the iteration traces from both runs as evidence.
