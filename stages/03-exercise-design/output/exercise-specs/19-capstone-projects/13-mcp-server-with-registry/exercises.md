## Exercises

1. **Implement a tool registry** as a Python dataclass or dictionary that stores at least five tools with the following metadata fields per tool: `name`, `description`, `input_schema`, `owner_team`, `permission_tier` (one of `public`, `internal`, `elevated`, `restricted`), `cost_units` (integer), and `rate_limit_per_minute`. Register three realistic GTM tools — for example, `apollo_person_search`, `clay_enrich_email`, and `crm_create_contact`. Write a `query_registry(permission_tier=None, max_cost=None)` function that filters the registry by the given criteria and prints the matching tool names as a JSON array to stdout. Verify by running `python registry.py` and confirming that filtering by `permission_tier="internal"` returns only the expected tools.

2. **Implement a governance middleware function** called `intercept_tool_call(registry, caller_id, caller_tier, tool_name, tool_input)` that looks up the tool in the registry, checks whether `caller_tier` meets or exceeds the tool's `permission_tier`, and returns either `{"allowed": True, "tool": tool_name}` or `{"allowed": False, "reason": "insufficient_permissions", "required_tier": ..., "caller_tier": ...}` as a JSON dict. Test it with four calls: a caller with `public` tier calling `apollo_person_search` (allowed), the same caller calling `crm_create_contact` (rejected), a caller with `restricted` tier calling `crm_create_contact` (allowed), and a call to a nonexistent tool (rejected with `reason: "tool_not_found"`). Print all four results to stdout and confirm each decision is correct.

3. **Build an append-only JSONL audit logger** that writes one line per tool-call interception to `outputs/audit_log.jsonl`. Each line must include: `timestamp` (ISO 8601 UTC), `caller_id`, `tool_name`, `decision` (`"permitted"` or `"rejected"`), `rejection_reason` (null if permitted), and `input_hash` (SHA-256 hex digest of the JSON-serialized tool input, truncated to 16 characters). Run your middleware from Exercise 2 twenty times across a mix of permitted and rejected calls with different inputs, then write a `replay_log(path)` function that reads the JSONL file and prints a summary table showing total calls, permit count, reject count, and the top rejected tool by frequency. No scaffold is provided — design the log format and hash function yourself.

4. **Compare stdio and StreamableHTTP transports** by configuring a minimal MCP server (with at least two registered tools) under both `stdio.Server` and a StreamableHTTP handler (using the `mcp` Python SDK or equivalent). Start the stdio variant and send a tool-list request through stdin; capture the response. Then start the HTTP variant on `localhost:8080` and send the same request via `curl`. Write the raw response from each transport to `outputs/stdio_response.txt` and `outputs/http_response.txt`. In your terminal, run `diff outputs/stdio_response.txt outputs/http_response.txt` and print the result. Document in a comment at the top of your script which transport preserves per-call session state and which requires external session storage for horizontal scaling.

5. **Deploy a complete MCP server** that combines the registry, governance middleware, audit logger, and human-approval gating. The server must expose at least four tools — two read-only (e.g., `apollo_company_search`, `clay_read_table_row`) and two destructive (e.g., `crm_update_contact`, `clay_overwrite_cell`). Destructive tools must require `permission_tier: "restricted"` AND an explicit `human_approval_token` field in the input; if the token is missing or invalid, the middleware returns `{"allowed": False, "reason": "human_approval_required"}` and logs the rejection. Start the server on a StreamableHTTP endpoint. Make five `curl` calls: two read-only calls (one with `internal` tier, one with `public` tier), one destructive call without an approval token, one destructive call with an invalid token, and one destructive call with a valid token. Confirm that the audit log at `outputs/audit_log.jsonl` contains all five entries with correct decisions. Save the server implementation as `handlers/governed_mcp_server.py`.

6. **Design a governance policy document** for a fictional GTM engineering team that has access to twelve tools spanning Apollo prospecting, Clay enrichment, CRM writes, Slack notifications, and a billing API. Define the permission tiers, per-tool rate limits, cost budgets per tier, and the escalation path for restricted operations. Specify which operations require human approval tokens and what constitutes a valid token issuance flow. Save the document as `outputs/skill-governance-policy.md`. Then write a companion Python script that loads this policy from the markdown (or a structured sidecar JSON you define) into a live registry, prints the full registry to stdout as a formatted table, and validates that every tool has all required metadata fields — exiting with code 1 and an error message if any field is missing.
