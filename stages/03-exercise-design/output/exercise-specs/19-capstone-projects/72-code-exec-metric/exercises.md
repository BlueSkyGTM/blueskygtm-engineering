# Exercises — Code Exec Metric

## Exercises

1. **Implement a subprocess-isolated runner that executes a Python file and prints a structured metrics line.** Write a function that takes a file path, runs it via `subprocess`, records wall-clock duration using `time.perf_counter()`, captures exit status and stderr, and classifies the error type (e.g., `TimeoutExpired`, `CalledProcessError`, clean exit). Create three small Python files — one that exits cleanly, one that raises `ValueError`, one that sleeps past a 1-second timeout — and run your function against all three. Output must be three JSON lines printed to stdout, each containing `file`, `duration_ms`, `exit_status`, and `error_type`.

2. **Build a metrics aggregator that consumes the JSON lines from Exercise 1 and reports summary statistics.** Write a script that reads N runs (re-run your runner 20 times across the three files to generate data) and computes: overall success rate, p50/p90/p99 latency in milliseconds, and a frequency table of error types. Print a formatted summary block to the terminal showing each metric with its value.

3. **Implement a pass-at-k scorer for code-exec tasks.** Given a directory of markdown files (each containing a problem statement followed by k=3 sampled Python solutions inside fenced code blocks), write a script that extracts each code block, executes it against a provided assertion harness (a separate `.py` file importing the candidate function and running assert statements), and computes `pass@k` — the fraction of problems where at least one of the k samples passed. Create two sample problem files yourself to validate. Print per-problem results and the aggregate pass@k score.

4. **Apply execution metrics to detect degradation in a GTM enrichment step.** Write a script that calls the Apollo People Search API (or a local mock endpoint returning JSON) 30 times in sequence, recording duration and HTTP status for each call. Compute the rolling success rate over windows of 5 calls and compare against two thresholds: a warning threshold (95% success) and a critical threshold (85% success). Print a timeline showing each window's success rate and flag any window that breaches a threshold with `[WARNING]` or `[CRITICAL]` annotations.

5. **Build a multi-dimension pipeline monitor that instruments all four observable dimensions and emits a persistent report.** Implement a runner that executes a Clay enrichment webhook payload against a mock CRM endpoint 50 times, collecting per-run duration, outcome (success/failure), error class, and throughput (requests per second). Produce a markdown report at `outputs/skill-exec-metrics.md` containing: a summary table with all four dimensions, a degradation verdict (healthy/degraded/critical) derived from threshold logic you define and justify in the report, and a recommendation for which single dimension an on-call engineer should investigate first if the verdict is not healthy.

6. **Design and prove a threshold-tuning methodology for execution metrics on a synthetic dataset.** Write a script at `signals/examples/threshold_proof.py` that generates 500 synthetic run records with injected degradation patterns (e.g., latency drift after run 200, error bursts after run 350). Implement three alerting strategies — static thresholds, rolling-window z-scores, and exponential moving average — and print a comparison table showing detection latency (how many runs after the injected fault before each strategy fires) and false-positive count. Conclude in terminal output which strategy you would deploy for a production enrichment pipeline and why.
