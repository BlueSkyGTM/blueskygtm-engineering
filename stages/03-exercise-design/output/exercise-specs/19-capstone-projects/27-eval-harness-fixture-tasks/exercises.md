# Exercises — Capstone Lesson 27: Eval Harness with Fixture Tasks

## Exercises

1. **Define fixture tasks in a portable format.** Create a directory `fixtures/icp_classification/` containing at least five YAML files. Each file must encode an `(input, expected_output, verifier)` triple — for example, a company description as input, an expected ICP label as output, and a verifier name (`exact_match` or `substring_match`). Write a loader function `load_fixtures(path: str) -> list[dict]` that reads every YAML file in the directory and returns a list of parsed fixture dicts. Print the count and first fixture to confirm the load worked.

2. **Implement interchangeable scoring callables.** Write three scorer functions — `exact_match(expected, actual) -> float`, `substring_match(expected, actual) -> float`, and `json_key_presence(expected_keys: list[str], actual: str) -> float` — each returning a score between 0.0 and 1.0. Build a `SCORERS` dict that maps string names to callables so a fixture's `verifier` field can be resolved by name. Write a small test block that passes sample inputs through each scorer and prints the score, confirming each returns the expected 0.0 or 1.0.

3. **Build the harness loop.** Implement `run_eval(fixtures, pipeline_fn, n_runs=1) -> list[dict]` that iterates over fixtures, invokes the provided `pipeline_fn` on each fixture's input `n_runs` times, resolves the scorer by name, applies it, and collects per-fixture results. Use a trivial `pipeline_fn` (e.g., one that echoes the input or returns a canned string) so the output is deterministic. Print a summary table showing fixture ID, scores per run, and the mean score. The harness must accept any callable as `pipeline_fn` without modification.

4. **Compute pass@1 and pass@k.** Extend the harness (or write a separate `compute_pass_at_k(results, k) -> dict`) so that it samples `k=5` runs per fixture and computes pass@1 (did any single run pass at threshold 0.6?) and pass@k (did at least one of the 5 runs pass?). Use a stochastic pipeline callable — one that randomly succeeds ~70% of the time — so pass@k is meaningfully higher than pass@1. Print a table comparing pass@1 and pass@k per fixture and the aggregate rate across the full set.

5. **Emit a structured results report for a CI gate.** Wire the full harness to a realistic GTM pipeline: load enrichment fixtures (company domain → expected industry + employee count) from `fixtures/enrichment/`, run the pipeline against a mock enrichment API callable, score with `json_key_presence`, and write a JSON results report to `outputs/eval-report.json`. The report must include timestamp, fixture count, per-fixture scores, aggregate pass rate, and a boolean `gate_passed` field set to `true` only if the aggregate mean exceeds 0.8. This file is the artifact a CI step or regression tracker consumes.

6. **Design a regression-tracker integration spec.** Write a Python module `handlers/regression_tracker.py` that reads two eval report JSON files (a baseline and a current run), diffs per-fixture scores, identifies regressions (fixtures whose score dropped by more than 0.1), and prints a human-readable summary to the terminal. The module must also exit with code 1 if any regression is detected, so it can serve as a CI gate. Document the file format contract (required fields, types) in a module-level docstring so other teams can produce compatible reports.
