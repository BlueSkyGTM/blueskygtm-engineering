# Exercises — Distributed Data Parallel and FSDP from Scratch

## Exercises

1. **Bring up a process group across 4 ranks with the `gloo` backend on CPU.** Launch 4 processes via `torch.multiprocessing.spawn` (or `mp.spawn`). Each rank must call `dist.init_process_group(backend="gloo", ...)`, print `rank`, `world_size`, and the hostname, then perform an all-reduce sum on a rank-specific scalar tensor. Verify every rank prints `10` (the sum of 1+2+3+4). Call `dist.destroy_process_group()` at the end.

2. **Implement a minimal DDP wrapper class** with two hooks: (a) at construction, iterate every parameter in the model and `dist.broadcast` it from rank 0; (b) register a backward hook on each parameter that calls `dist.all_reduce` on `.grad` and divides by `world_size`. Instantiate the wrapper on a small `nn.Linear(10, 5)` across 4 ranks, run a single forward + backward on random input, and print the gradient norm from each rank. Confirm all ranks report identical values to 6 decimal places.

3. **Compute and compare per-rank memory for DDP versus FSDP.** Write a script that builds a model with `sum(p.numel() for p in model.parameters())` equal to exactly 1,000,000 parameters. For DDP, report per-rank memory as `model_size_bytes`. For FSDP with `world_size` of 2, 4, 8, and 16, compute the per-rank shard size as `ceil(model_size_bytes / world_size)` and print a table showing the replication factor (`shard_size / full_model_size`) at each world size. No distributed launch required — this is a calculation script. Print the table to stdout.

4. **Trace the all-gather and reduce-scatter cycle in a minimal FSDP simulation.** Using 4 ranks, shard a length-16 tensor so each rank holds 4 elements (its shard). Implement the FSDP forward pass: `dist.all_gather` all shards into a full tensor, use it for a computation (e.g., matrix multiply or sum), print the assembled tensor. Implement the backward pass: each rank computes a gradient on the full tensor, then `dist.reduce_scatter` the gradients so each rank ends up with only its 4-element shard of the gradient. Print each rank's final gradient shard and confirm the full-tensor copy is deleted (`print(id(before) != id(after))` or similar).

5. **Implement gradient bucketing on top of your DDP wrapper from Exercise 2 and measure synchronization latency with and without bucketing.** Instead of all-reducing each gradient immediately, accumulate gradients into a flat buffer until the buffer exceeds a configurable `bucket_cap_mb` threshold, then fire a single all-reduce on the entire bucket. Benchmark 20 training steps with `bucket_cap_mb=0` (no bucketing) and `bucket_cap_mb=25` (default PyTorch bucketing) on a model with at least 5 linear layers. Print the mean step latency for each configuration and the percentage improvement. Produce a Markdown report with the latency table and your analysis of why bucketing helps at `outputs/skill-gradient-bucketing.md`.

6. **Build a distributed lead-scoring trainer that shards a contact dataset from an Apollo CSV export across N ranks and synchronizes a shared scoring model with DDP.** Each rank loads a different shard of `outputs/apollo_contacts.csv` (or a synthetic equivalent with columns like `title`, `company_size`, `industry`, `engagement_score`, `converted`). Train a small MLP for 50 steps with your DDP wrapper. After training, each rank must run inference on its local shard and write scored contacts to `outputs/rank_{rank}_scored.json`. Verify that the model weights are identical across all ranks at the end (print `norm` from each rank, confirm match). Artifact lands in `handlers/dist_lead_scoring.py`.
