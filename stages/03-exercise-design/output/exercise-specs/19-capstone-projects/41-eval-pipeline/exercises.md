# Exercises — Capstone Lesson 41: Full Evaluation Pipeline

## Exercises

1. **Implement three independent evaluators.** Write a Python module that defines three evaluator functions — `exact_match(output, reference)`, `schema_validator(output, reference_schema)`, and `keyword_coverage(output, reference_keywords)` — each returning a float in `[0, 1]`. Each evaluator must accept only the model output and its own reference data; none may read another evaluator's result. Run all three against a sample output `{"company": "Acme Corp", "domain": "acme.com"}` with an appropriate reference and print each score to the terminal.

2. **Implement the weighted-mean aggregator.** Write a function `aggregate(scores: dict[str, float], weights: dict[str, float]) -> float` that computes the weighted mean of evaluator scores. Then call it with the three scores from Exercise 1, apply weights `{"exact_match": 0.3, "schema_validator": 0.4, "keyword_coverage": 0.3}`, and print the aggregated score and a pass/fail verdict against a threshold of `0.75`. Verify the output by computing the same weighted mean by hand.

3. **Trace the aggregation logic for a regression scenario.** Given four evaluators with scores `exact_match=0.60`, `factual_accuracy=0.90`, `tone_check=0.85`, `format_compliance=0.40` and weights `0.25, 0.35, 0.15, 0.25` respectively, compute the weighted mean by hand. Then write a script that encodes these values and prints the result along with whether it passes a threshold of `0.78`. Confirm your hand calculation matches the script output.

4. **Calibrate thresholds from a golden dataset.** Create a JSON file containing 10 labeled examples (5 high-quality, 5 low-quality) of Apollo-style lead enrichment output — fields like `company_name`, `domain`, `employee_count`, `industry`. Run each evaluator from Exercise 1 against every example. For each evaluator, compute the score that best separates the high-quality set from the low-quality set (e.g., the midpoint between the lowest high-quality score and highest low-quality score). Write the resulting per-evaluator thresholds to a YAML config file at `signals/examples/calibration_config.yaml` and print the config contents to the terminal.

5. **Build the full pipeline with SQLite regression detection.** Implement a complete DAG pipeline that loads thresholds from the YAML config produced in Exercise 4, runs all evaluators against a new model output, aggregates scores, and compares the result against a rolling baseline stored in a SQLite database at `outputs/eval_baseline.db`. On the first run, store the aggregated score as the baseline. On subsequent runs, compute whether the current score has dropped more than 0.05 below the rolling average of the last 5 runs. Print the current score, the rolling baseline, and either `STATUS: PASS` or `STATUS: REGRESSION DETECTED`. Run the pipeline twice with different model outputs to demonstrate both outcomes. Write the pipeline code to `handlers/eval_pipeline.py`.

6. **Design an evaluator independence proof for a CRM enrichment pipeline.** Write a module at `handlers/crm_eval_pipeline.py` that evaluates enriched CRM records from a Clay export (CSV input with columns `company_domain`, `enriched_industry`, `enriched_employee_count`, `enriched_revenue_range`). Implement at least four evaluators, then include a function `verify_independence() -> bool` that programmatically inspects each evaluator's function signature and asserts no evaluator accepts the output of another. Call this function and print `INDEPENDENCE VERIFIED: True` or `False`. Run the full pipeline against a 20-row Clay export CSV and write a summary report to `outputs/skill-eval-summary.md` containing per-evaluator mean scores, the aggregated score, pass/fail status, and the independence verification result.
