# Exercises — Capstone Lesson 29: End-to-End Coding Agent on the Harness

## Exercises

1. **Run the agent loop end-to-end and capture per-iteration logs.** Using the agent loop built in the lesson, execute it against the provided sample task (`tasks/reverse_string.spec.md`). Your run must print, for every iteration: attempt number, exit code, truncated stdout (first 200 chars), and current token spend. Confirm that the loop terminates in the `Pass` state and that the printed exit code transitions from non-zero to zero across iterations.

2. **Force each terminal condition and verify the state machine exits correctly.** Configure three separate runs of the same agent against a deliberately unsolvable task (e.g., a spec requiring network access inside a sandboxed executor): (a) set `max_retries=2` and confirm the loop exits with `FailMaxRetries` after exactly 3 attempts; (b) set `token_budget=500` and confirm it exits with `FailBudget`; (c) set both limits high and confirm it still terminates via `FailMaxRetries`. Print the terminal state and final stats for each run.

3. **Implement a custom observation parser for structured stderr output.** Write a parser function that extracts a JSON error payload from the executor's stderr (e.g., `{"error": "TypeError", "line": 12, "detail": "unsupported operand"}`) and feeds a structured repair hint into the next generation call. Run the agent against a task with a planted type-error bug and verify that the repair hint appears in the generation prompt for iteration 2 and that the bug is resolved by iteration 3. Print the parsed hint and the resulting code diff between iterations.

4. **Apply the agent to a CRM data-cleaning task.** Point the agent at a spec file that describes reading a CSV of CRM contacts (`examples/crm_contacts.csv`), normalizing phone numbers to E.164, deduplicating by email, and writing cleaned rows to `outputs/cleaned_contacts.csv`. Run the full plan → generate → execute → observe → repair loop without manual intervention. Confirm the output CSV exists, has fewer rows than the input (deduplication worked), and that every phone column matches the regex `^\+\d{6,15}$`.

5. **Build a multi-task evaluation harness that produces a run report.** Extend the agent runner to accept a directory of task specs (`tasks/batch/`) and execute each one sequentially, collecting per-task metrics: task name, attempts used, terminal state, total tokens consumed, and wall-clock time. After all tasks complete, write a markdown summary table to `outputs/skill-agent-eval-report.md`. Include at least five tasks of varying difficulty (string manipulation, file I/O, API mock, error handling, data transformation). The report must render as a valid markdown table when opened.

6. **Design and prove a retry-budget policy that minimizes token waste across a 10-task suite.** Implement a configurable policy module that adjusts `max_retries` and `token_budget` per task based on a difficulty heuristic (e.g., spec line count, number of required tool calls). Run the full 10-task suite three times: once with flat `max_retries=3`, once with flat `max_retries=5`, and once with your adaptive policy. Write a comparison report to `outputs/skill-policy-comparison.md` showing total tokens spent, total passes, and average attempts-per-pass for each strategy. State which policy won and justify why in two sentences at the top of the file.
