# Exercises — Token and Positional Embeddings

## Exercises

1. **Implement a token-embedding lookup via row indexing.** Build a function that takes a list of integer token IDs and returns their corresponding dense vectors from a `(vocab_size, d_model)` matrix. Use `torch.nn.Embedding` or raw `torch.index_select` — your choice. Initialize the embedding matrix with `torch.manual_seed(42)` and `torch.randn(1000, 64)`. Look up token IDs `[7, 203, 999, 7]`. Print the shape of the result and confirm that the first and fourth rows are identical (same token ID, same vector). Also confirm that `result[0]` matches `embedding_matrix[7]` exactly by printing both side by side.

2. **Implement sinusoidal positional encoding from the mathematical formula.** Write a function `sinusoidal_pos_encoding(seq_len, d_model)` that returns a `(seq_len, d_model)` tensor with no learnable parameters. Use the standard formula: for position `pos` and dimension index `i`, `PE(pos, 2i) = sin(pos / 10000^(2i/d_model))` and `PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))`. Compute `sinusoidal_pos_encoding(50, 64)` and print row 0 and row 1. Verify that row 0 contains alternating sin/cos values starting with `sin(0)=0, cos(0)=1` and that row 1 differs. Print `torch.allclose(result[0, 0::2], torch.zeros(32))` — it should return `True`.

3. **Compose token and positional embeddings for a GTM sequence.** You have a vocabulary representing CRM action types: `{0: "page_view", 1: "email_open", 2: "meeting_booked", 3: "deal_created", 4: "deal_closed"}`. Create a `(5, 16)` token embedding matrix and a `(5, 16)` sinusoidal positional encoding for sequence length 5. Add them element-wise to produce the final input tensor. Then reverse the token order (`[4,3,2,1,0]` instead of `[0,1,2,3,4]`) and compute a new input tensor. Print the Frobenius norm of the difference between the two input tensors. Print the cosine similarity between the two representations of position 2 (index 2) in each ordering. The similarity should not be 1.0 — explain in a comment why.

4. **Measure cosine similarity decay across positional distances.** Using the sinusoidal encoding from Exercise 2 with `seq_len=100` and `d_model=64`, compute the cosine similarity between position 0 and every other position (1 through 99). Then do the same starting from position 50. Plot or print the similarities at distances `[1, 2, 5, 10, 20, 50]` from both anchor positions. Verify that nearby positions consistently have higher similarity than distant ones — this confirms the encoding's locality property. Print whether the decay pattern (monotonic or oscillating) is the same for both anchors, and explain in a brief comment what causes any differences.

5. **Build a reusable embedding pipeline for Clay enrichment data.** Create a Python module at `signals/examples/embedding_pipeline.py` that does the following: (a) defines a vocabulary mapping for 20 common enrichment fields (e.g., `"company_name"`, `"employee_count"`, `"industry"`, `"technologies"`), (b) generates a token embedding lookup matrix of shape `(20, 32)` seeded deterministically, (c) produces sinusoidal positional encodings for sequences up to length 50, (d) accepts a list of field-name strings, converts them to token IDs, and returns the composed (token + positional) input tensor. Include a `__main__` block that processes the sequence `["company_name", "employee_count", "industry", "technologies"]` and prints the final tensor shape, the norm of each row, and which field's representation has the highest cosine similarity to `"industry"`. The module must run with `python -m signals.examples.embedding_pipeline` and produce all output to the terminal.

6. **Prove that positional encoding preserves relative position information.** Using only the sinusoidal encoding formula, demonstrate mathematically (in code with printed computations) that the encoding at position `pos + k` can be expressed as a linear function of the encoding at position `pos`. Specifically: compute `PE(pos + k)` and `PE(pos)` for `d_model=8`, `pos=10`, `k=3`. Construct the appropriate rotation matrices `M_k` for each frequency pair such that `[PE(pos+ k, 2i), PE(pos+k, 2i+1)] = M_k @ [PE(pos, 2i), PE(pos, 2i+1)]`. Verify that `torch.allclose` returns `True` for each frequency pair. Save a markdown writeup of your derivation and results to `outputs/skill-positional-proof.md`, including the rotation angle for each frequency dimension and a statement on why this property matters for attention mechanisms.
