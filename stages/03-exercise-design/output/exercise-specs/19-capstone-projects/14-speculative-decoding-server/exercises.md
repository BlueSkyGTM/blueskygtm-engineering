## Exercises

1. **Implement the core speculative decoding loop.** Using a small draft model (e.g., `HuggingFaceTB/SmolLM-135M`) and a larger target model (e.g., `HuggingFaceTB/SmolLM-360M`), build a function `speculative_decode(prompt, K, max_new_tokens)` that runs the draft-then-verify cycle: draft generates K tokens autoregressively, target verifies in a single forward pass, and you resample the first mismatch position. Print the accepted token sequence and the rejection point for each iteration so the terminal shows exactly where the draft was accepted or rejected.

2. **Trace a single speculative iteration step-by-step.** Instrument your speculative decoding function so that for one target forward pass it prints: the K draft tokens, the target model's argmax at each of those K positions, the acceptance bitmask (e.g., `[True, True, False, ...]`), and the resampled token at the first rejection. Feed the prompt `"The key benefit of account-based marketing is"` with K=4 and confirm the trace output shows all intermediate values before the final appended tokens.

3. **Compare token-by-token latency: standard vs. speculative.** Implement both `autoregressive_decode(prompt, max_new_tokens)` and `speculative_decode(prompt, K, max_new_tokens)` using the same draft/target pair from Exercise 1. Run both on five identical prompts of varying length (50, 100, 200, 400, 800 tokens). Measure wall-clock time per generated token using `time.perf_counter()`. Print a table with columns: prompt_length, autoregressive_tok_per_sec, speculative_tok_per_sec, speedup_ratio.

4. **Apply speculative decoding to a real GTM enrichment workload.** Export a CSV of 20 enriched company leads from Apollo (or use a provided `apollo_leads_sample.csv` with company name, industry, and ICP fit score). For each lead, construct a personalized cold-outreach prompt and generate the email body using your speculative decoder. Log the mean draft acceptance rate per lead and the total wall-clock time. Compare against running the same batch through standard autoregressive decoding. Print which leads had acceptance rates below 0.4 and flag them as candidates where speculative overhead may exceed gains.

5. **Implement adaptive fallback for net-negative latency.** Build a wrapper class `AdaptiveSpeculativeDecoder` in `handlers/adaptive_speculative_decoder.py` that monitors a rolling mean of draft acceptance rate over the last N iterations. When the rolling acceptance rate drops below a threshold (default 0.3) or when measured speculative latency per token exceeds autoregressive latency per token, the class dynamically switches to standard autoregressive decoding for subsequent calls. Log every mode switch with a timestamp and the triggering metric. Run the class on a batch of mixed prompts (short technical, long creative, code-generation) and print the mode-switch log to demonstrate the adaptation.

6. **Design and validate draft-model sizing heuristics.** Benchmark speculative decoding across three draft-to-target parameter ratios (e.g., 135M→360M, 135M→1.7B, 360M→1.7B) and four K values (2, 4, 8, 16). For each configuration, measure acceptance rate, tokens-per-second, and net speedup over standard decoding. Write your findings, recommended heuristics for choosing K given a parameter ratio, and the breakeven acceptance rate formula to `outputs/skill-speculative-decoding-heuristics.md`. Include a markdown table of all benchmark results and a decision rule a GTM engineer could follow when selecting a draft model for a latency-sensitive inference endpoint.
