# Exercises — Function Call Dispatcher

## Exercises

1. **Build a registry and exercise it.** Create a Python dictionary called `REGISTRY` that maps at least three string tool names to callable handler functions. Write handlers for `lookup_company` (returns a company name and employee count), `score_lead` (takes a name and score, returns a formatted string), and `get_intent_signal` (returns a fixed intent topic). Print the registry keys, then call each handler directly and print its return value. Run the script and confirm all three handlers execute.
2. **Implement the dispatch function.** Write a `dispatch(tool_call)` function that accepts a dictionary with `name` and `arguments` keys, looks up the handler in your `REGISTRY`, calls it with the provided arguments, and returns a result string. On any failure (missing handler, bad arguments, runtime exception), return an error string prefixed with `ERROR:`. Test with three cases — a valid call, a name not in the registry, and a call missing required arguments — and print each result to the terminal.
3. **Implement the agent loop with a mock LLM.** Build an `agent_loop(messages, mock_responses)` function that iterates through a list of scripted LLM response dictionaries. Each response is either `{"tool_calls": [{"name": "...", "arguments": {...}}]}` or `{"content": "final answer text"}`. For tool-call responses, dispatch each call, append the result as a tool message to the conversation, and continue. For a plain-text response, print the final answer and break. Print each step: which tool was called, what result was injected, and when the loop terminates. Use your `dispatch` function from Exercise 2.
4. **Trace and guard against three failure modes.** Extend your `agent_loop` to detect and handle each of the following: (a) the LLM returns a tool name that is not in the registry — print `DISPATCH ERROR` and inject an error message so the LLM can recover; (b) a handler raises an exception due to bad arguments — print `ARG ERROR` and inject an error message; (c) the LLM keeps calling tools without terminating — implement a `max_iterations` cap (default 5) that prints `LOOP LIMIT REACHED` and exits. Construct a mock response sequence that triggers each guard and run all three through your loop to confirm the correct output.
5. **Build a permission-boundaried dispatcher with GTM tooling.** Create a dispatcher that registers handlers for real GTM operations: an Apollo person search (use the Apollo API or mock the response if no API key), a Clay-style enrichment lookup (mock web data lookup returning company domain and headcount), and a CRM record fetch (read from a local JSON file of mock contacts). Implement a `permissions` parameter on `agent_loop` that accepts a list of allowed tool names; any call to a tool outside that list must be blocked and return `PERMISSION DENIED`. Run two scenarios: one where all tools are permitted and the LLM completes its task, and one where the CRM tool is restricted and the LLM is forced to work without it. Print the blocked call, the error injected, and how the conversation differs. Persist your dispatcher as a runnable script at `handlers/gtm_dispatcher.py`.
6. **Compare hand-rolled dispatch to OpenAI's tool-calling API.** Implement the same three-tool agent two ways: first using your hand-rolled registry-and-dispatch loop with a mock LLM, and second using the OpenAI Python SDK's native `tools` parameter and `tool_calls` response parsing (you may use a mock or real API call). Run the same user query — "Find a person at Acme Corp and enrich their profile" — through both implementations. Write a Markdown analysis to `outputs/skill-dispatcher-comparison.md` that covers: (a) how each approach defines tool schemas, (b) how argument validation differs, (c) where error handling responsibility lives in each, and (d) one concrete scenario where the hand-rolled approach is preferable and one where the native API wins.
