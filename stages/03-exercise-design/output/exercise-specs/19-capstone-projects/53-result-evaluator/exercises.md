# Exercises — Result Evaluator

## Exercises

1. **Implement a rule-based evaluator for cold email subject lines.** Create a function that takes a list of generated subject lines and scores each on three deterministic checks: length between 30–60 characters, presence of at least one personalization token (e.g., `{{first_name}}`, `{{company}}`), and absence of banned spam words (`free`, `guarantee`, `act now`). Return `accept`, `reject`, or `rerank` for each subject line. Print a table to the terminal showing each input, per-dimension pass/fail, and the final decision.

2. **Build a model-based evaluator using embedding similarity.** Compute cosine similarity between each generated outreach email body and a reference "gold standard" email you define. Use any embedding library (e.g., `sentence-transformers`, OpenAI embeddings). Set an acceptance threshold of 0.75 and a rerank threshold of 0.60. Run the same set of email inputs through the rule-based evaluator from Exercise 1's approach (adapted for body text) and print a side-by-side comparison of decisions. Identify at least one case where the two evaluators disagree and note why.

3. **Build a composite evaluator with weighted dimensions.** Define four scoring dimensions — format compliance (rule-based), personalization depth (rule-based), semantic relevance (embedding similarity), and banned-word check (rule-based). Assign each dimension a weight (e.g., 0.2, 0.3, 0.3, 0.2) and compute an aggregate score. Configure thresholds: aggregate ≥ 0.85 routes to `accept`, 0.60–0.84 routes to `rerank`, below 0.60 routes to `reject`. Feed in at least eight sample AI-generated outreach emails and print each email's per-dimension scores, aggregate, and routing decision.

4. **Apply your evaluator to enriched Apollo lead data.** Use the Apollo API (or a provided CSV export of 20 leads with fields like name, company, title, industry) and generate a one-line personalization hook for each lead using an LLM call of your choice. Run each hook through your composite evaluator from Exercise 3. Print the full pipeline trace for each lead: input fields → generated hook → per-dimension scores → aggregate → routing decision. Count how many leads produced `accept` vs. `rerank` vs. `reject` and print the summary.

5. **Implement an asynchronous evaluation pipeline with fallback behavior.** Build a system that processes a batch of 50 AI-generated outreach messages asynchronously (using `asyncio` or a thread pool). The evaluator must wrap each scoring call in error handling: if an evaluator dimension fails (e.g., embedding API timeout, classifier error), log the failure, substitute a fallback score of 0.5 for that dimension, and continue processing. Write a trace log to `outputs/eval-trace.jsonl` with one JSON object per message containing: input, generated output, per-dimension scores, any fallback flags, aggregate score, and routing decision. Print a summary of total accepted/reranked/rejected and count of fallback invocations. Artifact: `handlers/async_evaluator.py`.

6. **Design and implement an LLM-as-judge evaluator with calibration.** Build an evaluator that sends each AI-generated outreach message to an LLM with a structured rubric prompt (covering tone, relevance, specificity, and compliance). Parse the LLM's returned scores, apply a calibration offset learned from a small labeled set of 10 messages you manually score, and produce a calibrated aggregate. Compare the LLM-as-judge decisions against your composite rule/embedding evaluator from Exercise 3 on the same 10-message set. Compute agreement rate and identify systematic disagreement patterns. Write the full comparison report including methodology, per-message scores from both evaluators, agreement analysis, and a recommendation for which evaluator to use in production based on cost, latency, and accuracy. Artifact: `outputs/skill-llm-judge-eval.md`.
