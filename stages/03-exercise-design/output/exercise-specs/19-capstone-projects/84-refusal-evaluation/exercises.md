## Exercises

1. **Implement** the refusal classifier from the lesson. Add three additional keyword patterns: one for hedging language (e.g., "generally," "typically," "in most cases") that indicates `PARTIAL_REFUSAL`, one for moralizing redirects that indicates `SOFT_REFUSAL`, and one for explicit compliance markers that indicates `FULL_COMPLIANCE`. Run the classifier over the five example responses bundled with the lesson. Print each response's predicted label and the matched pattern that triggered it. Confirm that all five labels match the expected taxonomy.

2. **Trace** the classifier's decision path through a set of edge cases you construct manually. Write six short response strings — two hard refusals, two soft refusals, two partial refusals — designed to test different keyword and heuristic triggers than the ones used in the lesson examples. Run each through your classifier and print the label and matched pattern. Verify that no response is misclassified as `FULL_COMPLIANCE` and that at least three distinct trigger patterns fire across the six inputs.

3. **Compute** over-refusal rate, under-refusal rate, and refusal F1 from a labeled evaluation run. Build a test suite of at least 20 prompt-response pairs with expected behavior labels (`SHOULD_COMPLY` or `SHOULD_REFUSE`). Run your classifier on each response. Classify a `SHOULD_COMPLY` pair predicted as any refusal type as an over-refusal, and a `SHOULD_REFUSE` pair predicted as `FULL_COMPLIANCE` as an under-refusal. Print a summary table showing total pairs, over-refusals, under-refusals, precision, recall, and F1. Confirm that at least one over-refusal or under-refusal appears in your output.

4. **Apply** your refusal classifier to responses from a GTM enrichment pipeline. Use sample prompts drawn from three categories — competitive intelligence (e.g., "Summarize this competitor's pricing model"), prospecting (e.g., "Draft a cold email for this ICP"), and market research (e.g., "Estimate TAM for this vertical"). Tag each prompt with an expected behavior label. Fetch or simulate model responses for each prompt, run the classifier, and print a per-category breakdown showing compliance rate, over-refusal count, and under-refusal count. Identify which category has the worst over-refusal rate and print it as a flagged diagnostic line.

5. **Build** a multi-model refusal comparison harness that evaluates refusal behavior across at least two different LLM backends (e.g., two different model versions or two providers). Feed each model the same set of 15 labeled prompts spanning benign, borderline, and policy-violating risk levels. Run your classifier on all responses. Write a per-category and per-model breakdown to `outputs/skill-refusal-comparison.md` containing: a results table with model name, prompt category, risk level, predicted label, and whether the prediction matched the expected behavior, followed by a summary section with each model's over-refusal rate, under-refusal rate, and refusal F1. Print a confirmation line with the output path and the model with the lowest F1.

6. **Diagnose** over-refusal patterns in a simulated GTM outreach pipeline. Construct a 30-prompt corpus covering realistic GTM tasks — Clay enrichment lookups, Apollo prospect queries, CRM data summarization, competitive battlecard generation, and cold email personalization — each tagged with domain and risk level. Some prompts should be benign but phrased in ways that trigger false refusals (e.g., mentioning competitor names, scraping intent, or "extract data"). Run your classifier on model responses for all 30 prompts. Implement a root-cause analysis function that groups over-refusals by domain and identifies the top three keyword or phrase patterns most frequently responsible for false refusals. Write the full diagnostic report — including the grouped over-refusal counts, the top offending patterns, and a recommended remediation for each — to `handlers/refusal_diagnostic.py` as a reusable function that accepts a prompt-response dataset and returns the diagnostic report. Print a summary table to the terminal showing per-domain over-refusal rates.
