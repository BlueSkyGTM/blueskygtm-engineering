# Exercises — Task Spec Format

## Exercises

1. **Construct** a complete task spec as a Python dictionary for a GTM use case: classifying whether a company matches an ICP. Include all seven components (`role`, `objective`, `context`, `constraints`, `input_schema`, `output_schema`, `few_shot_examples`). Print the dictionary, then print the result of `all(k in spec for k in REQUIRED_KEYS)` to confirm every required key is present. Run the script and verify the boolean prints `True`.

2. **Apply** the spec you built in Exercise 1 to a second use case — extracting a contact's seniority level from a job title. Change only `role`, `objective`, `input_schema`, `output_schema`, and `few_shot_examples`; reuse the same `constraints` block. Print both specs side by side (or diff them) and confirm visually which fields changed and which stayed identical. The terminal output should make the shared-vs-divergent components obvious.

3. **Implement** a `validate_task_spec(spec: dict) -> tuple[bool, list[str]]` function that returns `(True, [])` for a well-formed spec and `(False, [list_of_error_messages])` for a malformed one. Reject any spec that: omits a required key, has a non-dict `input_schema` or `output_schema`, contains an empty string for `role` or `objective`, or has fewer than one few-shot example. Test it against four cases: a valid spec, a spec missing `output_schema`, a spec with `role=""`, and a spec with zero few-shot examples. Print each result so a human can scan the output and confirm pass/fail correctness.

4. **Compare** two task spec versions (`spec_v1` and `spec_v2`) on a shared set of 10 input records (use a list of company descriptions or LinkedIn headlines). Version 1 should have no `constraints` field; Version 2 should add explicit constraints (`"output must be valid JSON"`, `"no hedging language"`, `"max 50 words"`). Simulate predictions by calling an LLM (or a deterministic stub function if no API key is available) with each spec against all 10 inputs. Compute and print the count of inputs where the two specs produce different outputs. The terminal output should show: `Inputs changed: N / 10`.

5. **Map** each of the seven task spec components to corresponding Claygent enrichment column fields for a research enrichment workflow (e.g., "Find the primary buying signal for this company"). For each component, identify the Claygent UI field or setting that controls it (prompt instructions, output column type, system prompt, etc.), and explain in one sentence which component controls output parseability and why. Write this as a structured Markdown document at `outputs/skill-task-spec-claygent-mapping.md`. The file must contain a table with columns: Spec Component | Claygent Field | Purpose. Verify the file exists and is non-empty by running `cat outputs/skill-task-spec-claygent-mapping.md | head -20`.

6. **Design** three concrete failure scenarios — one caused by a missing component, one caused by an over-constrained component, and one caused by a contradictory interaction between two components. For each scenario, write a short Python script at `signals/examples/<scenario_name>.py` that constructs the broken spec, runs it against a sample input (real LLM call or stub), prints the output, and prints a one-line diagnosis of the failure mode. Each script must produce terminal output showing the malformed spec JSON and the resulting bad output. Verify all three scripts run without errors (the *specs* are intentionally broken; the *scripts* must execute cleanly) by running `python signals/examples/<scenario_name>.py` for each.
