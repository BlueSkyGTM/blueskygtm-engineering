# Exercises — Hypothesis Generator

## Exercises

1. Implement a function `generate_hypothesis(observation: str)` that calls your local LLM with a structured prompt forcing four fields: `proposed_explanation`, `predicted_outcome`, `disconfirmation_condition`, and `confidence`. Parse the output into a dict and print it. Run it with the observation `"Enterprise SaaS trials in Q3 had 40% lower activation rates than Q2"` and confirm the disconfirmation condition is specific enough that a test runner could execute it.

2. Implement `score_hypothesis(hyp: dict) -> dict` that returns numeric scores (0–1) for falsifiability, specificity, and testability. Falsifiability checks whether `disconfirmation_condition` exists and contains a measurable threshold. Specificity penalizes vague language (e.g., words like "generally," "might," "various"). Testability checks whether `predicted_outcome` references something observable. Run the scorer on two hypotheses you write by hand — one strong, one deliberately weak — and print both score dicts so the difference is visible.

3. Build a temperature-ramped generator `generate_queue(observation: str, n: int = 6)` that produces `n` hypotheses, increasing temperature by 0.2 each pass starting at 0.0. Collect all candidates into a list and print each one with its temperature label. Use a real enrichment-adjacent observation such as `"Apollo data shows 70% of VP Engineering contacts at Series B fintechs changed roles in the last 90 days."` Verify that higher-temperature passes produce visibly different reasoning paths than the temp=0.0 pass.

4. Implement a novelty filter `filter_novel(candidates: list[dict], threshold: float = 0.75) -> list[dict]` that compares each candidate's `proposed_explanation` against all accepted survivors using cosine similarity over TF-IDF or sentence embeddings. Feed it the output from your Exercise 3 queue plus three hand-written near-duplicates you inject. Print which candidates survived and which were rejected, along with the similarity score that killed each rejected candidate.

5. Build a hypothesis re-ranker that ingests new evidence and reorders the queue. Start with 5 hypotheses about why a Clay-enriched outbound campaign had a 12% reply rate (above the 8% baseline). Then feed in a test-run signal: `"Reply rate for companies using Salesforce dropped to 3%, while non-Salesforce accounts stayed at 15%."` Re-rank all 5 hypotheses so that ones whose `predicted_outcome` aligns with the Salesforce signal rise to the top, and ones that are now disconfirmed sink or are removed. Write the final ranked queue to `outputs/skill-hypothesis-rerank.md` with each hypothesis, its old rank, new rank, and a one-line rationale for the move.

6. Design a complete hypothesis-generation pipeline for a GTM churn investigation. Accept a list of at-risk accounts (with enrichment fields from Apollo or Clay — industry, headcount, funding stage, last-touch channel) and generate a ranked queue of 8+ hypotheses explaining churn risk, each with a disconfirmation condition that maps to a query you could actually run against CRM data. Apply novelty filtering and temperature ramping. After a simulated evidence event (e.g., `"3 of 5 churned accounts had no CSM touch in 30 days"`), re-rank the queue. Persist the full pipeline as `signals/examples/churn_hypothesis_pipeline.py` and write the final ranked output to `outputs/skill-churn-hypotheses.md`. The Python file must be runnable end-to-end and produce the markdown artifact on execution.
