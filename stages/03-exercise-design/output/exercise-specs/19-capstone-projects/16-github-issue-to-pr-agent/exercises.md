## Exercises

1. **Implement** the fetch stage of the issue-to-PR state machine. Write a Python script that authenticates to the GitHub API using a personal access token, retrieves a specified issue number from a specified repo (owner/name format), and prints the issue title, body, labels, and a parsed list of acceptance criteria lines (any line beginning with `- [ ]` or `- [x]`). Run the script against a public repo of your choice and verify the terminal output contains all four fields.

2. **Compare** three stopping criteria by running a provided mock agent loop (you will build it from scratch — no starter code) that iterates up to 20 times on a synthetic issue. Implement all three termination conditions: (a) token budget of 5000 tokens consumed, (b) a mock test suite that passes after iteration 7, and (c) an LLM self-classification call that returns `"ready_to_pr"`. Print which stopping criterion fires first and at which iteration. Run the loop three times, shuffling the trigger order, and observe whether the winner changes.

3. **Build** a failure diagnostic tool that ingests a JSON array of agent stage logs — each entry containing `stage`, `status`, `error_message`, and `iteration` — and classifies each failure into one of four buckets: *fetch_rate_limit*, *plan_hallucinated_file*, *edit_diff_conflict*, *propose_test_failure*. For each failure, emit a boolean recommendation field `retry_would_recover` set to `True` only for rate-limit and diff-conflict failures (not for hallucinated files or test failures). Print a summary table to the terminal showing failure count per bucket and total retry-recoverable failures. Test with at least 8 synthetic log entries covering all four buckets.

4. **Configure** a GitHub Action workflow file (`.github/workflows/issue-agent.yml`) that triggers on `issue_comment` events containing the exact string `/agent-fix`. The workflow must: enforce a maximum of 3 API calls per run using a step-level environment variable, run the agent script from Exercise 1, and post a failure comment to the issue via `gh issue comment` if the agent exits non-zero. Include a circuit breaker step that checks a file `.agent-lock` in the repo root — if it exists, the workflow aborts early and posts "Agent circuit breaker tripped." Push the workflow to a test repo, trigger it with a comment, and verify the failure comment appears when the agent script is absent or errors.

5. **Extend** the four-stage agent into a GTM-aware pipeline that integrates with Apollo's people enrichment API. When a GitHub issue is filed, the agent fetches the issue author's GitHub profile, calls Apollo's enrichment endpoint to look up the author's company, and uses the company's employee count to set the agent's risk tolerance: companies with >500 employees get the conservative stopping strategy (test pass/fail required, max 5 iterations), while smaller companies get the aggressive strategy (LLM self-classification, max 3 iterations). Build the full agent in `handlers/issue_to_pr_agent.py`. Run it against a public issue and print: the enriched company name, the selected risk profile, and the final state-machine classification. The artifact must land at `handlers/issue_to_pr_agent.py`.

6. **Design** a proof-of-concept stopping-strategy selector documented in `outputs/skill-stopping-strategy.md`. The document must include: (a) a decision matrix mapping repo attributes — test coverage percentage, CI average runtime, number of contributors, presence of integration tests — to one of the three stopping strategies (token budget, test pass/fail, LLM self-classification); (b) a worked example applying the matrix to two real open-source repos of your choice, with the data you looked up and the strategy the matrix selects; (c) a terminal-runnable Python snippet (included in the markdown) that takes a JSON repo profile and prints the recommended strategy. Run the snippet on both example repos and paste the terminal output into the document. The artifact must land at `outputs/skill-stopping-strategy.md`.
