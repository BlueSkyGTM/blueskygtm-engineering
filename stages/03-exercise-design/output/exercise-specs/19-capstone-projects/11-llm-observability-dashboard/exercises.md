## Exercises

1. **Implement** a `SpanEmitter` class that accepts `(model_system, prompt, completion, input_tokens, output_tokens, latency_ms, metadata)` and appends each record as a JSON line to `outputs/spans.jsonl`. Emit five synthetic spans for a GTM email-drafting workflow (vary the `metadata.workflow` field across `"cold_outreach"`, `"follow_up"`, `"meeting_recap"`), then read the file back and print the count of spans per workflow. Confirm the terminal prints three non-zero counts.

2. **Implement** four pluggable eval functions — `eval_exact_match`, `eval_regex`, `eval_json_schema`, `eval_cosine_sim` — each accepting `(output: str, expected: dict) -> float` and returning a score in `[0.0, 1.0]`. Run all four against a set of six sample LLM outputs you define (two that pass cleanly, two that fail format checks, two that are semantically close but not exact). Print a table showing each output's score across all four evals.

3. **Extend** your `SpanEmitter` so that, after every emit, it automatically runs a list of configured eval functions and stores the resulting scores inside the span under a `"scores"` key. Wire in the four evals from Exercise 2 with appropriate `expected` configurations for a lead-enrichment workflow (e.g., expected JSON schema with `company_name` and `linkedin_url`, regex for a valid URL pattern). Emit ten spans where three outputs are intentionally malformed. Print only the spans with any score below `0.5` and verify exactly three appear.

4. **Compute** token cost attribution and latency percentiles from `outputs/spans.jsonl`. Assign per-model pricing (e.g., GPT-4o at `$2.50/M` input, `$10.00/M` output; GPT-4o-mini at `$0.15/M` input, `$0.60/M` output). Group spans by `metadata.workflow`, then print a summary table with columns: workflow, total_calls, total_cost_usd, p50_latency_ms, p95_latency_ms. Seed the store with at least twenty spans across three workflows and two model systems so the table shows meaningful differentiation.

5. **Build** a rolling-window regression detector that maintains two windows — a baseline of the last N spans before a checkpoint and a current window of the most recent M spans — and compares the mean composite score between them. If the delta exceeds a configurable threshold (e.g., `0.15`), emit an alert line to stdout and append it to `outputs/regression-alerts.jsonl`. Inject a regression by generating fifteen "good" spans (scores > `0.8`) followed by eight "bad" spans (scores < `0.5`) into the emitter, then run the detector and confirm the alert fires within the bounded window. Produce the detector as a persistent artifact at `handlers/regression_detector.py` and print the alert JSON from the file.

6. **Trace** an end-to-end regression scenario combining all components. Configure the emitter with evals for a real GTM enrichment pipeline — simulate Clay-style enrichment responses where each LLM call produces a JSON object with fields `company_name`, `domain`, `employee_count`, and `icp_fit_score`. Run three phases in sequence: (a) emit twenty well-formed enrichment spans, (b) swap in a "regressed" system prompt that produces malformed or semantically drifted outputs and emit twelve more spans, (c) invoke the regression detector, cost attribution, and latency percentile computations. Print a final dashboard summary to the terminal showing: per-workflow cost table, p50/p95 latency, any active regression alerts with the call index where detection triggered, and the five worst-scoring spans. Write the full dashboard script to `signals/examples/enrichment_dashboard.py` and confirm that running `python signals/examples/enrichment_dashboard.py` produces the summary with at least one active alert in under thirty seconds.
