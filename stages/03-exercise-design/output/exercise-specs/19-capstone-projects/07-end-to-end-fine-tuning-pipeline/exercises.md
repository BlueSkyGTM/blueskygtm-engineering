## Exercises

1. **Convert** raw CRM email logs from `data/raw/outreach_logs.jsonl` (fields: `thread_id`, `prospect_name`, `company`, `sent_email`, `replied`, `reply_sentiment`) into two derived datasets: an SFT file with `prompt`/`completion` keys (use only positive-outcome threads) and a DPO file with `prompt`/`chosen`/`rejected` keys (use `reply_sentiment` to pick chosen vs. rejected). Write both files to `data/processed/`. Print the row count and the first record of each file to the terminal so you can eyeball the schema.

2. **Implement** a `validate_contract(path, kind)` function — where `kind` is `"sft"` or `"dpo"` — that opens a JSONL file, asserts the required keys exist on every row, and exits non-zero with a precise error message (`row 14 missing key 'rejected'`) on the first violation. Run it against both files from Exercise 1 and against a deliberately corrupted file you hand-edit. The terminal must show `OK (N rows)` for the good files and a failing exit code for the bad one.

3. **Compare** outputs from three checkpoints — the base model, the SFT adapter, and the DPO adapter (adapters in `models/sft/` and `models/dpo/`) — on a fixed set of five GTM prompts: three cold-outbreak openers to prospects from an Apollo export in `data/apollo_sample.jsonl`, plus two objection-handling replies. Load each checkpoint with PEFT, generate with identical `temperature=0.3` and `max_new_tokens=128`, and print a three-column table (prompt | base | sft | dpo) to stdout. Note where DPO shifts wording relative to SFT.

4. **Configure** a local vLLM (or Ollama) serving endpoint that loads the DPO-merged model and expose it at `http://127.0.0.1:8000/v1/chat/completions`. Then write a `validate_served.py` script that POSTs five outreach-generation requests and asserts every response parses against a Pydantic schema requiring `subject_line` (≤60 chars) and `body` (≥40 chars). Print `PASS`/`FAIL` per request and a final summary line. Mismatched schema outputs must show the validation error.

5. **Build** a single reproducible script at `handlers/e2e_pipeline.py` that runs all four stages end-to-end on Clay-exported prospect data at `data/clay/export.jsonl`: (a) data prep producing SFT + DPO JSONL, (b) SFT via `trl.SFTTrainer`, (c) DPO via `trl.DPOTrainer` initialized from the SFT checkpoint, (d) a smoke-test inference call that prints one generated email for the first Clay prospect. The script must be re-runnable from a clean checkout and produce adapters under `models/sft/` and `models/dpo/`. The final terminal output is the generated email and a per-stage timing summary.

6. **Demonstrate empirically** why SFT must precede DPO. Run DPO directly on the base model (skip SFT) for the same number of steps as stage (c) in Exercise 5, then run SFT-then-DPO on the same data. Generate outputs on the five prompts from Exercise 3 with both checkpoints and compute a simple scoring rubric (format adherence, on-topic keyword overlap with the prospect's industry) by hand or via an LLM-as-judge call. Write a short markdown analysis — including the scores table and a 200-word explanation of the reference-policy initialization mechanism — to `outputs/skill-dpo-ordering.md`. The artifact must exist on disk and contain both the scores and the explanation.
