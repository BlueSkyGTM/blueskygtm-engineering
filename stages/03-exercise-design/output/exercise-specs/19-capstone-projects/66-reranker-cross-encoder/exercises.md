# Exercises — Cross-Encoder Reranker

## Exercises

1. **Compare input shapes and cost profiles.** Load `sentence-transformers/all-MiniLM-L6-v2` (bi-encoder) and `cross-encoder/ms-marco-MiniLM-L-6-v2` (cross-encoder) from the `sentence_transformers` library. For each model, pass a single query and a list of 5 short documents. Print the output shape of each model's prediction tensor and the wall-clock time for the forward pass using `time.perf_counter()`. Confirm that the bi-encoder returns one 384-dim vector per text (enabling pre-computation) while the cross-encoder returns one scalar per query-document pair. Run `python compare_shapes.py` and verify the printed shapes and timings match the expected pattern.

2. **Implement a minimal retrieve-then-rerank pipeline.** Using a corpus of 20 short product descriptions (you may generate them inline or load from a JSON file), encode them with a bi-encoder and build a FAISS or numpy cosine-similarity index. For the query `"wireless noise-cancelling headphones under $200"`, retrieve the top 10 candidates, then rerank those 10 with a cross-encoder and print the reordered top 5. Run `python retrieve_rerank.py` and confirm the top-5 ordering changes after reranking.

3. **Compute MRR and NDCG@5 before and after reranking.** Build a labeled evaluation set: 10 queries, each with 10 candidate documents and binary relevance labels (1 = relevant, 0 = not). Implement MRR and NDCG@k from scratch (no library imports for the metrics themselves). Run the bi-encoder-only retriever and the full retrieve-then-rerank pipeline. Print MRR and NDCG@5 for both approaches. Verify that reranking produces a measurable lift in both metrics.

4. **Apply retrieve-then-rerank to a GTM prospect-filtering task.** Create a dataset of 50 company records (fields: `name`, `description`, `industry`, `employee_count`, `funding_stage`) — you may hardcode these or pull from a sample Apollo/Clay export CSV. Given an ICP query such as `"Series B SaaS company building developer tools"`, retrieve top 20 with a bi-encoder, rerank to top 5 with a cross-encoder, and print the ranked company names with their cross-encoder scores. Run `python gtm_rerank.py` and confirm the output shows five companies sorted by descending relevance score.

5. **Benchmark candidate-set size and model variants against a latency budget.** Write a script that sweeps candidate-set sizes `[5, 10, 20, 50, 100]` and cross-encoder variants `[ms-marco-MiniLM-L-6-v2, ms-marco-TinyBERT-L-2-v2]`. For each combination, measure end-to-end pipeline latency (retrieval + reranking) and NDCG@5 on the labeled evaluation set from Exercise 3. Print a table showing candidate-set size, model, latency_ms, and NDCG@5. Then write a 3-sentence recommendation stating which configuration fits a 200 ms latency budget. Save the script as `handlers/benchmark_reranker.py` and the recommendation as `outputs/skill-reranker-benchmark.md`.

6. **Design and justify a production reranking configuration.** Given a latency budget of 500 ms per query, a corpus of 500 K prospects, and a requirement of NDCG@5 ≥ 0.65, design a three-tier pipeline: (1) a bi-encoder first-stage retriever, (2) a cross-encoder second-stage reranker, and (3) an optional third filtering or re-scoring stage of your choice (e.g., metadata filter, LLM relevance check). Write a proof-style markdown document that specifies the candidate-set size at each stage, the model variant selected, the estimated latency breakdown per stage, and a justification for why NDCG@5 will meet the threshold. Include any assumptions about hardware (CPU vs GPU). Save the artifact to `outputs/skill-reranker-design.md`.
