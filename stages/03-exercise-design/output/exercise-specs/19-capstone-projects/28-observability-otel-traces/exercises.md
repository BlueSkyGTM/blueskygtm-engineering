# Exercises — Capstone Lesson 28: Observability with OTel GenAI Spans and Prometheus Metrics

## Exercises

1. **Build** a `Span` dataclass with fields for `trace_id`, `span_id`, `parent_span_id`, `name`, `kind`, `start_time`, `end_time`, `status`, `events`, and `attributes`. Instantiate three spans: a root span named `"enrichment.run"`, a child span named `"apollo.lookup"` with a parent span ID pointing at the root, and a child of that child named `"llm.score"` carrying `gen_ai.request.model = "gpt-4o"` and `gen_ai.operation.name = "chat"`. Print all three spans and verify the `parent_span_id` chain is consistent from leaf to root by printing the resolved path (e.g., `root -> apollo.lookup -> llm.score`).

2. **Implement** a `JsonlSpanExporter` class with an `export(spans, path)` method that writes one JSON object per line to a `.jsonl` file. Each line must include every span field, with `start_time` and `end_time` serialized as ISO-8601 strings. After writing, read the file back with `json.loads` on each line and assert that `len(records) == len(spans)` and that every record round-trips successfully (no `datetime` objects leak through). Print a summary: `Exported 3 spans → /tmp/spans.jsonl | Roundtrip OK`.

3. **Build** `Counter` and `Histogram` classes. The `Counter` supports `.inc(value=1, **labels)` and the `Histogram` supports `.observe(value, **labels)`. Both classes must produce valid Prometheus text exposition format via a `.render()` method that returns a string containing `# HELP`, `# TYPE`, and labeled metric lines (e.g., `enrichment_requests_total{vendor="apollo",status="success"} 42`). Create a counter named `enrichment_requests_total` with labels `vendor` and `status`, increment it for three different vendors across both success and error statuses, create a histogram named `vendor_latency_seconds` with labels `vendor`, observe five latency values for one vendor, and print the full rendered output to the terminal.

4. **Implement** a `@span(name=..., kind=..., attributes={...})` context manager/decorator that wraps any callable. Inside the context manager, generate a random trace ID if none is active (use a thread-local or module-level current-span variable), generate a fresh span ID, record the wall-clock duration between entry and exit, capture any exception into the span's `events` list with an `"exception"` event, and set `status` to `"ERROR"` when an exception occurs or `"OK"` otherwise. Attach any attributes passed to the decorator plus `gen_ai.*` attributes when the wrapped function returns a dict containing a `"usage"` key. Write a small script that wraps an Apollo-style enrichment function (`def lookup_company(domain: str) -> dict`) and an LLM-call function (`def score_company(profile: dict) -> dict`) in nested spans, then export the resulting spans to `/tmp/nested_spans.jsonl` and print the trace tree.

5. **Build** a complete observability harness for a multi-vendor enrichment waterfall in `handlers/enrichment_observability.py`. The waterfall calls three vendors in sequence — Apollo (company lookup), Clearbit (logo + employee count), and an LLM call to produce an ICP fit score — each wrapped in a span sharing the same trace ID. Wire up a `Counter` for `enrichment_steps_total{step, status}` and a `Histogram` for `step_duration_seconds{step}`. Simulate a failure: Clearbit returns a 503 on one invocation. After the waterfall completes, print the trace tree, the Prometheus text output, and a one-line diagnostic that identifies which step failed and how long the prior steps took before the failure. The artifact must land at `handlers/enrichment_observability.py` and be runnable with `python handlers/enrichment_observability.py`.

6. **Design** a document in `outputs/skill-observability-design.md` that specifies how you would extend this harness to diagnose a production issue where the LLM scoring step intermittently takes 30+ seconds. The doc must cover: which additional spans you would add (e.g., separate spans for prompt construction, API call, response parsing), which new metrics you would instrument (names, types, labels), how the JSONL exporter would be extended to batch-flush every N seconds, and how you would correlate a slow-trace example with the histogram to determine whether the latency is at the network layer or the model layer. Include one concrete worked example with a sample trace tree and a sample Prometheus output block formatted as code fences.
