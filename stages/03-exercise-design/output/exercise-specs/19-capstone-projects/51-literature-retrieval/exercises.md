# Exercises — Literature Retrieval

## Exercises

1. **Implement a BM25 scorer** over a corpus of 10–15 short GTM documents (product descriptions, ICP definitions, case study summaries). Build a function that takes a query string and returns the top-5 ranked document IDs with their BM25 scores. Verify by querying for an exact product name that appears in only one document — that document must rank first. Print results to stdout.

2. **Implement dense retrieval** over the same corpus using a sentence-transformer model (e.g., `all-MiniLM-L6-v2`). Compute cosine similarity between the query embedding and each document embedding, then return the top-5 ranked document IDs. Verify by issuing a synonym query (e.g., "recurring revenue" when the document says "ARR") — dense retrieval should surface the correct document even with zero lexical overlap. Print results side by side with Exercise 1's BM25 output for the same query.

3. **Implement reciprocal rank fusion** from scratch — no library functions. Given two ranked lists of document IDs (from BM25 and dense retrieval), compute fused scores using the RRF formula with `k=60` and return the merged ranking. Run three queries: one where BM25 clearly wins (exact SKU or proper noun), one where dense clearly wins (semantic synonym), and one ambiguous query. Print all three ranked lists (BM25, dense, RRF) for each query and confirm that RRF places at least one relevant document in the top-3 for all three cases.

4. **Evaluate chunk size and overlap effects** on retrieval precision. Take a longer document (2,000+ words — a technical blog post or RFC), chunk it at four configurations: `(256 tokens, 0 overlap)`, `(256 tokens, 64 overlap)`, `(512 tokens, 128 overlap)`, `(1024 tokens, 256 overlap)`. For each configuration, build a hybrid index and issue 5 queries whose answers are known to reside in specific paragraphs. Compute precision@5 and recall@5 for each configuration. Print a summary table showing which chunking strategy maximizes each metric.

5. **Build a CLI-driven hybrid retrieval service** that ingests an entire directory of `.md` or `.txt` files, chunks them, builds both a BM25 index and a dense embedding index, and persists the combined index to disk (pickle, FAISS, or equivalent). The service must accept a query via command-line argument and return the top-k hybrid results. Place the implementation in `handlers/hybrid_retrieval.py`. Verify by running it against a directory of at least 20 documents and confirming that results persist across invocations without re-indexing.

6. **Design a retrieval-augmented enrichment signal** using real GTM data. Export 20–30 company records from Apollo (or use a provided CSV of company domains and descriptions). Build a knowledge base from your own product documentation or competitive intelligence notes. For each company, construct a query from their Apollo description and retrieve the top-3 matching chunks from your knowledge base. Output a ranked signal file at `signals/examples/company_intent_match.py` that maps each company domain to its top-matched internal document chunks with RRF scores, producing a persistent artifact that a GTM team could use to prioritize outbound.
