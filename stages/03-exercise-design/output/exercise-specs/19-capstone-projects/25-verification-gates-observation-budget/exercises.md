# Exercises — Capstone Lesson 25: Verification Gates and the Observation Budget

## Exercises

1. Implement a `VerificationGate` protocol with an `evaluate()` method and build two concrete gates: a `LengthGate` that returns `HALT` if a string field exceeds a max character count, and a `RegexGate` that returns `RETRY` if a phone number field does not match `^\+?[\d\s()-]{7,}$`. Define the decision type as an enum or literal with `PASS`, `RETRY`, `HALT`. Run both gates against three sample records (one that passes both, one that fails length, one that fails regex) and print the decision from each gate so the terminal shows six lines of output.

2. Implement an `ObservationLedger` class that initializes with a numeric budget and exposes a `debit(cost: int) -> bool` method returning `True` if funds were available and `False` if the debit would push the balance below zero. Write a short script that creates a ledger with `budget=100`, debits `30` three times, then attempts a fourth debit of `20`. After each debit, print the remaining balance or the string `BUDGET_EXHAUSTED`. The terminal must show the balance after each call and the final refusal.

3. Build a `GateChain` that accepts a list of gates (each implementing the protocol from Exercise 1) plus an `ObservationLedger`. Its `run(output)` method evaluates gates in order, debiting a fixed per-gate cost from the ledger before each evaluation, and short-circuits on the first non-`PASS` decision. If the ledger is exhausted before a gate runs, return `HALT` with reason `budget_exhausted`. Test the chain with four gates of cost 30 each on a budget of 100 so that the fourth gate never executes. Print the final decision and the remaining ledger balance.

4. Apply the verification gate pattern to enriched Apollo company data. Fetch 10 company records (use the Apollo People Search API or a local JSON fixture exported from Clay). Build three gates: one that checks the company domain resolves to a non-empty string, one that checks `employee_count` falls within an ICP range (e.g., 50–5000), and one that checks a `confidence_score` field exceeds `0.7`. Run all companies through a `GateChain` and print a table showing each company, the stopping gate, and the decision. Companies that fail should show which gate stopped them.

5. Extend your `GateChain` to emit a structured log entry on every gate evaluation containing: `timestamp`, `gate_name`, `decision`, `reason`, `budget_remaining_before`, `budget_remaining_after`. Implement a `FailureClassifier` function that reads these log entries and categorizes each pipeline run's termination as one of `budget_exhausted`, `confidence_shortfall`, `criteria_mismatch`, or `passed`. Run the chain against 20 synthetic agent outputs with deliberately varied failure modes (4 budget exhaustion, 6 confidence failures, 5 criteria mismatches, 5 passes). Save the classifier as `handlers/failure_classifier.py` and print a summary table of failure categories with counts.

6. Design and benchmark three verification strategies on a GTM lead-enrichment pipeline: (a) full-revalidation (every gate runs on every record), (b) sampling-based (gates run only on every Nth record, others pass through), and (c) heuristic-only (replace LLM-confidence gates with cheap regex and length checks). Using 100 Clay-exported company records, run each strategy and record total budget consumed, number of `HALT` decisions, number of `RETRY` decisions, and estimated downstream failure rate (simulate by flagging records that would have failed a held-out full-validation pass). Write a one-page comparison document with a recommendation for which strategy to use when budget is constrained to 500 units. Save the document to `outputs/skill-verification-strategy.md`.
