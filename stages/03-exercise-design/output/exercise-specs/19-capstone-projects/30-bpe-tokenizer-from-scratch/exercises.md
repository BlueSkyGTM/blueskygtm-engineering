# Exercises — BPE Tokenizer From Scratch

## Exercises

1. Implement the core BPE training loop from scratch: given a raw text file as input, convert it to byte sequences, then iteratively count adjacent symbol pairs, identify the most frequent pair, assign it a new token id, and merge all occurrences. Train on a corpus of at least 10,000 characters and print the first 20 merges to the terminal in the format `rank -> (left_id, right_id) -> new_id: count`. Verify your output shows merges like common English bigrams (e.g., `t` + `h`) appearing early.

2. Implement `encode(text) -> list[int]` and `decode(ids) -> str` functions that apply your trained merge table in rank order. Run a round-trip test on five inputs — plain ASCII, a Spanish phrase, a Japanese phrase, an emoji string, and a mix of all four — and assert that `decode(encode(text)) == text` for each. Print each input alongside `PASS` or `FAIL`.

3. Write a script that loads three different prompt phrasings for the same GTM outreach message (e.g., a terse version, a verbose version, and a heavily templated version with liquid-style variables). Encode each with your tokenizer, print the token count per phrasing, then compute the per-message cost and the monthly cost at 50,000 messages/month using a model priced at $0.005 per 1K input tokens. Print a cost comparison table to the terminal.

4. Export 100 rows of prospect data from Clay or Apollo (or use a provided sample CSV with company names, person names, and LinkedIn headlines containing non-Latin scripts and emoji). Write a script that encodes the `company_name` and `person_name` columns with your byte-level tokenizer and also with a character-level tokenizer (one that uses Unicode code points as its alphabet). Print the number of UNK tokens produced by each tokenizer across the dataset. Your byte-level implementation should produce zero UNK tokens; document what the character-level approach fails on and why.

5. Build `handlers/tokenizer.py` as a production-ready module that loads a pretrained merge table from disk (JSON or pickle), exposes `encode`, `decode`, and `estimate_cost(text, price_per_1k)` methods, and includes a `__main__` block that accepts text via stdin and prints the token count, the token id sequence, and the estimated cost. Train a merge table on a corpus of GTM-relevant text (sales emails, product descriptions, ICP definitions), save it to `handlers/merges.json`, and demonstrate the module end-to-end by piping a sample outreach email through it. The module must round-trip correctly and be importable from other handlers.

6. Write a proof-of-correctness artifact to `outputs/skill-bpe-roundtrip.md` that formally argues why byte-level BPE guarantees lossless round-trip for any UTF-8 input. Include three components: (a) a proof sketch showing that the merge table is a bijective function over byte sequences when applied in rank order, (b) an empirical test harness that round-trips 1,000 randomly generated byte strings of varying lengths and reports the pass rate, and (c) a worked example tracing a single non-ASCII input (e.g., `🚀`) through every encode step and every decode step, showing the intermediate id sequence at each merge rank. Run the harness and include the terminal output in the document.
