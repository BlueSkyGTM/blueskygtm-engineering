# Exercises — Paper Writer

## Exercises

1. **Implement** an outline generator that takes a topic string and a total word-count target (e.g., 2000 words) and returns a JSON outline where each section has a `title`, `target_word_count`, `role` (one of `intro`, `analysis`, `synthesis`), and `key_points` (list of 2–4 strings). Validate that the per-section word counts sum to no more than the total target plus 10%. Print the outline as a formatted table and print `BUDGET_OK` or `BUDGET_EXCEEDED` at the bottom.

2. **Extend** the outline from Exercise 1 by adding a `drafting_prompt` field to each section. This field should contain a fully-rendered prompt string that includes the section title, target word count, role, and key points as instructions for a drafting API call. Print each section's `title` alongside the token count of its `drafting_prompt` (use `tiktoken` or an equivalent tokenizer). Confirm that no single drafting prompt exceeds 500 tokens.

3. **Build** a section-by-section drafting loop that consumes the JSON outline from Exercise 1 and generates prose for each section independently. After each section is drafted, pass its final 200 words as `previous_context` into the next section's prompt. Write all sections to `outputs/draft_sections/` as individual `.md` files named `01_intro.md`, `02_analysis.md`, etc. At the end, print each section's filename alongside its actual word count and its target, flagging any section that deviates by more than 15%.

4. **Assemble** the individual section files from Exercise 3 into a single document at `outputs/full_draft.md`, inserting `---` separators between sections. Then **implement** a verification pass that reads the assembled document and reports: total word count, per-section word count, any section that missed its target by more than 15%, and whether the document's total word count is within 10% of the original target. Print a summary table and exit with code 1 if any budget violation is detected.

5. **Design** a LaTeX skeleton generator for a research paper that accepts a JSON outline (same schema as Exercise 1) and produces a `.tex` file at `outputs/skeleton.tex` containing `\section{}`, `\subsection{}`, `\begin{figure}[h]` placeholder slots (one per key point that mentions "figure" or "table"), and `\cite{}` citation keys derived from the key points. Write a companion validation script at `handlers/validate_latex_skeleton.py` that parses the generated `.tex` file and reports: number of sections, number of figure environments, number of unique citation keys, and whether all `\section` titles match the original outline. Run both scripts and print their output.

6. **Apply** the outline-then-fill pattern to a GTM asset: build a competitive battle card generator at `handlers/battlecard_writer.py` that takes a competitor name and your product name, calls an enrichment API (Apollo, Clay, or a public company-data API) to fetch competitor firmographics, generates a JSON outline with sections for Overview, Strengths, Weaknesses, Where We Win, and Talk Track, drafts each section using the forward-context loop, and writes the final battle card to `outputs/skill-battlecard-<competitor>.md`. The script must print the outline, per-section word counts, and the final artifact path. Use real API data for at least the Overview section's firmographics (company size, funding, headquarters).
