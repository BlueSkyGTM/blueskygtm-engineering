## Exercises

1. **Run the baseline pipeline and observe stage timings.** Execute the lesson's `pipeline.py` against `outputs/fixtures/sample_utterance.wav`. Print, for a single utterance, the millisecond timestamps of: (a) ASR final transcript, (b) LLM first token, (c) TTS first audio chunk. Then change the ASR frame size constant from 200ms to 100ms and re-run. Print the latency delta on first-audio-out. Both runs must produce terminal output that shows the timing breakdown.

2. **Verify LLM first-token latency in isolation.** Modify the LLM stage so it streams tokens to stdout the moment they arrive (no batching). Feed it the transcript string `"What does Acme's premium plan include?"` and print each token alongside its relative timestamp in milliseconds from when the prompt was submitted. Confirm first-token arrives under 300ms on your machine; if it does not, document which knob (max_tokens, model, prompt size) you changed to bring it under budget.

3. **Implement partial result forwarding without the lesson's helper.** You are given a mock ASR iterator that yields `{"text": "...", "is_final": bool, "confidence": float}` frames. Write a connector that triggers LLM generation as soon as a partial stabilizes (no text change for 2 consecutive frames) — without waiting for `is_final`. Run it on three sample utterances and print, for each, the gap in milliseconds between LLM-first-token and ASR-is_final. The gap must be positive (LLM started first) on at least two of three.

4. **Add barge-in handling with a shared State object.** Define a `PipelineState` class with a `cancel_event`. Wire the ASR stage to set the event when it detects new speech frames while the LLM or TTS is mid-flight. Confirm the LLM generation aborts and TTS playback stops. Ground the test response in a Clay-exported CSV of company names (`data/clay_accounts.csv`) so the interrupted response references a real account. Print a line at every state transition (`GENERATING → CANCELLED → IDLE`). Run the scenario: user begins asking about Acme, interrupts mid-response with a new question about Globex.

5. **Instrument every stage and produce a latency report.** Add timing probes at: ASR finalization, LLM first token, TTS first audio, and total end-of-speech-to-first-audio. Run the pipeline against five sample utterances of varying length (short greeting, mid-length product question, long multi-part question, etc.). Compute mean and p95 for each stage. Write the report — including pass/fail against the 800ms budget per utterance — to `outputs/skill-voice-pipeline-latency.md`. The markdown must include a table the student can diff against a known-good reference.

6. **Build the retrieval-augmented voice agent.** Index a directory of product docs and case studies (`data/product_docs/`, `data/case_studies/`) using any embeddings approach from prior lessons. Modify the LLM stage so that on receiving the transcript it retrieves the top-3 chunks and injects them into the system prompt before generation. Run end-to-end on five GTM-flavored questions (e.g., "How does Acme handle SSO?", "Tell me about a customer who migrated from LegacyCorp"). Persist the agent module to `handlers/voice_agent_rag.py` and print, for each question, the cited source filename alongside the spoken response — confirming the agent grounds answers in retrieved content rather than parametric memory.
