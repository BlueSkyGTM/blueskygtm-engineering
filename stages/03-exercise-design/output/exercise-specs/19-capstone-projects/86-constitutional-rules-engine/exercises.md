## Exercises

1. Implement a rules engine as a list of rule objects — each with an `id`, a `condition` callable that accepts a string and returns `bool`, a `severity` of `"block"` or `"warn"`, and a `reason` template. Write an `evaluate(text, rules)` function that iterates rules in list order, short-circuits on the first `block` match, and accumulates `warn` matches into a warnings list. Run the engine on three inputs: one that triggers a block, one that triggers two warns, and one that passes cleanly. Print the verdict dict for each.

2. Modify your `evaluate` function so that a `block` rule fires, evaluation stops, and any warnings collected *before* the block are still attached to the verdict under a `pre_block_warnings` key. Construct a ruleset where a warn rule precedes a block rule in priority order. Run an input that trips both. Confirm via terminal output that the block reason appears, the warning also appears, and no rules after the block were evaluated.

3. Build an override protocol. Write a function `authorize_override(verdict, rule_id, reason, authorizer)` that checks whether the verdict was blocked by `rule_id`, appends a bypass record `{"rule_id": rule_id, "reason": reason, "authorizer": authorizer, "timestamp": <ISO 8601>}` to the verdict, sets `override_applied = True`, and returns the modified verdict. If the rule_id does not match the blocking rule, raise a `ValueError`. Demonstrate it on a blocked verdict and on a mismatched rule_id — print both outcomes.

4. Construct an audit trail. Wrap your `evaluate` function in an `evaluate_with_audit(text, rules)` function that computes the SHA-256 hash of the input text, calls `evaluate`, and appends a record `{"input_hash": ..., "rule_id_fired": ..., "verdict": ..., "override": False}` to a JSON-lines file at `outputs/audit.jsonl`. Run five evaluations (mix of block, warn, pass) and inspect the file to confirm each line is a valid audit record. Do not read the file in your script — verify by running `cat outputs/audit.jsonl` in the terminal.

5. Compare rule-as-data enforcement against system-prompt enforcement. Write a system prompt that instructs an LLM to reject inputs containing competitor names and to warn on inputs containing pricing claims. Then write the same two rules as data objects in your engine. Run five crafted inputs — including edge cases like partial competitor names, ambiguous pricing language, and inputs with both issues — through both systems. Record which inputs each approach catches or misses in a markdown table. Write the comparison to `outputs/skill-rules-vs-prompt.md`. Conclude with a one-paragraph analysis of where each approach fails.

6. Design a constitutional rules engine for GTM data quality using enrichment data from Apollo or a mock CRM export. Define at least six rules covering: missing email format, disposable domain detection, job-title blocklist (e.g., "student", "intern"), persona-fit scoring thresholds, account blacklist, and duplicate detection. Each rule must have an `id`, `severity`, `condition` accepting a prospect dict, and a human-readable `reason`. Run a batch of 20 prospects (mix of valid and invalid) through the engine, apply your override protocol to two blocked prospects with documented reasons, and write the full audit trail. Save the engine to `signals/examples/gtm_constitution.py` and the audit output to `outputs/audit.jsonl`. Print a summary: total blocked, total warned, total passed, overrides applied.
