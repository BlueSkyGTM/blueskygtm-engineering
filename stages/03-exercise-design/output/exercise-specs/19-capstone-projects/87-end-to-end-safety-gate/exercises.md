## Exercises

1. **Implement** a two-stage safety gate with short-circuit behavior. Create a `SafetyGate` class that accepts a list of middleware callables, iterates through them, and stops the moment any stage returns a `"blocked"` verdict. Each middleware function takes a `context` dict and returns `{"verdict": "pass"}` or `{"verdict": "blocked", "reason": "..."}`. Add a print statement at the top of each stage so you can observe which stages execute. Run the gate with three test inputs: one that passes both stages, one that fails stage 1, and one that fails stage 2. Confirm from terminal output that stage 2 never runs when stage 1 blocks.

2. **Extend** the gate to four classifier checkpoints arranged in pre-gen / post-gen order: `input_validation` → `intent_classification` → *(LLM call placeholder)* → `output_scanning` → `policy_enforcement`. Insert a mock `generate_response` function between the two phases that simply uppercases the prompt. Trace the full execution for four inputs: a clean request, a malformed prompt, a prompt-injection attempt, and a request whose LLM response triggers the toxicity scanner. Print the final verdict and the stage that caused each failure.

3. **Build** structured JSONL audit logging for the four-stage gate. Every time a stage runs, append a JSON object to `outputs/gate_audit.jsonl` containing: timestamp (ISO 8601), stage name, verdict, reason (if blocked), and a truncated copy of the context dict (first 200 characters of the prompt). After running ten diverse inputs through the gate, inspect the log file with `head` or `jq` and verify that every evaluation—pass or fail—produced exactly one log line per stage that executed. Confirm that blocked requests have fewer log lines than passed requests, proving short-circuit propagation.

4. **Configure** all stage thresholds via an external `gate_config.yaml` file (or JSON if you prefer). Each stage entry must include: stage name, enabled/disabled flag, threshold values (e.g., max prompt length for input validation, toxicity score cutoff for output scanning), and a human-readable description. Load this file at gate initialization so that changing a threshold or disabling a stage requires zero edits to the gate's Python code. Demonstrate the config-driven approach by running the same input twice—once with the toxicity threshold set to 0.3 (blocked) and once set to 0.9 (passed)—using only a config-file change between runs.

5. **Compare** middleware-chain architecture against a monolithic if-else implementation of the same four-stage gate. Write both versions as standalone scripts: `gate_middleware.py` using the composable stage list from previous exercises, and `gate_monolithic.py` using a single function with nested if-else blocks and inline threshold logic. Benchmark both across 100 randomized inputs (vary prompt length, injection markers, toxicity scores) and measure: total wall-clock time, lines of code, and number of edits required to add a hypothetical fifth stage. Write the results—raw numbers plus a 3-paragraph analysis of which architecture is more maintainable for production safety pipelines—to `outputs/skill-safety-gate-comparison.md`.

6. **Apply** the full safety gate to a GTM automation scenario. Build a safety pipeline for outbound email generation that uses Clay-enriched prospect data (or a CSV of enriched leads with fields like `company`, `revenue`, `employee_count`, `contact_email`). The gate must run four stages on each generated email: (1) PII scan—block if the email contains a personal phone number or home address from the enrichment payload, (2) competitive mention check—block if the email references a competitor by name, (3) tone compliance—block if sentiment analysis flags aggressive or desperate language, (4) policy enforcement—block if the email makes a discount promise exceeding configured limits. Write blocked and passed emails to `outputs/gtm_email_audit.jsonl` with the stage verdict, prospect ID, and reason. Save the gate implementation as `handlers/gtm_email_safety_gate.py`. Run it against at least 20 enriched prospect records and print a summary: total emails processed, emails passed, emails blocked, and the most common blocking stage.
