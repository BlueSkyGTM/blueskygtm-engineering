## Exercises

1. **Implement the core search loop.** Build a `run_research_cycle` function that calls planner, executor, and evaluator sub-agents in sequence for a single experiment node. The planner receives a research question string, the executor runs a mock experiment (e.g., a web search or code snippet), and the evaluator returns a confidence score between 0 and 1. Print each sub-agent's output to the terminal so you can trace the full cycle. Run it against the research question *"Does Company X use multi-cloud infrastructure?"* and confirm you see three labeled outputs plus a final confidence score.

2. **Build the lab notebook data structure.** Implement a `LabNotebook` class with methods `add_experiment(hypothesis, plan, observations, confidence)` and `get_context()` that returns a formatted string of all prior experiments. Run three research cycles on the question *"Is accounts-receivable automation a pain point for mid-market SaaS companies?"*, feeding the notebook's accumulated context into each planner call. After all three cycles, serialize the notebook to JSON and print it. Verify that cycle 3's planner output references findings from cycles 1 and 2.

3. **Compare autonomous loop against single-query ReAct.** Implement two agents that answer the same research question: (a) a single ReAct call that reasons, acts, and observes once; (b) your best-first search loop with a confidence threshold of 0.8 or a max of 5 nodes. Run both against *"Does Apollo.io's technographic data cover Kubernetes adoption by company size?"* Print the final confidence score, total token usage, and wall-clock time for each approach. In a comment, state which conditions favored the autonomous loop and which favored the single query.

4. **Add circuit breakers to the search loop.** Extend your research cycle with three termination safeguards: a cost limit (track cumulative token estimates, abort at a threshold you set), a wall-clock budget (abort after N seconds), and a hypothesis-revisit guard (if the planner proposes a hypothesis that matches one already in the notebook above a string-similarity threshold, skip it and log a warning). Run the loop against a deliberately ambiguous question designed to cause repetition — *"What is the best CRM?"* — and confirm via terminal output that the circuit breakers fire before the loop runs indefinitely.

5. **Convert the lab notebook into an ICP validation brief.** Build a handler at `handlers/icp_validator.py` that runs the autonomous research loop against a list of 5 target accounts pulled from a Clay export or Apollo CSV. Each account becomes a research question (e.g., *"Does Acme Corp fit our ICP for data-lakehouse migration services?"*). The planner draws on your company's ICP definition, the executor queries enrichment APIs or scrapes the account's public tech stack, and the evaluator scores ICP fit. After all accounts are processed, write a Markdown brief to `outputs/skill-icp-validation-brief.md` containing each account's score, top evidence items from the notebook, and a recommend/disqualify verdict. Open the file and confirm it has entries for all 5 accounts.

6. **Design a priority function and prove convergence.** Implement a priority function for best-first node selection that combines the evaluator's predicted information gain with a novelty penalty for hypotheses too similar to prior notebook entries. Run the full search loop against a GTM question of your choice for 10 nodes with no confidence threshold, logging each node's priority score at selection time. Plot or print the sequence of confidence scores. Argue in a docstring or Markdown block at the top of `signals/examples/convergence_analysis.py` whether your priority function causes confidence to monotonically increase, and identify the conditions under which it fails to converge.
