## Exercises

1. **Implement a single `kubectl` tool wrapper.** Write a Python function that accepts a bounded kubectl subcommand (e.g. `get pods`, `describe pod <name>`, `logs <pod> --previous`), executes it via `subprocess`, and returns a dictionary with keys `command`, `stdout`, `stderr`, and `returncode`. Reject any command not on an explicit allowlist. Verify by running `kubectl get pods` against a local `minikube` or `kind` cluster (or a mock fixture that returns canned YAML) and printing the structured result to the terminal.

2. **Configure a structured incident-report schema.** Define a Pydantic `IncidentReport` model with fields for `pod_name`, `namespace`, `failure_class` (enum: `CrashLoopBackOff`, `ImagePullBackOff`, `OOMKilled`, `Pending`), `root_cause`, `recommended_remediation`, and `severity` (enum: `critical`, `warning`, `info`). Instantiate the model from a hardcoded sample diagnosis, serialize it to JSON, and print it. Confirm the JSON has every required key by piping the output through `jq .` and verifying no validation errors appear.

3. **Build the ReAct loop for a single failure type.** Implement an agent that receives an alert dict (`{"pod": "api-server-7d4", "namespace": "production", "status": "CrashLoopBackOff"}`), then autonomously calls `get_pods`, `logs --previous`, and `describe pod` in sequence using your tool wrapper. The system prompt must encode the reasoning steps from the decision tree shown in the lesson's Mermaid diagram. The agent terminates when it calls a `submit_diagnosis` function and prints the resulting `IncidentReport` JSON. Test against a synthetic fixture directory `fixtures/crashloop/` containing mock `kubectl` outputs and confirm the agent produces `failure_class: CrashLoopBackOff` with a non-empty `root_cause`.

4. **Extend the decision tree to classify all four failure modes.** Add reasoning instructions and tool-selection logic for `ImagePullBackOff`, `OOMKilled`, and `Pending` to your system prompt. Create four fixture directories under `fixtures/` — one per failure class — each containing the mock `kubectl` outputs an agent would encounter. Run your agent against each fixture and print the classified `failure_class`. All four runs must match their directory name with zero mismatches. No scaffold is provided; design the fixture YAML and the prompt additions yourself.

5. **Build a deterministic evaluation harness.** Implement a Python script that loads every fixture directory under `fixtures/`, runs the agent against each one, and validates the output `IncidentReport` against an expected-results JSON file (`fixtures/<name>/expected.json`) using exact-match on `failure_class` and substring-match on `root_cause`. The harness must print a pass/fail table to the terminal and write a Markdown summary to `outputs/skill-k8s-eval-report.md` containing per-fixture accuracy, agent token usage per diagnosis, and any misclassifications. Run it and confirm the report file is created with at least four fixture rows.

6. **Integrate incident reports with a real PagerDuty webhook.** Extend your `submit_diagnosis` tool so that when `severity == "critical"`, it POSTs the `IncidentReport` JSON to a PagerDuty Events API v2 endpoint (or a `webhook.site` URL for testing) with the correct `routing_key` and `event_action: trigger` payload structure. Add a CLI flag `--dry-run` that prints the webhook payload instead of sending it. Verify by running with `--dry-run` against a `CrashLoopBackOff` fixture and confirming the printed payload contains the diagnosis fields nested inside the PagerDuty event body. Save the webhook integration logic to `handlers/incident-webhook.py`.
