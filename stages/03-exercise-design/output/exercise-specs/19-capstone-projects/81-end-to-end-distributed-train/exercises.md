# Exercises — End-to-End Distributed Training

## Exercises

1. **Run the lesson's multi-rank training loop and verify the ZeRO-1 memory formula.** Launch the training script from the lesson across 4 simulated ranks (N=4) with a model of P=10M parameters. After step 1, compute the expected per-rank memory as `12P/N` bytes and compare it to the value printed by the training loop's memory tracker. The terminal output must show your hand-computed expected value alongside the measured value, and they must match within 1% — print `MATCH` or `MISMATCH` as the final line.

2. **Scale the model and recompute.** Modify the lesson's configuration to use P=40M parameters with N=8 simulated ranks. Before running, write down the expected per-rank optimizer memory. Run the loop and confirm the measured allocation matches your prediction. Then halve N to 4 and re-run — print a table showing P, N, expected bytes, measured bytes for both configurations so you can see the linear scaling in action.

3. **Implement sharded checkpoint resume from scratch.** Build a training loop that (a) runs for 5 steps, (b) saves optimizer state shards to disk — one file per rank — and (c) on restart, loads each rank's shard and reconstructs the full optimizer. After restart, compare the optimizer state tensor for rank 0 against the pre-checkpoint version byte-for-byte using `torch.equal()`. Print `BYTE-EQUAL: PASS` or `BYTE-EQUAL: FAIL`. No scaffold from the lesson — write the save and load logic yourself.

4. **Produce a side-by-side memory and communication comparison of DDP, ZeRO-1, and ZeRO-2.** Implement three training configurations for the same 100M parameter model across 8 simulated ranks: vanilla DDP (replicated optimizer), ZeRO-1 (sharded optimizer), and ZeRO-2 (sharded optimizer + sharded gradients). For each, instrument and print: peak per-rank memory, number of all-reduce calls per step, and volume of data communicated per step in bytes. Output a formatted table and identify which configuration is bandwidth-bound vs memory-bound.

5. **Build a parallelism strategy recommender that persists its recommendation.** Write a Python script that takes three inputs — model parameter count, number of GPUs, and per-GPU memory budget in GB — and outputs a recommended strategy (DDP, ZeRO-1, ZeRO-3, tensor-parallel, or pipeline-parallel) with a one-paragraph justification citing the memory math. Run it for these scenarios: (7B, 8, 80GB), (70B, 8, 80GB), (70B, 64, 40GB), (1.5B, 4, 24GB). Write the full recommendations with justification to `outputs/skill-parallelism-recommender.md`. The file must contain all four scenarios with distinct recommendations and memory calculations.

6. **Design a distributed fine-tuning pipeline for a GTM intent-classification model trained on enriched lead data.** You have 2M rows of Apollo-enriched contact data (job title, company size, industry, technographics) with binary intent labels. The base model is a 1.3B parameter transformer. You have 4 GPUs with 24GB each. Implement the full training loop with ZeRO-1 sharding, sharded checkpointing every 500 steps, and a resume capability. After training completes, load the final checkpoint, run inference on a held-out batch of 100 leads, and print precision, recall, and F1. Save the model card documenting the parallelism strategy, memory profile, and evaluation metrics to `outputs/skill-intent-model-card.md`. The training loop must run end-to-end and the metrics must print to the terminal.
