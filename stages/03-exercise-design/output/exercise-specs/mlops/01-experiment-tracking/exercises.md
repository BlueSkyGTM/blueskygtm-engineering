## Exercises

1. **Configure** an MLflow tracking server backed by SQLite. Set the tracking URI to `sqlite:///mlflow.db`, create an experiment named `lead-score-baseline`, then log a single run with three parameters (`n_estimators=50`, `max_depth=5`, `learning_rate=0.1`), two metrics (`accuracy=0.82`, `f1=0.76`), and one artifact (a text file containing your model summary). Print the run ID to the terminal. Verify by running `mlflow ui --backend-store-uri sqlite:///mlflow.db` and confirming the run appears in the UI.

2. **Query** the experiment you created in Exercise 1 using `MlflowClient.search_runs()`. Sort all runs by the `f1` metric in descending order and print the top run's ID, parameter dictionary, and metric values to the terminal as formatted output. Do not hard-code the run ID — your code must discover it dynamically by ranking.

3. **Implement** a hyperparameter sweep that logs 8 distinct training runs to the same `lead-score-baseline` experiment. Vary `n_estimators` across `[25, 50, 100, 200]` and `max_depth` across `[3, 7]` in a full grid. For each combination, simulate a metric by computing `f1 = 0.6 + 0.001 * n_estimators + 0.02 * max_depth + random_noise` where noise is drawn from `Normal(0, 0.01)`. After all runs complete, query the experiment and print the parameter组合 of the winning run. No scaffold — structure the loop, the logging, and the query yourself.

4. **Register** the best-performing model from your sweep to the MLflow Model Registry. Use `MlflowClient.create_registered_model()` to create a model named `lead-scorer-v1`, then `create_model_version()` to add the winning run's artifact as version 1. Transition the version to the `Staging` stage using `transition_model_version_stage()`. Print the model name, version number, current stage, and source run ID to the terminal. Verify with `mlflow models serve -m models:/lead-scorer-v1/Staging`.

5. **Build** a complete experiment tracking pipeline for a GTM lead prioritization model. Pull 100 sample company records from the Apollo API (or load a local CSV export from Clay containing `company_name`, `employee_count`, `industry`, `tech_stack_count`, `intent_score`). Train a simple logistic regression or gradient-boosted classifier where the target is `qualified` (derived from `intent_score > 0.7`). Log all feature names, model type, hyperparameters, accuracy, precision, recall, and a confusion-matrix plot as artifacts. Run the pipeline 3 times with different `random_state` values, then register the best run to the Model Registry as `gtm-lead-qualifier` version 1 in `Production` stage. Produce a runnable script at `handlers/lead_scorer_tracking.py` that executes end-to-end with `python handlers/lead_scorer_tracking.py` and prints the final registered model version and stage.

6. **Design** a written proof-of-concept that demonstrates the difference between experiment tracking semantics and model registry semantics. Write a script that performs the following sequence: (a) log 3 runs to an experiment, (b) register version 1 from run A and transition to `Production`, (c) register version 2 from run B and transition to `Production`, (d) attempt to re-query run A's metrics and confirm they are unchanged despite the registry transition. Print a table showing each operation, which system it touched, and whether state was appended (tracking) or mutated (registry). Produce the artifact at `outputs/skill-tracking-vs-registry.md` containing your table, a one-paragraph explanation of why the append-only guarantee matters for audit trails in a GTM context (e.g., proving which model version scored a sales lead), and the terminal output from your script.
