# Exercises — Git & Collaboration

## Exercises

1. **Initialize** a Git repository in a new directory called `gtm-ops`. Create a file `enrichment_config.yaml` with at least three fields (`provider`, `api_key`, `rate_limit`). Stage and commit it. Then modify one field value and run `git status`, `git diff`, and `git diff --staged` after staging — capturing output that shows the file passing through the working directory, staging area, and commit graph. Finish by running `git log --oneline --graph` to display the commit history.

2. **Create** a second GTM configuration file in JSON format representing a Clay table export schema with fields such as `company_name`, `email`, `linkedin_url`, and `title`. Commit it using a Conventional Commits message (e.g., `feat: add Clay export schema`). Then change one field name, commit that change, and use `git show HEAD~1:<filename>` to print the prior version alongside the current one so both are visible in your terminal.

3. **Build** a branching scenario from your `gtm-ops` repository. Create a branch `feature/apollo-integration` and add an `apollo_config.yaml` file. On `main`, make a non-conflicting edit to `enrichment_config.yaml`. Merge the feature branch and run `git log --oneline --graph --all`. Next, create a branch `hotfix/rate-limit` and edit the `rate_limit` line in `enrichment_config.yaml` on both branches with different values so a conflict occurs. Attempt the merge, resolve the conflict manually, complete it, and display the final file contents and merge commit graph.

4. **Apply** the branching workflow to a realistic GTM scenario. Assume your repo manages enrichment pipeline configs for both Apollo and Clay providers. On `main`, a teammate has committed a change bumping `rate_limit` from `100` to `200`. Create a branch `feature/switch-provider` that changes the `provider` field from `apollo` to `clay` and updates the corresponding `api_endpoint` URL on the same lines your teammate touched. Merge the branch, resolve the conflict so that both changes survive, and verify the result is valid YAML by running `python -c "import yaml; print(yaml.safe_load(open('enrichment_config.yaml')))"` in your terminal.

5. **Build** a pre-commit hook script at `handlers/validate_gtm_config.py` that validates any staged YAML config file in the repository before commit. The script must check that required fields (`provider`, `api_key`, `rate_limit`) are present and that `rate_limit` is a positive integer. If validation fails, exit with a non-zero status and print a clear error message so the commit is rejected. Install it via `.git/hooks/pre-commit` (make it executable with `chmod +x`). Demonstrate it works by attempting to commit a config missing `api_key` (rejected) and then a valid config (accepted), capturing both terminal outputs.

6. **Design** a branch protection and pull-request workflow specification for a shared GTM operations repository and write it to `outputs/skill-git-workflow.md`. The document must define: required reviewer count, required status checks (e.g., YAML lint, config validation hook from Exercise 5), branch naming conventions tied to GTM tooling (`feature/clay-*`, `hotfix/apollo-*`), commit message conventions, and the chosen merge strategy with rationale. Include a copy-pasteable `pull_request_template.md` section that a GTM ops engineer would fill out when proposing a config change to an enrichment pipeline.
