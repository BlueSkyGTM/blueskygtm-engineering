# Exercises — Debugging and Profiling

## Exercises

1. **Implement a profiling wrapper for a single LLM call.** Write a function `profiled_llm_call(prompt, mock_llm)` that wraps any callable mock LLM, records wall-clock latency, estimates token counts (split on whitespace as a proxy), and captures the raw prompt and raw response. Return a dictionary with keys `prompt`, `response`, `latency_ms`, `prompt_tokens`, `response_tokens`. Call it three times with prompts of different lengths and print all three profile dicts. Verify that latency and token counts vary across calls.

2. **Implement schema-validated retry with failure-signal injection.** Write a mock LLM function that returns valid JSON 70% of the time and malformed JSON 30% of the time. Build a wrapper that validates the response against a Pydantic schema (fields: `company_name: str`, `icp_score: int`, `summary: str`), retries up to 3 times on failure, and injects the previous error message into the retry prompt (e.g., `"Your previous response failed: <error>. Please return valid JSON."`). Print each attempt number, whether validation passed, and the final parsed object.

3. **Trace a non-deterministic failure in a 3-step prompt chain.** Build a chain of three mock LLM calls: (1) classify industry, (2) generate ICP score, (3) format outreach message. Introduce non-determinism by having step 2 randomly return a string instead of a number 20% of the time. Add structured intermediate-state logging that writes the full prompt, raw response, and parsed output for each step to a JSONL file at `outputs/trace.jsonl`. Run the chain 10 times in a loop. After the runs complete, read the trace file and print the step number and raw response for every run that failed. Confirm that the trace file pinpoints which step and which raw response caused each failure.

4. **Compare per-step token costs and latencies for a GTM enrichment workflow.** Build a 4-step enrichment pipeline that processes a list of 5 company names: (1) fetch company description via mock, (2) classify technographics via mock, (3) compute ICP fit score via mock, (4) generate personalized cold email via mock. Make each mock consume different amounts of tokens (vary prompt/response lengths realistically — step 4 should be the longest). Profile every call, then compute and print a table showing per-step totals: total tokens, total latency, cost per step (assume $0.01 per 1K tokens), and percentage of total pipeline cost. Identify and print the bottleneck step.

5. **Build a production-grade debugging handler for an enrichment pipeline.** Combine all three mechanisms — profiling, schema validation with failure-signal retry, and intermediate-state tracing — into a single reusable handler class. The handler must accept any callable step function, wrap it with profiling (tokens, latency, raw I/O), validate output against a caller-provided Pydantic schema, retry on failure with injected error context, and append a structured trace record to `outputs/trace.jsonl`. Demonstrate it on a 3-step enrichment workflow that pulls company data from the Apollo API (or a local mock that returns realistic Apollo-format JSON), enriches with technographics, and generates a personalized email. Run it across 10 companies and print a summary report showing total tokens, total latency, retry count, and success rate per step. Save the handler to `handlers/enrichment_debugger.py`.

6. **Design a bottleneck-optimization report for a multi-step GTM enrichment workflow.** Build a 5-step pipeline: Apollo company search → technographic enrichment → ICP scoring → email personalization → CRM-ready payload formatting. Profile every call across 20 target companies. Compute per-step token cost, latency, cost-per-company, and failure rate. Identify the top bottleneck and write a concrete optimization recommendation (e.g., prompt compression, caching, batch API, model downgrade for low-stakes steps). Produce a markdown report at `outputs/skill-enrichment-bottleneck.md` containing: the profiling methodology, the data table, the bottleneck analysis, and the optimization plan with expected savings.
