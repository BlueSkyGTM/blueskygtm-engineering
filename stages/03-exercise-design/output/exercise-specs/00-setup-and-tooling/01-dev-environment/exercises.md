# Exercises — Dev Environment

## Exercises

1. **Trace your `$PATH` resolution for Python and pip.** Run `which -a python3` and `which -a pip3` to list every matching executable on your system. Then run `python3 -c "import sys; print(sys.executable)"` to confirm which interpreter actually launches. Compare the two outputs: if `which -a` lists more than one path, determine which directory in `$PATH` caused that binary to win. Write a one-line shell command that prints just the winning directory for `python3` by parsing `$PATH` manually (no use of `which`). Your terminal output should show the full list of candidates and the winning directory string.

2. **Configure dotenv-based secret isolation from scratch.** Create a `.env` file containing a dummy key `TEST_API_KEY=fake-key-12345`. Write a `.gitignore` that excludes `.env`. Write a Python script that uses `python-dotenv` to load the file and prints only the first 6 characters of `TEST_API_KEY` followed by `...`. Run the script and confirm the masked output appears. Then run `git status` (after `git init` if needed) and confirm `.env` does not appear in the untracked files list.

3. **Verify authenticated HTTPS connectivity against a live API.** Obtain a free SerperDev API key (or use an existing one). Store it in your `.env` file as `SERPER_API_KEY`. Write a Python script that loads the key via dotenv, sends a POST request to `https://google.serper.dev/search` with a JSON body `{"query": "B2B SaaS companies in San Francisco"}`, and prints the HTTP status code plus the first three result titles. Run it and confirm you see `200` and three company names in your terminal.

4. **Diagnose and fix a simulated PATH collision.** Create a shell script named `python3` in a temporary directory that simply echoes `FAKE PYTHON`. Prepend that directory to your `PATH` in the current shell session (do not modify your profile). Run `python3 --version` and observe the fake output. Now write a command that bypasses the collision by invoking the real Python binary using its absolute path (found via your Exercise 1 technique or by checking common install locations). Print the real version number. Restore your `PATH` afterward and confirm the real `python3` resolves again.

5. **Build a reusable enrichment client for Apollo's people search API.** Store your `APOLLO_API_KEY` in `.env`. Write a script at `handlers/apollo_search.py` that accepts a company domain as a command-line argument, constructs the authenticated request to `https://api.apollographql.com/api/v1/people/match` (or Apollo's current people search endpoint), and outputs the name, title, and email of the first matching contact as formatted JSON. Include input validation: if the key is missing from the environment, print a clear error and exit with code 1. Run `python3 handlers/apollo_search.py stripe.com` and confirm you get structured JSON with a person's details.

6. **Design a multi-provider comparison harness for GTM enrichment.** Create `signals/examples/provider_comparison.py` that queries both SerperDev and Apollo for a list of 5 target company domains you define inline. For each domain, capture: search result count (SerperDev), first contact name and title (Apollo), and response latency in milliseconds for each provider. Output a Markdown table to `outputs/skill-enrichment-benchmark.md` comparing all three metrics across providers and domains. Run the script and open the artifact file to verify the table renders correctly with data for all 5 domains.
