# Exercises — Editor Setup

## Exercises

1. Install `ruff` (`pip install ruff`) and create a file `signals/examples/lead_scoring.py` containing at least three lint violations: an unused import, an undefined variable reference, and a line over 100 characters. Run `ruff check signals/examples/lead_scoring.py` from the terminal and capture the diagnostic output. Then run `ruff check --fix signals/examples/lead_scoring.py` and re-run the check to confirm zero violations remain.

2. Install `black` (`pip install black`) and create a file `signals/examples/formatting_target.py` with intentionally inconsistent formatting: mixed indentation, missing blank lines around functions, long single-line dictionaries. Run `black --check --diff signals/examples/formatting_target.py` to view the formatting diff without modifying the file. Then run `black signals/examples/formatting_target.py` and confirm the check command now passes cleanly.

3. Build a JSON schema for a Clay enrichment webhook payload. The schema must require `email` (string, email format), `company_name` (string), `linkedin_url` (string, URI format), and `confidence_score` (number, 0–1 range). Save the schema as `handlers/clay_schema.json`. Write a Python script that loads a sample payload from `signals/examples/clay_payload.json`, validates it against the schema using `jsonschema.validate`, and prints either `PASS: payload is valid` or `FAIL:` followed by the validation error message. Run the script against two payloads: one valid and one missing the `email` field.

4. Create a `.vscode/settings.json` file that associates `signals/examples/*.json` files with a JSON schema stored at `handlers/clay_schema.json` using the `json.schemas` setting. Create `signals/examples/test_association.json` with an intentional schema violation (e.g., `confidence_score` set to `"high"` instead of a number). Open the file in VS Code and confirm the editor surfaces a schema-validation diagnostic. Then run `npx ajv-cli validate -s handlers/clay_schema.json -d signals/examples/test_association.json` from the terminal to produce the same validation result outside the editor.

5. Build a single-command workflow at `handlers/check_all.py` that, when run as `python handlers/check_all.py`, executes three steps across every `.py` and `.json` file in `signals/` and `handlers/`: (a) runs `ruff check` and collects the result, (b) runs `black --check` and collects the result, (c) validates every JSON file in `signals/examples/` against the schema at `handlers/clay_schema.json`. The script must print a summary table showing pass/fail for each check per file and exit with code 1 if any check failed. Run the command and redirect the summary output to `outputs/skill-editor-setup.md`. Confirm the exit code changes based on whether violations exist.

6. Create a Python module `signals/examples/error_comparison.py` that contains one error a language server catches before runtime (e.g., calling a function with the wrong number of arguments) and one error only surfaced at runtime (e.g., a `KeyError` from accessing a missing dict key inside a try/except). Add a docstring at the top of the file that documents which error the language server flags, which it does not, and why the language server's static analysis has a blind spot in the second case. Run `ruff check` and `python signals/examples/error_comparison.py` to demonstrate both detection points, then write a two-paragraph analysis to `outputs/skill-error-comparison.md` explaining the tradeoff between editor-time and runtime error detection for GTM data pipelines handling untrusted enrichment payloads.
