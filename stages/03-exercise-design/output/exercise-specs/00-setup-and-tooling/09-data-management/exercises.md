# Exercises — Data Management

## Exercises

1. **Implement a schema validation function** that accepts a list of company records and enforces the following schema: `domain` (str, required), `company_name` (str, required), `employee_count` (int, required), `founded_year` (int, optional). The function must return a list of valid records and a list of rejection reasons for invalid ones. Test it against the sample dataset embedded in the lesson. Print the count of valid records, the count of rejected records, and the first three rejection reasons to the terminal.

2. **Build a normalization function** that converts three different input shapes — an Apollo API response (`{"organization_name": "Acme", "website_url": "acme.com", "headcount": 250}`), a Clay export (`{"Company Name": "Acme", "Website": "acme.com", "# Employees": "250"}`), and a custom scraper (`{"name": "acme", "domain": "acme.com", "size": "250"}`) — into the canonical form `{"domain": str, "company_name": str, "employee_count": int}`. Feed all three through the function and print each normalized record. Verify that `employee_count` is an `int` in every output.

3. **Implement an append-only JSONL writer with hash-based deduplication.** Write a function that receives a stream of lead records, computes a SHA-256 hash of the record's canonical key (domain + email), checks whether that hash already exists in the log file, and appends only new records. Seed the log with five records, then attempt to write seven records (three duplicates, four new). Print the total lines in the file after the operation and the number of duplicates skipped.

4. **Construct a quarantine pipeline for CRM-exported contact data.** Given a CSV file with 50 rows of CRM contacts (include rows with missing emails, non-string phone numbers, and malformed domains), apply your schema validator from Exercise 1. Write valid records to `outputs/contacts_valid.jsonl` and invalid records to `outputs/contacts_quarantine.jsonl`, each invalid record annotated with a `rejection_reason` field. Print the valid count, quarantine count, and the distribution of rejection reasons.

5. **Build a complete data ingestion handler** at `handlers/ingest_company_data.py` that combines schema validation, normalization, deduplication, and quarantine into a single CLI tool. The handler should accept an input file path (CSV or JSON), a source profile flag (`apollo`, `clay`, or `scraper`), and write four outputs: `outputs/companies_valid.jsonl`, `outputs/companies_quarantine.jsonl`, `outputs/ingest_report.json` (counts, dedup stats, timing), and an append-only dedup log at `outputs/companies_dedup.jsonl`. Run the handler against two different source profiles and print the report JSON to the terminal for each run.

6. **Use the Hugging Face `datasets` library to load a GTM-relevant CSV dataset** (e.g., a company enrichment export or a lead list), convert it to Parquet and JSON formats, and stream it record-by-record through your schema validator from Exercise 1. Write a summary of the conversion — row counts, file sizes for each format, and schema pass/fail rates — to `outputs/skill-dataset-conversion.md`. Print the summary markdown to the terminal as well.
