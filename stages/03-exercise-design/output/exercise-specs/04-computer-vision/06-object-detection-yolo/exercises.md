## Exercises

1. **Compute IoU between two axis-aligned bounding boxes.** Implement a function `iou(box_a, box_b)` where each box is a tuple `(x1, y1, x2, y2)`. Compute intersection area, union area, and return the ratio. Test with three cases and print results: identical boxes (expect `1.0`), non-overlapping boxes (expect `0.0`), and partially overlapping boxes where you can verify the value by hand. Your terminal should print each test case name, the two input boxes, and the computed IoU rounded to four decimal places.

2. **Decode a raw YOLO output tensor slice.** Given a pre-built `7×7×30` tensor (create one with known values using NumPy — set a few cells to contain simulated detections with realistic confidence and class-probability values), write code that iterates over every grid cell, extracts the `B=2` bounding boxes and `C=20` class probabilities, and prints a human-readable summary: cell coordinates, box parameters `(x, y, w, h, confidence)` for each box, and the top-3 class probabilities. Print only cells where at least one box has confidence above `0.3`. This confirms you understand what every value in `S×S×(B×5+C)` represents.

3. **Implement non-maximum suppression from scratch.** Write a function `nms(detections, iou_threshold)` that takes a list of detections — each a dict with `box`, `score`, and `class_id` — and returns the filtered list after NMS. Sort by confidence descending, greedily keep the highest-scoring detection, remove all remaining detections of the same class whose IoU with the kept box exceeds the threshold, and repeat. Test on a hand-crafted list of 8–10 overlapping detections across two classes with known expected survivors. Print the input list, then the output list, so you can visually confirm suppression worked.

4. **Apply NMS to multi-class detections on a synthetic GTM dataset.** Generate 50 random bounding boxes across 3 classes (`"logo"`, `"cta_button"`, `"form_field"`) with random confidence scores between 0.2 and 0.95, clustered around 5 spatial hotspots so many boxes overlap. Run your NMS function with `iou_threshold=0.4` per class. Print the count of input detections vs. surviving detections per class, and verify that the kept detections are spatially diverse (no two same-class survivors overlapping above the threshold).

5. **Build a YOLO-style detection pipeline for website screenshots and emit per-account signal counts.** Download screenshots for 10 company websites (use a screenshot API like `screenshotone.com`, `urlbox.io`, or a headless browser). Run a pre-trained YOLO model (e.g., `ultralytics` YOLOv8 with COCO weights, or a fine-tuned model if available) on each screenshot. For each company, count detected objects by class (e.g., `person`, `laptop`, `tv`, `cell_phone`) and classify each account's visual signal profile. Write the results as a handler that outputs a JSON report to `outputs/skill-website-vision-signals.json` containing per-account detection counts and a `high_contact_density` boolean flag for accounts with more than 5 person detections. This simulates enrichment of CRM accounts with visual intent signals — the kind of data you would feed back into Clay or a CRM scoring workflow.

6. **Benchmark batched vs. sequential detection and identify the production bottleneck.** Collect 50 website screenshots (reuse Exercise 5's or download fresh). Run the detector two ways: (a) one image at a time in a loop, (b) all images passed as a single batched forward pass. Measure wall-clock time for each approach using `time.perf_counter()`, compute throughput in images/second, and print a comparison table. Then profile the full pipeline — download time, preprocessing time, inference time, postprocessing time — using `cprofile` or manual timing, and identify which stage dominates. Write your findings to `outputs/skill-detection-latency-report.md` including a recommendation for whether batched detection is viable in a real-time enrichment API (latency budget: 2 seconds per account) versus a bulk backfill job (10,000 accounts overnight).
