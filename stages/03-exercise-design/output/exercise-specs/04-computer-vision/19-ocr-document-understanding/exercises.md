# Exercises — OCR & Document Understanding

## Exercises

1. **Implement** basic text extraction with Tesseract. Download or generate a simple document image (e.g., `tesseract input.png stdout` from the CLI, or `pytesseract.image_to_string()` from Python). Run OCR on two images — one clean screenshot and one photographed receipt — and print the raw text output for each to the terminal. Print the character count and word count of each result so you can see the difference in extraction quality.

2. **Compare** OCR accuracy before and after preprocessing. Take a low-contrast or rotated image, run Tesseract directly, and record the output. Then apply grayscale conversion, Otsu thresholding, and deskew correction using OpenCV (`cv2.cvtColor`, `cv2.threshold`, `cv2.warpAffine` with a skew-angle estimate). Re-run OCR and print both versions side by side. Compute and print the Levenshtein edit distance between the two outputs to quantify the improvement.

3. **Reconstruct** reading order from spatial data. Call `pytesseract.image_to_data()` with `output_type=Output.DICT` on a multi-column or table-style image. Using the `left`, `top`, `width`, and `height` fields, sort the detected words into a logical top-to-bottom, left-to-right reading order — grouping words on the same line (within a vertical tolerance) and ordering lines by their `top` coordinate. Print the reconstructed text block in reading order to the terminal so you can visually verify it matches the document's flow.

4. **Build** a structured-field extractor for invoices. Run OCR on a receipt or invoice image, then parse the raw text into a JSON object with named fields: `vendor`, `date`, `total`, and `line_items`. Use regex patterns to locate each field (e.g., `r"total[:\s]*\$?([\d,]+\.\d{2})"` for the total). Print the resulting JSON to the terminal. Test your extractor against at least two different invoice layouts and print both results so you can see where your patterns break.

5. **Extend** your invoice extractor into a persistent document-processing handler. Build a script in `handlers/invoice_processor.py` that accepts a directory of invoice images, runs OCR (Tesseract or PaddleOCR), extracts structured fields with confidence scores, flags any field below a 70% confidence threshold, and writes all results to `outputs/invoice_results.json` as an array of objects. Include a summary at the end of stdout showing total documents processed, average confidence, and count of low-confidence fields. This artifact should be runnable as `python handlers/invoice_processor.py --input ./invoices/`.

6. **Design** a document-routing system for a GTM workflow. A sales team receives three document types via email attachments: vendor contracts (multi-page PDFs with signature blocks), business cards (images with name/title/company/phone), and enrichment CSV exports. Implement a classifier in `handlers/doc_router.py` that inspects each file, selects the appropriate processing strategy (layout parsing for contracts, OCR + field extraction for business cards, direct parsing for CSVs), and outputs structured records to `outputs/routed_contacts.json` formatted for CRM import. Print a routing log to stdout showing each file, the detected type, the tool selected, and a one-line justification for the choice. Demonstrate the distinction between OCR, layout parsing, and document understanding in your tool selection logic.
