# Exercises — POS Tagging and Syntactic Parsing

## Exercises

1. **Assign Universal POS tags and verify against grammatical expectations.** Load `en_core_web_sm`, process the sentence *"Our platform automates outbound sales workflows for B2B teams."*, and print each token with its `.pos_` and `.tag_` labels. Then write a function that takes a sentence and returns `True` only if every token tagged `VERB` in the Universal POS layer is also tagged with a PTB verb tag (`VB`, `VBD`, `VBG`, `VBN`, `VBP`, `VBZ`). Run it on three sentences of your own and print the per-token tag table and the verification result for each.

2. **Extract head-modifier dependency pairs and classify arc labels.** Parse the sentence *"The revenue operations team uses Clay to enrich leads automatically"* and print every `(head_text, head_pos, modifier_text, modifier_pos, dep_label)` tuple. Then group the tuples into four buckets — `nsubj`, `dobj`, `amod`, `compound` — and print each bucket. If a bucket is empty, print `NONE FOUND` for that label so the output is machine-checkable.

3. **Implement Viterbi decoding on a toy HMM.** Define a Python module that hard-codes a 3-state HMM (`NOUN`, `VERB`, `DET`) with explicit transition and emission probability matrices for a 5-word observation sequence of your choice. Implement the Viterbi algorithm from scratch (no library calls) and print the full trellis — every cell's max probability and backpointer — followed by the best state path. Your terminal output must show how each cell's value was derived from the previous column so a reader can trace the combination of transition and emission scores.

4. **Compare `en_core_web_sm` vs `en_core_web_trf` throughput and accuracy.** Export 1,000 company-description strings from an Apollo or Clay table (or use the `companies` sample in `data/` if you do not have API access). Run both models over the batch and print: (a) tokens-per-second for each model, (b) the percentage of tokens where the two models disagree on the Universal POS tag, and (c) five example sentences where they disagree most, with the differing tags printed side by side. Time the batch with `time.perf_counter` and report wall-clock seconds.

5. **Build a company-description enrichment function.** Implement `enrich_company(description: str) -> dict` in `handlers/company_enricher.py` that uses only POS tags and dependency-parsed features (no LLM calls, no external APIs) to return a dictionary with three keys: `industry` (proper nouns and compound nouns near industry keywords), `capabilities` (verb–dobj pairs indicating what the company does), and `sentiment` (polarity derived from adjective modifiers attached to the root verb or product noun). Run the function on five real company descriptions (e.g., pulled from LinkedIn About pages) and write the results as a Markdown table to `outputs/skill-company-enricher.md`. The artifact must include the input description, the extracted fields, and a short note on which dependency arcs drove each extraction.
