## Exercises

1. **Implement a word-level tokenizer and a regex-based tokenizer.** Write a Python script that reads a multi-paragraph string of raw text (a few sentences is fine) and produces two token lists: one from a whitespace-plus-punctuation split and one from a regex pattern of your design (e.g., `\b\w+\b` or a pattern that keeps contractions intact). Print both lists and their token counts side by side. Run the script and confirm the two lists differ in length for at least one sentence containing a contraction or an abbreviation.

2. **Compare stemming and lemmatization on the same corpus.** Using NLTK's Porter stemmer and WordNet lemmatizer (or spaCy's lemmatizer), process a list of 15–20 GTM-relevant phrases (e.g., *"generating leads"*, *"organizations scaling revenue"*, *"ran outbound campaigns"*). Print each original phrase alongside its stemmed and lemmatized forms. Identify and print the count of tokens where the stemmer produced a non-dictionary word and the lemmatizer did not. Run the script and verify the output shows at least 3 such cases.

3. **Build a text normalization pipeline function.** Implement `normalize_text(text, method="stem"|"lemma"|"none")` that lowercases input, removes punctuation, tokenizes, removes English stopwords, then applies the specified normalization method. The function returns the processed token list. Test it on three company description strings (e.g., *"We provide innovative solutions for enterprise customers seeking to optimize their workflows."*) and print the resulting token lists and vocabulary size for each method. Confirm the vocabulary size shrinks as you move from `"none"` → `"stem"` → `"lemma"`.

4. **Evaluate normalization choice for a lead-scoring keyword matching task.** You are given a list of target buyer-intent phrases (e.g., `["data pipeline", "data pipelines", "pipelining data", "data integration"]`) and a list of 10 short prospect bio strings scraped from LinkedIn via Apollo's people search export. Implement a matching function that scores each prospect bio against the target phrases using (a) exact substring match, (b) stemmed match, and (c) lemmatized match. Print a per-prospect score table showing all three columns. Identify which method surfaces the most matches and which produces a false positive (a match that isn't semantically relevant). Write your observation as a printed summary string at the end of the script.

5. **Build a reusable normalization module and produce a comparison artifact.** Implement a module at `handlers/text_normalizer.py` that exposes a `Normalizer` class. The class accepts a config dict specifying tokenizer type (`word` or `regex`), stopword removal (bool), and normalization method (`none`, `porter`, `snowball`, `wordnet`). Load the 20 Newsgroups dataset (or any labeled text corpus with ≥100 documents across ≥3 categories) via `sklearn.datasets`. Run the normalizer across all four method combinations on the same document set and write a Markdown report to `outputs/skill-normalization-eval.md` containing: a table of vocabulary sizes per method, a table of the top-10 most frequent tokens per method, and a one-paragraph recommendation of which configuration to use for a downstream text classifier on this corpus. Run the script and open the generated Markdown file to verify all sections are populated.

6. **Design a proof-of-concept that demonstrates the recall/precision tradeoff on Clay-enriched company data.** Export (or simulate) a CSV of 30 company descriptions enriched via Clay's firmographic enrichment. Design an experiment that constructs a stemmed index and a lemmatized index of these descriptions, then issues 5 search queries targeting business-category concepts (e.g., "payment processing", "cloud infrastructure", "customer success tools"). For each query, compute precision@5 and recall@10 against a hand-labeled relevance set. Write the full results — including a short analysis of which index wins on recall, which wins on precision, and why — into `outputs/skill-stem-lemma-tradeoff.md`. Run the script and confirm the Markdown file contains the metrics table and the analysis paragraph.
