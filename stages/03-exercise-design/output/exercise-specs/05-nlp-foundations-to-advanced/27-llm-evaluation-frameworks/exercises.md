## Exercises

1. **Compute** RAGAS faithfulness and answer relevance scores for a set of five pre-built `(query, context, answer)` triples loaded from a JSON file. Use the `ragas` library with GPT-4o-mini as the judge model. Print a table showing each sample's faithfulness score, answer relevance score, and whether each passes a 0.7 threshold. Confirm your terminal output lists all five rows with numeric scores.

2. **Configure** a DeepEval test suite containing at least three `LLMTestCase` objects built from RAG pipeline outputs. Define a `faithfulness_metric` with a strict threshold of 0.8 and run `assert_test` on each case. Run the suite from the command line and verify that at least one case fails, producing a visible traceback with the metric name and actual score.

3. **Implement** a G-Eval custom metric that scores whether a sales outreach email is "persuasive without being pushy." Define the evaluation criteria in a chain-of-thought prompt, integrate it into DeepEval's `GEval` class, and score a batch of ten cold emails sourced from your CRM or a sample CSV. Print each email's score alongside the judge's one-sentence reasoning. Verify scores appear on a 1–5 scale and that the reasoning strings differ across samples.

4. **Compare** RAGAS, DeepEval's default faithfulness metric, and a custom G-Eval by running the same ten GTM RAG samples — queries about account fit, enrichment data, and ICP matching — through all three. Write a script that outputs a comparison table: `sample_id | ragas_faithfulness | deepeval_faithfulness | geval_score`. Identify the sample with the largest score divergence across frameworks and print its query, answer, and context for manual inspection.

5. **Build** a diagnostic harness that detects the three core evaluation failure modes covered in this lesson. Given a set of scored samples, the script must: (a) run each sample through the judge model three times and report any sample whose standard deviation across runs exceeds 0.15 — flagging it as **scoring variance**; (b) detect **length bias** by correlating answer word count against faithfulness score and flagging if Pearson correlation exceeds 0.6; (c) detect **threshold misconfiguration** by flagging any threshold where more than 95% of samples pass or more than 95% fail. Save the full diagnostic report to `outputs/skill-llm-eval-diagnostics.md` and print a summary table of flagged samples to the terminal.

6. **Design** and implement an end-to-end CI/CD evaluation pipeline for a Clay enrichment workflow. Fetch twenty enriched company records (revenue, headcount, tech stack, ICP score) from a real or simulated Clay export. For each record, generate a short "why this company fits our ICP" explanation using a lightweight LLM call. Score every explanation using RAGAS faithfulness against the raw enrichment fields as context, and a G-Eval metric evaluating "actionability for an SDR." Write the pipeline as a pytest suite in `handlers/clay_eval_suite.py` that exits non-zero if any test case falls below threshold, and produces a persistent report at `outputs/skill-clay-eval-report.md` summarizing pass rate, per-record scores, and flagged failures. Run the suite and confirm the exit code and artifact file both appear.
