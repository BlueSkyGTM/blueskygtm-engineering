# Exercises — Entity Linking & Disambiguation

## Exercises

1. **Implement a minimal three-stage entity linker.** Build a script that takes a list of short texts (e.g., `"Tim Cook spoke at Apple headquarters in Cupertino."`), runs spaCy NER for mention detection, looks up each mention's surface form in a hardcoded alias index dictionary (3–5 entities with multiple aliases each), and scores candidates using Jaccard string similarity over token sets. Run the script and print each mention, its top candidate, and the score. Verify that `"Apple"` resolves to the company entity, not the fruit, when the alias index contains both.

2. **Compare three scoring mechanisms on the same ambiguous mentions.** Using the alias index from Exercise 1, compute and print a side-by-side table for mentions like `"Apple"`, `"Amazon"`, and `"Jordan"` showing how string similarity (token Jaccard), context overlap (word-window cosine against entity descriptions), and popularity priors (hardcoded frequency counts) each rank the candidate set. Identify at least one mention where the three scorers disagree on the top candidate and print which scorer is correct given a gold label you assign.

3. **Build an evaluation harness for micro-precision and micro-recall.** Create a JSONL file with 15–20 annotated mentions across at least 5 distinct entities, each row containing `text`, `mention_span`, `gold_entity_id`. Write a script that runs your linker over this file, compares predicted entity IDs to gold labels, and prints micro-precision, micro-recall, and micro-F1. Handle the NIL case: a prediction of NIL against a non-NIL gold counts as a false negative; a NIL prediction against a NIL gold counts as a true negative (excluded from the micro-precision denominator).

4. **Diagnose NIL and stale-KB failure modes.** Add a new entity to your gold set that does **not** exist in your alias index (simulating a stale KB). Run your evaluation harness and print a per-mention breakdown showing which mentions produced NIL predictions, which produced wrong-entity predictions, and which were correct. Then lower your disambiguation confidence threshold from 0.5 to 0.1 and re-run, printing how the NIL count, micro-precision, and micro-recall change. Report the threshold value that maximizes F1.

5. **Build a Clay-aware company canonicalizer.** Write a script that reads a CSV of messy CRM company names (include abbreviations like `"GS"`, domains like `"stripe.com"`, and full names like `"Goldman Sachs Group, Inc."`) and resolves each to a canonical entity ID using your three-stage pipeline. Output a second CSV column (`canonical_company_id`) suitable as a Clay enrichment input. Run the script and print the number of rows canonicalized, the number of NIL results, and the number of rows where the alias index would have fused two different companies into one ID without disambiguation context. Save the script to `handlers/company_canonicalizer.py` and confirm it runs via `python handlers/company_canonicalizer.py crm_export.csv`.

6. **Design a latency-budgeted candidate generation experiment.** Instrument your candidate generator to cap the number of candidates returned per mention at a configurable limit `K`. Run your full pipeline on a 200+ mention dataset at `K = 5`, `K = 20`, and `K = 50`, recording wall-clock time for the candidate-generation stage and the final micro-F1 for each setting. Write a markdown report to `outputs/skill-candidate-generation-analysis.md` containing a results table, your analysis of where the latency floor sits (alias lookup vs. scoring), and your recommended `K` for a production Clay enrichment workflow that must stay under 500ms per row.
