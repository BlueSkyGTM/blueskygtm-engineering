## Exercises

1. **Build** a passage-level retrieval system using `sentence-transformers/all-MiniLM-L6-v2`. Create a corpus of 12 short GTM documents (3 about outbound prospecting, 3 about lead scoring, 3 about churn signals, 3 about CRM hygiene). Embed all 12, store vectors in a NumPy array, and implement cosine similarity search. Run 5 queries (e.g., "how do I find buyers ready to switch vendors?") and print the top-3 matching documents with their cosine scores for each query. Verify that each query returns documents from the expected topic cluster.

2. **Construct** three adversarial query pairs that test embedding failure modes and measure similarity collapse between them. Pair 1 (negation): "I want to upgrade my plan" vs "I want to cancel my plan." Pair 2 (temporal order): "We deployed the tool then churn decreased" vs "Churn decreased then we deployed the tool." Pair 3 (long-tail jargon): "Our CAC payback is under 6 months" vs "Customer acquisition cost recoupment period falls below half a year." Print the cosine similarity for each pair and flag any pair scoring above 0.85 as a model blind spot. Re-run all three pairs with a second model (e.g., `BAAI/bge-small-en-v1.5`) and print the score deltas side by side.

3. **Implement** a lead-text classifier that routes inbound messages to categories using centroid similarity — no regex, no keyword lists. Create 5 category labels (Sales, Support,Billing, Partnership, Spam). Write 4 example messages per category. Compute a centroid vector (element-wise mean of embeddings) for each category. Classify 10 held-out test messages by computing cosine similarity to each centroid and picking the argmax. Print each test message, the predicted category, the winning centroid's score, and the runner-up score. Report accuracy at the end.

4. **Compare** two embedding model families empirically on the same retrieval task. Run your 12-document GTM corpus from Exercise 1 through both `all-MiniLM-L6-v2` and `BAAI/bge-small-en-v1.5`. For each model, run the same 5 queries and record the top-3 results. Print a comparison table showing: query, rank-1 document for each model, and whether the two models agree on the top result. Compute mean cosine similarity of the top-1 result across all queries for each model and note which model produces higher-confidence (but not necessarily more correct) scores.

5. **Deploy** an embedding-backed vector search system in SQLite with metadata pre-filtering. Create a SQLite database with a table that stores the document text, a JSON metadata column (topic, region, company_stage), and the embedding as a serialized float array. Insert your 12 GTM documents with metadata. Implement two search functions: (1) unfiltered cosine search across all documents, and (2) pre-filtered search that first runs a SQL `WHERE` clause on metadata (e.g., `topic = 'churn_signals' AND region = 'NA'`) then computes cosine similarity only on the survivors. Time 100 iterations of each using `time.perf_counter` and print the average latency in milliseconds for filtered vs. unfiltered. Save the implementation as `handlers/filtered_vector_search.py`.

6. **Design** and **build** a GTM triage pipeline that combines retrieval, centroid classification, and failure-mode awareness into a single artifact. Ingest 20 real-looking inbound lead messages (mix of LinkedIn DMs, form fills, and email replies — include at least 2 adversarial examples that exploit negation or jargon ambiguity). For each message: (a) classify it against category centroids from Exercise 3, (b) retrieve the top-2 relevant knowledge-base documents from your SQLite store, (c) check whether the message triggers any known failure-mode pattern (cosine score suspiciously high between semantically opposite texts), and (d) emit a routing recommendation with a confidence score. Print a structured report for each message. Save the full pipeline as `signals/examples/gtm_triage.py`.
