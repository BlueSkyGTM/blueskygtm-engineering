# Exercises — Dialogue State Tracking

## Exercises

1. **Implement** a `DialogueStateTracker` class with an `update(turn)` method that accepts a parsed user turn (a list of `(slot, value, action)` tuples, where `action ∈ {"inform", "negate", "retract"}`) and mutates an internal `belief_state` dict accordingly. An `inform` sets a slot to a value, a `negate` marks a slot as explicitly excluded (store as `None` with a `negated=True` flag), and a `retract` removes the slot entirely from the belief state. Feed the following turn sequence through your tracker and print the belief state after each turn:
   - Turn 1: `[("budget", "50000", "inform"), ("team_size", "5", "inform")]`
   - Turn 2: `[("budget", "75000", "inform")]` (correction)
   - Turn 3: `[("team_size", "5", "retract")]`
   - Turn 4: `[("industry", "fintech", "negate")]`

2. **Build** an ontology as a Python dict named `LEAD_QUALIFICATION_ONTOLOGY` with at least five slots relevant to B2B lead qualification: `company_size` (closed-vocab: `1-10`, `11-50`, `51-200`, `201-500`, `500+`), `budget_range` (closed-vocab: `<10k`, `10k-50k`, `50k-100k`, `100k+`), `decision_maker` (closed-vocab: `yes`, `no`, `unknown`), `pain_point` (open-vocab free text), and `timeline` (closed-vocab: `immediate`, `1-3_months`, `3-6_months`, `6+months`). Then write a function `validate_value(slot, value)` that returns the value if it passes ontology validation, or the string `"OUT_OF_ONTOLOGY"` with a suggestion to the closest valid value (simple Levenshtein or substring match) for closed-vocab slots, and passes through any string for open-vocab slots. Print the output of `validate_value("company_size", "51-200")`, `validate_value("company_size", "55-200")`, and `validate_value("pain_point", "slow onboarding flow")`.

3. **Extend** your tracker from Exercise 1 to log the full belief-state history as a list of dicts, one snapshot per turn. Then write a function `trace_evolution(history)` that prints, for each turn, which slots changed and how (added, updated with old→new, negated, or retracted). Run your tracker over a 6-turn conversation where the user corrects their budget twice and retracts a slot they previously informed, then call `trace_evolution` and verify the printed trace matches your expectations.

4. **Implement** two DST formulations over the same 5-turn dialogue and compare their outputs. Formulation A (classification-based): for each turn, iterate over each slot in the ontology and classify the user utterance into one of the slot's valid values or `dontcare` using keyword matching. Formulation B (generation-based): pass each utterance to an LLM with a prompt instructing it to return a JSON object of slot-value pairs, then parse the JSON into the belief state. Use the first 5 turns of a sales-qualification conversation (you may author the transcript yourself). Print the final belief state from each formulation side-by-side, and write 2–3 sentences observing where they diverge. Use `openai` or a local model client of your choice.

5. **Build** a complete DST evaluation harness that computes Joint Goal Accuracy (JGA) and per-slot accuracy. Create a dataset of 10 dialogues, each with a gold-standard final belief state (you may author these by hand). Run your rule-based tracker over all 10 and compute: (a) JGA — the fraction of dialogues where the predicted final belief state exactly matches the gold state, and (b) per-slot accuracy — for each slot in the ontology, the fraction of dialogues where the predicted value matches the gold value. Write the results to `outputs/skill-dst-evaluation.md` as a markdown table with one row per slot plus a final JGA row. Include a short section at the bottom listing which dialogue indices failed JGA and which slots were most error-prone.

6. **Design and implement** a lead-qualification dialogue state tracker that ingests real prospect data enriched via Clay or Apollo. Configure the tracker's ontology to capture the five slots from Exercise 2 plus `company_name` (open-vocab). Write a script that: (a) takes a list of 5 company domains as input, (b) calls the Apollo People Search API (or Clay enrichment endpoint) to fetch company-size and industry data for each, (c) maps the enriched fields into your ontology's closed-vocab buckets, (d) seeds the belief state for each prospect, and (e) prints a formatted table of all five prospects with their seeded slot values. Save the tracker module to `handlers/lead_qual_tracker.py` so it can be imported by downstream CRM-routing code. Use environment variables for your API key and handle rate limits with a retry decorator.
