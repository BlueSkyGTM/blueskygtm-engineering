# Exercises — Relation Extraction & Knowledge Graph Construction

## Exercises

1. **Implement** a rule-based triple extractor using spaCy dependency parsing. Write a function that takes a list of sentences about company acquisitions (e.g., "Google acquired DeepMind", "Salesforce purchased Slack", "Microsoft was bought by IBM" — note the passive voice trap) and extracts `(subject, relation, object)` triples by matching `nsubj` → `ROOT` → `dobj` patterns. Print each extracted triple to the terminal. Your output should show which sentences produced triples and which failed silently. Count and print the total number of triples extracted versus sentences provided — this ratio is your first recall baseline.

2. **Build** a directed knowledge graph from a hardcoded list of at least 15 triples using NetworkX. Use companies and relationships from the CRM domain: `(Stripe, competes_with, Square)`, `(Sequoia, invested_in, Stripe)`, `(Stripe, employs, Patrick_Collison)`. Add nodes with a `type` attribute (Organization, Person, Investor) and edges with a `relation` attribute. Print the in-degree and out-degree of every node, then use `nx.single_source_shortest_path` to find the connection path between two seemingly unrelated entities. Your terminal output should display the full graph statistics and the discovered path.

3. **Implement** entity resolution for a noisy set of entity mentions drawn from Apollo lead-export data. You are given surface forms like `["Stripe Inc.", "Stripe", "STRIPE", "stripe.com", "Stripe, Inc.", "Square", "SQ"]`. Build a canonicalizer that normalizes case, strips legal suffixes (Inc., LLC, Corp.), lowercases domains, and uses string-similarity matching (Levenshtein ratio or Jaccard on tokens) to cluster mentions into canonical entities. Print each cluster with its canonical form and the surface variants it absorbed. Inject the resolved entities into a NetworkX graph and confirm the node count is lower than the raw mention count — the difference is your deduplication rate.

4. **Apply** two extraction strategies to the same corpus of 20 LinkedIn company-description strings and evaluate both against a gold-standard triple set you hand-label. Strategy A: your rule-based dependency-parser extractor from Exercise 1. Strategy B: an LLM-based extractor using structured output prompting (ask an LLM to return JSON triples with a fixed schema of `competes_with`, `acquired`, `invested_in`, `partners_with`). Compute precision, recall, and F1 for each strategy. Print a comparison table showing which strategy wins on precision, which wins on recall, and list the specific triples each strategy missed or hallucinated.

5. **Design and build** an end-to-end KG construction pipeline that processes enriched Clay export rows (company name, website, employees, funding round, competitors field — all unstructured). Your pipeline must: parse each row with spaCy, run rule-based triple extraction, apply entity resolution to merge duplicate company mentions, insert resolved triples into a NetworkX directed graph, and export the graph to `outputs/skill-gtm-knowledge-graph.json` in a format loadable by `nx.node_link_graph`. Additionally, write a handler at `handlers/triple_extractor.py` that accepts a raw text string and returns a list of extracted triples — this handler will be composed into a larger GTM enrichment workflow. Run the pipeline on at least 50 Clay rows and print: total triples extracted, post-resolution triple count, graph density, top-5 nodes by degree centrality, and any disconnected subgraphs detected.

6. **Prove** or disprove the following hypothesis with evidence from your pipeline: "LLM-based extraction strictly dominates rule-based extraction on F1 for all relation types in GTM corpora." Build a per-relation-type evaluation harness that breaks down precision, recall, and F1 by relation (`competes_with`, `acquired`, `invested_in`, `partners_with`) across both strategies on a 100-row dataset. Write your findings, including a counterexample relation type where rules outperform LLMs, to `outputs/skill-extraction-analysis.md`. The document must include a markdown table of metrics, at least three concrete failure examples per strategy, and a recommendation for which extraction method to use per relation type in a production GTM enrichment pipeline.
