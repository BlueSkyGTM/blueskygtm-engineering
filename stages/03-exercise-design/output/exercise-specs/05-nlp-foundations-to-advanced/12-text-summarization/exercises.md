# Exercises — Text Summarization

## Exercises

1. **Implement** a TextRank extractive summarizer from scratch. Given a multi-paragraph input (use any news article of 8+ sentences), split the text into sentences, compute TF-IDF vectors for each sentence, build a cosine-similarity graph, run PageRank for 50 iterations (damping factor 0.85), then extract and print the top 3 sentences in their original document order. Print each sentence's TextRank score alongside it so you can verify that structurally central sentences rank highest.

2. **Build** an abstractive summarizer using an LLM API with token-bounded chunking. Take a document that is at least 1,500 tokens, implement a tokenizer-based chunking function that splits the document into chunks no larger than 80% of the model's context window, summarize each chunk with a single API call, and print the concatenated result. Verify your chunker produces correctly sized chunks by printing the token count of each chunk before the API call.

3. **Compare** extractive and abstractive summaries on the same 1,000+ word document. Run your TextRank summarizer from Exercise 1 and your abstractive summarizer from Exercise 2 on the same input. Print both summaries, then print a side-by-side comparison: word count, number of named entities preserved (use spaCy or an NER library), and any sentences in the abstractive output that do not appear in the original document (potential hallucinations). Write one sentence at the end stating which method performed better for this document and why.

4. **Implement** a map-reduce summarization pipeline for a document that exceeds a 4,000-token context window. Split the document into overlapping chunks of 1,500 tokens with 200-token overlap, summarize each chunk independently via an LLM API call (the map step), concatenate the chunk summaries, then run a final summarization pass on the concatenated summaries (the reduce step). Print the number of chunks, each chunk summary, and the final reduced summary. Use a company earnings call transcript or a long product documentation page as your input document.

5. **Build** a named entity precision scorer that evaluates summarization quality across multiple documents. Fetch 5 company profiles from the Apollo API (or use 5 local CRM records in JSON format if you do not have an API key). For each company, generate a summary using either TextRank or an LLM API. Extract named entities from both the original text and the summary using spaCy, compute entity-level precision (entities in summary that also appear in original / total entities in summary), and print a table showing per-document precision and the mean precision across all documents. Write the full pipeline and results to `outputs/skill-entity-precision.md`.

6. **Design** a hybrid summarization pipeline that combines extractive pre-filtering with abstractive refinement for a GTM use case. Take a batch of 10 enriched lead records (from Clay, Apollo, or a local dataset of company + contact data). Use TextRank to extract the 5 most central sentences from each lead's combined description and notes, then pass those sentences to an LLM API to produce a 2-sentence actionable account brief. Output a JSON array where each element contains the company name, the TextRank-selected sentences, and the final LLM-generated brief. Save the pipeline to `handlers/account-briefer.py` and the output JSON to `signals/examples/account-briefs.json`.
