# Exercises — Named Entity Recognition

## Exercises

1. **Extract named entities from a raw prospect email and print each entity's text, label, and character offsets.** Load `en_core_web_sm`, run the NLP pipeline on a multi-sentence sales email of your choice (at least 3 sentences), and iterate over `doc.ents`. For each entity, print a row in the format `LABEL | text | start_char–end_char`. Verify that the character offsets correctly bracket the entity text when you index back into `doc.text`.

2. **Implement BIO tagging for an arbitrary spaCy `Doc`.** Write a function `bio_tag(doc)` that returns a list of `(token_text, bio_label)` tuples. Use the entity type from `token.ent_type_` for `B-` and `I-` prefixes; assign `O` to non-entity tokens. Run it on the sentence `"Stripe acquired TaxJar and laid off 200 employees in Dublin."` and print the token-by-token BIO table. Confirm that multi-token entities chain `I-` tags correctly and that punctuation receives `O`.

3. **Compare statistical and rule-based extraction on the same document.** Choose a short press release or company blog post. Run spaCy's statistical NER to collect the entity set. Then build a `spaCy.Matcher` with at least three pattern groups: one for monetary amounts (e.g., `$5M`, `$1.2 billion`), one for job titles from a small hard-coded list (`VP of Sales`, `Chief Revenue Officer`, `Account Executive`), and one for product/technology keywords. Run both systems, print every entity/match each system finds that the other does not, and write a two-sentence summary of where each approach excels.

4. **Evaluate NER precision against hand-labeled ground truth on 10 job postings.** Fetch 10 real job postings from a public source (e.g., the Common Crawl job postings dataset, an RSS feed, or copy-paste from LinkedIn/AngelList into a local JSON file). Manually annotate the `ORG` and `LOCATION` entities in each posting and store them in a JSON file as `{"text": "...", "labels": [[start, end, "ORG"], ...]}`. Run `en_core_web_sm` on each posting, compute precision as `correct_predictions / total_predictions` (a prediction is correct if its span and label match a ground-truth span), and print the per-posting precision along with the overall average. Report which posting had the lowest precision and inspect what the model misclassified.

5. **Build a production-style batched NER handler with confidence-based fallback to rule-based matchers.** Implement a handler in `handlers/ner_fallback.py` that accepts a list of raw text strings (e.g., CRM notes, Apollo-enriched company descriptions, or Clay-enriched prospect bios), runs them through spaCy's `nlp.pipe()` in batch mode, and checks each extracted entity's confidence. Since `en_core_web_sm` does not expose per-entity scores, use `doc.cats` from a textcat component or wrap `EntityRuler` patterns and treat matched entities as high-confidence (1.0) and statistical entities as lower-confidence (0.5). For any document where the statistical model extracts zero `ORG` entities, fall back to an `EntityRuler` with regex patterns for company suffixes (`Inc.`, `LLC`, `Corp`, `GmbH`, `Ltd`) and capitalization heuristics. Output a JSON report to `outputs/ner-batch-report.json` with one entry per input document containing: the extracted entities, their source (`statistical` vs `rule_fallback`), and the fallback trigger flag. Run it on at least 20 CRM-style text records and verify that the JSON report is valid and that at least one document triggered the fallback path.
