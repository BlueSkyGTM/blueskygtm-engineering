## Exercises

1. **Implement a single-cell NIAH probe.** Write a Python script that constructs a prompt by inserting the sentence `"The secret access code is BLUE-FALCON-77."` at exactly 50% depth within padding text (use repeated paragraphs of public-domain text or lorem ipsum to reach ~8000 tokens). Send the prompt to any model API and print the model's response alongside a boolean indicating whether the exact code string appears in the response. Run the script three times and print the pass/fail result for each trial. You should see three lines of terminal output, each showing `PASS` or `FAIL`.

2. **Build a depth sweep over five positions.** Extend the single-cell probe into a loop that places the same needle at 0%, 25%, 50%, 75%, and 100% depth within a fixed 8000-token context. For each depth, call the model once and store the result. Print a table to the terminal showing depth percentage, whether the model retrieved the needle, and the raw response truncated to 100 characters. Visually inspect the output to identify which positions, if any, show failures.

3. **Implement a full NIAH grid and classify the decay pattern.** Write a script that sweeps both context length (use four sizes: 2k, 8k, 16k, 32k tokens) and needle depth (0%, 25%, 50%, 75%, 100%) — producing a 4×5 grid of pass/fail cells. After running all 20 cells, print the grid as an ASCII table and then implement logic that examines the grid rows to classify each context length's decay pattern as `linear`, `u_shaped`, `threshold`, or `flat`. Print the classification next to each row. A model with the "lost in the middle" effect should show `u_shaped` at longer context lengths.

4. **Implement a two-needle retrieval test.** Place two independent needles — `"The renewal date is March 15th"` and `"The billing contact is Sarah Chen"` — at different depths within a 16k-token context. Test all four combinations of positions: (25%, 75%), (50%, 50%), (10%, 90%), and (75%, 25%). For each combination, ask the model to retrieve both facts in a single response. Print a table showing needle positions, whether each needle was retrieved, and whether both were retrieved successfully. Compute and print the correlation between needle-1 accuracy and needle-2 accuracy across the four trials — if retrieval failures are position-dependent rather than independent, the correlation will be high.

5. **Evaluate usable context window for GTM enrichment payloads.** Build a test that simulates a realistic Clay enrichment workflow: construct a context window containing 50 enriched company profiles (name, industry, employee count, tech stack, recent funding, contact email) as structured records, then insert a target company with a specific attribute at varying depths. Sweep across context sizes of 4k, 16k, and 64k tokens by adding more enrichment records. At each size, test whether the model can correctly answer a multi-hop question that requires extracting one fact from the target company and joining it with a fact from a different profile in the context (e.g., "What is the email domain of the company that shares the same primary CRM vendor as the company whose Series B was led by Sequoia?"). Print a results table and write a markdown summary of your findings — including whether the model's advertised context window matches its practical usability for enrichment joins — to `outputs/skill-niah-gtm-evaluation.md`.

6. **Design a benchmark comparison study.** Write a Python module at `signals/examples/benchmark-comparison.py` that implements a minimal version of each of the four benchmark types discussed (NIAH, a RULER-style multi-keyword aggregation task, a LongBench-style multi-document QA task, and an MRCR-style multi-target retrieval task). Run all four against the same model at a single context length (8k tokens). Print a comparison table showing each benchmark's task type, the specific failure mode it isolates, and the model's accuracy. In the module's docstring, write a 3–4 sentence analysis of which failure modes the model is most susceptible to and which benchmark you would prioritize for evaluating models intended for GTM data-processing pipelines.
