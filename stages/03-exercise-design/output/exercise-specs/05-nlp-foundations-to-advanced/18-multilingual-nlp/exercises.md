# Exercises — Multilingual NLP

## Exercises

1. **Compute tokenization inflation factors across five languages.** Load `xlm-roberta-base` and its tokenizer from Hugging Face. Prepare the same firmographic sentence — "Our company provides cloud infrastructure solutions for enterprise clients" — translated into German, Japanese, Arabic, Hindi, and Portuguese (write the translations yourself or use any reference). Tokenize each version and print the token count and the ratio relative to the English token count. Run the script from the terminal and confirm you see a table with language, token count, inflation ratio, and the first 10 subword tokens for each language.

2. **Trace the three failure modes on raw input samples.** Create a Python script that loads three short text snippets: one Japanese company description containing kanji, one Arabic firm name with right-to-left script, and one Hindi product tagline with Devanagari. For each snippet, run it through `bert-base-uncased` (an English-trained tokenizer) and print the tokenized output with token IDs. Classify each snippet as suffering primarily from tokenization mismatch, embedding space fragmentation, or script/encoding edge cases, and print your classification with a one-sentence justification next to each. Run from the terminal and verify each sample prints its tokens, IDs, diagnosed failure mode, and reasoning.

3. **Implement a language detection and routing pipeline for firmographic data.** Build a script that takes a JSON array of company records where fields like `name`, `description`, and `industry` may be in any of 10 languages. Use `langdetect` or `fasttext`'s language identification model to detect the dominant language of each record's `description` field. Route each record to a normalizer function that lowercases, strips punctuation appropriate to the detected script (Latin, CJK, Cyrillic, Arabic, Devanagari), and standardizes common firmographic abbreviations (GmbH, 株式会社, S.A., OOО, شرکت) into a canonical English field `entity_type` — all without translating the original text. Print the routed and normalized records as formatted JSON to the terminal.

4. **Build a classify-original-vs-translate-first decision engine.** Write a function that accepts a text input, its detected language code, a latency budget in milliseconds, and a per-language model-confidence lookup table (you define this table for at least 8 languages using plausible F1 scores). The function should return a routing decision: `"classify_original"`, `"translate_then_classify"`, or `"insufficient_signal"`. Use these decision factors: (a) if the per-language F1 for that language is above 0.80, classify in original language; (b) if F1 is below 0.60, translate first; (c) if translation is required but the latency budget is under 50ms and the text exceeds 200 tokens, return `"insufficient_signal"`. Feed the function five test records covering each branch and print the decision and reasoning for each.

5. **Build a multilingual firmographic enrichment handler that integrates Clay-style person data.** Create a handler module at `handlers/multilingual_enrichment.py` that accepts a list of lead records — each containing `name`, `title`, `company`, and `bio` fields potentially in any of 15 languages. The handler must: detect language per field independently, normalize entity suffixes and titles without translation (e.g., map "Directeur Général", "取締役", "Geschäftsführer", "مدير تنفيذي" to `seniority: executive`), flag records where tokenization inflation ratio exceeds 4x relative to an English baseline (using `xlm-roberta-base`), and output an enriched JSON with normalized fields plus diagnostics. Simulate enrichment API data by generating 20 multilingual lead records programmatically (varying scripts and entity types). The handler must be importable and runnable as `python -m handlers.multilingual_enrichment` from the project root, printing the enriched records and a summary of flagged high-inflation records. Persist the final enriched output to `outputs/multilingual_enrichment_results.json`.

6. **Design and benchmark a polyglot GTM classification architecture.** Produce a design document at `outputs/skill-multilingual-gtm.md` that specifies a production pipeline for classifying multilingual inbound leads into ICP tiers. Your document must include: (1) a diagram (ASCII or Mermaid) showing the routing logic across detect → normalize → classify-or-translate → score; (2) a table comparing latency, entity preservation, and accuracy tradeoffs for classify-original vs translate-then-classify across at least 8 languages with representative benchmark numbers; (3) a failure-mode mitigation matrix mapping each of the three failure modes to at least two concrete engineering mitigations; (4) a recommended threshold policy for when translation is worth the latency cost, with quantitative justification. Run a script that validates your document renders as well-formed Markdown (parse it and print section headings and table row counts to the terminal).
